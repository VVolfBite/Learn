# Go 笔记修订执行规范

## 目标

将现有 Go 笔记整理为一份：

- 标准
- 覆盖全面
- 不求展开详细
- 适合复习
- 结构统一
- 代码风格统一

核心要求是：

**知识点要尽量全，易错点和需要知道的小点都要出现，但不做教程式展开。**

------

# 一、标题层级规则

## 一级标题

一级标题只能是：

# GO

不得改动，不新增其他一级标题。

------

## 二级标题

二级标题的大范围结构保持你当前已有设计，不改动。
 例如你已经写好的这类范围：

- 基础
- 设计与继承
- 并发与网络

后续整理时只能在你现有框架内修订内容，**不能重划二级结构**。

------

## 三级标题

三级标题的大体框架也视为固定。
 后续修改时：

- 不随意增删三级标题
- 不重组三级标题的大框架
- 只在其下修正内容密度、层次和表达方式

也就是说，三级标题以下才是主要修订空间。

------

## 四级标题

四级标题用于落实到一个**明确知识领域或技术对象**。

例如：

- 字符串
- 切片
- map
- 方法
- 接口
- goroutine
- channel

四级标题要能够稳定承接该主题下的具体知识点。

------

## 五级标题

五级标题用于落实到该知识点下的**具体用法、特点、行为或子项**。

例如字符串下可以是：

- 复制
- 遍历
- 拼接
- 比较
- 转换

切片下可以是：

- 创建
- 追加
- 复制
- 截取
- 扩容

也就是说，五级标题控制在“具体操作 / 具体行为 / 具体特点”这一层，不再展开成教程式小节。

------

## 六级标题

六级标题只作为备用层级。
 只有在某个主题确实需要额外展开时才使用，例如：

- 并发
- channel 行为
- 接口实现细节
- 切片底层共享
- defer / panic / recover

六级标题不是常规层级，不主动大量使用。

------

# 二、内容定位规则

## 1. 这是复习笔记，不是教程

后续所有内容都要服从这一点：

- 覆盖尽量全面
- 但不追求详细讲解
- 不写成长篇教学说明
- 不把一个知识点拆成多个教学式小例子

写法应该更接近：

- 简洁定义
- 标准语法
- 核心要点
- 注意边界
- 少量综合示例

而不是：

- 大段解释概念
- 一个 API 配一个程序
- 一个现象配一段输出
- 一个小点展开成教程段落

------

## 2. 必须出现但不必展开的内容

对于这些内容，要尽量覆盖到：

- 易错点
- 默认行为
- 零值
- 常见边界
- 语义差异
- 使用限制
- 相关小知识点

但出现方式应尽量短，不写成大段说明。
 也就是说：

**该提到的都提到，但点到为止。**

------

## 3. 不使用教导式口吻

不写这种表达：

- 这是必会的
- 这是常用的
- 这个很重要
- 很容易错
- 一定要掌握

统一改为中性表达。
 需要强调时只用结构和标记，而不是讲课语气。

允许使用的强调方式只有：

- `**注意**`
- `**补充**`

必要时可少量用：

- `**对比**`
- `**示例**`

------

# 三、每个知识点的写法规则

每个知识点尽量按以下顺序写，但不强制每项都完整出现：

## 定义

一句或几句，说明：

- 是什么
- 有什么作用
- 和相近概念的区别是什么

要求简洁，不铺陈。

------

## 语法

只写核心语法形式，用标准模板表达。
 不在这里展开长篇解释。

------

## 要点

将关键点紧凑排在一起，要求：

- 连续
- 简短
- 聚焦区别、行为、边界
- 不把每一点拆成单独大段

------

## **注意**

用于写：

- 易错点
- 默认行为
- 边界情况
- 常见误解

要求短而明确。

------

## **补充**

用于写：

- 扩展知识
- 低频但应知道的点
- 与其他概念的关系
- 实际使用中的补足说明

也要尽量收短。

------

## 综合代码

每个主题尽量只保留一个综合代码块，或者少量几个代码块。
 代码块的职责是：

- 把前面的要点串起来
- 展示典型写法
- 顺带体现注意点和行为差异

而不是为每一个小 API 单独服务。

------

# 四、代码规范

## 1. 代码的核心目标

代码用于：

- 统一表达
- 快速复习
- 明确语法模板
- 承载多个知识点

不要求所有代码都能直接复制运行。
 优先保证的是：

- 标准
- 表意明确
- 结构一致

------

## 2. 命名风格

统一采用严格表意命名，不用生活化例子，不用随意变量名。

优先使用这类名称：

- `condition`
- `statement`
- `expression`
- `value`
- `result`
- `source`
- `target`
- `index`
- `element`
- `key`
- `mappedValue`
- `sliceValue`
- `arrayValue`
- `stringValue`
- `channelValue`
- `typeName`

不使用这类名称：

- `apple`
- `banana`
- `foo`
- `bar`
- `aaa`
- `bbb`
- `x`
- `y`
- `tmp`

除非该名称本身正是要说明的对象。

------

## 3. 语法模板风格

代码优先写成标准语法模板形式，例如：

```
if condition {
    statement
}
if condition {
    statement
} else {
    statement
}
for initialStatement; condition; postStatement {
    statement
}
for index, element := range collectionValue {
    statement
}
func functionName(parameterType) returnType {
    statement
}
value := make(typeName, length, capacity)
result := append(sliceValue, element)
copiedCount := copy(target, source)
mappedValue, exists := mapValue[key]
go functionName(argument)
value := <-channelValue
channelValue <- value
```

------

## 4. 注释规范

代码块内部必须承担解释作用。
 说明信息应尽量写在代码注释中，包括：

- 参数含义
- 返回值含义
- 对应分点
- 行为结果
- 输出示意
- 文件说明
- 命令说明

不再把这些内容拆成代码块外的独立“输出如下”“文件如下”。

例如：

```
sliceValue := make([]ElementType, length, capacity)
// length: 初始长度
// capacity: 初始容量
// len(sliceValue) == length
// cap(sliceValue) == capacity
```

------

## 5. 综合代码块原则

如果某个主题下存在多个零碎示例，应优先合并成一个代码块，前提是：

- 合并后依然清晰
- 每个片段有注释标识用途
- 不因为强行合并而让代码难读

总原则是：

**减少碎片代码，增加综合代码。**

------

# 五、内容处理规则

## 优先保留

后续修改时优先保留：

- 定义
- 核心语法
- 常见行为
- 易错点
- 关键差异
- 默认值 / 零值
- 典型用法
- 能串联多个知识点的代码

------

## 优先压缩

后续修改时优先压缩：

- 过长解释
- 教程式展开
- 重复描述
- 单个 API 的独立样例
- 独立输出块
- 独立命令块
- 来源痕迹明显的文本

------

## 优先合并

优先合并这些内容：

- 同主题下多个零碎代码块
- 相近 API 的独立说明
- 同一类行为的重复解释
- 输出与代码分离的内容

------

# 六、修改流程规则

## 1. 修改前先汇报方向

除了你明确要求我只是做：

- 合并零碎代码块为一个统一代码块

否则我在正式修改前都要先向你汇报：

- 本次准备改哪一部分
- 为什么改
- 准备怎么改
- 保留什么
- 压缩什么
- 合并什么

然后再执行修改。

------

## 2. 我的两个固定任务

### 任务 1：生成统一串起来的代码

当你让我整理代码时，我需要：

- 用标准语法模板
- 用严格表意命名
- 用注释解释参数、行为、输出
- 尽量把多个点合并到一个综合代码块中

------

### 任务 2：给文章提出重新规划建议

当你给我某一段或某一章内容时，我需要先告诉你：

- 当前结构有什么问题
- 建议如何重排
- 哪些标题不动
- 哪些内容可以压缩
- 哪些内容适合合并成综合代码
- 哪些位置适合加 `**注意**` 或 `**补充**`

然后再按你的要求修改。

------

# 七、最终执行核心

后续所有修改都围绕这一句执行：

**这份笔记要标准、覆盖全面、适合复习，不追求详细展开；易错点和需要知道的小点必须出现，但只做紧凑表达。**
