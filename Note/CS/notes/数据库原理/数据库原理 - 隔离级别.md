## 隔离级别是什么

隔离级别定义了事务之间的隔离程度，是数据库在并发性能和数据一致性之间做权衡的机制。

不同的隔离级别，能解决不同的并发一致性问题，但隔离级别越高，并发性能越低。

**隔离级别是目标，锁和 MVCC 是实现手段**。

SQL 标准定义了四种隔离级别，从低到高：
1. 读未提交 Read Uncommitted (RU)
2. 读已提交 Read Committed (RC)
3. 可重复读 Repeatable Read (RR)
4. 串行化 Serializable

## 读未提交 Read Uncommitted

### 特点

事务可以读到其他事务未提交的数据。

### 能解决什么

什么都解决不了，所有并发问题都可能发生。

### 不能解决什么

- 脏读 错误
- 不可重复读 错误
- 幻读 错误

### 场景

几乎不使用，因为会读到脏数据，风险太大。

## 读已提交 Read Committed

### 特点

事务只能读到其他事务已提交的数据。

### 能解决什么

- 脏读 正确

### 不能解决什么

- 不可重复读 错误
- 幻读 错误

### 实现原理

每次读取数据时，都生成一个新的 ReadView，只读取已提交的数据。

### 场景

Oracle、PostgreSQL 的默认隔离级别。适合对一致性要求不高、但需要较高并发的场景。

### 例子

```sql
-- 事务 A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 读到 100

-- 事务 B
START TRANSACTION;
UPDATE account SET balance = 200 WHERE id = 1;
COMMIT;  -- 提交

-- 事务 A
SELECT balance FROM account WHERE id = 1;  -- 读到 200（不可重复读）
COMMIT;
```

事务 A 两次读取结果不同，因为事务 B 在中间提交了修改。

## 可重复读 Repeatable Read

### 特点

保证在同一个事务中多次读取同一数据的结果一致。

### 能解决什么

- 脏读 正确
- 不可重复读 正确

### 不能解决什么

- 幻读 错误（但 MySQL InnoDB 通过 Next-Key Lock 解决了）

### 实现原理

事务开始时生成一个 ReadView，整个事务期间都使用这个 ReadView，读取的是事务开始时的快照。

### 场景

MySQL InnoDB 的默认隔离级别。适合大多数业务场景。

### 例子

```sql
-- 事务 A
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 读到 100

-- 事务 B
START TRANSACTION;
UPDATE account SET balance = 200 WHERE id = 1;
COMMIT;  -- 提交

-- 事务 A
SELECT balance FROM account WHERE id = 1;  -- 还是读到 100（可重复读）
COMMIT;
```

事务 A 两次读取结果相同，因为读的是事务开始时的快照。

## 串行化 Serializable

### 特点

强制事务串行执行，完全避免并发。

### 能解决什么

- 脏读 正确
- 不可重复读 正确
- 幻读 正确

### 实现原理

对所有读取的数据都加锁，事务之间完全串行执行。

### 场景

极少使用，因为并发性能极低。只在对数据一致性要求极高的场景使用。

## 四种隔离级别对比

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|---------|------|-----------|------|---------|
| Read Uncommitted | 错误 | 错误 | 错误 | 不加锁 |
| Read Committed | 正确 | 错误 | 错误 | MVCC（每次读生成 ReadView） |
| Repeatable Read | 正确 | 正确 | 错误 | MVCC（事务开始时生成 ReadView） |
| Serializable | 正确 | 正确 | 正确 | 加锁 |

**注意**：MySQL InnoDB 在 RR 级别下，通过 Next-Key Lock 解决了幻读问题。

## 隔离级别的权衡

### 一致性 vs 并发性

- **隔离级别越高**：数据一致性越好，但并发性能越低
- **隔离级别越低**：并发性能越高，但数据一致性越差

### 如何选择

这一节看起来往往容易变成一串结论，其实先抓住“如何选择”在数据库原理里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**Read Committed**：
- 适合读多写少、对一致性要求不高的场景
- 并发性能好，但可能出现不可重复读

**Repeatable Read**：
- 适合大多数业务场景
- 平衡了一致性和并发性
- MySQL 默认级别

**Serializable**：
- 适合对一致性要求极高的场景（如金融系统）
- 并发性能差，慎用

## MySQL 的隔离级别

### 默认隔离级别

MySQL InnoDB 默认使用 **Repeatable Read**。

### 查看隔离级别

```sql
-- 查看当前会话的隔离级别
SELECT @@transaction_isolation;

-- 查看全局隔离级别
SELECT @@global.transaction_isolation;
```

### 设置隔离级别

```sql
-- 设置当前会话的隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置全局隔离级别
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

## RC 和 RR 的核心区别

| 维度 | Read Committed | Repeatable Read |
|------|---------------|-----------------|
| ReadView 生成时机 | 每次读取时生成 | 事务开始时生成 |
| 能否读到其他事务的提交 | 能 | 不能 |
| 不可重复读 | 会发生 | 不会发生 |
| 幻读 | 会发生 | 会发生（InnoDB 通过 Next-Key Lock 解决） |
| 并发性能 | 较高 | 较低 |

**记忆技巧**：
- RC：每次读都是"已提交"的最新数据
- RR：每次读都是"可重复"的快照数据

## 不可重复读 vs 幻读（再强调）

在隔离级别的语境下，这两个概念经常一起出现：

**不可重复读**：
- 同一条数据，多次读取值不同
- RR 级别解决

**幻读**：
- 同一个范围查询，多次读取行数不同
- Serializable 级别解决（MySQL InnoDB 在 RR 级别通过 Next-Key Lock 解决）

## 追问

**Q: MySQL 默认隔离级别是什么？**
Repeatable Read（可重复读）。

**Q: 为什么 MySQL 选择 RR 作为默认级别？**
平衡了一致性和并发性能，适合大多数场景。而且 MySQL 通过 MVCC 和 Next-Key Lock 优化了 RR 的性能。

**Q: RC 和 RR 的区别？**
RC 每次读都生成新的 ReadView，能读到其他事务的提交；RR 事务开始时生成 ReadView，读的是快照。

**Q: RR 能解决幻读吗？**
SQL 标准中 RR 不能解决幻读，但 MySQL InnoDB 通过 Next-Key Lock 解决了。

**Q: 如何选择隔离级别？**
大多数场景用 RR。如果对一致性要求不高、需要更高并发，可以用 RC。

**Q: Serializable 为什么并发性能差？**
因为要对所有读取的数据加锁，事务之间完全串行执行，失去了并发能力。

**Q: 隔离级别能解决丢失更新吗？**
不能。丢失更新需要通过应用层加锁（乐观锁或悲观锁）来解决。

**Q: 为什么 RC 不能解决不可重复读？**
因为 RC 每次读都生成新的 ReadView，能读到其他事务的提交，所以同一事务内多次读取结果可能不同。

**Q: 为什么 RR 能解决不可重复读？**
因为 RR 事务开始时生成 ReadView，整个事务期间都用这个 ReadView，读的是快照，不受其他事务提交的影响。

**Q: 提高隔离级别一定能提高数据一致性吗？**
是的，但会降低并发性能。需要根据业务场景权衡。
