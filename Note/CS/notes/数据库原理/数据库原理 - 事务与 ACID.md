# 数据库原理 - 事务与 ACID

## 事务是什么

事务是一组操作的集合，这些操作要么全部成功，要么全部失败。

一个转账场景：A 给 B 转 100 元，需要两步操作：
1. A 账户减 100
2. B 账户加 100

如果第 1 步成功、第 2 步失败，A 的钱少了但 B 没收到，数据就错了。事务保证这两步要么都执行，要么都不执行。

```sql
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE user = 'A';
UPDATE account SET balance = balance + 100 WHERE user = 'B';
COMMIT;
```

如果中途出错，可以用 `ROLLBACK` 回滚，让数据回到事务开始前的状态。

## ACID 四个特性

### 原子性 Atomicity

事务是不可分割的最小单元，要么全部成功，要么全部失败。

转账例子中，两个 UPDATE 要么都生效，要么都不生效，不能只执行一半。

**实现机制**：undo log（回滚日志）。数据库会记录每个修改操作的反向操作，回滚时反向执行即可。

### 一致性 Consistency

数据库从一个一致性状态转换到另一个一致性状态。一致性状态是指数据满足所有完整性约束。

转账前 A + B 的总金额是 1000，转账后总金额还是 1000。如果转账后总金额变了，就违反了一致性约束。

**一致性的两个层面**：
- **数据库层面**：满足完整性约束（主键、外键、唯一性、非空等）
- **业务层面**：满足业务规则（如转账前后总金额不变）

一致性是目标，原子性、隔离性、持久性都是为了保证一致性。

**实现机制**：
- 数据库层面：数据库自身的约束检查
- 业务层面：应用程序逻辑保证

### 隔离性 Isolation

多个事务并发执行时，一个事务的执行不应被其他事务干扰。

如果 A 给 B 转账的同时，C 也在给 B 转账，两个事务不应该互相影响。隔离性保证每个事务感觉像是独占数据库。

**实现机制**：锁、MVCC、隔离级别。

### 持久性 Durability

事务一旦提交，对数据库的修改就是永久的，即使系统崩溃也不会丢失。

转账成功后，即使数据库服务器立刻断电，重启后这笔转账记录依然存在。

**实现机制**：redo log（重做日志）。数据库会先把修改写入日志，再写入磁盘。崩溃恢复时，根据 redo log 重新执行已提交的操作。

## ACID 之间的关系

ACID 不是平级关系，而是有层次的：

**一致性是核心目标**，其他三个特性都是手段：

- **无并发场景**：只要满足原子性，就能保证一致性。因为事务串行执行，不会互相干扰。
- **有并发场景**：需要同时满足原子性和隔离性，才能保证一致性。因为多个事务可能同时修改同一数据。
- **持久性**：保证已提交的事务在系统崩溃后依然有效，是一致性在时间维度上的延伸。

举例说明：
- 如果没有原子性，转账只执行一半，总金额就不对了（破坏一致性）
- 如果没有隔离性，两个事务同时修改同一账户，可能互相覆盖（破坏一致性）
- 如果没有持久性，提交后崩溃导致数据丢失（破坏一致性）

## 为什么需要事务

**核心原因**：保证数据正确性。

没有事务的情况下：
- 操作执行一半失败，数据处于中间状态
- 多个操作并发执行，互相干扰导致数据错误
- 系统崩溃后，已完成的操作可能丢失

有了事务：
- 原子性保证操作要么全做要么全不做
- 隔离性保证并发操作不互相干扰
- 持久性保证已提交的操作不会丢失

## MySQL 的自动提交

MySQL 默认开启自动提交（AUTOCOMMIT），每条 SQL 语句都是一个独立事务，执行完自动提交。

```sql
-- 默认情况下，这条语句执行完就自动提交了
UPDATE account SET balance = balance - 100 WHERE user = 'A';
```

如果需要多条语句作为一个事务，必须显式开启：

```sql
START TRANSACTION;  -- 或者 BEGIN
-- 多条 SQL
COMMIT;  -- 或者 ROLLBACK
```

也可以关闭自动提交：

```sql
SET AUTOCOMMIT = 0;  -- 关闭自动提交
-- 之后每个事务都需要手动 COMMIT
```

## 追问

**Q: ACID 中哪个最重要？**
一致性是目标，其他三个是手段。但实际实现中，原子性和隔离性最难保证。

**Q: 原子性和一致性有什么区别？**
原子性是操作层面的"全或无"，一致性是数据层面的"符合约束"。原子性是实现一致性的手段之一。

**Q: 隔离性和一致性有什么区别？**
隔离性关注并发场景下事务之间的影响，一致性关注数据是否符合业务规则。隔离性是并发场景下保证一致性的手段。

**Q: 如果只有一个事务在执行，还需要隔离性吗？**
不需要。单事务场景下，只要保证原子性就能保证一致性。隔离性是为了解决并发问题。

**Q: redo log 和 undo log 分别保证什么？**
redo log 保证持久性（已提交的不丢失），undo log 保证原子性（失败能回滚）。

**Q: 事务提交后还能回滚吗？**
不能。COMMIT 之后事务就结束了，修改已经持久化。只有 COMMIT 之前可以 ROLLBACK。
