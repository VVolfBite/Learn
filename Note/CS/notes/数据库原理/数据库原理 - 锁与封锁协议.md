## 锁的作用

锁是解决并发一致性问题的核心机制。

通过锁，可以控制多个事务对同一数据的访问顺序，避免互相干扰。就像多个人要用同一个房间，需要用钥匙（锁）来协调谁先用、谁后用。

## 共享锁与排他锁

### 共享锁 S 锁（读锁）

多个事务可以同时持有同一数据的共享锁。

- 加了 S 锁后，可以读数据，但不能写
- 其他事务可以继续加 S 锁，但不能加 X 锁

**场景**：多个事务同时读取同一数据，互不影响。

### 排他锁 X 锁（写锁）

只有一个事务可以持有某个数据的排他锁。

- 加了 X 锁后，可以读也可以写
- 其他事务不能加任何锁（既不能加 S 锁，也不能加 X 锁）

**场景**：一个事务正在修改数据，其他事务必须等待。

**记忆口诀**：读读共享，读写互斥，写写互斥。

## 行锁与表锁

### 行锁

锁定某一行数据。

- 粒度小，并发度高
- 开销大（需要为每一行维护锁信息）

**场景**：`UPDATE account SET balance = 100 WHERE id = 1` 只锁 id=1 这一行。

### 表锁

锁定整张表。

- 粒度大，并发度低
- 开销小（只需要一个锁）

**场景**：`LOCK TABLE account WRITE` 锁住整张表。

### 如何选择

**锁粒度权衡**：
- 粒度越小，并发度越高，但开销越大
- 粒度越大，开销越小，但并发度越低

MySQL InnoDB 默认使用行锁，MyISAM 只支持表锁。

## 意向锁

### 为什么需要意向锁

假设事务 T1 对某一行加了 X 锁，事务 T2 想对整张表加 X 锁。

**没有意向锁**：T2 需要检查表中每一行是否被加锁，非常耗时。

**有意向锁**：T1 在对行加锁前，先对表加意向锁。T2 只需要检查表级锁，就知道表中有行被锁了。

### 意向锁的类型

- **意向共享锁 IS**：表示事务想要对表中某些行加 S 锁
- **意向排他锁 IX**：表示事务想要对表中某些行加 X 锁

### 加锁规则

- 对某行加 S 锁前，必须先对表加 IS 锁或更强的锁
- 对某行加 X 锁前，必须先对表加 IX 锁

- 表级 IX 和行级 X 兼容（两个事务可以锁不同的行）

## 封锁协议

封锁协议规定了何时加锁、何时释放锁，用于解决不同的并发问题。

### 一级封锁协议

**规则**：事务修改数据前必须加 X 锁，直到事务结束才释放。

**解决问题**：丢失更新。

**原理**：两个事务不能同时修改同一数据，后来的事务必须等前一个事务结束。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207220440451.png"/> </div><br>

### 二级封锁协议

**规则**：在一级基础上，读取数据前必须加 S 锁，读完立即释放。

**解决问题**：脏读。

**原理**：如果数据被加了 X 锁（正在修改），就不能加 S 锁（不能读），避免读到未提交的数据。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207220831843.png"/> </div><br>

### 三级封锁协议

**规则**：在二级基础上，读取数据前必须加 S 锁，直到事务结束才释放。

**解决问题**：不可重复读。

**原理**：读数据时加 S 锁并持有到事务结束，其他事务不能加 X 锁修改数据，保证多次读取结果一致。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207221313819.png"/> </div><br>

### 协议对比

| 封锁协议 | 读锁何时释放 | 写锁何时释放 | 解决的问题 |
|---------|------------|------------|-----------|
| 一级 | 不需要读锁 | 事务结束 | 丢失更新 |
| 二级 | 读完立即释放 | 事务结束 | 丢失更新、脏读 |
| 三级 | 事务结束 | 事务结束 | 丢失更新、脏读、不可重复读 |

## 两段锁协议 2PL

### 协议内容

事务的加锁和解锁分为两个阶段：
1. **加锁阶段**：只能加锁，不能解锁
2. **解锁阶段**：只能解锁，不能加锁

### 作用

保证事务的可串行化调度。

**可串行化**：并发执行的事务结果，等价于某种串行执行的结果。

### 例子

**满足两段锁协议**：
```
lock-x(A) ... lock-s(B) ... lock-s(C) ... unlock(A) ... unlock(C) ... unlock(B)
```
前三个操作是加锁阶段，后三个操作是解锁阶段。

**不满足两段锁协议**（但仍可能是可串行化的）：
```
lock-x(A) ... unlock(A) ... lock-s(B) ... unlock(B) ... lock-s(C) ... unlock(C)
```
加锁和解锁交替进行，不符合两段锁协议。

### 两段锁协议的局限

- 是可串行化的充分条件，但不是必要条件
- 可能导致死锁

## 乐观锁与悲观锁

这两个不是具体的锁，而是并发控制的思想。

### 悲观锁

**思想**：假设冲突一定会发生，所以每次操作都加锁。

**实现**：数据库的 S 锁、X 锁就是悲观锁。

**场景**：写操作频繁，冲突概率高。

**例子**：
```sql
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 加 X 锁
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

### 乐观锁

**思想**：假设冲突很少发生，不加锁，提交时检查是否有冲突。

**实现**：通过版本号或时间戳实现。

**场景**：读操作频繁，写操作少，冲突概率低。

**例子**：
```sql
-- 读取数据和版本号
SELECT balance, version FROM account WHERE id = 1;  -- balance=100, version=1

-- 更新时检查版本号
UPDATE account SET balance = 50, version = 2 
WHERE id = 1 AND version = 1;  -- 如果 version 已经不是 1，说明被别人改过了
```

### 对比

| 维度 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 加锁时机 | 操作前 | 不加锁 |
| 冲突处理 | 等待 | 提交时检查，冲突则重试 |
| 适用场景 | 写多读少 | 读多写少 |
| 性能 | 并发度低 | 并发度高 |

## MySQL 的隐式与显式锁定

### 隐式锁定

InnoDB 会根据隔离级别自动加锁，事务结束时自动释放。

```sql
UPDATE account SET balance = 100 WHERE id = 1;  -- 自动加 X 锁
```

### 显式锁定

手动指定加锁。

```sql
SELECT * FROM account WHERE id = 1 LOCK IN SHARE MODE;  -- 加 S 锁
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 加 X 锁
```

## 死锁

### 什么是死锁

两个或多个事务互相等待对方释放锁，导致都无法继续执行。

**例子**：
- T1 锁住 A，等待 B
- T2 锁住 B，等待 A
- 两个事务互相等待，形成死锁

### 死锁的处理

**检测**：数据库定期检测是否有死锁（检测等待图中是否有环）。

**解决**：选择一个事务回滚（通常选择代价小的），释放它持有的锁。

**预防**：
- 按固定顺序访问资源
- 设置锁超时时间
- 减少事务持有锁的时间

## 锁、隔离级别、MVCC 的关系

这三者不是平级关系，而是目标与手段的关系：

**隔离级别**：目标，定义了事务之间的隔离程度
- RU：不隔离
- RC：隔离未提交的修改
- RR：隔离所有修改（同一事务内可重复读）
- Serializable：完全隔离

**锁**：实现手段之一，通过互斥来保证隔离
- 用于当前读和写操作
- 会阻塞其他事务
- 并发性能较低

**MVCC**：实现手段之一，通过多版本来保证隔离
- 用于快照读
- 不阻塞其他事务
- 并发性能较高

**配合使用**：
- RC/RR 级别：快照读用 MVCC，当前读和写操作用锁
- Serializable 级别：所有操作都用锁
- RU 级别：不需要 MVCC 和锁

## 追问

**Q: 共享锁和排他锁的区别？**
共享锁用于读，多个事务可以同时持有；排他锁用于写，只有一个事务可以持有。

**Q: 为什么需要意向锁？**
为了快速判断表中是否有行被锁，避免逐行检查。

**Q: 两段锁协议能保证什么？**
保证可串行化调度，但可能导致死锁。

**Q: 乐观锁和悲观锁哪个好？**
看场景。读多写少用乐观锁，写多读少用悲观锁。

**Q: 如何避免死锁？**
按固定顺序访问资源、设置超时、减少锁持有时间。

**Q: 行锁和表锁的区别？**
行锁粒度小、并发高、开销大；表锁粒度大、并发低、开销小。

**Q: 三级封锁协议分别解决什么问题？**
一级解决丢失更新，二级解决脏读，三级解决不可重复读。
