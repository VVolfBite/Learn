# Security Basics

> 对后端来说，安全不是“上线前再加一层”的附件，而是接口、状态和数据在整个生命周期里的基本约束。

## 背景与作用

Web 安全和 API 安全关注的是系统如何识别请求、保护数据、限制权限、抵御滥用，并在出问题时留下可追踪的证据。它和具体框架无关，和业务是否大厂也无关；只要系统对外提供接口、处理身份、保存敏感信息，就已经进入安全问题的范围。

安全首先解决的是“系统能否被轻易滥用”。这包括冒充身份、越权访问、注入恶意输入、窃取数据、伪造请求和放大流量等问题。很多攻击看上去花样很多，本质上都来自几个基础薄弱点：输入不可信、边界不清楚、权限太宽、传输裸奔、日志不足。

## 场景与边界

面试问安全，不是要求候选人像安全专家那样谈漏洞利用，而是想知道你有没有基础安全直觉。工作里更是如此：登录、权限、接口设计、文件上传、日志、密钥管理，几乎每一个后端模块都和安全相关。懂基础安全，很多设计会在一开始就更稳；不懂基础安全，补洞会越来越被动。身份体系里常见的 Basic Auth、2FA、OAuth、OIDC、SAML，本质上也都是在回答“怎么证明你是谁、怎么把身份安全地交给系统或第三方”这个大问题。

最典型的误区是把安全理解成“登录做了就差不多了”。身份认证只是入口，权限控制、输入校验、传输保护、最小权限、审计和速率控制一样重要。另一个误区是以为安全只和框架配置有关。框架能提供工具，但真正决定安全上限的仍然是系统边界和设计习惯。

## 面试表达

对后端来说，安全是一组基础约束，包括身份确认、权限控制、输入校验、传输保护和审计追踪。它不是单点能力，而是贯穿接口设计、数据存储和系统运维的整体问题。面试里如果只谈 JWT 或 HTTPS，通常还不够，最好能把这些能力串成一个完整视角。

## 快速回忆

- 安全是系统性问题，不是某一个中间件。
- 身份、权限、输入、传输、审计是主线。
- 框架能帮忙，但设计习惯更关键。
- 安全基础薄弱时，系统功能越多风险越大。
