# CSRF

> CSRF 的关键不在“伪造了内容”，而在“浏览器帮攻击者自动带上了原本属于受害者的身份”。

## 背景与作用

CSRF 是跨站请求伪造。它利用的是浏览器在某些场景下会自动携带 Cookie 或其他凭证这一特性，使受害者在已登录目标站点的情况下，被诱导向目标站点发出一个并非自己本意的请求。攻击者不一定能读到响应，但只要请求真的被执行，风险就已经发生。

严格说，CSRF 不是“解决什么问题”，而是提醒我们一个认证方案的边界：只要身份凭证会被浏览器自动带上，服务端就不能只凭“带了凭证”就默认这是用户真实意图。防范 CSRF，就是要让服务端额外确认这个请求是否真的来自受信任的交互上下文。

## 场景与边界

CSRF 是浏览器安全和认证机制交叉处的经典问题。面试问它，往往是在看你是否理解 Cookie 场景的风险。工作里只要应用使用基于浏览器自动携带凭证的方案，表单提交、状态修改、后台管理操作就都有必要考虑 CSRF。

最常见的误区是把 CSRF 和 XSS 混在一起。两者都危险，但成因不同。另一个误区是觉得“用了 Token 就绝对没有 CSRF”。更准确地说，风险是否存在取决于凭证是否会被浏览器自动携带以及服务端如何验证请求来源。常见防护思路包括 CSRF Token、SameSite Cookie、校验来源等，但重点始终是理解攻击原理。

## 面试表达

CSRF 利用的是浏览器自动携带身份凭证的行为。受害者在已登录站点的情况下，被诱导访问恶意页面后，浏览器可能替攻击者向目标站点发出带有合法凭证的请求。防护的核心不是“隐藏接口”，而是让服务端能区分真正的用户操作和被伪造的跨站请求。

## 快速回忆

- CSRF 借的是浏览器自动带凭证。
- 风险常见于基于 Cookie 的认证场景。
- 重点是区分真实用户意图和伪造请求。
- CSRF Token 和 SameSite 是常见防护思路。
