# OAuth 2.0

> OAuth 2.0 的核心不是“登录”，而是授权委托：让第三方应用在用户同意的前提下，代表用户访问受保护资源。

## 背景与作用

OAuth 2.0 是一种授权框架。它解决的问题不是用户如何证明自己是谁，而是用户如何把某部分访问权交给别的应用，而不用直接把账号密码交出去。比如一个第三方日历应用希望读取你的邮箱日程，它更合理的做法不是索要邮箱密码，而是让你在授权页面中明确同意它访问特定资源。

OAuth 2.0 最大的价值是把“身份所有者”和“客户端应用”之间的信任关系拆开管理。用户可以只授权一部分能力，令牌也可以设置范围、有效期和撤销方式。这种做法既减少了直接暴露账号密码的风险，也让平台更容易控制第三方访问边界。

## 场景与边界

第三方登录、开放平台、企业集成、服务对服务授权，这些场景都可能碰到 OAuth 2.0。面试问它，往往是想确认候选人是否知道它的真正定位。工作里如果把 OAuth 误解成“登录协议”，后续在设计身份认证、用户信息获取和权限范围时就很容易混乱。

最大的误区就是把 OAuth 2.0 直接等同于登录。OAuth 解决的是授权，不是身份认证本身。另一个误区是死记所有授权流程却不知道核心思想。对后端入门来说，真正要掌握的是授权委托的意义，以及为什么现代系统更推荐安全边界更明确的授权方式，而不是沉迷于流程名词。

## 面试表达

OAuth 2.0 是一个授权框架，用来让第三方应用在用户同意的前提下获取受限资源的访问权限，而不必直接掌握用户密码。它的核心是授权委托，不是身份认证。实际系统里如果还需要确认用户是谁，通常要结合更偏认证语义的方案，比如 OpenID Connect。

## 快速回忆

- OAuth 2.0 解决授权委托。
- 它不是登录协议。
- 重点是权限范围、令牌和第三方访问控制。
- 记住思想比背流程更重要。
