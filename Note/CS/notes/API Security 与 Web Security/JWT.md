# JWT

> JWT 的价值在于便于传递和验证，不在于“更现代”。它适合某些场景，但并不天然优于其他凭证方案。

## 背景与作用

JWT 是一种结构化的令牌格式，通常由头部、载荷和签名三部分组成。它最常见的用途是把身份相关信息封装成一个可验证的字符串，让服务端能够在不查会话存储的情况下完成基本验证。这里要注意，JWT 的常见形态是签名而不是加密，所以其中的内容并不适合随意放敏感信息。

JWT 主要解决的是跨服务或跨节点传递身份时的便利性。服务端不必总是依赖集中式 Session 存储，就能根据签名验证令牌是否被篡改，并读取其中声明的信息。这对网关鉴权、服务间透传身份、前后端分离应用中的认证流程都很有吸引力。

## 场景与边界

JWT 太常见了，以至于很多人把它当成默认答案。面试会用它看你是否知道它的优势和代价。工作里真正重要的不是会不会签发 JWT，而是是否知道它适合什么场景、不适合什么场景，以及为什么“无状态”会把吊销、主动失效和权限实时变更变得更难。

一个常见误区是把 JWT 当成加密容器。多数 JWT 只是签名，能防篡改，不代表内容不可见。另一个误区是把 JWT 当成天然更先进的认证方式。它确实便于扩展和分布式传递，但也带来了令牌泄露后更难控制、主动吊销不方便、载荷容易膨胀等问题。是否使用 JWT，应当根据架构需要决定，而不是把它当成技术时髦度的象征。

## 面试表达

JWT 是一种结构化令牌格式，适合在分布式环境中携带身份声明并通过签名做验证。它的优点是便于传递、降低对集中会话存储的依赖，但代价是吊销和即时失效处理更麻烦，而且默认并不加密。所以它适合某些场景，不是所有系统都必须使用的默认方案。

## 快速回忆

- JWT 常见是签名，不是加密。
- 适合传递身份声明，不代表适合存敏感信息。
- 优点是方便扩展，代价是吊销和失效更难。
- 选不选 JWT 取决于场景，不取决于流行度。
