# Session Cookie Token

> Session、Cookie 和 Token 不是三个同义词，而是三个不同层次的概念：Session 是服务端状态，Cookie 是浏览器携带方式，Token 是身份凭证形式。

## 背景与作用

Session 是服务端保存的一段会话状态，常用于记录某个用户已经登录、拥有哪些上下文信息。Cookie 是浏览器保存并在后续请求中自动携带的小块数据，它本身不是认证方案，只是一个传输和存储机制。Token 则是身份凭证，可以是随机字符串，也可以是带签名的结构化内容，比如 JWT。

HTTP 天生无状态，服务端默认无法知道前后两次请求是否属于同一个用户。Session 解决的是服务端如何记住会话，Cookie 解决的是浏览器怎样自动把会话标识带回来，Token 解决的是“拿什么表示身份”以及“服务端如何验证这个身份”。三者既可组合使用，也可以分开使用，所以必须先分清层次，才能讨论方案优劣。

## 场景与边界

这三个词是 Web 认证里最容易混淆的一组概念。面试经常拿它们做基本功判断。工作里更明显：传统网页常见 Session 加 Cookie，前后端分离项目常见 Token 方案，而安全问题又会进一步涉及 Cookie 属性、Session 存储和 Token 生命周期。概念一混，后续很多判断都会偏。

最常见的误区是把 Cookie 和 Session 当成一个东西。更准确地说，Cookie 常常只是用来携带 Session ID。另一个误区是把 Token 理解成天然无状态、天然更先进。Token 只是凭证形式，是否无状态取决于服务端怎样验证和管理它。还有一种误区是只谈“存在哪里”，不谈“怎样保护”。其实 Cookie 的 HttpOnly、Secure、SameSite 等属性，以及服务端的过期与吊销策略，同样重要。

## 面试表达

Session 是服务端保存的会话状态，Cookie 是浏览器自动存储和携带数据的机制，Token 是用于表达身份的凭证。常见组合是把 Session ID 放进 Cookie，也可以把某种 Token 放进 Cookie 或请求头。区分这三者的关键是分清状态在哪一侧、凭证是什么形式、由谁负责自动携带。

## 快速回忆

- Session 是服务端状态。
- Cookie 是浏览器携带机制。
- Token 是身份凭证形式。
- Cookie 不等于 Session，Token 也不等于 JWT。
