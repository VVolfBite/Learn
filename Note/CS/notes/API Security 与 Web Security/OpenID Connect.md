# OpenID Connect

> 如果说 OAuth 2.0 更偏授权，OpenID Connect 则是在它之上补上“我到底是谁”这一层身份语义。

## 背景与作用

OpenID Connect 通常简称 OIDC，它建立在 OAuth 2.0 之上，用来提供身份认证能力。简单理解，OAuth 解决“第三方能拿到什么资源”，OIDC 解决“系统如何确认当前用户身份，并把身份信息安全地交给客户端”。

很多实际场景并不只是想拿访问资源的授权，还要完成单点登录、第三方登录、统一身份接入等能力。仅靠 OAuth 2.0，客户端未必能可靠判断用户身份；OIDC 则通过额外的身份令牌和标准化信息约定，把这件事补齐。

## 场景与边界

现代互联网产品和企业内部系统都大量依赖统一登录和第三方身份接入，所以 OIDC 非常常见。面试问这部分，通常是为了看你能否把授权和认证分清。工作里如果把 OIDC 和 OAuth 2.0 混为一谈，系统设计就容易在身份来源和访问授权两条线之间打结。

一个误区是认为 OIDC 是 OAuth 2.0 的替代品。更准确地说，OIDC 是建立在 OAuth 2.0 之上的身份层扩展。另一个误区是只记“第三方登录会用到”，却不知道它真正解决的问题是身份确认标准化和单点登录协作。

## 面试表达

OpenID Connect 是基于 OAuth 2.0 的身份认证协议，用来补足“用户是谁”这一层能力。OAuth 2.0 更偏授权，OIDC 更偏认证，所以第三方登录、单点登录等场景里通常会看到它们一起出现。

## 快速回忆

- OAuth 偏授权，OIDC 偏认证。
- OIDC 不是替代 OAuth，而是建立在其上的扩展。
- 第三方登录和 SSO 是典型使用场景。
- 分清“资源授权”和“身份确认”是关键。
