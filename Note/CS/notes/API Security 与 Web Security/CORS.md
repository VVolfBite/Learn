# CORS

> CORS 不是后端鉴权方案，而是浏览器为了限制跨源读取而引入的一套协商机制。

## 背景与作用

CORS 的全称是跨源资源共享。浏览器默认会限制一个源中的脚本去读取另一个源返回的数据，这就是同源策略的一部分。CORS 的作用不是帮服务端判断“谁能访问”，而是让服务端通过响应头明确告诉浏览器：哪些跨源读取是被允许的。

前后端分离、子域拆分、静态站点调用 API，这些场景常常都会跨源。如果没有 CORS，浏览器端脚本即使请求成功，也可能无法读取响应内容。CORS 解决的是这种“浏览器是否允许前端代码拿到跨源响应”的问题，而不是“请求有没有发到服务器”。

## 场景与边界

CORS 是前后端协作里极高频的基础问题。面试喜欢问它，因为太多人会把它和鉴权混为一谈。工作里如果理解错了，经常会出现两种极端：一种是把 CORS 放得过宽，另一种是把明明该由认证和授权解决的问题，误交给 CORS 去兜底。

最大的误区是把 CORS 当成安全总开关。它主要约束浏览器环境中的跨源读取，不是服务端鉴权机制，也挡不住非浏览器客户端。另一个误区是认为“服务器收到了请求，就说明 CORS 没问题”。实际上 CORS 关注的是浏览器能否把响应交给脚本读取，其中还会涉及预检请求等机制。

## 面试表达

CORS 是浏览器跨源资源共享机制，用来控制前端脚本在跨源场景下能否读取响应。它是浏览器安全策略的一部分，不是服务端鉴权方案。真正决定请求是否应被允许的，仍然是认证、授权和接口本身的访问控制。

## 快速回忆

- CORS 管的是浏览器跨源读取。
- 它不是后端鉴权机制。
- 能否发出请求和能否读取响应不是一回事。
- 认证授权问题不能拿 CORS 代替。
