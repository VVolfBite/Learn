# XSS SQL Injection and Input Risks

> 后端最稳的输入安全直觉只有一条：外部输入默认不可信。很多漏洞表面不同，本质上都是把不可信输入当成了可直接执行、拼接或展示的内容。

## 背景与作用

输入风险指的是系统在接收用户、浏览器、第三方服务或其他系统传来的数据时，没有正确地区分“数据”和“指令”。XSS 发生在不安全地输出内容到页面时，SQL 注入发生在把输入直接拼进 SQL 时，而更广义的输入校验问题则出现在任何把外部值直接带入关键逻辑的地方。

讨论这些风险的目的，是建立统一的处理原则。只要输入未经约束、未经编码或未经参数化就直接进入页面、数据库、命令行或权限逻辑，攻击面就出现了。不同漏洞的具体触发点不同，但防守思路往往高度一致：识别边界、验证格式、参数化执行、在合适的位置做编码。

## 场景与边界

这是安全基础里最实用的一组知识。面试经常用来确认候选人是否具备最基本的输入安全意识。工作里文件名、搜索条件、富文本、筛选参数、排序字段、第三方回调内容都可能成为输入风险入口。懂得“输入不可信”这件事，很多设计会自然更保守、更稳。

一个误区是把输入校验理解成只检查长度和非空。真正有效的校验是围绕业务允许的格式和范围建立白名单思维。另一个误区是认为“前端校验过了，后端就可以相信”。后端必须把所有进入系统边界的数据当成不可信来源。至于 XSS 和 SQL 注入，也不要只记漏洞名，要知道它们分别发生在输出上下文和查询拼接上下文。

## 面试表达

XSS、SQL 注入以及更广义的输入风险，本质上都和不可信输入被错误地解释有关。后端设计里应当默认外部输入不可信，根据场景做严格校验，并通过参数化查询、正确输出编码等方式，把数据和指令区分开。与其死记漏洞名，不如先建立统一的输入边界意识。

## 快速回忆

- 外部输入默认不可信。
- SQL 注入的核心是拼接执行。
- XSS 的核心是危险内容被当作页面脚本解释。
- 校验、参数化、输出编码是主线。
