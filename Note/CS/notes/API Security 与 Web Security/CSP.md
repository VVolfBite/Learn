# CSP

> CSP 不是用来替代输入校验的，它更像浏览器端的一层额外约束，用来减少注入成功后的破坏面。

## 背景与作用

CSP 全称内容安全策略。它通过响应头或页面策略，告诉浏览器页面允许加载哪些资源、允许执行哪些脚本，从而限制恶意内容的执行空间。它经常和 XSS 一起被讨论，因为它最直观的价值就是在脚本注入类问题发生时，帮助浏览器少做“危险的事”。

即使开发者已经尽量避免 XSS，现实系统里仍可能存在遗漏。CSP 提供的是一道额外防线：当页面只允许受信任来源的脚本和资源时，攻击者注入的内容就更难直接执行。它不是根治措施，但能在很多场景下显著降低攻击成功率和影响范围。

## 场景与边界

CSP 在大型前端应用、后台系统和高敏页面中都很常见。面试问它，更多是看你是否知道浏览器端也能参与安全防护。工作里虽然配置会偏前端或平台侧，但后端需要知道它解决什么、不解决什么，这样在安全方案讨论里才不会把责任全推给某一层。

最大的误区是把 CSP 当成 XSS 的替代品。更准确地说，输入校验和输出编码仍然是主防线，CSP 是附加保护层。另一个误区是以为配置上了就能覆盖所有脚本风险。策略配置本身也需要精细设计，过宽的策略意义有限，过严的策略又可能影响业务。

## 面试表达

CSP 是浏览器端的内容执行约束机制，常用于限制脚本和资源的来源，从而降低 XSS 等注入问题的攻击面。它不是替代输入校验的万能方案，而是和输出编码、模板安全一起组成多层防护。

## 快速回忆

- CSP 降低注入后的破坏面。
- 它是附加防线，不是主防线。
- 常与 XSS 防护一起讨论。
- 浏览器也能参与安全约束。
