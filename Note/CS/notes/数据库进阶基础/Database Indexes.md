# Database Indexes

> 索引的核心价值是用额外空间和写入成本，换更快的定位与排序。索引从来不是“加上就会更快”的免费优化。

## 背景与作用

索引可以理解为数据库为特定列建立的快速查找结构。没有索引时，数据库常常只能扫描大量数据才能找到目标；有了索引之后，就能更快地缩小范围、定位记录，某些排序和范围查询也会因此受益。它的本质和书的目录很像，只是维护成本比目录高得多。

索引主要解决查询慢的问题，尤其是按主键、唯一键、时间、状态、业务编号这类高频条件检索时效果最明显。它也经常帮助数据库避免无谓排序或大范围扫描，所以一旦接口核心性能取决于检索条件，索引通常就是最先考虑的工具。

## 场景与边界

索引是数据库性能调优最常见、最容易被误用的主题。面试问索引，往往想看你是否理解成本交换；工作里很多慢 SQL 最终也会回到“有没有合适索引、索引是否命中、索引设计是否和查询条件匹配”这些问题。懂一点索引，性能排查会更有抓手。

最常见的误区是认为索引越多越好。索引越多，写入、更新、删除的维护成本越高，占用的空间也越大。另一个误区是只知道“建索引”，不知道“索引如何被查询使用”。联合索引、覆盖索引这些概念之所以重要，是因为索引是否有效并不只取决于有没有，而取决于查询条件、排序方式和扫描范围是否与索引结构匹配。

## 面试表达

索引是数据库为查询建立的快速定位结构，它通过空间和写入成本换取读性能。设计索引时要结合高频查询条件、排序方式和返回字段来考虑，而不是看到慢查询就随手加。好的索引能明显减少扫描，差的索引则可能只是增加维护负担。

## 快速回忆

- 索引提升读性能，但增加写成本。
- 不是索引越多越好。
- 索引是否有效取决于查询模式是否匹配。
- 联合索引和覆盖索引是理解“为什么命中”的入口。
