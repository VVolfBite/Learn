# Failure Modes

> 可靠数据库不是“不会出故障”的数据库，而是“故障发生时行为仍然可理解、可恢复、可控制”的数据库。

## 背景与作用

Failure modes 指系统可能以哪些方式失败。数据库不是只有“挂了”这一种失败，常见情况包括主机宕机、磁盘损坏、网络分区、主从延迟、写入超时、锁等待、死锁、流量突增导致的资源耗尽。这些失败模式决定了系统应该怎样做复制、切换、重试和降级。

如果只按“理想数据库永远可用”来设计，一旦遇到真实故障，系统就会出现读到旧数据、写入丢失、请求堆积、切换放大事故等连锁问题。理解故障模式的价值在于，设计时就知道哪些风险必须正面处理，哪些后果需要业务接受，哪些地方要提前留下观测和恢复手段。

## 场景与边界

面试问数据库故障，通常是在看你是否已经超出“单机表设计”的理解。工作里只要服务进入生产环境，就一定会和故障打交道。数据库是状态核心，一旦故障处理没有预案，应用层往往很难兜住。因此即使你不负责 DBA 工作，也需要知道复制延迟意味着什么，锁等待会把系统拖成什么样，以及为什么故障切换常常不是一键魔法。

最大的误区是把高可用理解成“数据库永不宕机”。更现实的理解是：允许故障发生，但故障时系统仍有边界，有降级路径，有恢复方案。另一个误区是把所有数据库失败都交给重试。某些失败确实能靠重试恢复，某些失败会因为盲目重试而放大冲突和拥塞，必须结合失败原因判断。

## 面试表达

数据库常见的 failure modes 不只是宕机，还包括网络分区、复制延迟、锁冲突、死锁、磁盘和资源耗尽等。理解它们的意义不在于记名词，而在于知道架构设计为什么要做复制、故障切换、限流和观测，以及为什么应用层不能把数据库当成永远稳定的黑盒。

## 快速回忆

- 故障模式是设计输入，不是事故总结。
- 宕机只是失败的一种。
- 复制延迟、锁等待、网络分区都可能改变业务语义。
- 可靠性来自预案和边界，不来自幻想“不会出事”。
