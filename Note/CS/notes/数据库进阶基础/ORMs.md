# ORMs

> ORM 提升的是开发效率，不是数据库理解力。写得顺手不代表代价不存在，很多性能问题本质上都是对 ORM 行为缺乏感知。

## 背景与作用

ORM 是对象关系映射。它把表、行、关系这些数据库概念映射到语言里的对象、属性和关联，让开发者可以用更接近业务对象的方式做增删改查。这样做的直接好处是减少样板 SQL，统一数据访问风格，也让应用代码更容易和领域对象放在一起理解。

在没有 ORM 的时候，开发者要手写 SQL、拼参数、映射结果集、处理连接和事务边界。ORM 把这些重复劳动抽象起来，最适合解决的是日常 CRUD、简单关联查询和标准化的数据访问。它让大部分业务代码不必直接暴露在 SQL 细节里，所以在中后台系统、管理系统、常规业务服务里几乎是默认配置。

## 场景与边界

面试问 ORM，通常不是想听框架 API，而是想看你是否理解“抽象给了你什么，也遮住了什么”。工作里 ORM 确实能让迭代更快，但一旦查询复杂、数据量上来、关联过深，问题就会很快暴露出来。到那时，真正决定上限的不是你会不会调用 ORM，而是你知不知道它背后会生成怎样的 SQL，是否能读懂执行代价。

最常见的误区是把 ORM 当成数据库的替代品。ORM 只能帮你表达访问意图，不能替你理解索引、事务、锁、执行计划和数据分布。另一个误区是为了“纯粹”而拒绝原生 SQL。实际工程里，简单查询用 ORM 可以提高效率，复杂报表、批量操作、高性能热点场景回到 SQL 往往更稳，也更容易控制成本。

## 面试表达

ORM 是对象和关系表之间的映射层，它的价值是减少重复 SQL 和结果映射，提高日常开发效率。但 ORM 不是没有代价的，它会带来抽象泄漏、N+1、生成 SQL 不可控、复杂查询表达笨重等问题。所以真实项目里通常是以 ORM 为主，在复杂和热点场景里补原生 SQL，而不是二选一。

## 快速回忆

- ORM 适合常规 CRUD，不适合所有查询。
- ORM 隐藏 SQL，但 SQL 成本不会消失。
- 会用 ORM 不等于理解数据库。
- 性能热点、复杂查询、批量操作常常要回到 SQL。
