# Migrations

> 数据库迁移不是“改一下表结构”这么简单，它本质上是线上状态演进。真正难的不是写 DDL，而是让旧应用、新应用和旧数据、新结构平稳过渡。

## 背景与作用

Migration 指数据库结构和相关数据随应用版本一起演进的过程。它通常包括新增表、修改字段、建立索引、迁移历史数据、清理旧结构等。只要应用上线后仍在持续迭代，数据库迁移就是不可避免的日常动作。

业务会变化，数据结构也必须跟着变化。如果没有明确的迁移过程，开发环境和生产环境就会迅速失去一致性，版本发布也会越来越危险。Migration 的价值在于把结构变更从“手工操作”变成有序、可追踪、可回滚或至少可补救的演进过程。

## 场景与边界

面试问迁移，是想看你是否意识到数据库变更的风险远高于普通代码改动。工作里很多事故并不是代码逻辑写错，而是迁移步骤不兼容、锁表过久、索引创建方式不合适，或者新旧版本同时运行时对字段语义理解不一致。

最常见的误区是把 migration 理解成只要“能改成功”就行。真实上线要考虑兼容性、发布窗口、回滚预案、历史数据修复和双版本共存。另一个误区是一次性做破坏性大改。更稳的做法通常是分阶段演进：先加新结构并兼容旧逻辑，再切流量，再清理旧字段和旧路径。

## 面试表达

数据库迁移是应用和数据一起演进的过程。它的重点不是 DDL 语法，而是兼容性和发布顺序。安全的迁移通常会分阶段做，尽量避免长时间锁表和破坏性变更，并且提前想好失败时怎样恢复或回退。

## 快速回忆

- Migration 是线上状态演进。
- 重点在兼容性，不在命令本身。
- 大改尽量拆成多步。
- 数据库变更比普通代码变更更危险。
