# Data Replication

> 复制让数据库更能扛读流量和单点故障，但复制从来不是“白送副本”。只要有复制，就一定要接受延迟、切换和一致性方面的新问题。

## 背景与作用

数据复制是把一份数据变化同步到其他副本上。最常见的理解是主从或主副本结构：一个节点负责主要写入，其他节点同步变更后提供读能力或故障接管。复制的目标不是把数据库“多开几份”，而是让服务在读取、容灾和高可用上获得更大的弹性。

复制首先解决的是单点风险。只有一个实例时，一旦故障，服务就很容易整体受影响。复制还可以分担读流量，把大量查询从写主库分出去。对于灾备和恢复而言，副本也是非常重要的基础，因为它提供了更好的恢复选择和切换空间。

## 场景与边界

复制是数据库高可用和读扩展的基本手段，所以面试里很常出现。工作里即使你不直接维护数据库，也会受到复制行为的影响，比如刚写入的数据在从库上暂时读不到，故障切换后连接目标发生变化，或者应用需要区分强一致读和普通读。

最常见的误区是觉得有了复制就天然高可用。真实系统还要考虑复制延迟、脑裂风险、切换策略、读写路由和恢复流程。另一个误区是认为副本上的数据永远和主库同步。异步复制下，延迟是常态而不是例外，这意味着某些读请求必须明确知道自己能不能接受旧数据。

## 面试表达

数据复制是把主数据库的数据变化同步到副本，用来提升读能力、容灾能力和故障恢复能力。它的收益很明显，但代价也很明确，包括复制延迟、切换复杂度以及一致性语义变弱。设计时最关键的是想清楚哪些读场景可以接受延迟，哪些场景必须读到最新结果。

## 快速回忆

- 复制带来读扩展和高可用基础。
- 有复制就有延迟问题。
- 副本不代表永远实时一致。
- 复制方案的难点在切换和一致性语义。
