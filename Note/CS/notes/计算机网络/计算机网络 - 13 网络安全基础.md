## 中间人攻击

**中间人攻击（Man-in-the-Middle Attack，MITM）**是指攻击者插入到通信双方之间，截获、监听甚至篡改通信内容的攻击方式。正常情况下，客户端和服务器直接通信。中间人攻击时，攻击者处于两者之间，对客户端伪装成服务器，对服务器伪装成客户端。双方都以为在直接通信，实际上所有数据都经过攻击者，可以被查看、修改、伪造。

```
正常通信：
客户端 ←→ 服务器

中间人攻击：
客户端 ←→ 攻击者 ←→ 服务器
         (伪装成服务器) (伪装成客户端)
```

### 攻击过程

**典型的中间人攻击流程**：

**1. 拦截通信**：攻击者通过各种手段拦截客户端和服务器之间的通信，常见方式有ARP 欺骗、DNS 劫持、恶意 WiFi 热点

**2. 建立双向连接**：攻击者与客户端建立连接（伪装成服务器），攻击者与服务器建立连接（伪装成客户端）

**3. 转发和篡改**：攻击者接收客户端的请求，可以查看、修改请求内容，然后将请求转发给服务器；当接收服务器的响应，可以查看、修改响应内容，然后将响应转发给客户端

**具体示例**：

```
1. 用户输入用户名和密码
   ↓
2. 客户端发送登录请求（HTTP）
   POST /login
   username=alice&password=123456
   ↓
3. 攻击者拦截请求
   - 记录用户名和密码
   - 可以修改密码
   ↓
4. 攻击者转发请求给服务器
   ↓
5. 服务器返回响应
   ↓
6. 攻击者拦截响应
   - 可以修改响应内容
   ↓
7. 攻击者转发响应给客户端
```

**攻击者可以**：窃取敏感信息（密码、信用卡号等）、篡改通信内容、注入恶意代码、伪造响应、劫持会话

### 防御措施

最有效的防御是使用 HTTPS。通过 SSL/TLS 加密通信，即使被拦截也无法解密；通过数字证书验证服务器身份，防止伪装。浏览器访问 HTTPS 网站时会检查证书的有效性，如果证书有问题会显示警告，这时千万不要忽略。

在公共 WiFi 环境下特别危险，因为攻击者很容易在同一网络中进行中间人攻击。这种情况下最好使用 VPN 建立加密隧道，或者直接用移动数据网络。不要在公共 WiFi 下进行敏感操作，也不要连接来历不明的 WiFi 热点。

对于网站来说，可以使用 HSTS（HTTP Strict Transport Security）强制浏览器使用 HTTPS，防止降级攻击。对于移动应用，可以使用证书固定（Certificate Pinning），在应用程序中内置服务器证书，只信任特定的证书。

**相关案例**：2015年Lenovo Superfish事件 2014年9月至2015年2月，联想在数百万台消费级笔记本电脑上预装了Superfish VisualDiscovery广告软件。该软件为了在用户浏览网页时插入针对性广告，主动在系统里安装了一个自签名的根证书（非唯一CA证书）。当用户访问任何HTTPS网站时，浏览器会信任这个根证书，Superfish软件就成为“中间人”：它解密HTTPS流量、查看/修改内容、插入广告，再重新加密转发给用户。影响全球数百万台联想电脑，FTC（美国联邦贸易委员会）最终对联想处以巨额和解金，并要求其停止此类行为。用户隐私被系统性暴露，即使使用HTTPS也形同虚设。

## 重放攻击

**重放攻击（Replay Attack）**是指攻击者截获合法的通信数据，然后将其重新发送给接收方，以达到欺骗的目的。

攻击者不需要破解加密，只需要重新发送截获的数据。比如你转账 100 元，攻击者截获这个请求，然后重放 10 次，你就转了 1000 元。即使请求是加密的，攻击者也不需要知道内容是什么，只要重新发送就行。

### 攻击过程

```
1. 客户端 → 服务器：合法请求（被攻击者截获）
2. 攻击者 → 服务器：重放相同的请求
3. 服务器：再次执行操作
```

**场景 1：重复转账**

```
1. 用户 A 向用户 B 转账 100 元
   请求：POST /transfer
   { "from": "A", "to": "B", "amount": 100 }
   ↓
2. 攻击者截获这个请求
   ↓
3. 攻击者重放这个请求 10 次
   ↓
4. 如果没有防护，用户 A 会转账 1000 元
```

**场景 2：重复登录**

```
1. 用户登录成功，获得 Session ID
   响应：Set-Cookie: SESSIONID=abc123
   ↓
2. 攻击者截获 Session ID
   ↓
3. 攻击者使用这个 Session ID 访问系统
   ↓
4. 如果 Session 未过期，攻击者可以冒充用户
```

**场景 3：重复提交订单**

```
1. 用户提交订单
   请求：POST /order
   { "product_id": 123, "quantity": 1 }
   ↓
2. 攻击者截获请求
   ↓
3. 攻击者重放请求多次
   ↓
4. 如果没有防护，会创建多个订单
```

### 防御措施

最常用的方法是**时间戳**。在请求中包含时间戳，服务器检查时间戳是否在有效期内（比如 60 秒），超过有效期就拒绝。这样即使攻击者截获了请求，过一会儿就失效了。另一种方法是**随机数（Nonce）**。每次请求包含一个唯一的随机数，服务器记录已使用的随机数，拒绝重复的。这样每个请求只能用一次。还可以用**序列号**。每次请求包含递增的序列号，服务器检查序列号是否递增，拒绝序列号不正确的请求。

更安全的做法是组合使用：HTTPS 防止截获 + 时间戳限制有效期 + 签名防止篡改。签名用密钥对请求内容进行哈希，服务器验证签名是否正确。

```json
{
  "action": "transfer",
  "amount": 100,
  "timestamp": 1678176000,
  "signature": "abc123xyz..."
}
```

签名计算：`signature = HMAC(secret_key, action + amount + timestamp)`

**相关案例**：汽车无钥匙进入系统重放攻击（2016年起多款车型曝光） 攻击者使用廉价无线电设备（如HackRF或Yard Stick One）在车主按下钥匙扣解锁按钮时捕获合法无线信号（包含加密的授权数据包），然后在车主离开后原样重放该信号，车辆就会再次解锁/启动。研究人员在2016-2021年间公开演示了针对Honda、Acura等多款车型的攻击，甚至Tesla早期Model S也曾存在类似漏洞（无需破解加密，只需重放）。

## 常见网络安全概念

### XSS 攻击

**XSS（Cross-Site Scripting，跨站脚本攻击）**是指攻击者在网页中注入恶意脚本，当其他用户浏览该网页时，恶意脚本会在用户浏览器中执行。

**攻击类型**：

* **反射型 XSS**：恶意脚本通过 URL 参数传递，服务器将参数直接输出到页面，这种攻击需要诱导用户点击恶意链接。

```
URL: http://example.com/search?q=<script>alert('XSS')</script>
页面显示：搜索结果：<script>alert('XSS')</script>
```

* **存储型 XSS**：恶意脚本存储在服务器（如数据库），其他用户访问时会执行恶意脚本，其危害更大，影响所有访问用户

```
用户在评论中输入：<script>steal_cookie()</script>
其他用户查看评论时，脚本会执行
```

**DOM 型 XSS**：恶意脚本通过修改 DOM 执行，其不经过服务器，完全在客户端发生

**防御措施**：输入验证和过滤、输出编码（HTML 实体编码）、使用 Content Security Policy（CSP）、设置 Cookie 的 HttpOnly 属性、使用安全的 JavaScript 框架

**相关案例**：2005年MySpace Samy蠕虫,19岁黑客Samy Kamkar在自己MySpace个人资料页嵌入一段仅4KB的恶意JavaScript（利用反射型+存储型XSS漏洞）。当其他用户浏览他的页面时，脚本自动执行：在后台向MySpace服务器发送好友请求（添加Samy为好友），并在受害者“我的英雄”栏插入“but most of all, samy is my hero”文字，同时把这段恶意代码复制到受害者自己的资料页。仅20小时内感染超过100万用户，MySpace服务器因好友列表爆炸而瘫痪两天，成为历史上传播最快的XSS蠕虫。

### CSRF 攻击

**CSRF（Cross-Site Request Forgery，跨站请求伪造）**是指攻击者诱导用户在已登录的网站上执行非本意的操作。

**攻击过程**：

```
1. 用户登录 bank.com，获得 Cookie
   ↓
2. 用户访问恶意网站 evil.com
   ↓
3. evil.com 的页面包含恶意请求
   <img src="http://bank.com/transfer?to=attacker&amount=1000">
   ↓
4. 浏览器自动携带 bank.com 的 Cookie 发送请求
   ↓
5. bank.com 认为这是用户的合法请求，执行转账
```

**防御措施**：使用 CSRF Token、检查 Referer 头部、使用 SameSite Cookie 属性、重要操作需要二次验证、使用 POST 而不是 GET 进行状态修改

**相关案例**：2008年YouTube CSRF漏洞（及同期ING Direct银行案例）,攻击者构造一个恶意网页，里面嵌入\<img src="http://youtube.com/add_favorite?video_id=恶意视频">或类似POST请求。当已登录YouTube的用户（Cookie有效）不小心访问该恶意页面时，浏览器会自动携带Cookie发送“添加视频到收藏夹”“发送私信”等请求，用户毫不知情。类似地，ING Direct银行曾被研究员演示：恶意链接可让已登录用户“转账给攻击者”。YouTube当时被迫紧急修复；银行案例直接证明CSRF能造成真实金钱损失，成为早期CSRF防护研究的标志性事件。

### SQL 注入

**SQL 注入**是指攻击者通过在输入中插入恶意 SQL 代码，欺骗服务器执行非预期的 SQL 命令。

**攻击示例**：

```sql
-- 正常查询
SELECT * FROM users WHERE username = 'alice' AND password = '123456';

-- 注入攻击
输入用户名：admin' OR '1'='1
输入密码：任意

-- 实际执行的 SQL
SELECT * FROM users WHERE username = 'admin' OR '1'='1' AND password = '任意';
-- '1'='1' 永远为真，绕过密码验证
```

**防御措施**：使用参数化查询（Prepared Statement）、使用 ORM 框架、输入验证和过滤、最小权限原则、错误信息不要暴露数据库结构

**相关案例**：2008年Heartland Payment Systems支付系统泄露，攻击者通过Heartland公司一个旧版Web表单的SQL注入漏洞（类似文档示例：输入' OR '1'='1绕过登录），入侵支付处理服务器。恶意代码植入后，持续数月窃取信用卡磁条数据（卡号、有效期、CVV），最终影响约1.3亿张信用卡。当时最大规模支付卡泄露事件，公司直接经济损失超2亿美元，股价暴跌77%，CEO下台，并引发全行业PCI-DSS合规风暴。

### DDoS 攻击

**DDoS（Distributed Denial of Service，分布式拒绝服务攻击）**是指攻击者控制大量计算机同时向目标发送请求，耗尽目标的资源，使其无法提供正常服务。

**攻击类型**：

* **流量型攻击**：发送大量数据包、耗尽带宽，如 UDP Flood、ICMP Flood

* **协议型攻击**：利用协议漏洞、耗尽服务器资源，如 SYN Flood、ACK Flood

* **应用层攻击**：针对应用层，发送大量看似正常的请求如 HTTP Flood、Slowloris

**防御措施**：使用 CDN 和负载均衡、限流和熔断、使用防火墙和 IDS/IPS、使用专业的 DDoS 防护服务、增加带宽和服务器资源

**相关案例**：2016年Mirai僵尸网络攻击Dyn DNS，黑客利用Mirai恶意软件感染全球数十万台IoT设备（摄像头、路由器，默认密码未改），组成僵尸网络，向Dyn DNS服务发起应用层+流量型混合DDoS（峰值1.2Tbps）。导致Twitter、Netflix、Reddit、PayPal、Amazon等美国与欧洲主要网站瘫痪数小时。

### DNS 劫持

**DNS 劫持**是指攻击者篡改 DNS 解析结果，将用户导向恶意网站。

**攻击方式**：

* **本地 DNS 劫持**：修改用户的 hosts 文件，修改路由器的 DNS 设置，从而引导用户 DNS 解析向错误网站

* **DNS 服务器劫持**：攻击 DNS 服务器，以篡改 DNS 记录

* **DNS 缓存投毒**：向 DNS 服务器发送伪造的 DNS 响应，污染 DNS 缓存

**防御措施**：使用可信的 DNS 服务器、使用 DNSSEC（DNS 安全扩展）、定期检查 hosts 文件和 DNS 设置、使用 HTTPS 防止被导向恶意网站

**相关案例**：2018年MyEtherWallet加密货币钱包劫持，攻击者通过BGP路由劫持（结合DNS缓存投毒），将Amazon Route 53 DNS解析服务指向俄罗斯的恶意服务器。当用户访问myetherwallet.com时，被重定向到假冒钓鱼网站（证书无效但用户常忽略）。短短2小时内窃取约15万美元以太坊。

### ARP 欺骗

**ARP 欺骗**是指攻击者发送伪造的 ARP 消息，将自己的 MAC 地址与其他主机的 IP 地址关联，从而截获网络流量。

**攻击过程**：

```
正常情况：
主机 A (IP: 192.168.1.10, MAC: AA:AA:AA:AA:AA:AA)
主机 B (IP: 192.168.1.20, MAC: BB:BB:BB:BB:BB:BB)

ARP 欺骗：
攻击者 (IP: 192.168.1.30, MAC: CC:CC:CC:CC:CC:CC)
1. 攻击者向主机 A 发送伪造的 ARP 响应
   "192.168.1.20 的 MAC 地址是 CC:CC:CC:CC:CC:CC"
2. 主机 A 更新 ARP 缓存
3. 主机 A 发送给主机 B 的数据包会发送给攻击者
4. 攻击者可以查看、修改数据包，然后转发给主机 B
```

**防御措施**：使用静态 ARP 表、使用 ARP 防火墙、使用交换机的端口安全功能、定期检查 ARP 缓存、使用加密通信（HTTPS、VPN）

**相关案例**：攻击者在咖啡店或大学宿舍同一局域网内，用Ettercap等工具发送伪造ARP响应（“路由器IP的MAC地址是我”），让受害者所有流量经过攻击者电脑。攻击者可窃听未加密流量、注入恶意JS，甚至实施MITM。真实案例包括2019年某企业内部员工利用ARP欺骗监控同事流量，以及2020年大学校园数百学生受影响导致临时断网。

## 高频判断

### 1. 中间人攻击

**Q：什么是中间人攻击？**
A：攻击者插入到通信双方之间，截获、监听甚至篡改通信内容的攻击方式。

**Q：如何防御中间人攻击？**
A：使用 HTTPS、验证证书、使用 VPN、避免使用不安全的网络。

**Q：HTTPS 如何防止中间人攻击？**
A：通过加密保护通信内容，通过数字证书验证服务器身份。

### 2. 重放攻击

**Q：什么是重放攻击？**
A：攻击者截获合法的通信数据，然后重新发送给接收方，以达到欺骗的目的。

**Q：如何防御重放攻击？**
A：使用时间戳、随机数、序列号、挑战-响应机制、短期有效的会话令牌。

**Q：时间戳和随机数哪个更好？**
A：各有优劣。时间戳简单但依赖时钟同步，随机数更安全但需要存储。通常结合使用。

### 3. XSS 攻击

**Q：什么是 XSS 攻击？**
A：攻击者在网页中注入恶意脚本，当其他用户浏览该网页时，恶意脚本会在用户浏览器中执行。

**Q：XSS 有哪些类型？**
A：反射型 XSS、存储型 XSS、DOM 型 XSS。

**Q：如何防御 XSS 攻击？**
A：输入验证和过滤、输出编码、使用 CSP、设置 HttpOnly 属性。

### 4. CSRF 攻击

**Q：什么是 CSRF 攻击？**
A：攻击者诱导用户在已登录的网站上执行非本意的操作。

**Q：CSRF 和 XSS 的区别？**
A：
- XSS：注入恶意脚本，在用户浏览器中执行
- CSRF：利用用户的登录状态，执行非本意的操作

**Q：如何防御 CSRF 攻击？**
A：使用 CSRF Token、检查 Referer、使用 SameSite Cookie、重要操作二次验证。

### 5. SQL 注入

**Q：什么是 SQL 注入？**
A：攻击者通过在输入中插入恶意 SQL 代码，欺骗服务器执行非预期的 SQL 命令。

**Q：如何防御 SQL 注入？**
A：使用参数化查询、使用 ORM 框架、输入验证和过滤、最小权限原则。

**Q：为什么参数化查询可以防止 SQL 注入？**
A：因为参数化查询将 SQL 代码和数据分离，数据不会被当作 SQL 代码执行。

### 6. DDoS 攻击

**Q：什么是 DDoS 攻击？**
A：攻击者控制大量计算机同时向目标发送请求，耗尽目标的资源，使其无法提供正常服务。

**Q：DDoS 有哪些类型？**
A：流量型攻击、协议型攻击、应用层攻击。

**Q：如何防御 DDoS 攻击？**
A：使用 CDN 和负载均衡、限流和熔断、使用防火墙、使用专业的 DDoS 防护服务。

### 7. 综合问题

**Q：HTTPS 能防止哪些攻击？**
A：中间人攻击、窃听、篡改、部分重放攻击。

**Q：HTTPS 不能防止哪些攻击？**
A：XSS、CSRF、SQL 注入、DDoS（这些是应用层的问题）。

**Q：最重要的安全措施是什么？**
