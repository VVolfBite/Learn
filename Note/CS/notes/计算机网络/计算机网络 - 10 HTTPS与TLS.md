## HTTPS

**HTTPS（HyperText Transfer Protocol Secure）**是 HTTP 的安全版本，通过在 HTTP 和 TCP 之间加入 SSL/TLS 层来提供安全保障。**HTTPS = HTTP + SSL/TLS**

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/ssl-offloading.jpg" width="700"/> </div><br>

- **SSL（Secure Sockets Layer）**：安全套接层，已被 TLS 取代
- **TLS（Transport Layer Security）**：传输层安全协议，SSL 的继任者
- HTTPS 使用端口 **443**，而 HTTP 使用端口 80

**HTTPS 的三大特性**：**加密（Encryption）**：防止数据被窃听，**认证（Authentication）**：防止通信方身份被伪装，**完整性保护（Integrity）**：防止数据被篡改

### HTTP 的安全性问题

HTTP 协议存在以下安全性问题：

**1. 使用明文通信，内容可能被窃听**：数据在传输过程中是明文的，任何人都可以截获并查看通信内容，敏感信息（如密码、信用卡号）容易泄露

**2. 不验证通信方的身份，身份可能被伪装**：无法确认服务器的真实身份，可能遭遇中间人攻击，无法防止钓鱼网站

**3. 无法证明报文的完整性，报文可能被篡改**：数据在传输过程中可能被修改，接收方无法验证数据是否被篡改，可能导致严重的安全问题

### 加密方式

#### 对称加密

**对称密钥加密（Symmetric-Key Encryption）**：加密和解密使用同一个密钥。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/7fffa4b8-b36d-471f-ad0c-a88ee763bb76.png" width="600"/> </div><br>

**工作原理**：通信双方共享同一个密钥，发送方使用密钥加密数据，接收方使用相同的密钥解密数据

**常见算法**：

- **AES（Advanced Encryption Standard）**：高级加密标准，最常用
- **DES（Data Encryption Standard）**：数据加密标准，已不安全
- **3DES**：三重 DES，安全性提高但速度慢

**特点**：运算速度快，加密效率高，适合大量数据的加密；但是密钥分发困难，无法安全地将密钥传输给通信方，密钥管理复杂（n 个人通信需要 n(n-1)/2 个密钥）

#### 非对称加密

**非对称密钥加密（Asymmetric-Key Encryption）**，又称**公开密钥加密（Public-Key Encryption）**：加密和解密使用不同的密钥。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/39ccb299-ee99-4dd1-b8b4-2f9ec9495cb4.png" width="600"/> </div><br>

**密钥对**：

- **公钥（Public Key）**：公开的，任何人都可以获得
- **私钥（Private Key）**：保密的，只有持有者知道

**工作原理**：接收方生成一对密钥（公钥和私钥）；接收方公开公钥，保密私钥；发送方使用接收方的公钥加密数据；接收方使用自己的私钥解密数据；其中公钥加密的数据只能用对应的私钥解开，私钥加密的数据只能用对应的公钥解开，而从公钥推导私钥在计算上不可行。

**常见算法**：

- **RSA**：最常用的非对称加密算法
- **ECC（Elliptic Curve Cryptography）**：椭圆曲线加密，安全性高且密钥短
- **DSA（Digital Signature Algorithm）**：数字签名算法

**两种用途**：

**1. 加密通信**：发送方用接收方的公钥加密，接收方用自己的私钥解密，从而保证数据的机密性

**2. 数字签名**：发送方用自己的私钥签名，接收方用发送方的公钥验证，保证数据的真实性和完整性

**特点**：密钥分发简单（只需公开公钥），密钥管理方便（n 个人通信只需 n 对密钥），可以实现数字签名；但是运算速度慢（比对称加密慢 100-1000 倍），不适合大量数据的加密

HTTPS 采用**混合加密机制**，结合了对称加密和非对称加密的优点。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/How-HTTPS-Works.png" width="600"/> </div><br>

**混合加密的工作流程**：

**1. 使用非对称加密传输对称密钥**：客户端生成一个随机的对称密钥（会话密钥）；客户端使用服务器的公钥加密这个对称密钥；服务器使用自己的私钥解密，得到对称密钥；这样就安全地完成了对称密钥的传输

**2. 使用对称加密传输数据**：双方使用协商好的对称密钥加密通信内容，保证了加密效率和通信速度

简单来说就是使用非对称加密来完成对称加密的钥匙，然后之后的会话基于这个对称加密钥匙来完成。

### 认证方式

**数字证书（Digital Certificate）**用于解决公钥的可信性问题。

思考如何确保收到的公钥确实是服务器的公钥，而不是中间人的公钥？可以使用数字证书认证机构（CA）颁发的证书。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/2017-06-11-ca.png" width=""/> </div><br>

**数字证书认证机构（CA，Certificate Authority）**：是客户端与服务器双方都信赖的第三方机构，负责颁发和管理数字证书，常见的 CA有DigiCert、Let's Encrypt、GlobalSign 等

**证书申请和使用流程**：

**1. 服务器申请证书**：服务器运营者向 CA 提出公钥申请，CA 验证申请者的身份，CA 对服务器的公钥进行数字签名，CA 将签名后的公钥放入证书中，CA 将证书颁发给服务器

**2. 客户端验证证书**：服务器将证书发送给客户端，客户端使用 CA 的公钥验证证书的数字签名，如果验证通过，说明证书是可信的，客户端提取证书中的服务器公钥，使用服务器公钥进行后续通信

**证书内容**：服务器的公钥、服务器的域名、证书的有效期、证书的颁发机构、数字签名

**证书链**：
- 根证书（Root Certificate）：操作系统或浏览器内置
- 中间证书（Intermediate Certificate）：由根 CA 签发
- 服务器证书（Server Certificate）：由中间 CA 签发

### TLS 握手基础

**TLS 握手**是建立 HTTPS 连接的过程，主要目的是协商加密参数和验证身份。

**TLS 握手的四个阶段**：

**Client Hello（客户端问候）**：客户端发送支持的 TLS 版本，客户端发送支持的加密套件列表，客户端生成一个客户端随机数（Client Random）

**Server Hello（服务器问候）**：服务器选择 TLS 版本和加密套件，服务器生成一个服务器随机数（Server Random），服务器发送数字证书，服务器发送公钥

**客户端验证和密钥交换**：客户端验证服务器证书，客户端生成一个预主随机数（Pre-Master Secret），客户端使用服务器公钥加密并发送加密后的预主随机数 Pre-Master Secret

**生成会话密钥**：双方使用三个随机数生成会话密钥（Session Key），会话密钥 = Function(Client Random, Server Random, Pre-Master Secret)，双方使用会话密钥进行对称加密通信

**简化流程图**：

```
客户端                                服务器
  |                                     |
  |-------- Client Hello -------------->|
  |  (Client Random, 加密套件列表)      |
  |                                     |
  |<------- Server Hello ---------------|
  |  (Server Random, 选定的加密套件)    |
  |<------- Certificate ---------------|
  |  (服务器证书)                       |
  |                                     |
  |-- Client Key Exchange ------------>|
  |  (加密的 Pre-Master Secret)         |
  |                                     |
  |-- Change Cipher Spec ------------->|
  |  (开始使用会话密钥加密)             |
  |                                     |
  |<- Change Cipher Spec --------------|
  |  (服务器也开始使用会话密钥)         |
  |                                     |
  |====== 加密通信开始 ================|
```

## HTTPS 的特点

**加密通信，防止窃听**

- HTTP：明文传输，任何人都可以看到通信内容
- HTTPS：加密传输，即使被截获也无法解密

**身份认证，防止伪装**

- HTTP：无法验证服务器身份，容易遭遇钓鱼网站
- HTTPS：通过数字证书验证服务器身份，确保连接到正确的服务器

**完整性保护，防止篡改**

- HTTP：数据可能在传输过程中被修改，无法检测
- HTTPS：使用消息认证码（MAC）保证数据完整性，任何篡改都会被发现

**防止中间人攻击**

- HTTP：中间人可以截获、查看、修改通信内容
- HTTPS：中间人无法解密通信内容，无法伪装服务器身份

**安全性对比**：

| 安全特性 | HTTP | HTTPS |
| :---: | :---: | :---: |
| 数据加密 | 错误 | 正确 |
| 身份认证 | 错误 | 正确 |
| 完整性保护 | 错误 | 正确 |
| 防窃听 | 错误 | 正确 |
| 防篡改 | 错误 | 正确 |
| 防伪装 | 错误 | 正确 |



**性能开销**：加密解密需要消耗 CPU 资源，TLS 握手增加了额外的往返时间（RTT），通常比 HTTP 慢 2-100 倍（取决于具体场景）

**证书成本**：需要向 CA 购买证书（虽然现在有免费的 Let's Encrypt），证书需要定期更新，证书管理需要额外的运维成本

**部署复杂**：需要配置 SSL/TLS，需要管理证书，需要处理证书过期、更新等问题

**缓存效率降低**：加密的内容不容易被中间代理缓存，可能影响 CDN 的缓存效果

**兼容性问题**：老旧的浏览器可能不支持新的 TLS 版本，某些网络设备可能不支持 HTTPS

**优化方案**：

- 使用 HTTP/2（HTTPS 的性能更好）
- 使用会话复用（Session Resumption）
- 使用 OCSP Stapling（减少证书验证时间）

## 高频判断

### HTTPS 基础

**Q：HTTPS 是什么？**
A：HTTPS 是 HTTP 的安全版本，通过在 HTTP 和 TCP 之间加入 SSL/TLS 层来提供加密、认证和完整性保护。

**Q：HTTPS 使用什么端口？**
A：443 端口（HTTP 使用 80 端口）。

**Q：SSL 和 TLS 的关系？**
A：TLS 是 SSL 的继任者，SSL 已经被废弃。现在说的 SSL 通常指的就是 TLS。

### 加密方式

**Q：对称加密和非对称加密的区别？**
A：
- 对称加密：加密和解密使用同一个密钥，速度快但密钥分发困难
- 非对称加密：加密和解密使用不同的密钥（公钥和私钥），密钥分发简单但速度慢

**Q：HTTPS 使用什么加密方式？**
A：混合加密。使用非对称加密传输对称密钥，使用对称加密传输数据。

**Q：为什么 HTTPS 不全程使用非对称加密？**
A：因为非对称加密速度太慢，不适合大量数据的加密。

### 数字证书

**Q：数字证书的作用是什么？**
A：验证服务器的身份，确保公钥的可信性，防止中间人攻击。

**Q：CA 是什么？**
A：数字证书认证机构（Certificate Authority），是客户端和服务器都信赖的第三方机构，负责颁发和管理数字证书。

**Q：如何验证证书的真实性？**
A：使用 CA 的公钥验证证书的数字签名。CA 的公钥通常内置在操作系统或浏览器中。

### TLS 握手

**Q：TLS 握手的目的是什么？**
A：协商加密参数（TLS 版本、加密套件）、验证服务器身份、安全地交换对称密钥。

**Q：TLS 握手需要几次往返？**
A：通常需要 2 次往返（4 个消息）。TLS 1.3 优化为 1 次往返。

**Q：会话密钥是如何生成的？**
A：由客户端随机数、服务器随机数和预主密钥（Pre-Master Secret）三个随机数生成。

### 安全性

**Q：HTTPS 能防止哪些攻击？**
A：窃听、篡改、伪装、中间人攻击、重放攻击（部分）。

**Q：HTTPS 是绝对安全的吗？**
A：不是。HTTPS 只能保证传输层的安全，无法防止应用层的漏洞（如 SQL 注入、XSS 等）。

**Q：为什么 HTTPS 比 HTTP 安全？**
A：因为 HTTPS 提供了加密（防窃听）、认证（防伪装）和完整性保护（防篡改）。

### 性能和成本

**Q：HTTPS 比 HTTP 慢多少？**
A：通常慢 2-100 倍，取决于具体场景。主要开销在 TLS 握手和加密解密。

**Q：如何优化 HTTPS 性能？**
A：使用 HTTP/2、会话复用、OCSP Stapling、硬件加速等。

**Q：HTTPS 证书需要付费吗？**
A：不一定。Let's Encrypt 提供免费的证书，但企业级证书通常需要付费。

### 对比问题

**Q：HTTP 和 HTTPS 的区别？**
A：
- HTTP：明文传输，不安全，端口 80
- HTTPS：加密传输，安全，端口 443

**Q：什么时候必须使用 HTTPS？**
A：涉及敏感信息（密码、支付、个人信息）的场景，以及需要 SEO 优化的网站（搜索引擎优先收录 HTTPS 网站）。
