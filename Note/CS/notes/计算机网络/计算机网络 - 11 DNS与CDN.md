## DNS

### DNS 是什么

**DNS（Domain Name System，域名系统）**是互联网的一项核心服务，它将人类可读的域名（如 www.example.com）转换为计算机可识别的 IP 地址（如 192.0.2.1）。

**为什么需要 DNS**：
- IP 地址难以记忆（如 172.217.160.78）
- 域名易于记忆（如 www.google.com）
- IP 地址可能会变化，但域名可以保持不变
- 一个域名可以对应多个 IP 地址（负载均衡）

**DNS 的特点**：
- **分布式数据库**：每个 DNS 服务器只保存一部分域名信息
- **层次化结构**：域名和 DNS 服务器都是层次化的
- **使用 UDP 协议**：端口号 53（超过 512 字节时使用 TCP）

### 域名结构

域名具有**层次结构**，从右到左依次为：根域名、顶级域名、二级域名、三级域名等。

**域名示例**：`www.example.com.`

```
www.example.com.
 |    |      |  |
 |    |      |  └─ 根域名（通常省略）
 |    |      └──── 顶级域名（TLD）
 |    └─────────── 二级域名
 └──────────────── 三级域名（主机名）
```

**域名层次**：

**1. 根域名（Root Domain）**
- 用 `.` 表示，通常省略
- 全球只有 13 组根域名服务器

**2. 顶级域名（TLD，Top-Level Domain）**
- **通用顶级域名（gTLD）**：.com、.net、.org、.edu、.gov 等
- **国家顶级域名（ccTLD）**：.cn、.us、.uk、.jp 等
- **新通用顶级域名**：.app、.dev、.cloud 等

**3. 二级域名（Second-Level Domain）**
- 由组织或个人注册
- 如 example.com 中的 example

**4. 三级域名（Third-Level Domain）**
- 也称为子域名或主机名
- 如 www.example.com 中的 www

### DNS 服务器层次结构

DNS 服务器也是层次化的，与域名结构对应：

**1. 根域名服务器（Root DNS Server）**
- 全球 13 组（A-M），每组有多个镜像服务器
- 负责返回顶级域名服务器的地址
- 是 DNS 查询的起点

**2. 顶级域名服务器（TLD DNS Server）**
- 负责管理顶级域名（如 .com、.cn）
- 返回二级域名服务器的地址

**3. 权威域名服务器（Authoritative DNS Server）**
- 负责管理特定域名的 DNS 记录
- 返回域名对应的 IP 地址
- 由域名所有者或其 DNS 服务提供商管理

**4. 本地域名服务器（Local DNS Server）**
- 也称为递归解析器
- 通常由 ISP 提供
- 代理客户端进行 DNS 查询
- 缓存查询结果

### 域名解析过程

**完整的域名解析过程**（以查询 www.example.com 为例）：

```
1. 用户输入 www.example.com
   ↓
2. 浏览器检查自身缓存
   ↓ (未命中)
3. 操作系统检查 hosts 文件和系统缓存
   ↓ (未命中)
4. 向本地 DNS 服务器查询
   ↓
5. 本地 DNS 服务器检查缓存
   ↓ (未命中)
6. 本地 DNS 向根域名服务器查询
   ← 返回 .com 顶级域名服务器地址
   ↓
7. 本地 DNS 向 .com 顶级域名服务器查询
   ← 返回 example.com 权威域名服务器地址
   ↓
8. 本地 DNS 向 example.com 权威域名服务器查询
   ← 返回 www.example.com 的 IP 地址
   ↓
9. 本地 DNS 将结果返回给客户端并缓存
   ↓
10. 客户端获得 IP 地址，建立连接
```

**关键步骤**：
1. 客户端向本地 DNS 服务器发起查询
2. 本地 DNS 服务器逐级向上查询（根 → TLD → 权威）
3. 权威 DNS 服务器返回最终结果
4. 本地 DNS 服务器缓存结果并返回给客户端

### DNS 查询方式

这一节主要把“DNS 查询方式”放回“计算机网络 - 10 DNS与CDN”这张卡片的主线里看。先抓住它在整个计算机网络知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

**1. 递归查询（Recursive Query）**
- 客户端向本地 DNS 服务器发起递归查询
- 本地 DNS 服务器负责完成整个查询过程
- 客户端只需等待最终结果

**特点**：
- 客户端只发送一次查询请求
- 本地 DNS 服务器负责所有后续查询
- 对客户端来说最简单

**2. 迭代查询（Iterative Query）**
- 本地 DNS 服务器向其他 DNS 服务器发起迭代查询
- 每次查询返回下一个应该查询的服务器地址
- 本地 DNS 服务器需要多次查询

**特点**：
- 需要多次查询
- 每次查询返回部分结果或下一步的指引
- 减轻了上层 DNS 服务器的负担

**典型场景**：
- 客户端 → 本地 DNS：递归查询
- 本地 DNS → 根/TLD/权威 DNS：迭代查询

### DNS 记录类型

DNS 记录存储在权威 DNS 服务器中，常见的记录类型：

| 记录类型 | 全称 | 作用 | 示例 |
| :---: | :---: | :---: | :---: |
| A | Address | 域名 → IPv4 地址 | example.com → 192.0.2.1 |
| AAAA | IPv6 Address | 域名 → IPv6 地址 | example.com → 2001:db8::1 |
| CNAME | Canonical Name | 域名别名 | www.example.com → example.com |
| MX | Mail Exchange | 邮件服务器 | example.com → mail.example.com |
| NS | Name Server | 域名服务器 | example.com → ns1.example.com |
| TXT | Text | 文本信息 | 用于 SPF、DKIM 等 |
| PTR | Pointer | IP → 域名（反向解析） | 192.0.2.1 → example.com |

**A 记录**：
- 最常用的记录类型
- 将域名映射到 IPv4 地址
- 一个域名可以有多个 A 记录（负载均衡）

**CNAME 记录**：
- 创建域名别名
- 便于管理（修改一处即可）
- 不能与其他记录类型共存

**MX 记录**：
- 指定邮件服务器
- 有优先级（数字越小优先级越高）
- 用于邮件路由

### DNS 缓存

**DNS 缓存**用于提高查询效率，减少网络流量。

**缓存位置**：
1. **浏览器缓存**：浏览器自身的 DNS 缓存
2. **操作系统缓存**：系统级的 DNS 缓存
3. **本地 DNS 服务器缓存**：ISP 的 DNS 服务器缓存
4. **其他 DNS 服务器缓存**：根、TLD 服务器也可能缓存

**TTL（Time To Live）**：
- 每条 DNS 记录都有 TTL 值
- 表示记录可以被缓存的时间
- TTL 过期后需要重新查询

**缓存的优缺点**：

**优点**：
- 加快域名解析速度
- 减少 DNS 服务器负载
- 降低网络流量

**缺点**：
- 可能返回过期的信息
- DNS 记录更新后需要等待 TTL 过期
- 可能被 DNS 缓存投毒攻击

## CDN

### CDN 是什么

**CDN（Content Delivery Network，内容分发网络）**是一种分布式网络架构，通过在全球各地部署边缘节点服务器，将内容缓存到离用户最近的节点，从而加速内容传输。

**核心思想**：
- 将内容分发到离用户最近的地方
- 减少数据传输距离
- 提高访问速度和用户体验

### CDN 的作用

这一节主要把“CDN 的作用”放回“计算机网络 - 10 DNS与CDN”这张卡片的主线里看。先抓住它在整个计算机网络知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

**1. 加速内容传输**
- 用户从最近的 CDN 节点获取内容
- 减少网络延迟
- 提高下载速度

**2. 减轻源站压力**
- 大部分请求由 CDN 节点处理
- 源站只需处理少量请求
- 降低源站带宽成本

**3. 提高可用性**
- 多个节点提供服务
- 单个节点故障不影响整体服务
- 提高系统容错能力

**4. 防御 DDoS 攻击**
- 分散攻击流量
- 保护源站安全
- 提供安全防护

**5. 跨地域访问优化**
- 解决跨国访问慢的问题
- 优化不同运营商之间的访问
- 提供更好的全球访问体验

### CDN 工作原理

**CDN 的基本架构**：

```
用户 → 本地 DNS → CDN DNS → 最近的 CDN 节点 → 源站
                      ↓
                  智能调度
                      ↓
              选择最优 CDN 节点
```

**详细工作流程**：

**1. 用户发起请求**
- 用户访问 www.example.com
- 浏览器向本地 DNS 查询域名

**2. DNS 解析**
- 域名的 CNAME 记录指向 CDN 域名
- 如：www.example.com → www.example.cdn.com
- 本地 DNS 向 CDN 的 DNS 服务器查询

**3. CDN 智能调度**
- CDN DNS 根据用户的 IP 地址、网络状况等因素
- 选择最优的 CDN 节点
- 返回该节点的 IP 地址

**4. 访问 CDN 节点**
- 用户向选定的 CDN 节点发起请求
- 如果节点有缓存，直接返回内容
- 如果节点无缓存，向源站请求内容

**5. 回源（如果需要）**
- CDN 节点向源站请求内容
- 缓存内容到本地
- 返回内容给用户

**6. 后续请求**
- 内容已缓存在 CDN 节点
- 直接从 CDN 节点返回
- 无需访问源站

**智能调度策略**：
- **地理位置**：选择离用户最近的节点
- **网络状况**：选择网络质量最好的节点
- **负载情况**：选择负载较低的节点
- **内容可用性**：选择已缓存内容的节点

### CDN 的优势

读到“CDN 的优势”时，先别急着记结论。更稳的方式是先弄清它在“计算机网络 - 10 DNS与CDN”这张卡片里到底想解决什么问题，这样后面的分类和判断就不会显得太散。

**1. 性能优势**
- 减少延迟：用户访问最近的节点
- 提高速度：减少数据传输距离
- 并发能力：多个节点分担流量

**2. 成本优势**
- 降低带宽成本：减少源站带宽消耗
- 降低服务器成本：减少源站服务器数量
- 按需付费：根据实际使用量付费

**3. 可靠性优势**
- 高可用性：多节点冗余
- 容错能力：单点故障不影响服务
- 负载均衡：自动分配流量

**4. 安全性优势**
- DDoS 防护：分散攻击流量
- WAF 防护：Web 应用防火墙
- HTTPS 加速：SSL/TLS 卸载

**5. 运维优势**
- 简化运维：CDN 服务商负责节点维护
- 全球部署：无需自建全球服务器
- 弹性扩展：自动应对流量波动

### CDN 的应用场景

这一节主要把“CDN 的应用场景”放回“计算机网络 - 10 DNS与CDN”这张卡片的主线里看。先抓住它在整个计算机网络知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

**1. 静态资源加速**
- 图片、CSS、JavaScript 文件
- 字体文件、图标
- 适合：电商网站、新闻网站

**2. 视频点播加速**
- 视频文件分发
- 流媒体传输
- 适合：视频网站、在线教育

**3. 直播加速**
- 实时视频流分发
- 低延迟传输
- 适合：直播平台、游戏直播

**4. 下载加速**
- 软件下载
- 大文件分发
- 适合：软件厂商、游戏厂商

**5. 动态内容加速**
- API 请求加速
- 动态页面加速
- 适合：SaaS 应用、Web 应用

**6. 全站加速**
- 静态 + 动态内容加速
- 全方位优化
- 适合：大型网站、跨国企业

**常见 CDN 服务商**：
- 国际：Cloudflare、Akamai、Amazon CloudFront
- 国内：阿里云 CDN、腾讯云 CDN、七牛云

## 高频判断

### 1. DNS 基础

**Q：DNS 是什么？**
A：域名系统，将域名转换为 IP 地址的分布式数据库系统。

**Q：为什么需要 DNS？**
A：因为域名比 IP 地址更容易记忆，且 IP 地址可能变化但域名可以保持不变。

**Q：DNS 使用什么协议和端口？**
A：主要使用 UDP 协议，端口 53。超过 512 字节时使用 TCP。

### 2. 域名解析

**Q：域名解析的完整过程？**
A：
1. 检查浏览器和系统缓存
2. 向本地 DNS 服务器查询
3. 本地 DNS 依次查询根、TLD、权威 DNS 服务器
4. 获得 IP 地址并缓存
5. 返回给客户端

**Q：递归查询和迭代查询的区别？**
A：
- 递归查询：客户端只发一次请求，DNS 服务器负责完成所有查询
- 迭代查询：每次查询返回下一步的指引，需要多次查询

**Q：DNS 缓存的作用？**
A：提高查询效率、减少网络流量、降低 DNS 服务器负载。

### 3. DNS 记录

**Q：常见的 DNS 记录类型有哪些？**
A：A（IPv4 地址）、AAAA（IPv6 地址）、CNAME（别名）、MX（邮件服务器）、NS（域名服务器）、TXT（文本信息）。

**Q：A 记录和 CNAME 记录的区别？**
A：
- A 记录：域名直接指向 IP 地址
- CNAME 记录：域名指向另一个域名（别名）

**Q：一个域名可以有多个 A 记录吗？**
A：可以，用于负载均衡和容错。

### 4. CDN 基础

**Q：CDN 是什么？**
A：内容分发网络，通过在全球部署边缘节点，将内容缓存到离用户最近的地方，加速内容传输。

**Q：CDN 的核心作用是什么？**
A：加速内容传输、减轻源站压力、提高可用性、防御攻击。

**Q：CDN 适合什么场景？**
A：静态资源加速、视频点播、直播、下载、全站加速等。

### 5. CDN 工作原理

**Q：CDN 如何选择最优节点？**
A：根据用户的地理位置、网络状况、节点负载、内容可用性等因素智能调度。

**Q：什么是回源？**
A：当 CDN 节点没有缓存内容时，向源站请求内容的过程。

**Q：CDN 如何与 DNS 配合工作？**
A：通过 CNAME 记录将域名指向 CDN 域名，CDN 的 DNS 服务器根据用户位置返回最优节点的 IP 地址。

### 6. DNS 与 CDN 对比

**Q：DNS 和 CDN 的关系？**
A：DNS 负责域名解析，CDN 负责内容分发。CDN 通常利用 DNS 进行智能调度，选择最优节点。

**Q：没有 CDN 可以用 DNS 做负载均衡吗？**
A：可以，通过配置多个 A 记录实现简单的负载均衡，但不如 CDN 智能。

### 7. 实际应用

**Q：如何判断网站是否使用了 CDN？**
A：
1. 查看域名的 CNAME 记录
2. 使用 ping 命令查看 IP 地址（CDN 通常返回不同的 IP）
3. 查看 HTTP 响应头（可能包含 CDN 相关信息）

**Q：DNS 解析慢怎么办？**
A：
1. 更换更快的 DNS 服务器（如 8.8.8.8、1.1.1.1）
2. 使用 DNS 缓存
3. 减少 DNS 查询次数（减少域名数量）

**Q：CDN 缓存更新怎么办？**
A：
1. 等待 TTL 过期自动更新
2. 使用 CDN 提供的缓存刷新功能
3. 使用版本号或哈希值（如 style.css?v=1.2.3）
