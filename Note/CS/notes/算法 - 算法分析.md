# 算法 - 算法分析

复杂度分析是评估算法性能的基础。在面试中，写完代码后通常需要分析时间和空间复杂度，这不仅是对算法效率的量化，也是判断算法是否可行的重要依据。

## 时间复杂度的数学模型

对于一个包含多项的运行时间表达式，比如 N³/6 - N²/2 + N/3，我们只保留最高次项并忽略系数，记作 O(N³)。这种表示法叫大 O 记号，它将算法与具体实现隔离开来——无论用什么语言实现，在什么机器上运行，增长数量级为 O(N³) 的算法在处理大规模数据时的表现都是相似的。

在分析时间复杂度时，重点关注算法的内循环——执行最频繁的那部分代码。比如在冒泡排序中，内层循环的比较和交换操作决定了整体的时间复杂度。

使用成本模型来评估算法时，可以选择具体的操作作为度量单位。比如对于排序算法，可以用比较次数和交换次数作为成本；对于图算法，可以用边的访问次数作为成本。

## 实际应用中的考量

在实际分析算法时，有几个容易被忽视但很重要的因素。

如果低级项的常数系数很大，那么在数据规模不够大时，理论上更优的算法可能反而更慢。比如某个 O(N) 算法的实际运行时间是 1000N，而某个 O(N²) 算法是 N²，那么在 N < 1000 时后者反而更快。

现代计算机的缓存机制对性能影响很大。访问数组相邻元素会比访问不相邻元素快得多，这就是为什么归并排序虽然和快速排序都是 O(NlogN)，但快速排序通常更快——它总是顺序访问数据，能更好地利用缓存。

对于某些关键系统（核反应堆控制、心脏起搏器、刹车系统），最坏情况下的性能保证比平均性能更重要。这时候需要选择最坏情况也能保证性能的算法。

随机化算法通过打乱输入来消除对特定输入模式的依赖。快速排序在排序前随机打乱数组，就是为了避免在已排序数组上退化为 O(N²)。

均摊分析用于评估一系列操作的平均代价。动态数组的扩容虽然单次代价高，但均摊到每次插入操作只需要常数时间。对一个空栈进行 N 次连续 push() 调用，总共访问数组 N+4+8+16+...+2N=5N-4 次，均摊后每次操作只需要常数次访问。

## ThreeSum 问题分析

ThreeSum 是一个经典的算法分析案例：给定一个数组，统计有多少组三个数的和为 0。通过这个问题可以看到不同算法思路带来的性能差异。

暴力解法用三层循环枚举所有可能的三元组，内循环的判断语句 `if (nums[i] + nums[j] + nums[k] == 0)` 总共执行 N(N-1)(N-2)/6 次，时间复杂度为 O(N³)。

```java
public class ThreeSumSlow implements ThreeSum {
    @Override
    public int count(int[] nums) {
        int N = nums.length;
        int cnt = 0;
        for (int i = 0; i < N; i++) {
            for (int j = i + 1; j < N; j++) {
                for (int k = j + 1; k < N; k++) {
                    if (nums[i] + nums[j] + nums[k] == 0) {
                        cnt++;
                    }
                }
            }
        }
        return cnt;
    }
}
```

第一个优化是先排序，然后对两个元素求和，用二分查找判断是否存在该和的相反数。这样可以将时间复杂度降低到 O(N²logN)。需要注意的是，这种方法要求数组中没有重复元素，否则二分查找的结果会出错。

```java
public class ThreeSumBinarySearch implements ThreeSum {
    @Override
    public int count(int[] nums) {
        Arrays.sort(nums);
        int N = nums.length;
        int cnt = 0;
        for (int i = 0; i < N; i++) {
            for (int j = i + 1; j < N; j++) {
                int target = -nums[i] - nums[j];
                int index = BinarySearch.search(nums, target);
                if (index > j) {
                    cnt++;
                }
            }
        }
        return cnt;
    }
}
```

更高效的方法是使用双指针。先排序，然后固定一个数，用两个指针从两端向中间移动寻找另外两个数。这样时间复杂度降为 O(N²)。

```java
public class ThreeSumTwoPointer implements ThreeSum {
    @Override
    public int count(int[] nums) {
        int N = nums.length;
        int cnt = 0;
        Arrays.sort(nums);
        for (int i = 0; i < N - 2; i++) {
            int l = i + 1, h = N - 1, target = -nums[i];
            while (l < h) {
                int sum = nums[l] + nums[h];
                if (sum == target) {
                    cnt++;
                    l++;
                    h--;
                } else if (sum < target) {
                    l++;
                } else {
                    h--;
                }
            }
        }
        return cnt;
    }
}
```

## 倍率实验验证

倍率实验是验证复杂度分析的实用方法。如果算法的时间复杂度为 T(N) ~ aN^b，那么当输入规模翻倍时，运行时间的比值应该接近 2^b。

对于 O(N³) 的 ThreeSum 暴力算法，理论上 T(2N)/T(N) ~ 8。实际测试结果：

| N | Time(ms) | Ratio |
| :---: | :---: | :---: |
| 500 | 48 | / |
| 1000 | 320 | 6.7 |
| 2000 | 555 | 1.7 |
| 4000 | 4105 | 7.4 |
| 8000 | 33575 | 8.2 |
| 16000 | 268909 | 8.0 |

可以看到，当 N 足够大时，比值确实稳定在 8 左右，验证了 O(N³) 的分析。前几次比值不稳定是因为数据规模较小时，常数项和低阶项的影响还比较明显。

```java
public class RatioTest {
    public static void main(String[] args) {
        int N = 500;
        int loopTimes = 7;
        double preTime = -1;
        while (loopTimes-- > 0) {
            int[] nums = new int[N];
            StopWatch.start();
            ThreeSum threeSum = new ThreeSumSlow();
            int cnt = threeSum.count(nums);
            double elapsedTime = StopWatch.elapsedTime();
            double ratio = preTime == -1 ? 0 : elapsedTime / preTime;
            System.out.println(N + "  " + elapsedTime + "  " + ratio);
            preTime = elapsedTime;
            N *= 2;
        }
    }
}
```

## 高频判断

**如何快速判断一个算法的时间复杂度？**
看循环嵌套层数和每层的迭代次数。单层循环通常是 O(N)，两层嵌套是 O(N²)。如果每次迭代都将问题规模减半，通常是 O(logN)。递归算法需要分析递归树的深度和每层的工作量。

**为什么二分查找是 O(logN)？**
每次比较都将搜索范围减半，最多需要 log₂N 次比较就能找到目标或确定不存在。

**递归算法的时间复杂度怎么分析？**
分析递归树：树的深度乘以每层的工作量。比如归并排序的递归树深度为 logN，每层需要合并 N 个元素，所以时间复杂度是 O(NlogN)。

**O(NlogN) 和 O(N²) 差距有多大？**
当 N = 1000 时，NlogN ≈ 10000，而 N² = 1000000，相差 100 倍。当 N = 1000000 时，差距会更加明显。这就是为什么对于大规模数据，必须使用高效算法。

**空间换时间值得吗？**
要看具体场景。哈希表用 O(N) 的空间将查找从 O(N) 降到 O(1)，在需要频繁查找时非常值得。但如果内存受限或只查找一次，就不一定划算。
