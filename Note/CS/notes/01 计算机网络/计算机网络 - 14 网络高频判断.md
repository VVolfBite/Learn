## TCP vs UDP

TCP 和 UDP 是传输层的两个核心协议，面试必问。

TCP 面向连接，需要三次握手建立连接，四次挥手断开连接。UDP 无连接，发送数据前不需要建立连接。TCP 保证可靠传输，通过确认应答、超时重传、序号机制等保证数据不丢失、不乱序。UDP 不保证可靠性，尽最大努力交付，可能丢包、乱序。

TCP 慢但可靠，UDP 快但不可靠。TCP 有流量控制和拥塞控制，UDP 没有。TCP 首部 20-60 字节，UDP 首部只有 8 字节。TCP 只能一对一通信，UDP 可以一对一、一对多、多对多。

什么时候用 TCP？需要可靠传输的场景，比如文件传输、邮件、HTTP。什么时候用 UDP？实时性要求高、可以容忍少量丢包的场景，比如视频直播、语音通话、在线游戏、DNS 查询。

UDP 不可靠为什么还要用？因为速度快、开销小、支持广播和多播。而且应用层可以自己实现可靠性机制，比如 QUIC 协议就是基于 UDP 实现的可靠传输，是 HTTP/3 的基础。

TCP 如何保证可靠传输？六个机制：序号和确认应答、超时重传、流量控制（滑动窗口）、拥塞控制（慢开始、拥塞避免、快重传、快恢复）、校验和、顺序控制。

## HTTP vs HTTPS

HTTP 和 HTTPS 的核心区别是安全性。HTTP 明文传输，HTTPS 加密传输。HTTP 使用 80 端口，HTTPS 使用 443 端口。HTTPS 需要 SSL/TLS 证书，HTTP 不需要。

HTTPS 通过对称加密 + 非对称加密保证数据安全。对称加密速度快，但密钥分发困难；非对称加密密钥分发简单，但速度慢。HTTPS 用非对称加密传输对称密钥，然后用对称密钥加密数据，兼顾了安全和效率。

HTTPS 为什么安全？三个保障：加密（防窃听）、认证（通过数字证书验证服务器身份，防伪装）、完整性（使用 MAC 保证数据未被篡改）。

HTTPS 一定安全吗？不一定。HTTPS 只能保证传输层的安全，无法防止应用层漏洞（XSS、SQL 注入等）、服务器被攻破、用户主动信任恶意证书等情况。

HTTPS 为什么比 HTTP 慢？因为需要 TLS 握手（额外的往返时间）、加密解密（CPU 计算）、证书验证。但可以通过会话复用、HTTP/2、OCSP Stapling、硬件加速等方式优化。

什么时候必须使用 HTTPS？涉及敏感信息（密码、支付、个人信息）、需要用户登录的网站、需要 SEO 优化的网站、需要使用某些浏览器 API（如地理位置、摄像头）的场景。

## Cookie vs Session vs Token

这三个都是用来管理用户会话状态的，但存储位置和工作方式不同。

Cookie 存储在客户端（浏览器），大小限制 4KB，安全性较低，容易被窃取。Session 存储在服务器端，可以存储任意数据，安全性较高，但需要服务器存储和查询。Token 也存储在客户端，但包含签名，服务器不需要保存，只需验证签名，适合分布式系统。

Cookie 和 Session 的区别？Cookie 在客户端，Session 在服务器端；Cookie 大小受限，Session 无限制；Cookie 安全性低，Session 安全性高；Cookie 不占用服务器资源，Session 需要服务器存储。

Session 和 Token 的区别？Session 有状态（服务器保存），Token 无状态（服务器不保存）；Session 需要查询存储，Token 只需验证签名；Session 适合传统应用，Token 适合分布式系统和移动应用。

为什么 Token 适合分布式系统？因为无状态，服务器不需要保存 Token，任何服务器都可以验证；不需要共享存储（如 Redis）；水平扩展容易，增加服务器不需要同步 Session；跨域友好，不依赖 Cookie。

JWT 的优缺点？优点是无状态、跨域友好、移动端友好、性能好（不需要查询数据库）。缺点是无法主动失效（除非维护黑名单）、Token 较长（每次请求都需要传输）、无法更新用户信息（需要重新签发）。

如何选择？传统 Web 应用用 Cookie + Session，前后端分离应用用 Token（JWT），移动应用用 Token（JWT），微服务架构用 Token（JWT），需要主动失效会话用 Session，需要存储大量用户数据用 Session。

## 三次握手 / 四次挥手高频问法

### 三次握手

为什么需要三次握手？核心是确认双方的收发能力。第一次握手，服务器确认客户端能发；第二次握手，客户端确认服务器能收能发；第三次握手，服务器确认客户端能收。另一个重要原因是防止失效的连接请求到达服务器。如果只有两次握手，客户端发送的失效连接请求（因网络延迟滞留）可能导致服务器错误打开连接并等待数据，浪费资源。

三次握手的过程：第一次，客户端发送 SYN，进入 SYN_SENT 状态；第二次，服务器收到 SYN，发送 SYN+ACK，进入 SYN_RCVD 状态；第三次，客户端收到 SYN+ACK，发送 ACK，进入 ESTABLISHED 状态，服务器收到 ACK 也进入 ESTABLISHED 状态。

为什么不能是两次握手？无法确认客户端的接收能力，失效的连接请求可能导致服务器错误打开连接。想象这个场景：客户端发送连接请求，但因网络延迟滞留；客户端超时重传，建立新连接并完成通信；旧的连接请求到达服务器；如果只有两次握手，服务器会打开连接并等待数据，但客户端不会发送数据，浪费服务器资源。

第三次握手可以携带数据吗？可以。第三次握手时连接已经建立，可以携带数据。但前两次握手不能携带数据，防止 SYN 洪泛攻击。

SYN 洪泛攻击是什么？攻击者发送大量 SYN 请求，服务器为每个请求分配资源并等待第三次握手，攻击者不发送第三次握手，服务器资源被耗尽。防御方法：SYN Cookie、增加半连接队列大小、减少超时时间。

### 四次挥手

为什么需要四次挥手？因为 TCP 是全双工通信，双方都需要关闭连接。客户端发送 FIN 表示不再发送数据，但仍可以接收数据。服务器可能还有数据要发送，所以 ACK 和 FIN 分开发送。四次挥手确保双方都完成数据传输。

四次挥手的过程：第一次，客户端发送 FIN，进入 FIN_WAIT_1 状态；第二次，服务器收到 FIN，发送 ACK，进入 CLOSE_WAIT 状态，客户端收到 ACK 进入 FIN_WAIT_2 状态；第三次，服务器发送完数据后发送 FIN，进入 LAST_ACK 状态；第四次，客户端收到 FIN，发送 ACK，进入 TIME_WAIT 状态，服务器收到 ACK 进入 CLOSED 状态，客户端等待 2MSL 后进入 CLOSED 状态。

为什么不能是三次挥手？可以是三次，如果服务器没有数据要发送，可以将 ACK 和 FIN 合并发送。但通常情况下，服务器收到 FIN 后可能还有数据要发送，所以需要四次挥手。

TIME_WAIT 状态是什么？为什么需要等待 2MSL？TIME_WAIT 是客户端发送最后一个 ACK 后进入的状态，2MSL 是 2 倍的最大报文生存时间。原因有两个：一是确保最后一个 ACK 到达服务器，如果 ACK 丢失，服务器会重传 FIN，客户端需要等待并重新发送 ACK；二是让旧连接的数据包在网络中消失，防止旧连接的数据包影响新连接。

大量 TIME_WAIT 状态怎么办？调整系统参数（如 tcp_tw_reuse、tcp_tw_recycle）、使用长连接减少连接建立和关闭、使用连接池、让服务器主动关闭连接（服务器的 TIME_WAIT 影响较小）。

## DNS / CDN 高频问法

### DNS

**Q1：DNS 的完整解析过程？**

A：
1. 浏览器缓存 → 系统缓存 → hosts 文件
2. 本地 DNS 服务器缓存
3. 本地 DNS 向根域名服务器查询
4. 根域名服务器返回顶级域名服务器地址
5. 本地 DNS 向顶级域名服务器查询
6. 顶级域名服务器返回权威域名服务器地址
7. 本地 DNS 向权威域名服务器查询
8. 权威域名服务器返回 IP 地址
9. 本地 DNS 缓存结果并返回给客户端

**Q2：递归查询和迭代查询的区别？**

A：
- **递归查询**：客户端向本地 DNS 发起，本地 DNS 负责完成所有查询
- **迭代查询**：本地 DNS 向其他 DNS 服务器发起，每次返回下一步的指引

**Q3：DNS 使用 TCP 还是 UDP？**

A：
- 主要使用 **UDP**，端口 53
- 当响应超过 512 字节时使用 **TCP**
- 区域传送（主从同步）使用 **TCP**

**Q4：如何优化 DNS 解析速度？**

A：
- 使用更快的 DNS 服务器（如 8.8.8.8、1.1.1.1）
- 使用 DNS 缓存
- 减少域名数量（减少 DNS 查询次数）
- 使用 DNS 预解析（`<link rel="dns-prefetch">`）
- 使用 CDN（CDN 通常有更快的 DNS）

### CDN

**Q1：CDN 的工作原理？**

A：
1. 用户访问域名
2. DNS 解析，CNAME 指向 CDN 域名
3. CDN 的 DNS 根据用户位置返回最近的 CDN 节点 IP
4. 用户访问 CDN 节点
5. 如果节点有缓存，直接返回；否则回源获取
6. CDN 节点缓存内容并返回给用户

**Q2：CDN 如何选择最优节点？**

A：
- **地理位置**：选择离用户最近的节点
- **网络状况**：选择网络质量最好的节点
- **负载情况**：选择负载较低的节点
- **内容可用性**：选择已缓存内容的节点

**Q3：CDN 的优势？**

A：
- **加速**：用户访问最近的节点，减少延迟
- **减轻源站压力**：大部分请求由 CDN 处理
- **提高可用性**：多节点冗余
- **防御 DDoS**：分散攻击流量
- **节省带宽**：减少源站带宽消耗

**Q4：什么内容适合用 CDN？**

A：
- **静态资源**：图片、CSS、JavaScript、字体
- **视频**：点播、直播
- **下载文件**：软件、游戏
- **动态内容**：通过动态加速技术

**Q5：CDN 缓存更新怎么办？**

A：
- 等待 TTL 过期自动更新
- 使用 CDN 提供的缓存刷新功能
- 使用版本号或哈希值（如 `style.css?v=1.2.3`）
- 使用 Cache-Control 头部控制缓存策略

## 高频易混淆点

### 1. 传输时延 vs 传播时延

**传输时延**：
- 数据从主机进入链路所需的时间
- 公式：数据长度 / 传输速率
- 与带宽有关，与距离无关

**传播时延**：
- 信号在链路上传播所需的时间
- 公式：链路长度 / 传播速度
- 与距离有关，与带宽无关

### 2. 网络层 vs 传输层

**网络层**：
- 主机到主机的通信
- IP 地址
- 路由选择
- 协议：IP、ICMP、ARP

**传输层**：
- 进程到进程的通信
- 端口号
- 可靠传输（TCP）或快速传输（UDP）
- 协议：TCP、UDP

### 3. MAC 地址 vs IP 地址

**MAC 地址**：
- 物理地址，固化在网卡中
- 48 位，如 `AA:BB:CC:DD:EE:FF`
- 用于同一链路的通信
- 数据链路层

**IP 地址**：
- 逻辑地址，可以改变
- 32 位（IPv4）或 128 位（IPv6）
- 用于跨网络的通信
- 网络层

### 4. 对称加密 vs 非对称加密

**对称加密**：
- 加密和解密使用同一个密钥
- 速度快
- 密钥分发困难
- 如：AES、DES

**非对称加密**：
- 加密和解密使用不同的密钥（公钥和私钥）
- 速度慢
- 密钥分发简单
- 如：RSA、ECC

### 5. GET vs POST

**GET**：
- 获取资源
- 参数在 URL 中
- 可以缓存
- 幂等
- 安全（不修改服务器状态）

**POST**：
- 传输数据
- 参数在请求体中
- 不可缓存
- 不幂等
- 不安全（可能修改服务器状态）

### 6. 状态码 301 vs 302

**301 Moved Permanently**：
- 永久重定向
- 浏览器会缓存
- SEO 权重会转移

**302 Found**：
- 临时重定向
- 浏览器不会缓存
- SEO 权重不会转移

### 7. Cookie 的 Secure vs HttpOnly

**Secure**：
- 只能通过 HTTPS 传输
- 防止中间人攻击

**HttpOnly**：
- 不能被 JavaScript 访问
- 防止 XSS 攻击

### 8. XSS vs CSRF

**XSS（跨站脚本攻击）**：
- 注入恶意脚本
- 在用户浏览器中执行
- 窃取 Cookie、会话等

**CSRF（跨站请求伪造）**：
- 利用用户的登录状态
- 执行非本意的操作
- 不需要获取 Cookie

### 9. 长连接 vs 短连接

**短连接**：
- 每次请求建立新连接
- 请求完成后关闭连接
- HTTP/1.0 默认

**长连接**：
- 多个请求复用同一个连接
- 减少连接建立和关闭的开销
- HTTP/1.1 默认（Connection: keep-alive）

### 10. 流量控制 vs 拥塞控制

**流量控制**：
- 控制发送方的发送速率
- 防止接收方溢出
- 使用滑动窗口机制
- 端到端的控制

**拥塞控制**：
- 控制发送方的发送速率
- 防止网络拥塞
- 使用慢开始、拥塞避免等算法
- 全局性的控制

### 11. HTTP/1.0 vs HTTP/1.1 vs HTTP/2

**HTTP/1.0**：
- 短连接（每次请求建立新连接）
- 无 Host 头部
- 不支持断点续传

**HTTP/1.1**：
- 长连接（Connection: keep-alive）
- 支持 Host 头部（虚拟主机）
- 支持断点续传（Range）
- 支持流水线（但浏览器默认关闭）

**HTTP/2**：
- 二进制分帧
- 多路复用（一个连接多个请求）
- 服务器推送
- 头部压缩

### 12. 同步 vs 异步、阻塞 vs 非阻塞

**同步 I/O**：
- 发起 I/O 操作后，必须等待完成才返回
- 包括：阻塞 I/O、非阻塞 I/O、I/O 复用、信号驱动 I/O

**异步 I/O**：
- 发起 I/O 操作后，立即返回
- 操作完成后通过回调通知

**阻塞**：
- 调用后一直等待，直到操作完成
- 如：阻塞 I/O

**非阻塞**：
- 调用后立即返回，可能未完成
- 如：非阻塞 I/O、异步 I/O

### 13. 正向代理 vs 反向代理

**正向代理**：
- 代理客户端
- 客户端知道代理的存在
- 用途：访问控制、缓存、匿名访问
- 例子：VPN、翻墙工具

**反向代理**：
- 代理服务器
- 客户端不知道代理的存在
- 用途：负载均衡、缓存、安全防护
- 例子：Nginx、CDN

### 14. URI vs URL vs URN

**URI（统一资源标识符）**：
- 用于标识资源
- 包括 URL 和 URN

**URL（统一资源定位符）**：
- 用于定位资源
- 包含访问方式和位置
- 例子：`http://www.example.com/index.html`

**URN（统一资源名称）**：
- 用于命名资源
- 不包含位置信息
- 例子：`urn:isbn:0451450523`

### 15. 网关 vs 路由器 vs 交换机

**网关**：
- 协议转换器
- 可以在不同协议之间转换
- 工作在应用层或传输层

**路由器**：
- 连接不同网络
- 根据 IP 地址转发数据包
- 工作在网络层

**交换机**：
- 连接同一网络的设备
- 根据 MAC 地址转发帧
- 工作在数据链路层

**记忆技巧总结**：

**TCP vs UDP 记忆**：
- TCP = 可靠的快递（有签收、有追踪）
- UDP = 广播电台（发出去就不管了）

**三次握手记忆**：
- 第一次：你能听到我吗？（SYN）
- 第二次：我能听到，你能听到我吗？（SYN+ACK）
- 第三次：我能听到（ACK）

**四次挥手记忆**：
- 第一次：我说完了（FIN）
- 第二次：我知道了（ACK）
- 第三次：我也说完了（FIN）
- 第四次：我知道了（ACK）

**HTTPS 记忆**：
- 对称加密 = 一把钥匙开一把锁（快但不安全传输）
- 非对称加密 = 公钥加密私钥解密（慢但安全传输）
- HTTPS = 用非对称加密传输对称密钥，然后用对称加密传输数据

**Cookie vs Session vs Token 记忆**：
- Cookie = 会员卡（客户端保存）
- Session = 会员档案（服务器保存）
- Token = 令牌（客户端保存，服务器验证签名）

**DNS 解析记忆**：
- 浏览器缓存 → 系统缓存 → hosts 文件 → 本地 DNS → 根域名服务器 → 顶级域名服务器 → 权威域名服务器

**面试常见追问**：

**Q：为什么 HTTPS 比 HTTP 慢？**
A：因为需要 TLS 握手（额外的 RTT）、加密解密（CPU 计算）、证书验证。但可以通过会话复用、HTTP/2、硬件加速等方式优化。

**Q：Token 如何实现失效？**
A：Token 本身无法主动失效。解决方案：1) 设置短过期时间 2) 维护黑名单 3) 使用 Refresh Token 机制。

**Q：为什么需要 TIME_WAIT 状态？**
A：1) 确保最后一个 ACK 能到达服务器（如果丢失，服务器会重传 FIN）2) 让旧连接的数据包在网络中消失，避免影响新连接。

**Q：大量 TIME_WAIT 怎么办？**
A：1) 调整系统参数（tcp_tw_reuse、tcp_tw_recycle）2) 使用长连接 3) 使用连接池 4) 让服务器主动关闭连接。

**Q：CDN 如何加速？**
A：1) 就近访问（减少传播时延）2) 缓存静态资源（减少源站压力）3) 负载均衡（提高可用性）4) 智能路由（选择最优路径）。

**Q：如何防止 XSS 攻击？**
A：1) 输入验证和过滤 2) 输出编码（HTML 实体编码）3) 使用 HttpOnly Cookie 4) 使用 CSP（内容安全策略）。

**Q：如何防止 CSRF 攻击？**
A：1) 使用 CSRF Token 2) 验证 Referer 3) 使用 SameSite Cookie 4) 重要操作二次验证。

**易错点提醒**：

**错误 1**：认为 POST 比 GET 安全
- 正确理解：POST 和 GET 都不安全，都可以被抓包。真正的安全需要使用 HTTPS。

**错误 2**：认为 HTTPS 绝对安全
- 正确理解：HTTPS 只保证传输层安全，无法防止应用层漏洞（XSS、SQL 注入等）。

**错误 3**：认为 Cookie 只能存储 4KB
- 正确理解：单个 Cookie 限制 4KB，但可以设置多个 Cookie。

**错误 4**：认为 Session 一定比 Cookie 安全
- 正确理解：Session ID 也是通过 Cookie 传输的，如果 Cookie 被窃取，Session 也不安全。

**错误 5**：认为长连接一定比短连接好
- 正确理解：长连接适合频繁通信的场景，但会占用服务器资源。短连接适合偶尔通信的场景。
