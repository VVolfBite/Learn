# CD Basics

> CD 关心的是把通过验证的软件稳定地送到下一个环境，而不是让代码“改完就立刻上生产”。

## 背景与作用

CD 一般指持续交付或持续部署。持续交付强调系统始终处于可发布状态，需要上线时能稳定推出去；持续部署则更进一步，让通过流水线验证的改动自动进入生产。两者都围绕“如何把软件可靠地往前推”，区别主要在最后一步是否自动完成。

没有清晰的交付机制时，上线往往依赖手工操作和临场经验，容易引入新的不确定性。CD 的价值就是把打包、发布、环境变更和验证过程做成可重复动作，让发布从“高风险活动”慢慢变成“有流程、有证据的常规动作”。

## 场景与边界

面试问 CD，是因为很多人会把它和 CI 混在一起。工作里则更现实：功能做好只是第一步，真正把变更稳定地送到测试环境、预发环境和生产环境，同样是系统质量的一部分。懂得交付过程，才能更好地理解灰度、回滚、版本兼容这些实际问题。

一个误区是把 CD 理解成“自动上线生产”。持续交付和持续部署需要区分，是否自动推生产取决于业务风险和组织成熟度。另一个误区是只看发布动作，不看发布后的验证。真正稳的 CD 不只是把东西发出去，还要确认发出去以后系统仍然正常。

## 面试表达

CD 关注的是把经过验证的软件稳定地交付到下一个环境。持续交付强调随时可发布，持续部署则是通过流水线后自动上线生产。它的核心价值是让发布过程可重复、可追踪、可回退，而不是靠人工经验硬推版本。

## 快速回忆

- CD 关注交付和发布，不等于 CI。
- 持续交付和持续部署要区分。
- 自动化发布不等于无脑上线。
- 稳定交付同样属于质量保障。
