# Common Pipeline Stages

> 流水线分阶段，不是为了看起来完整，而是为了让问题在最便宜的位置暴露，并让发布过程更容易回溯。

## 背景与作用

一条常见流水线通常会经历构建、验证、制品生成、部署和部署后验证等阶段。不同团队顺序和细节会有差别，但整体思路很一致：先确认代码能被正确构建，再确认行为和规范没问题，再产出可追踪的制品，最后把制品发布到目标环境并做必要验证。

如果所有动作都混成一步，出问题时很难知道是编译失败、测试失败、环境差异还是部署脚本问题。分阶段的意义就在于让每一层职责更清晰。构建阶段负责确认软件能被生产出来，验证阶段负责确认质量门禁通过，部署阶段负责把同一份制品送到环境，最后的验证则确认部署结果确实可用。

## 场景与边界

流水线阶段几乎是 CI/CD 讨论里的共通知识。面试借它判断候选人是否理解自动化交付的基本结构。工作里如果没有这种阶段意识，很多团队会把“能跑”和“能发”、“发了”和“可用”混在一起，最终导致排查和回滚都困难。

一个误区是认为流水线阶段越多越专业。阶段应服务于定位和治理，而不是制造流程负担。另一个误区是不同环境重新构建不同产物。更稳的做法通常是尽量复用同一份制品，只改变部署目标和配置，从而减少“环境间行为不一致”的机会。

## 面试表达

常见流水线一般包括构建、质量验证、产出制品、部署和部署后验证。它们的价值不是形式完整，而是让问题分层暴露，让发布过程可重复、可追踪，并尽量保证不同环境使用同一份经过验证的产物。

## 快速回忆

- 先构建，再验证，再交付，再验证结果。
- 阶段服务于定位，不服务于表面流程感。
- 同一份制品跨环境更稳。
- “发出去”不等于“已经可用”。
