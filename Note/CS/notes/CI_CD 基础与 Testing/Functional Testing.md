# Functional Testing

> 功能测试更像从系统行为和用户目标出发提问：当用户做这件事时，系统整体表现是否符合预期。

## 背景与作用

功能测试关注的是系统对外呈现的功能是否完成了应有行为。它不像单元测试那样只盯局部逻辑，也不像集成测试那样强调协作边界本身，而是从更接近用户视角的层面去验证一个完整功能是否成立，比如下单、登录、审批、导出报表这些流程。

系统里很多问题并不是局部逻辑错误，而是功能路径没有真正闭环。比如表单提交成功但状态没更新，接口返回正确但页面展示不完整，某个功能在理想路径可用但异常路径断裂。功能测试的意义，就是从业务视角去确认这些流程是否真的跑通。

## 场景与边界

功能测试帮助团队把“技术正确”转换成“业务可用”。面试中它常被拿来和单元测试、集成测试做区分。工作里如果没有功能测试，团队很容易只知道局部没坏，却不知道用户真正关心的那条业务链有没有被新改动打断。

一个误区是把功能测试等同于 UI 自动化。实际上它关心的是功能行为，可以通过 API、服务编排或更高层测试来实现。另一个误区是希望它替代低层测试。功能测试更贵、更慢，适合压住关键业务闭环，但不适合承担所有细节验证。

## 面试表达

功能测试从业务行为和用户目标出发，验证系统对外提供的完整功能是否符合预期。它和单元测试、集成测试的区别在于视角更高，更关心业务流程是否闭环，而不是某个局部逻辑或某段依赖协作本身。

## 快速回忆

- 功能测试关注“业务是否真的可用”。
- 它是系统行为视角，不一定等于 UI 自动化。
- 适合压关键业务闭环。
- 不能替代低层测试，但能补足业务视角。
