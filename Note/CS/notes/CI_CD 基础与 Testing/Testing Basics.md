# Testing Basics

> 测试的核心价值不是“显得专业”，而是降低改动风险。系统越常变化，越需要一套能快速告诉你“这次改动有没有把旧行为弄坏”的机制。

## 背景与作用

测试是用明确输入和预期结果去验证系统行为的一种方法。它既可以很小，聚焦单个函数的边界，也可以很大，从用户视角验证完整流程。测试的本质不是增加仪式感，而是把“我觉得没问题”变成“我有证据说明它在这些场景下还能工作”。

任何持续迭代的系统都会面对回归风险。你修了一个 bug，可能顺手打坏另一个分支；你优化了一段逻辑，可能改变边界行为；你重构了代码结构，可能让旧接口悄悄失效。测试就是用来把这些不确定性尽可能提前暴露出来，避免问题等到线上才第一次出现。

## 场景与边界

面试问测试，通常是在看候选人有没有基本工程质量意识。工作里更明显：没有测试，系统一复杂就会陷入“不敢改”；有了测试，团队至少有机会更稳地重构和迭代。测试不是保证零 bug，但它能显著提高改动的可控性和定位效率。

最常见的误区是把测试目标说成“追求覆盖率”。覆盖率是一个信号，但不是最终目的。另一个误区是把测试理解成只对 QA 有用。其实开发者最直接受益，因为测试能给重构和上线提供最基本的安全网。

## 面试表达

测试的核心价值是降低改动风险，帮助系统在持续演进中保持可验证性。不同层次的测试关注点不同，但共同目标都是在尽可能早的位置发现问题，并让开发团队对代码改动更有信心。

## 快速回忆

- 测试是证据，不是安慰剂。
- 主要价值是降低回归风险。
- 覆盖率不是测试的最终目标。
- 没有测试，系统越复杂越不敢改。
