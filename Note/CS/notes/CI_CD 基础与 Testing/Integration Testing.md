# Integration Testing

> 集成测试的价值不在“覆盖更多代码”，而在验证模块之间真的能一起工作。很多问题单看每个单元都没错，组合起来才暴露。

## 背景与作用

集成测试关注的是多个组件、模块或外部依赖之间的协作行为。它可能涉及数据库、消息队列、缓存、文件系统，或者服务与服务之间的接口配合。和单元测试相比，它离真实运行环境更近，但执行速度和维护成本也通常更高。

很多 bug 不是单个函数逻辑错误，而是对象映射不一致、事务边界不对、序列化格式不匹配、查询结果和业务假设不同。单元测试很难覆盖这些协作问题，集成测试正是用来补这层空白。它让团队确认：不仅每个零件单独没问题，而且这些零件拼在一起时行为仍然正确。

## 场景与边界

面试用集成测试区分“只懂局部逻辑”和“理解系统协作”的开发者。工作里只要系统接上数据库和外部依赖，集成问题就迟早会出现。许多线上故障并不是业务分支判断错了，而是底层协作假设错了，集成测试能更早把这种问题拉出来。

最常见的误区是把“只要不是单元测试”都叫集成测试。更准确地说，集成测试关心的是协作边界，而不是单纯地“运行得慢一些”。另一个误区是希望它替代所有测试。集成测试很重要，但它不可能像单元测试那样高频、细粒度，因此更适合验证关键协作路径。

## 面试表达

集成测试验证的是模块或依赖之间能否正确协作，比如应用和数据库、服务和服务、消息生产和消费是否一致。它比单元测试更接近真实环境，能发现协作层的问题，但执行更慢、成本更高，所以通常要围绕关键路径有选择地建设。

## 快速回忆

- 集成测试看的是协作，不只是范围更大。
- 它能抓到很多单元测试看不到的问题。
- 成本高于单元测试，更适合关键路径。
- 慢不等于集成，协作才是关键。
