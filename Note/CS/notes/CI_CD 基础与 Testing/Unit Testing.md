# Unit Testing

> 单元测试关心的是小范围逻辑在各种输入下是否仍然成立，它之所以重要，不是因为“小”，而是因为它能最快、最稳定地给出反馈。

## 背景与作用

单元测试通常围绕一个函数、一个方法、一个类中的局部逻辑展开。它强调边界清晰、依赖可控、执行快速，适合验证计算、判断、转换、状态机这类相对独立的逻辑单元。真正好的单元测试，并不是测试对象越小越好，而是被测边界足够明确，结果足够可预测。

当业务逻辑不断迭代时，最容易先出问题的往往是局部规则变化。单元测试的价值就在于把这些规则固定下来：输入是什么，输出应该是什么，边界值该怎么处理，异常情况会怎样表现。这样一来，改动之后一跑测试，就能很快看出行为有没有偏。

## 场景与边界

单元测试是测试分层中的基础层。它数量通常最多，运行也最快，所以非常适合支撑日常开发反馈。面试问它，是想看你是否知道什么适合用单元测、什么不适合，以及为什么“快、稳、可重复”对这类测试这么重要。

一个误区是把单元测试缩成“测 getter/setter”。真正值得测的是有业务价值的逻辑和边界，而不是毫无判断的样板代码。另一个误区是让单元测试依赖数据库、网络或真实时间。那样做不仅会变慢，还会把它拖向集成测试的范畴，失去快速反馈的优势。

## 面试表达

单元测试是围绕最小可测试逻辑边界建立的测试，适合验证函数或局部规则在各种输入和边界下的行为。它的关键特征是快、稳、可重复，尽量不依赖外部资源。这样它才能在重构和日常开发中持续提供高频反馈。

## 快速回忆

- 单元测试先看边界，不先看大小。
- 快、稳、可重复是它的生命线。
- 值得测的是逻辑和规则，不是样板代码。
- 依赖外部资源太多时，它就不再是纯单元测试。
