## 局部性原理

局部性原理(Principle of Locality)是指程序在执行过程中,在某个时间段内,访问的存储器地址往往集中在一个较小的范围内。**两种局部性:**

- **时间局部性(Temporal Locality)**: 最近被访问的数据,在不久的将来很可能再次被访问
- **空间局部性(Spatial Locality)**: 如果某个地址被访问,那么它附近的地址也很可能被访问

### 存在性

源自**程序结构的特点:**

* **循环结构:**循环体内的指令和数据被反复访问，产生时间局部性

* **顺序执行:**指令通常顺序存储，顺序执行，产生空间局部性

* **数据结构:**数组、结构体等数据连续存储，访问时产生空间局部性

* **函数调用:**函数内的局部变量集中在栈上，产生空间局部性

### 有效性

* **Cache 的有效性完全依赖于局部性原理。**虽然 Cache 只缓存千分之一的数据，但这千分之一恰好是最热的数据,覆盖了 90%-99% 的访问。

**性能影响:**

```
局部性好的程序(顺序访问数组):
  Cache 命中率 95%
  平均访问时间 = 0.95 × 2ns + 0.05 × 100ns = 6.9ns

局部性差的程序(随机访问链表):
  Cache 命中率 50%
  平均访问时间 = 0.50 × 2ns + 0.50 × 100ns = 51ns

性能差距: 51 / 6.9 ≈ 7.4 倍
```

**这就是为什么:**

- 顺序访问数组比随机访问链表快得多
- 按行访问二维数组比按列访问快
- 循环展开、数据结构布局优化能显著提升性能

局部性原理是整个存储层次设计的基础。没有局部性,Cache 就没有意义,CPU 性能会下降 10-100 倍。

### 时间局部性

**时间局部性:** 如果某个数据被访问,那么在不久的将来,它很可能再次被访问。

其主要来自**循环**、**频繁调用的函数**、**全局变量**、**累加器:**

**场景 1: 循环变量**

```c
for (int i = 0; i < 1000; i++) {
    // i 被访问 1000 次
    // 具有很强的时间局部性
    sum += arr[i];
}
```

**分析:**
- 变量 `i` 在循环中被反复访问
- 变量 `sum` 在循环中被反复访问
- 这两个变量应该保持在寄存器或 Cache 中

**场景 2: 累加器**

```c
int sum = 0;
for (int i = 0; i < 1000; i++) {
    sum += arr[i];  // sum 被访问 1000 次
}
```

**分析:**
- `sum` 在每次循环中都被读取和写入
- 具有极强的时间局部性
- 编译器通常会将 `sum` 分配到寄存器

**场景 3: 函数调用**

```c
int add(int a, int b) {
    return a + b;
}

for (int i = 0; i < 1000; i++) {
    result = add(x, y);  // add 函数被调用 1000 次
}
```

**分析:**
- `add` 函数的代码被反复执行
- 函数代码应该保持在 I-Cache 中

**场景 4: 热点数据**

```c
struct Config {
    int max_connections;
    int timeout;
};

Config config;  // 全局配置

void handle_request() {
    if (connections > config.max_connections) {
        // config 被频繁访问
        return;
    }
}
```

**分析:**
- 全局配置 `config` 被多个请求访问
- 具有时间局部性,应该保持在 Cache 中

**场景 5: 递归调用**

```c
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 递归调用
}
```

**分析:**
- `factorial` 函数代码被反复执行
- 函数代码具有时间局部性

### 空间局部性

如果某个地址被访问,那么它附近的地址也很可能在不久的将来被访问。

其主要来自于**顺序执行指令:**、**数组访问**、**结构体访问**、**栈操作:**

**场景 1: 顺序访问数组**

```c
int arr[1000];
int sum = 0;
for (int i = 0; i < 1000; i++) {
    sum += arr[i];  // 顺序访问 arr[0], arr[1], arr[2]...
}
```

**分析:**
- 访问 `arr[0]` 后,接下来访问 `arr[1]`
- 数组元素连续存储,具有空间局部性
- Cache Line(通常 64 字节)可以一次加载多个元素

**Cache 效果:**
```
假设 int 占 4 字节,Cache Line 64 字节
一次加载可以包含 16 个 int

访问 arr[0]:
  - Cache 未命中,从内存加载 arr[0] 到 arr[15]
访问 arr[1] 到 arr[15]:
  - Cache 命中,无需访问内存
访问 arr[16]:
  - Cache 未命中,从内存加载 arr[16] 到 arr[31]
...

命中率: 15/16 = 93.75%
```

**场景 2: 二维数组按行访问**

```c
int matrix[100][100];

// 按行访问(好的空间局部性)
for (int i = 0; i < 100; i++) {
    for (int j = 0; j < 100; j++) {
        sum += matrix[i][j];
    }
}
```

**分析:**
- C 语言中,二维数组按行存储
- `matrix[0][0], matrix[0][1], matrix[0][2]...` 连续存储
- 按行访问具有良好的空间局部性

**场景 3: 二维数组按列访问(差的空间局部性)**

```c
// 按列访问(差的空间局部性)
for (int j = 0; j < 100; j++) {
    for (int i = 0; i < 100; i++) {
        sum += matrix[i][j];  // 跳跃访问
    }
}
```

**分析:**
- 访问 `matrix[0][0]` 后,接下来访问 `matrix[1][0]`
- 两者在内存中相距 100 个 int(400 字节)
- 空间局部性差,Cache 命中率低

**性能对比:**

```c
// 测试代码
int matrix[1000][1000];

// 按行访问
for (int i = 0; i < 1000; i++) {
    for (int j = 0; j < 1000; j++) {
        sum += matrix[i][j];
    }
}
// 时间: 约 10ms

// 按列访问
for (int j = 0; j < 1000; j++) {
    for (int i = 0; i < 1000; i++) {
        sum += matrix[i][j];
    }
}
// 时间: 约 50ms(慢 5 倍!)
```

**场景 4: 结构体访问**

```c
struct Point {
    int x;
    int y;
    int z;
};

Point p;
p.x = 10;  // 访问 x
p.y = 20;  // 访问 y,与 x 相邻
p.z = 30;  // 访问 z,与 y 相邻
```

**分析:**
- `x`, `y`, `z` 连续存储
- 访问 `x` 时,Cache Line 可能已经包含 `y` 和 `z`
- 具有空间局部性

**场景 5: 指令顺序执行**

```c
int a = 10;     // 指令 1
int b = 20;     // 指令 2
int c = a + b;  // 指令 3
```

**分析:**
- 指令 1、2、3 在内存中连续存储
- CPU 顺序执行,具有空间局部性
- 指令 Cache 可以一次加载多条指令

**场景 6: 链表访问(差的空间局部性)**

```c
struct Node {
    int data;
    Node* next;
};

Node* head = ...;
for (Node* p = head; p != NULL; p = p->next) {
    sum += p->data;  // 节点可能分散在内存中
}
```

**分析:**
- 链表节点通过指针连接,不一定连续存储
- 空间局部性差,Cache 命中率低
- 这是链表相比数组的性能劣势之一

### 局部性原理的应用

#### Cache 设计

**Cache Line:**Cache 以 Cache Line 为单位加载数据(通常 64 字节)，利用空间局部性,一次加载多个数据

**LRU 替换策略:**替换最久未使用的 Cache Line，利用时间局部性，保留最近访问的数据

#### 虚拟内存

**页面置换:**操作系统使用 LRU 等策略选择要换出的页面，利用时间局部性，保留最近访问的页面

**工作集:**程序在某个时间段内频繁访问的页面集合，如果工作集能全部放入内存,缺页率低

#### 预取技术

**硬件预取:**CPU 检测到顺序访问模式，自动预取后续数据到 Cache

```
for (int i = 0; i < 1000; i++) {
    sum += arr[i];
}

访问 arr[0]:
  - 加载 arr[0] - arr[15]
  - 硬件检测到顺序访问
  - 预取 arr[16] - arr[31] 到 Cache

访问 arr[16] 时:
  - 已经在 Cache 中,命中!
```

**软件预取:**程序员使用预取指令

```c
for (int i = 0; i < 1000; i++) {
    __builtin_prefetch(&arr[i + 16]);  // 预取后续数据
    sum += arr[i];
}
```

#### 高效编程

##### 提高时间局部性

**技巧 1: 重用数据**

```c
// 差的时间局部性
for (int i = 0; i < n; i++) {
    result1 += compute(arr[i]);
}
for (int i = 0; i < n; i++) {
    result2 += compute(arr[i]);  // arr[i] 再次访问,可能已不在 Cache
}

// 好的时间局部性
for (int i = 0; i < n; i++) {
    int val = compute(arr[i]);
    result1 += val;
    result2 += val;  // 重用 val,在寄存器或 Cache 中
}
```

**技巧 2: 循环融合**

```c
// 差的时间局部性
for (int i = 0; i < n; i++) {
    a[i] = b[i] + c[i];
}
for (int i = 0; i < n; i++) {
    d[i] = a[i] * 2;  // a[i] 可能已不在 Cache
}

// 好的时间局部性
for (int i = 0; i < n; i++) {
    a[i] = b[i] + c[i];
    d[i] = a[i] * 2;  // a[i] 刚刚计算,在 Cache 中
}
```

##### 提高空间局部性

**技巧 1: 按行访问二维数组**

```c
// 好的空间局部性
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        sum += matrix[i][j];
    }
}

// 差的空间局部性
for (int j = 0; j < n; j++) {
    for (int i = 0; i < n; i++) {
        sum += matrix[i][j];
    }
}
```

**技巧 2: 使用数组而非链表**

```c
// 差的空间局部性
struct Node* list = ...;
for (Node* p = list; p != NULL; p = p->next) {
    process(p->data);
}

// 好的空间局部性
int arr[n];
for (int i = 0; i < n; i++) {
    process(arr[i]);
}
```

**技巧 3: 数据结构布局优化**

```c
// 差的空间局部性(AoS: Array of Structures)
struct Point {
    float x, y, z;
};
Point points[1000];

for (int i = 0; i < 1000; i++) {
    sum += points[i].x;  // 只需要 x,但加载了整个 Point
}

// 好的空间局部性(SoA: Structure of Arrays)
struct Points {
    float x[1000];
    float y[1000];
    float z[1000];
};
Points points;

for (int i = 0; i < 1000; i++) {
    sum += points.x[i];  // 只加载 x 数组,空间局部性好
}
```

**技巧 4: 分块(Blocking/Tiling)**

```c
// 矩阵乘法,差的局部性
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        for (int k = 0; k < n; k++) {
            C[i][j] += A[i][k] * B[k][j];
        }
    }
}

// 分块优化,好的局部性
int block_size = 64;
for (int ii = 0; ii < n; ii += block_size) {
    for (int jj = 0; jj < n; jj += block_size) {
        for (int kk = 0; kk < n; kk += block_size) {
            // 在小块内计算,提高 Cache 命中率
            for (int i = ii; i < ii + block_size; i++) {
                for (int j = jj; j < jj + block_size; j++) {
                    for (int k = kk; k < kk + block_size; k++) {
                        C[i][j] += A[i][k] * B[k][j];
                    }
                }
            }
        }
    }
}
```

## 高频判断

### 判断 1: 时间局部性是指最近访问的数据很可能再次被访问

**正确。** 这是时间局部性的定义。

### 判断 2: 空间局部性是指访问某个地址后,附近的地址也可能被访问

**正确。** 这是空间局部性的定义。

### 判断 3: 局部性原理是 Cache 有效性的基础

**正确。** 如果程序访问完全随机,Cache 命中率会很低,局部性保证了 Cache 的有效性。

### 判断 4: 循环变量具有时间局部性

**正确。** 循环变量在循环中被反复访问,具有很强的时间局部性。

### 判断 5: 顺序访问数组具有空间局部性

**正确。** 数组元素连续存储,顺序访问具有良好的空间局部性。

### 判断 6: 按列访问二维数组比按行访问快

**错误。** C 语言中二维数组按行存储,按行访问空间局部性好,比按列访问快。

### 判断 7: 链表访问具有良好的空间局部性

**错误。** 链表节点通过指针连接,不一定连续存储,空间局部性差。

### 判断 8: Cache Line 的设计利用了空间局部性

**正确。** Cache Line 一次加载多个连续数据,利用了空间局部性。

### 判断 9: LRU 替换策略利用了时间局部性

**正确。** LRU 保留最近访问的数据,利用了时间局部性。

### 判断 10: 局部性好的程序 Cache 命中率高

**正确。** 局部性好意味着访问集中,Cache 能有效缓存热点数据,命中率高。

### 判断 11: 全局变量具有时间局部性

**正确。** 全局变量通常被多个函数访问,具有时间局部性。

### 判断 12: 指令执行具有空间局部性

**正确。** 指令通常连续存储,顺序执行,具有空间局部性。

### 判断 13: 提高局部性可以提高程序性能

**正确。** 局部性好的程序 Cache 命中率高,访问速度快,性能好。

### 判断 14: 所有程序都具有良好的局部性

**错误。** 不同程序的局部性差异很大,随机访问的程序局部性差。

### 判断 15: 数组比链表有更好的空间局部性

**正确。** 数组元素连续存储,空间局部性好;链表节点分散,空间局部性差。
