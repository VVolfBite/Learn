# 数据库进阶基础 - 06 缓存与读模型

缓存的目标是降低延迟与数据库压力，但一致性与故障模式必须被显式设计。核心主题：旁路缓存、一致性策略、三大故障、热点治理。

## 缓存定义

### 语义定位

缓存可以理解为“**读路径上的近端副本**”。“近端”表示它离请求更近、访问更快；“副本”表示它保存的是数据库某个时刻的数据拷贝，而不是唯一真实来源。
 这一定义非常重要，因为它直接说明了缓存的边界：缓存负责提速，但不天然负责比数据库更强的一致性。相反，只要使用缓存，就必须接受这样一个现实——缓存中的数据在某些时刻可能比数据库旧，而且这种旧值不一定是 bug，而是缓存体系的一部分代价。

因此，缓存的设计重点不只是“怎么命中”，还包括：

- 允许旧到什么程度
- 什么时候失效
- 写入后如何避免明显读旧
- 缓存异常时如何止损

只要这些问题没有提前回答，缓存就不是系统能力，而只是一个潜在风险点。

### 典型收益

缓存最直接的收益是降低延迟，尤其是 P95、P99 这类尾延迟。相比数据库，缓存通常更轻、更近，访问路径更短，因此热点请求可以更快返回。第二个收益是减轻数据库 QPS 压力。大量重复读取如果都命中缓存，数据库就可以把资源留给真正需要计算、聚合和写入的请求。第三个收益是承接突发流量。在热点事件、促销活动、榜单刷新、内容爆发等场景里，请求量常常会突然上升，如果没有缓存，数据库很容易直接被打到瓶颈。

但收益越大，也意味着系统越依赖缓存命中率。一旦缓存失效、雪崩或回源策略设计不好，数据库会瞬间接住原本被缓存挡住的全部流量，因此**缓存故障时的止损能力**和**缓存命中时的性能收益**一样重要。

## 缓存位置

缓存可以放在不同层，位置不同，收益和一致性难度也不同。

**客户端缓存**离用户最近，速度最快，适合一些相对静态或弱一致的数据；但客户端分散、不可控，失效与刷新最难统一，因此一致性最难保证。

**应用内缓存**位于服务进程内部，访问速度很快，适合本机热点数据；但一旦服务有多个实例，每个实例都有自己的本地缓存，就会出现实例间数据不一致的问题。某个实例刚更新，本机是新的，其他实例仍然是旧的。

**分布式缓存**是最常见的通用方案。它位于应用和数据库之间，多实例共享，扩展性也较好，适合承接大部分高频读请求。但它同样需要解决失效、更新、热点与回源问题。

**CDN / 边缘缓存**更适合静态资源或弱一致内容，例如图片、静态页面、公共内容分发。它能极大降低回源压力，但对实时强一致场景不适合。

因此，缓存位置并没有“最好”的统一答案，只有“离用户更近但一致性更难”与“更集中更可控但速度略慢”之间的不同平衡。

## 旁路缓存

### 读路径

旁路缓存（Cache Aside）是最常见的缓存模式。读请求先查缓存，命中就直接返回；如果没命中，再去查数据库，并把结果写回缓存，供后续请求复用。

这种方式的优点是职责清晰：数据库仍然是真实数据源，缓存只是加速层；当缓存不存在时，系统依然可以通过数据库得到正确结果。也正因为如此，旁路缓存在工程上最常用、最容易理解。

### 写路径

写请求通常先写数据库，再删除缓存，或者更新缓存。
 实践里更常见的是“**先写库，再删缓存**”，因为删除缓存比直接更新缓存更容易统一读路径，也更少出现并发更新竞态。
 如果选择直接更新缓存，就必须非常仔细地处理“数据库写成功但缓存更新失败”“多个并发写把缓存覆盖成旧值”等问题；而删除缓存后，后续读请求自然会统一走“查库再回填”的路径，逻辑更稳定。

所以“写后删缓存”之所以常见，不是因为它完美，而是因为它在一致性和实现复杂度之间相对更稳。

## 写后读一致性

写数据库之后，如果缓存中还是旧值，那么紧接着发生的读取就可能看到旧数据。这类问题在用户体验上往往非常刺眼：刚改完昵称刷新还是旧的，刚支付完查订单还是未支付，刚更新资料结果页面没变。

因此，缓存系统必须回答“**写后多久允许看到旧值**”这个问题。不同场景允许的答案不同。

对于关键路径，常见做法是**写后读主库或一致性读**，保证刚写完就能读到最新状态。对于一般路径，可以接受短暂旧值，但要通过版本戳、失效顺序控制、延迟双删等方式尽量缩小旧值窗口。
 本质上，写后读一致性并不是缓存自身天然具备的能力，而是缓存系统与数据库读写策略共同协作出来的结果。

## 三大故障

### 穿透

缓存穿透表示大量请求查询的是**根本不存在的数据**。因为缓存里没有，数据库里也没有，所以每次请求都会一路打到数据库。
 如果攻击流量或异常流量专门构造这种不存在的 key，数据库就会被无意义请求持续消耗。

常见对策包括：

- **布隆过滤器**：提前挡掉明显不存在的 key
- **缓存空值**：数据库查不到时也短时间缓存“空结果”
- **参数校验与限流**：从入口减少无效请求进入核心链路

穿透的本质是：缓存挡不住“从没存在过的数据”，所以必须在缓存之前再加一层“存在性判断”或“无效请求过滤”。

### 击穿

缓存击穿通常发生在**热点 key 失效的瞬间**。这个 key 平时一直被大量请求读取，一旦它过期，大量并发请求会同时发现 miss，然后一起回源数据库。
 由于这个 key 本来就是热点，所以回源压力可能非常集中，甚至直接把数据库压垮。

常见对策包括：

- **互斥回源**：同一时刻只允许一个请求回源数据库，其余等待结果
- **永不过期 + 后台刷新**：热点 key 不在业务请求路径上自然失效，而是后台提前更新
- **分层缓存**：本地缓存 + 分布式缓存，减少单层 miss 冲击

击穿的核心不是“缓存少了一个值”，而是“这个值太热，失效后回源压力太集中”。

### 雪崩

缓存雪崩表示**大量 key 在同一时间段集中失效**，导致原本被缓存挡住的请求同时落到数据库上。和击穿不同，雪崩不是某一个热点 key 的问题，而是很多 key 一起失效造成的整体回源洪峰。

最典型的触发方式是：

- 批量设置了相同 TTL
- 缓存集群整体故障
- 大促前后统一预热、统一过期

常见对策包括：

- **TTL 抖动**：给过期时间加随机值，避免同一时刻大面积失效
- **分批失效**：把集中失效拆开
- **预热**：提前把关键数据放入缓存
- **限流、熔断、降级**：缓存失效时保护数据库不要被拖死

雪崩的本质是：原本被缓存层平滑掉的流量，突然整体穿透到数据库，形成系统级压力峰值。

## 热点 key

热点 key 指的是某个或某类 key 的访问量远高于其他数据，导致它成为缓存系统中的局部热点。
 典型来源包括：爆款商品、首页榜单、热门房间、超级大 V 主页、活动入口等。

热点 key 的风险在于：即使缓存命中率很高，也可能因为**单个 key 过热**而把某一层资源打满。例如：

- 某个缓存分片因为这个 key 过热而 CPU 飙高
- 某个应用实例因为本地缓存回源该 key 太频繁而负载不均
- 这个 key 一旦失效，就形成典型击穿

常见治理手段包括：

- **分桶 / 加盐**：把一个热点 key 拆成多个逻辑 key 分散流量
- **多级缓存**：本地缓存先挡一层，再访问分布式缓存
- **读写隔离**：把热点读和普通读分开处理
- **热点保护与限流**：对异常高频访问做节流，避免把后端拖垮

热点治理的核心思路不是“彻底消灭热点”，而是**让热点不要集中打在同一个点上**。
