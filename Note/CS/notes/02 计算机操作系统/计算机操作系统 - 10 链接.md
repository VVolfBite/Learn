## 编译系统

以下是一个 hello.c 程序：

```c
#include <stdio.h>

int main()
{
    printf("hello, world\n");
    return 0;
}
```

在 Unix 系统上，由编译器把源文件转换为目标文件。

```bash
gcc -o hello hello.c
```

这个过程大致如下：

![](https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/b396d726-b75f-4a32-89a2-03a7b6e19f6f.jpg)

从源代码到可执行文件，需要经历四个阶段：预处理、编译、汇编、链接。

**预处理阶段**把源文件中的预处理命令展开。预处理器会处理所有以 # 开头的指令，比如展开宏定义（#define）、处理条件编译（#ifdef）、包含头文件（#include）。预处理后生成 hello.i 文件，这是一个纯粹的 C 源文件，不再包含任何预处理指令。

**编译阶段**把预处理后的源文件翻译成汇编代码。编译器会进行词法分析、语法分析、语义分析，然后生成中间代码，最后进行代码优化，输出汇编文件 hello.s。汇编代码是人类可读的，但已经非常接近机器指令了。

**汇编阶段**把汇编文件翻译成机器指令。汇编器会把每条汇编指令翻译成对应的机器指令，生成可重定位目标文件 hello.o。这个文件已经是二进制格式，但还不能直接运行，因为它可能引用了其他文件中的函数或变量。

**链接阶段**把多个目标文件合并成一个可执行文件。链接器会解析符号引用（比如 printf 函数），重定位代码和数据（分配内存地址），最终生成可执行文件 hello。因为程序通常由多个源文件组成，而且会调用标准库函数（如 printf）。编译器只能处理单个源文件，生成单个目标文件。链接器的作用是把这些目标文件组合起来，生成最终的可执行文件。

## 静态链接

静态链接器以一组可重定位目标文件为输入，生成一个完全链接的可执行目标文件作为输出。链接器主要完成两个任务：符号解析和重定位。

**符号解析**是将每个符号引用与一个符号定义关联起来。每个符号对应于一个函数、一个全局变量或一个静态变量。例如，main 函数调用了 printf，链接器需要找到 printf 的定义（在 libc.a 中），并将 main 中对 printf 的引用指向 printf 的定义。

**重定位**是把每个符号定义与一个内存位置关联起来，然后修改所有对这些符号的引用，使得它们指向这个内存位置。编译器生成的目标文件中，函数和变量的地址都是相对地址（从 0 开始）。链接器需要把这些相对地址转换为绝对地址。

简单来说，符号解析就是把所有符号引用和对应的符号定义关联起来；重定位则是在确定各个符号最终装入地址后，修改代码和数据中的引用，使其指向正确的内存地址。

![](https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/47d98583-8bb0-45cc-812d-47eefa0a4a40.jpg)

链接器首先读取所有目标文件和静态库，然后解析所有符号引用，找到对应的符号定义。接着，链接器会合并相同类型的节（.text、.data、.bss），为每个符号分配内存地址，修改所有符号引用使其指向正确的内存地址，最后生成可执行文件。

## 目标文件

目标文件有三种类型：

1. **可重定位目标文件（Relocatable Object File）**：不能直接执行，需要经过链接器与其他目标文件或库进行链接，用来生成可执行文件或共享库，比如`.o` 文件（Linux）、`.obj` 文件（Windows）
2. **可执行目标文件（Executable Object File）**：可以直接加载到内存中执行，包含完整的程序代码和数据，比如ELF 文件（Linux）、.exe 文件（Windows）
3. **共享目标文件（Shared Object File）**：特殊的可重定位目标文件，可以在加载时或运行时被动态加载到内存并被多个进程共享，比如.so 文件（Linux）、.dll 文件（Windows）

**目标文件的结构**：

- **ELF 头部**：描述文件的基本信息
- **.text 节**：已编译程序的机器代码
- **.data 节**：已初始化的全局变量和静态变量
- **.bss 节**：未初始化的全局变量和静态变量
- **.symtab 节**：符号表，包含函数和全局变量的信息
- **.rel.text 节**：.text 节的重定位信息
- **.rel.data 节**：.data 节的重定位信息

## 动态链接

静态库有以下两个问题：

1. **更新问题**：当静态库更新时，整个程序都要重新链接；即使只修改了一个函数，也要重新链接整个程序，这样十分浪费时间和资源

2. **空间浪费问题**：静态库的问题在于更新困难和空间浪费。如果库函数有 bug 或需要升级，所有使用这个库的程序都需要重新链接。更严重的是，对于 printf 这种标准函数库，如果每个程序都包含一份代码，会极大浪费磁盘空间和内存空间。例如，100 个程序都使用 printf，就有 100 份 printf 的代码。

共享库（Shared Library）是为了解决静态库的这两个问题而设计的。共享库在磁盘上只有一份，所有引用该库的可执行文件都共享这个文件，不会被复制到可执行文件中。在内存中，一个共享库的 .text 节的一个副本可以被不同的进程共享，多个进程可以同时使用同一份代码，节省内存空间。更重要的是，更新共享库不需要重新链接程序，程序下次运行时会自动使用新版本的库。

![](https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/76dc7769-1aac-4888-9bea-064f1caa8e77.jpg)

动态链接有两种方式。**加载时动态链接**是程序启动时，动态链接器加载所需的共享库，解析符号引用，进行重定位，然后将控制权交给程序。**运行时动态链接**是程序运行过程中，通过 dlopen 等函数加载共享库，通过 dlsym 获取符号地址，通过 dlclose 卸载共享库。

动态链接的优点是节省磁盘空间和内存空间，可以独立更新库而不需要重新链接程序，还可以实现插件机制。缺点是启动时间稍长（需要加载和链接共享库），可能出现版本兼容性问题（DLL Hell）。

## 静态链接 vs 动态链接

静态链接和动态链接的主要区别在于链接时机和资源占用。

## 静态链接与动态链接对比

| 对比维度           | 静态链接                                                 | 动态链接                                           |
| ------------------ | -------------------------------------------------------- | -------------------------------------------------- |
| **链接时机**       | 在编译/链接阶段完成                                      | 在程序加载时或运行时完成                           |
| **可执行文件大小** | 较大，包含所需库代码                                     | 较小，只保留对共享库的引用信息                     |
| **内存占用**       | 每个程序各自保存一份库代码，占用较多内存                 | 多个程序可共享同一份库代码，内存占用较少           |
| **启动速度**       | 通常较快，不需要额外装载共享库                           | 通常稍慢，需要装载并解析共享库                     |
| **库更新**         | 库更新后通常需要重新链接并重新生成可执行文件             | 只要接口兼容，一般不需要重新链接程序               |
| **外部依赖**       | 依赖较少，程序自包含                                     | 依赖外部共享库                                     |
| **可移植性**       | 较好，拷贝到目标环境更容易运行                           | 较差，目标环境需要存在对应共享库                   |
| **运行稳定性**     | 较稳定，不容易受外部库变化影响                           | 可能受共享库版本变化影响                           |
| **磁盘空间利用**   | 多个程序若都静态链接同一库，会重复占用磁盘空间           | 多个程序共享同一库，磁盘空间利用率更高             |
| **适用场景**       | 独立运行程序、启动速度要求高、嵌入式系统、发布单文件工具 | 大型应用、公共库频繁更新、多个程序共享库、插件系统 |

使用静态链接：需要生成**独立运行**的程序，不希望依赖外部库，或者对**启动速度**要求较高；比如用于**嵌入式系统**或部署环境固定的场景，希望降低运行时因库缺失或版本不匹配导致的问题

使用动态链接：程序较大，希望**减小可执行文件体积**，且多个程序需要**共享同一份库**或者库需要**频繁更新**；需要支持**插件化、模块化**扩展；希望节省内存和磁盘空间

## 面试高频问题

### 什么是链接？为什么需要链接？

**链接**是将多个目标文件和库文件组合成一个可执行文件的过程。

**为什么需要链接？**
- 程序通常由多个源文件组成，编译器只能处理单个源文件
- 程序会调用标准库函数（如 printf），需要将库代码链接进来
- 需要解析符号引用，将函数调用指向正确的函数定义
- 需要为代码和数据分配内存地址

### 静态链接和动态链接的区别？

读到“静态链接和动态链接的区别？”时，先别急着记结论。更有效的方式是先弄清它在“计算机操作系统 - 链接”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**静态链接**：
- 编译时将库代码复制到可执行文件中
- 可执行文件大，但无外部依赖
- 更新库需要重新链接

**动态链接**：
- 运行时加载共享库
- 可执行文件小，多个程序共享库代码
- 更新库不需要重新链接

### 为什么动态链接可以节省内存？

因为多个进程可以共享同一份库代码（.text 节）。

例如，100 个进程都使用 libc.so，内存中只有一份 libc.so 的代码，所有进程共享这份代码。如果是静态链接，内存中会有 100 份 libc 的代码。

注意：只有代码（.text 节）可以共享，数据（.data 节）不能共享，每个进程都有自己的数据副本。

### 什么是符号解析？什么是重定位？

**符号解析**：将符号引用与符号定义关联起来。

例如，main 函数调用了 printf，链接器需要找到 printf 的定义，并将 main 中对 printf 的引用指向 printf 的定义。

**重定位**：为符号分配内存地址，并修改所有对这些符号的引用。

例如，编译器生成的目标文件中，函数和变量的地址都是相对地址。链接器需要把这些相对地址转换为绝对地址。

### 编译、汇编、链接的区别？

**编译**：将高级语言（C/C++）翻译成汇编语言

**汇编**：将汇编语言翻译成机器指令（目标文件）

**链接**：将多个目标文件组合成一个可执行文件

### 为什么 Linux 下是 .so，Windows 下是 .dll？

这只是命名约定不同：
- Linux：Shared Object（.so）
- Windows：Dynamic Link Library（.dll）

本质上都是共享库，功能相同。

### 什么是 DLL Hell？

DLL Hell 是指动态链接库的版本兼容性问题。

**问题**：
- 程序 A 依赖 lib.dll 1.0 版本
- 程序 B 安装时覆盖了 lib.dll，升级到 2.0 版本
- 程序 A 运行时使用 2.0 版本，可能不兼容，导致崩溃

**解决方法**：
- 使用版本号管理（如 lib-1.0.dll、lib-2.0.dll）
- 使用私有部署（每个程序使用自己的库副本）
- 使用包管理器（如 apt、yum）统一管理依赖

## 快速回忆

这一节看起来往往容易变成一串要点，其实先理解“快速复习要点”在计算机操作系统里的作用会更顺。后面的内容可以理解成围绕这个问题展开的几种常见情况。

这一节看起来往往容易变成一串结论，其实先抓住“快速复习要点”在计算机操作系统里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

1. 编译系统四个阶段：预处理 → 编译 → 汇编 → 链接
2. 链接的两个任务：符号解析、重定位
3. 静态链接：编译时链接，可执行文件大，无外部依赖
4. 动态链接：运行时链接，可执行文件小，多个程序共享库代码
5. 动态链接节省内存：多个进程共享同一份库代码（.text 节）
6. 静态链接适合：独立运行、启动速度要求高、嵌入式系统
7. 动态链接适合：大型应用、需要频繁更新、多个程序共享库
8. 目标文件三种类型：可重定位、可执行、共享
9. 符号解析：将符号引用与符号定义关联
10. 重定位：为符号分配内存地址，修改引用
