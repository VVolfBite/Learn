
# 计算机操作系统 - 上下文切换

上下文切换（Context Switching）是指 CPU 从一个进程或线程切换到另一个进程或线程的过程。

在切换过程中，操作系统需要：
1. 保存当前进程/线程的上下文（CPU 寄存器、程序计数器、栈指针等）
2. 加载下一个进程/线程的上下文
3. 恢复执行

上下文包括：CPU 寄存器的值、程序计数器（PC）、栈指针（SP）、进程状态、内存管理信息（页表等）。

## 为什么需要上下文切换

上下文切换是实现多任务操作系统的基础，主要原因包括：

**时间片轮转**：每个进程分配一个时间片，时间片用完后切换到下一个进程，实现进程的并发执行。

**进程阻塞**：进程等待 I/O 操作完成或等待某个资源时，CPU 切换到其他就绪进程，提高利用率。

**优先级调度**：高优先级进程到达时，抢占当前低优先级进程，保证重要任务及时响应。

**中断处理**：硬件中断发生时，需要立即处理中断，暂停当前进程执行。

## 上下文切换的过程

典型的上下文切换过程：

```
1. 保存当前进程状态
   - 保存 CPU 寄存器到 PCB
   - 保存程序计数器
   - 保存栈指针
   
2. 更新进程状态
   - 将当前进程状态改为就绪或阻塞
   - 选择下一个要运行的进程
   
3. 加载新进程状态
   - 从新进程的 PCB 加载寄存器
   - 加载程序计数器
   - 加载栈指针
   - 切换页表（如果是进程切换）
   
4. 恢复执行
   - 跳转到新进程的指令位置
   - 继续执行
```

## 上下文切换的类型

### 进程上下文切换

需要切换虚拟内存空间和页表，开销最大。

包含的操作：
- 保存/恢复 CPU 寄存器
- 切换页表和虚拟内存
- 刷新 TLB（Translation Lookaside Buffer）
- 切换内核栈

典型场景：
- 时间片用完
- 进程主动让出 CPU（sleep、wait）
- 高优先级进程抢占

### 线程上下文切换

同一进程内的线程切换：
- 不需要切换虚拟内存空间
- 不需要切换页表
- 只需要切换线程私有数据（寄存器、栈）
- 开销较小

不同进程的线程切换：
- 等同于进程切换
- 需要切换虚拟内存空间

### 中断上下文切换

由硬件中断触发，只保存必要的寄存器，不涉及进程调度，开销最小。

过程：
1. 硬件自动保存部分寄存器
2. 跳转到中断处理程序
3. 处理中断
4. 恢复寄存器并返回

## 上下文切换的成本

### 直接成本

读到“直接成本”时，先别急着记结论。更有效的方式是先弄清它在“计算机操作系统 - 上下文切换”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

CPU 时间消耗：
- 保存和恢复寄存器：几十到几百个 CPU 周期
- 切换页表：需要刷新 TLB
- 内核调度算法执行时间

内存访问：
- 访问 PCB 保存/加载状态
- 访问页表
- 可能导致缓存失效

### 间接成本

这是最大的性能损失来源。

缓存失效：
- CPU 缓存（L1、L2、L3）失效
- TLB 缓存失效
- 新进程需要重新加载数据到缓存

流水线停顿：
- CPU 流水线被清空
- 分支预测失效
- 需要重新填充流水线

典型数据：
- 线程切换：1-2 微秒
- 进程切换：5-10 微秒（包括缓存失效的影响可能达到几十微秒）

## 如何减少上下文切换

### 减少线程数量
- 使用线程池，避免频繁创建销毁线程
- 合理设置线程池大小
- 避免创建过多线程

### 使用协程
- 用户态调度，无需内核参与
- 切换开销极小
- 适合 I/O 密集型任务

### 使用无锁数据结构
- 减少锁竞争导致的阻塞
- 使用 CAS（Compare-And-Swap）等原子操作
- 减少线程等待时间

### 合理使用多线程
- CPU 密集型任务：线程数 ≈ CPU 核心数
- I/O 密集型任务：可以适当增加线程数
- 避免过度并发

### 绑定 CPU
- 将进程/线程绑定到特定 CPU 核心
- 减少跨核心调度
- 提高缓存命中率

### 批处理
- 批量处理任务，减少切换次数
- 使用消息队列缓冲请求

## 常见问题

**上下文切换 vs 模式切换**

上下文切换是 CPU 从一个进程/线程切换到另一个，需要保存和恢复完整的执行环境，开销较大。

模式切换是同一进程内的特权级切换（用户态 ↔ 内核态），不涉及进程调度，开销相对较小。系统调用会触发模式切换，但不一定触发上下文切换。

关系：上下文切换一定包含模式切换，模式切换不一定包含上下文切换。

**进程切换 vs 线程切换**

| 对比项 | 进程切换 | 线程切换（同进程） |
|--------|----------|-------------------|
| 虚拟内存 | 需要切换 | 不需要切换 |
| 页表 | 需要切换 | 不需要切换 |
| TLB | 需要刷新 | 不需要刷新 |
| 开销 | 大 | 小 |
| 典型时间 | 5-10 微秒 | 1-2 微秒 |

**什么情况会触发上下文切换**

主动切换：
- 进程调用 sleep()、wait() 等系统调用
- 进程等待 I/O 操作完成
- 进程主动让出 CPU

被动切换：
- 时间片用完
- 高优先级进程抢占
- 硬件中断发生

**上下文切换一定会进入内核态吗**

是的。上下文切换由操作系统内核完成，必须在内核态执行，用户态程序无法直接进行上下文切换。

例外：协程切换在用户态完成，但协程切换不是真正的上下文切换，只是用户态的执行流切换。

**上下文切换越少越好吗**

不一定。

切换太少的问题：
- 某些进程可能长时间得不到执行
- 响应时间变长
- 系统吞吐量可能下降

切换太多的问题：
- CPU 时间浪费在切换上
- 缓存频繁失效
- 系统整体性能下降

需要平衡：根据应用场景调整。CPU 密集型应该减少切换，I/O 密集型可以适当增加切换提高并发。

**多线程一定比单线程快吗**

不一定。

多线程更快的场景：
- 多核 CPU 上的 CPU 密集型任务
- I/O 密集型任务（等待时可以切换）
- 任务可以并行执行

单线程可能更快的场景：
- 单核 CPU 上的 CPU 密集型任务
- 上下文切换开销大于并行收益
- 任务之间依赖性强，无法并行

**如何查看系统的上下文切换次数**

Linux 系统：

```bash
# 查看整体上下文切换次数
vmstat 1

# 查看每个进程的上下文切换
pidstat -w 1

# 查看详细信息
cat /proc/interrupts
cat /proc/<pid>/status | grep ctxt
```

关键指标：
- cs（context switch）：每秒上下文切换次数
- in（interrupt）：每秒中断次数
- 自愿切换（voluntary）：进程主动让出 CPU
- 非自愿切换（involuntary）：被动切换（时间片用完）
