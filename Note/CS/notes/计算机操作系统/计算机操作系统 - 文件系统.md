
# 计算机操作系统 - 文件系统

## 文件系统

文件系统（File System）是操作系统用于管理和组织存储设备上数据的方法和数据结构。它将物理存储空间抽象为文件和目录，提供统一的文件访问接口，管理存储空间的分配和回收，保证数据的完整性和安全性。

### 文件系统的组成

以 Linux 的 ext2/ext3/ext4 文件系统为例，文件系统主要由四部分组成：

```
┌─────────────────────────────────────────────────┐
│                  文件系统                        │
├─────────────┬──────────┬──────────┬─────────────┤
│  超级块      │  inode   │  数据块   │  其他元数据  │
│ (Superblock)│  表      │  (Data   │             │
│             │          │  Blocks) │             │
└─────────────┴──────────┴──────────┴─────────────┘
```

#### 超级块（Superblock）

存储文件系统的元信息，是文件系统的"身份证"。超级块损坏，整个文件系统无法使用，所以通常会有多个备份。包含的信息：文件系统的类型和版本、文件系统的大小、块的大小、inode 的数量、空闲块的数量、空闲 inode 的数量、挂载时间、最后写入时间

#### inode（索引节点）

存储文件的元数据，每个文件对应一个 inode。详见下一节。

#### 数据块（Data Blocks）

存储文件的实际内容，是文件系统的最小存储单位。固定大小（通常是 4KB）、一个文件可能占用多个数据块、小文件可能浪费空间

#### 目录

组织文件的层次结构，提供文件名到 inode 的映射。

目录也是一种特殊的文件，内容是目录项的列表，每个目录项包含文件名和 inode 号。
## Inode

inode（Index Node）是文件系统中存储文件元数据的数据结构。

每个文件都有一个 inode；inode 存储文件的所有信息，除了文件名和内容；文件名存储在目录中；文件内容存储在数据块中。

### inode 信息

```c
struct inode {
    mode_t    i_mode;      // 文件类型和权限
    uid_t     i_uid;       // 所有者 ID
    gid_t     i_gid;       // 组 ID
    off_t     i_size;      // 文件大小
    time_t    i_atime;     // 最后访问时间
    time_t    i_mtime;     // 最后修改时间
    time_t    i_ctime;     // inode 最后改变时间
    nlink_t   i_nlink;     // 硬链接计数
    blkcnt_t  i_blocks;    // 占用的块数
    blksize_t i_blksize;   // 块大小
    dev_t     i_rdev;      // 设备号（设备文件）
    
    // 数据块指针
    blk_t     i_block[15]; // 数据块索引
};
```

注意：**inode 不包含文件名！**

### inode 号

inode 号（inode number）：每个 inode 都有一个唯一的编号；操作系统通过 inode 号来识别文件；文件名只是 inode 号的别名。

查看 inode 号：
```bash
ls -i filename
stat filename
```

### inode 的大小

典型大小：
- ext2/ext3：128 字节
- ext4：256 字节

inode 数量：
- 文件系统创建时确定
- 通常按照一定比例分配（如每 16KB 一个 inode）
- inode 用完，即使有空闲空间也无法创建新文件

## 目录与文件组织

### 目录的实现

目录是特殊的文件：有自己的 inode；内容是目录项的列表；每个目录项包含文件名和 inode 号

目录项结构：
```c
struct dirent {
    ino_t  d_ino;           // inode 号
    off_t  d_off;           // 偏移量
    unsigned short d_reclen; // 记录长度
    unsigned char  d_type;   // 文件类型
    char   d_name[256];     // 文件名
};
```

### 目录项示例

目录 /home/user 的内容：

```
文件名        inode 号
.            12345      (当前目录)
..           67890      (父目录)
file1.txt    11111
file2.txt    22222
subdir       33333
```

### 路径解析

查找文件 /home/user/file1.txt 的过程：

```
1. 从根目录 / 开始（inode 号通常是 2）
   ↓
2. 读取根目录的 inode
   ↓
3. 读取根目录的数据块，查找 "home"
   ↓
4. 找到 "home" 的 inode 号
   ↓
5. 读取 home 目录的 inode
   ↓
6. 读取 home 目录的数据块，查找 "user"
   ↓
7. 找到 "user" 的 inode 号
   ↓
8. 读取 user 目录的 inode
   ↓
9. 读取 user 目录的数据块，查找 "file1.txt"
   ↓
10. 找到 "file1.txt" 的 inode 号
   ↓
11. 读取 file1.txt 的 inode，获取文件信息
```

优化：使用目录项缓存（dentry cache）减少磁盘访问次数。

## 硬链接与软链接

### 硬链接（Hard Link）

定义：多个文件名指向同一个 inode，本质上是同一个文件的不同名字。

特点：
- 共享同一个 inode
- 共享同一份数据
- 删除一个硬链接，不影响其他硬链接
- inode 的硬链接计数（nlink）记录链接数
- 只有当硬链接计数为 0 时，文件才真正被删除

创建硬链接：
```bash
ln source.txt hardlink.txt
```

示例：
```
创建硬链接前：
文件名        inode 号    nlink
file1.txt    12345       1

创建硬链接后：
文件名        inode 号    nlink
file1.txt    12345       2
file2.txt    12345       2  (指向同一个 inode)
```

限制：
- 不能跨文件系统
- 不能链接目录（防止循环）

### 软链接（Soft Link / Symbolic Link）

定义：一个特殊的文件，内容是另一个文件的路径，类似于 Windows 的快捷方式。

特点：
- 有自己的 inode
- 内容是目标文件的路径
- 可以跨文件系统
- 可以链接目录
- 目标文件删除后，软链接失效（悬空链接）

创建软链接：
```bash
ln -s source.txt symlink.txt
```

示例：
```
文件名        inode 号    类型        内容
file1.txt    12345       普通文件    "Hello World"
symlink.txt  67890       软链接      "/path/to/file1.txt"
```

### 两者对比

| 特性 | 硬链接 | 软链接 |
|------|--------|--------|
| inode | 共享同一个 | 有自己的 inode |
| 数据 | 共享同一份 | 存储路径字符串 |
| 跨文件系统 | 不可以 | 可以 |
| 链接目录 | 不可以 | 可以 |
| 原文件删除 | 不影响 | 失效 |
| 磁盘空间 | 不占用额外空间 | 占用少量空间 |
| 性能 | 直接访问 | 需要解析路径 |

使用建议：
- 需要备份：硬链接
- 需要跨文件系统：软链接
- 需要链接目录：软链接
- 需要知道原文件位置：软链接

## 文件的存储方式

### 连续分配

方法：文件占用连续的磁盘块，目录项记录起始块号和长度。

优点：
- 顺序访问速度快
- 随机访问简单（起始块号 + 偏移量）

缺点：
- 外部碎片严重
- 文件增长困难
- 需要预先知道文件大小

应用：光盘文件系统（ISO 9660）

### 链式分配

方法：每个数据块包含指向下一个块的指针，目录项记录第一个块的位置。

优点：
- 没有外部碎片
- 文件可以动态增长

缺点：
- 随机访问慢（需要遍历链表）
- 指针占用空间
- 可靠性差（一个指针损坏，后续数据丢失）

改进：FAT（File Allocation Table）
- 将所有指针集中存储在 FAT 表中
- FAT 表可以缓存在内存中
- 提高访问速度

### 索引分配

方法：每个文件有一个索引块，索引块包含所有数据块的指针，目录项记录索引块的位置。

优点：
- 支持随机访问
- 没有外部碎片
- 文件可以动态增长

缺点：
- 索引块占用空间
- 小文件浪费空间

多级索引（Unix/Linux）：

```
inode 包含 15 个块指针：
- 12 个直接指针：直接指向数据块
- 1 个一级间接指针：指向索引块，索引块包含数据块指针
- 1 个二级间接指针：指向索引块，索引块包含一级索引块指针
- 1 个三级间接指针：指向索引块，索引块包含二级索引块指针
```

支持的文件大小（假设块大小 4KB，指针 4 字节）：
- 直接块：12 × 4KB = 48KB
- 一级间接：1024 × 4KB = 4MB
- 二级间接：1024 × 1024 × 4KB = 4GB
- 三级间接：1024 × 1024 × 1024 × 4KB = 4TB
- 总计：约 4TB

## 空闲空间管理

### 位图（Bitmap）

方法：用一个位表示一个块的状态，0 表示空闲，1 表示已占用。

优点：
- 简单高效
- 容易找到连续的空闲块

缺点：
- 需要额外的存储空间
- 大文件系统的位图可能很大

示例：
```
位图：0 1 1 0 0 1 0 0
块号：0 1 2 3 4 5 6 7

空闲块：0, 3, 4, 6, 7
已占用：1, 2, 5
```

### 空闲链表（Free List）

方法：将所有空闲块链接成链表，超级块记录链表头。

优点：不需要额外的位图空间。

缺点：
- 分配和回收需要遍历链表
- 难以找到连续的空闲块

### 分组链接（Grouping）

方法：将空闲块分组，每组的第一个块存储该组所有空闲块的地址。

优点：结合了位图和链表的优点，可以快速分配多个块。

## 常见问题

**inode 包含什么信息**

包含：
- 文件类型和权限
- 所有者和组
- 文件大小
- 时间戳（atime, mtime, ctime）
- 硬链接计数
- 数据块指针

不包含：
- 文件名（存储在目录中）
- 文件内容（存储在数据块中）

**硬链接 vs 软链接**

| 特性 | 硬链接 | 软链接 |
|------|--------|--------|
| inode | 共享 | 独立 |
| 跨文件系统 | 否 | 是 |
| 链接目录 | 否 | 是 |
| 原文件删除 | 不影响 | 失效 |

记忆技巧：硬链接硬绑定（共享 inode，限制多），软链接软引用（独立 inode，灵活）。

**删除文件的过程**

步骤：
1. 从目录中删除文件名和 inode 号的映射
2. inode 的硬链接计数减 1
3. 如果硬链接计数为 0：
   - 释放 inode
   - 释放数据块
   - 更新空闲空间管理结构

注意：
- 如果文件还有其他硬链接，文件不会被删除
- 如果文件被进程打开，即使硬链接计数为 0，文件也不会立即删除

**为什么 inode 用完了无法创建文件**

原因：
- 每个文件都需要一个 inode
- inode 数量在文件系统创建时确定
- inode 用完，无法创建新文件

解决方法：
- 删除不需要的文件
- 重新创建文件系统（会丢失数据）
- 使用支持动态 inode 的文件系统

**目录也是文件吗**

是的。

目录是特殊的文件：
- 有自己的 inode
- 内容是目录项的列表
- 每个目录项包含文件名和 inode 号

区别：
- 普通文件：内容是任意数据
- 目录文件：内容是目录项列表

**文件名存储在哪里**

存储在目录中，不是存储在 inode 中。

原因：
- 一个文件可以有多个文件名（硬链接）
- 文件名和 inode 是多对一的关系
- 目录负责维护文件名到 inode 的映射

**如何恢复删除的文件**

原理：删除文件只是删除了目录项，如果数据块没有被覆盖，数据仍然存在。

方法：
1. 立即停止写入操作
2. 使用数据恢复工具扫描磁盘
3. 根据文件特征恢复数据

注意：不保证 100% 恢复，越早恢复成功率越高。

**为什么移动文件很快，复制文件很慢**

移动文件（同一文件系统）：
- 只需要修改目录项
- 不需要移动数据
- 速度很快

复制文件：
- 需要读取原文件的所有数据
- 需要写入新文件的所有数据
- 速度取决于文件大小

跨文件系统移动：
- 等同于复制 + 删除
- 速度慢
