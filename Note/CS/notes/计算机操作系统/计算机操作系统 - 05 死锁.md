## 死锁

### 必要条件

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/c037c901-7eae-4e31-a1e4-9d41329e5c3e.png"/> </div><br>

- **互斥**：每个资源要么已经分配给了一个进程，要么就是可用的。
- **占有和等待**：已经得到了某个资源的进程可以再请求新的资源。
- **不可抢占**：已经分配给一个进程的资源不能强制性地被抢占，它只能被占有它的进程显式地释放。
- **环路等待**：有两个或者两个以上的进程组成一条环路，该环路中的每个进程都在等待下一个进程所占有的资源。

### 处理方法

#### 鸵鸟策略

把头埋在沙子里，假装根本没发生问题。因为解决死锁问题的代价很高，因此鸵鸟策略这种**不采取任何措施**的方案会获得**更高的性能**。

当发生死锁时不会对用户造成多大影响，或发生死锁的概率很低，可以采用鸵鸟策略。大多数操作系统，包括 Unix，Linux 和 Windows，处理死锁问题的办法仅仅是忽略它。

#### 死锁检测与死锁恢复

不试图阻止死锁，而是当检测到死锁发生时，采取措施进行恢复。

**一个资源的死锁检测**

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/b1fa0453-a4b0-4eae-a352-48acca8fff74.png"/> </div><br>

上图为资源分配图，其中方框表示资源，圆圈表示进程。资源指向进程表示该资源已经分配给该进程，进程指向资源表示进程请求获取该资源。图 a 可以抽取出环，如图 b，它满足了环路等待条件，因此会发生死锁。

每种类型一个资源的死锁检测算法是通过**检测有向图是否存在环**来实现，从一个节点出发进行**深度优先搜索**，对访问过的节点进行标记，如果访问了已经标记的节点，就表示有向图存在环，也就是检测到死锁的发生。

**多个资源的死锁检测**

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/e1eda3d5-5ec8-4708-8e25-1a04c5e11f48.png"/> </div><br>

上图中，有三个进程四个资源，每个数据代表的含义如下：

- E 向量：资源总量
- A 向量：资源剩余量
- C 矩阵：每个进程所拥有的资源数量，每一行都代表一个进程拥有资源的数量
- R 矩阵：每个进程请求的资源数量

进程 P<sub>1</sub> 和 P<sub>2</sub> 所请求的资源都得不到满足，只有进程 P<sub>3</sub> 可以，让 P<sub>3</sub> 执行，之后释放 P<sub>3</sub> 拥有的资源，此时 A = (2 2 2 0)。P<sub>2</sub> 可以执行，执行后释放 P<sub>2</sub> 拥有的资源，A = (4 2 2 1) 。P<sub>1</sub> 也可以执行。所有进程都可以顺利执行，没有死锁。

算法总结如下：

```
每个进程最开始时都不被标记，执行过程有可能被标记。当算法结束时，任何没有被标记的进程都是死锁进程。

1. 寻找一个没有标记的进程 Pi，它所请求的资源小于等于 A。
2. 如果找到了这样一个进程，那么将 C 矩阵的第 i 行向量加到 A 中，标记该进程，并转回 1。
3. 如果没有这样一个进程，算法终止。
```

#### 死锁恢复

- 利用抢占恢复
- 利用回滚恢复
- 通过杀死进程恢复

#### 死锁预防

**破坏互斥条件**: 例如假脱机打印机技术允许若干个进程同时输出，唯一真正请求物理打印机的进程是打印机守护进程。

**破坏占有和等待条件**:一种实现方式是规定所有进程在开始执行前请求所需要的全部资源。

**破坏不可抢占条件**

**破坏环路等待**:给资源统一编号，进程只能按编号顺序来请求资源。

#### 死锁避免

在程序运行时避免发生死锁。

##### 安全状态

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/ed523051-608f-4c3f-b343-383e2d194470.png"/> </div><br>

图 a 的第二列 Has 表示已拥有的资源数，第三列 Max 表示总共需要的资源数，Free 表示还有可以使用的资源数。从图 a 开始出发，先让 B 拥有所需的所有资源（图 b），运行结束后释放 B，此时 Free 变为 5（图 c）；接着以同样的方式运行 C 和 A，使得所有进程都能成功运行，因此可以称图 a 所示的状态时安全的。

定义：如果没有死锁发生，并且即使所有进程突然请求对资源的最大需求，也仍然存在某种调度次序能够使得每一个进程运行完毕，则称该状态是安全的。

安全状态的检测与死锁的检测类似，因为安全状态必须要求不能发生死锁。下面的银行家算法与死锁检测算法非常类似，可以结合着做参考对比。

##### 单个资源的银行家算法

一个小城镇的银行家，他向一群客户分别承诺了一定的贷款额度，算法要做的是判断对请求的满足是否会进入不安全状态，如果是，就拒绝请求；否则予以分配。

##### 多个资源的银行家算法

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/62e0dd4f-44c3-43ee-bb6e-fedb9e068519.png"/> </div><br>

上图中有五个进程，四个资源。左边的图表示已经分配的资源，右边的图表示还需要分配的资源。最右边的 E、P 以及 A 分别表示：总资源、已分配资源以及可用资源，注意这三个为向量，而不是具体数值，例如 A=(1020)，表示 4 个资源分别还剩下 1/0/2/0。

检查一个状态是否安全的算法如下：

- 查找右边的矩阵是否存在一行小于等于向量 A。如果不存在这样的行，那么系统将会发生死锁，状态是不安全的。
- 假若找到这样一行，将该进程标记为终止，并将其已分配资源加到 A 中。
- 重复以上两步，直到所有进程都标记为终止，则状态时安全的。

如果一个状态不是安全的，需要拒绝进入这个状态。

## 补充

### 死锁的四个必要条件

死锁的发生必须同时满足互斥、占有和等待、不可抢占、环路等待这四个条件。破坏其中任何一个条件都可以预防死锁，但实际中最常用的是破坏环路等待条件，即给资源编号并要求按序申请。

### 死锁预防 vs 死锁避免

死锁预防是在系统设计时就破坏死锁的必要条件，简单但资源利用率低。死锁避免是在运行时动态检查资源分配是否安全，如银行家算法，但需要预知进程的最大资源需求，实际应用较少。

### 银行家算法的局限

银行家算法虽然能有效避免死锁，但需要预知每个进程的最大资源需求，且进程数和资源数必须固定，这在实际系统中很难满足。因此大多数操作系统（如 Unix、Linux、Windows）采用鸵鸟策略，即忽略死锁问题。

### 死锁检测的时机

死锁检测可以在每次资源请求时进行，也可以定期进行。频繁检测能及时发现死锁，但开销大；定期检测开销小，但可能延迟发现死锁。实际系统通常在 CPU 利用率较低时进行检测。

### 哲学家进餐问题的本质

哲学家进餐问题的死锁本质是环路等待：每个哲学家都持有左边的筷子，等待右边的筷子。解决方法是破坏环路等待，最简单的方式是让奇数号哲学家先拿左边筷子，偶数号哲学家先拿右边筷子，这样就不会形成环路。

### Q1：死锁的四个必要条件是什么？

死锁的发生必须同时满足以下四个条件：

| 条件 | 说明 | 示例 |
|------|------|------|
| **互斥（Mutual Exclusion）** | 资源不能被多个进程同时使用 | 打印机同时只能被一个进程使用 |
| **占有和等待（Hold and Wait）** | 进程持有资源的同时等待其他资源 | 进程 A 持有资源 1，等待资源 2 |
| **不可抢占（No Preemption）** | 资源不能被强制剥夺 | 进程持有的资源只能主动释放 |
| **环路等待（Circular Wait）** | 存在进程资源的循环等待链 | P1 等 P2，P2 等 P3，P3 等 P1 |

**记忆口诀：**
互占不环（互斥、占有等待、不可抢占、环路等待）

**关键理解：**
- 四个条件缺一不可
- 破坏任意一个条件就能预防死锁
- 但破坏某些条件代价很大

### Q2：死锁的处理方法有哪些？

| 方法 | 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **鸵鸟策略** | 忽略死锁 | 无开销 | 可能死锁 | 死锁概率低的系统 |
| **死锁预防** | 破坏必要条件 | 不会死锁 | 资源利用率低 | 对可靠性要求高 |
| **死锁避免** | 动态检查安全性 | 资源利用率较高 | 需要预知资源需求 | 资源需求已知 |
| **死锁检测与恢复** | 检测后恢复 | 资源利用率高 | 检测开销大 | 死锁概率中等 |

**对比：**

```
资源利用率：检测恢复 > 避免 > 预防 > 鸵鸟
实现复杂度：鸵鸟 < 预防 < 检测恢复 < 避免
```

**实际应用：**
- Unix/Linux/Windows：鸵鸟策略
- 数据库：死锁检测与恢复
- 实时系统：死锁预防

### Q3：如何预防死锁？

**破坏四个必要条件：**

| 破坏条件 | 方法 | 优点 | 缺点 | 可行性 |
|---------|------|------|------|--------|
| **破坏互斥** | 资源共享化 | 彻底解决 | 很多资源无法共享 | 低 |
| **破坏占有等待** | 一次性申请所有资源 | 简单 | 资源利用率低、可能饥饿 | 中 |
| **破坏不可抢占** | 允许抢占资源 | 灵活 | 实现困难、可能丢失工作 | 低 |
| **破坏环路等待** | 资源有序分配 | 实用 | 限制灵活性 | 高 |

**详细说明：**

**1. 破坏互斥条件：**
- 方法：将独占资源改为共享资源
- 示例：假脱机打印（SPOOLing）
- 问题：很多资源本质上无法共享（如打印机）

**2. 破坏占有和等待：**
- 方法 1：一次性申请所有资源
- 方法 2：申请新资源前释放已有资源
- 问题：资源利用率低、可能饥饿

**3. 破坏不可抢占：**
- 方法：允许抢占已分配的资源
- 问题：实现困难、可能丢失工作进度

**4. 破坏环路等待：**
- 方法：给资源编号，按序申请
- 示例：资源 R1, R2, R3，只能按 1→2→3 申请
- 优点：实用、有效
- 缺点：限制灵活性

**最常用：破坏环路等待**

### Q4：什么是银行家算法？

**背景：**
银行家向客户承诺贷款额度，但不能让银行资金不足。

**核心思想：**
在分配资源前，检查分配后系统是否处于安全状态。

**安全状态：**
存在一个进程序列，使得每个进程都能获得所需资源并完成。

**算法步骤：**

1. **检查请求是否合法**
   - 请求 ≤ 需要
   - 请求 ≤ 可用

2. **试探性分配**
   - 可用资源 -= 请求
   - 已分配 += 请求
   - 还需要 -= 请求

3. **安全性检查**
   - 寻找能完成的进程（需要 ≤ 可用）
   - 假设该进程完成，释放资源
   - 重复直到所有进程完成

4. **决策**
   - 安全：真正分配
   - 不安全：拒绝请求，恢复原状态

**示例：**

```
进程  已分配  最大需求  还需要  可用资源
P0     0       7        7
P1     2       3        1       3
P2     3       9        6
P3     2       2        0
P4     0       4        4

安全序列：P1 → P3 → P0 → P2 → P4
```

**检查过程：**
1. P1 需要 1，可用 3，满足 → P1 完成，释放 2，可用 = 5
2. P3 需要 0，可用 5，满足 → P3 完成，释放 2，可用 = 7
3. P0 需要 7，可用 7，满足 → P0 完成，释放 0，可用 = 7
4. P2 需要 6，可用 7，满足 → P2 完成，释放 3，可用 = 10
5. P4 需要 4，可用 10，满足 → P4 完成

**优缺点：**

优点：
- 能有效避免死锁
- 资源利用率较高

缺点：
- 需要预知最大资源需求（不现实）
- 进程数和资源数固定（不灵活）
- 检查开销大

**实际应用：**
很少使用，主要用于教学。

### Q5：死锁检测算法是什么？

**目标：**
检测系统是否已经发生死锁。

**每种资源一个实例：**

方法：检测资源分配图是否有环

```
资源分配图：
- 方框：资源
- 圆圈：进程
- 资源 → 进程：已分配
- 进程 → 资源：请求

有环 → 死锁
无环 → 无死锁
```

算法：深度优先搜索（DFS）检测环

**每种资源多个实例：**

类似银行家算法，但不需要最大需求。

算法步骤：
1. 寻找未标记的进程，其请求 ≤ 可用资源
2. 标记该进程，释放其资源
3. 重复直到无法继续
4. 未标记的进程 = 死锁进程

**示例：**

```
进程  已分配  请求  可用资源
P1    2 0     0 2
P2    0 1     2 0     2 2 2 0
P3    0 0     0 1

检查过程：
1. P3 请求 (0,1) ≤ 可用 (2,2,2,0)，满足
   → 标记 P3，释放 (0,0,0,0)，可用 = (2,2,2,0)
2. P1 请求 (0,2) ≤ 可用 (2,2,2,0)，满足
   → 标记 P1，释放 (2,0,0,0)，可用 = (4,2,2,0)
3. P2 请求 (2,0) ≤ 可用 (4,2,2,0)，满足
   → 标记 P2，释放 (0,1,0,0)，可用 = (4,3,2,0)

所有进程都能完成 → 无死锁
```

### Q6：死锁恢复的方法有哪些？

检测到死锁后，如何恢复？

| 方法 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **进程终止** | 杀死进程释放资源 | 简单 | 丢失工作 |
| **资源抢占** | 抢占资源给其他进程 | 不需要杀进程 | 实现困难 |
| **回滚** | 回滚到之前的检查点 | 可以恢复工作 | 需要检查点机制 |

**进程终止策略：**

1. **终止所有死锁进程**
   - 优点：简单、彻底
   - 缺点：代价大

2. **逐个终止直到解除死锁**
   - 优点：代价小
   - 缺点：需要多次检测

**选择终止哪个进程？**

考虑因素：
- 进程优先级
- 已运行时间
- 还需要多久完成
- 使用的资源数量
- 是否交互式进程

**资源抢占策略：**

1. **选择受害者**：选择代价最小的进程
2. **回滚**：回滚到安全状态
3. **防止饥饿**：不要总是选择同一个进程

### Q7：死锁 vs 饥饿 vs 死循环？

| 概念 | 定义 | 原因 | 能否自行解决 |
|------|------|------|-------------|
| **死锁** | 进程互相等待资源 | 资源分配不当 | 否 |
| **饥饿** | 进程长期得不到资源 | 调度不公平 | 可能（优先级提升） |
| **死循环** | 程序逻辑错误 | 代码 bug | 否 |

**详细对比：**

**死锁：**
- 多个进程互相等待
- 涉及资源分配
- 需要外部干预

**饥饿：**
- 单个进程长期等待
- 其他进程在运行
- 可能通过优先级提升解决

**死循环：**
- 单个进程内部问题
- 不涉及其他进程
- 需要终止进程

**示例：**

```c
// 死锁
P1: lock(A); lock(B);
P2: lock(B); lock(A);

// 饥饿
低优先级进程一直被高优先级进程抢占

// 死循环
while(1) {
    // 无退出条件
}
```

### Q8：如何避免哲学家进餐问题的死锁？

**问题回顾：**
5 个哲学家，5 根筷子，每人需要 2 根筷子才能吃饭。

**死锁场景：**
所有哲学家同时拿起左边筷子，等待右边筷子。

**解决方案：**

| 方案 | 说明 | 破坏的条件 |
|------|------|-----------|
| **限制人数** | 最多 4 人同时拿筷子 | 占有和等待 |
| **奇偶策略** | 奇数号先拿左边，偶数号先拿右边 | 环路等待 |
| **原子操作** | 同时拿起两根筷子 | 占有和等待 |
| **服务生** | 请求许可后才能拿筷子 | 占有和等待 |

**最优方案：奇偶策略**

```c
void philosopher(int i) {
    while(TRUE) {
        think();
        if (i % 2 == 0) {
            take(i);           // 偶数先拿左边
            take((i+1)%5);
        } else {
            take((i+1)%5);     // 奇数先拿右边
            take(i);
        }
        eat();
        put(i);
        put((i+1)%5);
    }
}
```

**为什么有效？**
破坏了环路等待：不会形成循环等待链。

## 快速回忆

### 死锁四个必要条件

**记忆口诀：**
互占不环

**详细：**
1. 互斥：资源独占
2. 占有和等待：持有资源等待其他资源
3. 不可抢占：资源不能被强制剥夺
4. 环路等待：循环等待链

**关键：**
四个条件同时满足才会死锁。

### 死锁处理方法

这一节看起来往往容易变成一串结论，其实先抓住“死锁处理方法”在计算机操作系统里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**四种方法：**
1. 鸵鸟策略：忽略（最常用）
2. 死锁预防：破坏必要条件
3. 死锁避免：银行家算法
4. 死锁检测与恢复：检测后处理

**选择建议：**
- 死锁概率低：鸵鸟策略
- 可靠性要求高：死锁预防
- 资源需求已知：死锁避免
- 死锁概率中等：死锁检测与恢复

### 死锁预防

**最实用：破坏环路等待**

方法：资源有序分配
- 给资源编号
- 按序申请资源
- 不允许逆序申请

### 银行家算法

**核心：**
分配前检查是否安全。

**安全状态：**
存在一个进程序列，每个进程都能完成。

**步骤：**
1. 检查请求合法性
2. 试探性分配
3. 安全性检查
4. 决策（分配或拒绝）

### 死锁检测

**单实例资源：**
检测资源分配图是否有环（DFS）

**多实例资源：**
类似银行家算法，找能完成的进程

### 死锁恢复

**三种方法：**
1. 进程终止（简单但丢失工作）
2. 资源抢占（复杂但不丢失进程）
3. 回滚（需要检查点）

## 易错点

### 1. 死锁 vs 饥饿

**常见混淆：**
认为饥饿就是死锁。

**正确理解：**
- 死锁：多个进程互相等待，都无法继续
- 饥饿：一个进程长期得不到资源，但其他进程在运行

**关键区别：**
- 死锁：所有相关进程都停止
- 饥饿：只有一个进程停止

### 2. 安全状态 vs 不安全状态

**常见误解：**
不安全状态 = 死锁状态

**正确理解：**
- 安全状态：一定不会死锁
- 不安全状态：可能死锁，但不一定死锁
- 死锁状态：一定是不安全状态

**关系：**
```
安全状态 ⊂ 无死锁状态
死锁状态 ⊂ 不安全状态
```

### 3. 银行家算法的前提

**常见错误：**
认为银行家算法可以解决所有死锁问题。

**正确理解：**
银行家算法需要：
- 预知最大资源需求（不现实）
- 进程数和资源数固定（不灵活）
- 进程最终会释放资源（不一定）

实际系统很少使用。

### 4. 破坏互斥条件

**常见误解：**
破坏互斥条件可以预防死锁。

**正确理解：**
很多资源本质上无法共享：
- 打印机：不能同时打印多个文档
- 临界区：不能同时访问

破坏互斥条件通常不可行。

### 5. 资源有序分配

**常见错误：**
认为资源有序分配会降低并发度。

**正确理解：**
- 只是限制申请顺序，不限制使用顺序
- 不同进程可以同时持有不同资源
- 并发度影响不大

### 6. 死锁检测的时机

**问题：**
什么时候进行死锁检测？

**选择：**
1. 每次资源请求时检测（开销大，及时）
2. 定期检测（开销小，可能延迟）
3. CPU 利用率低时检测（折中）

**实际应用：**
根据死锁概率和检测开销权衡。

### 7. 哲学家进餐问题

**常见错误方案：**
```c
// 错误：会死锁
take(left);
take(right);
```

**正确方案：**
- 奇偶策略（最优）
- 限制人数
- 原子操作

**关键：**
破坏环路等待条件。
