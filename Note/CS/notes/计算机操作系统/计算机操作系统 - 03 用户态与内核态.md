
# 计算机操作系统 - 用户态与内核态

## 用户态与内核态

用户态（User Mode）和内核态（Kernel Mode）是 CPU 的两种运行模式，也称为用户模式和内核模式。用户态是应用程序运行的模式，权限受限，只能访问受限的资源，不能直接访问硬件，不能执行特权指令，只能访问用户空间的内存。内核态是操作系统内核运行的模式，拥有最高权限，可以访问所有硬件资源，可以执行所有 CPU 指令，可以访问所有内存空间。

### CPU 区分

现代 CPU 通过特权级别（Privilege Level）来区分：

x86 架构使用 Ring 0-3 四个特权级：
- Ring 0：内核态（最高权限）
- Ring 3：用户态（最低权限）
- Ring 1-2：很少使用

ARM 架构使用 EL0-EL3 四个异常级别：
- EL0：用户态
- EL1：内核态
- EL2：虚拟化
- EL3：安全监控

### 区分用户态和内核态的作用

#### 系统安全

保护操作系统：防止用户程序破坏操作系统、防止恶意程序直接操作硬件、隔离不同应用程序

如果没有用户态/内核态区分，任何程序都可以直接读写磁盘，恶意程序可以删除系统文件，程序可以访问其他程序的内存。

#### 资源管理

统一管理硬件资源：所有硬件访问必须通过内核、内核协调多个程序对资源的访问、防止资源冲突

比如打印机资源管理。多个程序同时打印，内核统一调度打印任务，避免打印内容混乱。

#### 稳定性

隔离故障：用户程序崩溃不会影响内核、一个应用崩溃不会影响其他应用、系统整体保持稳定
#### 权限控制

分级管理：普通用户只能执行受限操作、管理员可以执行更多操作、内核拥有最高权限
### 用户态和内核态的区别

| 对比项 | 用户态 | 内核态 |
|--------|--------|--------|
| 权限 | 受限 | 最高权限 |
| 可执行指令 | 非特权指令 | 所有指令 |
| 内存访问 | 只能访问用户空间 | 可以访问所有内存 |
| 硬件访问 | 不能直接访问 | 可以直接访问 |
| 运行代码 | 应用程序 | 操作系统内核 |
| 崩溃影响 | 只影响当前进程 | 可能导致系统崩溃 |
| 典型操作 | 计算、逻辑处理 | I/O、内存管理、进程调度 |

### 特权指令示例

只能在内核态执行的指令：I/O 指令（读写硬盘、网卡等）、修改页表寄存器、修改中断寄存器、关闭中断、停机指令

用户态和内核态都能执行：算术运算指令、逻辑运算指令、数据传送指令、分支跳转指令
### 模式切换

#### 切换时机

用户态 → 内核态（陷入内核）：

* **系统调用**：用户程序主动请求内核服务，如 read()、write()、fork() 等。这是最常见的切换方式。

* **异常（Exception）**：程序执行出错，如除零错误、缺页异常、非法指令。被动触发。

* **中断（Interrupt）**：外部硬件设备触发，如时钟中断、I/O 中断、网络中断。异步发生。

内核态 → 用户态：
- **系统调用返回**
- **异常处理完成**
- **中断处理完成**
- **进程调度切换到用户进程**

#### 切换过程

用户态 → 内核态：

```
1. 触发切换（系统调用/异常/中断）
   ↓
2. 硬件自动完成：
   - 保存当前 PC（程序计数器）
   - 保存当前 PSW（程序状态字）
   - 切换到内核栈
   - 修改 CPU 特权级为内核态
   ↓
3. 跳转到内核代码：
   - 系统调用：跳转到系统调用处理程序
   - 异常：跳转到异常处理程序
   - 中断：跳转到中断处理程序
   ↓
4. 内核处理请求
   ↓
5. 返回用户态
```

内核态 → 用户态：

```
1. 内核处理完成
   ↓
2. 恢复用户态上下文：
   - 恢复用户栈
   - 恢复寄存器
   - 修改 CPU 特权级为用户态
   ↓
3. 返回用户程序继续执行
```

#### 切换开销

直接开销：保存和恢复寄存器、切换栈（用户栈 ↔ 内核栈）、修改 CPU 特权级、通常需要几十到几百个 CPU 周期

间接开销：CPU 流水线可能被清空、缓存可能失效、TLB 可能失效，这是最大的开销

- 简单系统调用：0.1-0.5 微秒
- 复杂系统调用：可能达到几微秒

## 系统调用

系统调用（System Call）是用户程序请求操作系统内核服务的接口。

- 用户程序不能直接访问硬件
- 需要通过系统调用请求内核帮忙
- 内核验证权限后执行操作

系统调用 vs 普通函数调用：

| 对比项 | 普通函数调用 | 系统调用 |
|--------|-------------|----------|
| 调用对象 | 用户空间函数 | 内核函数 |
| 特权级切换 | 不切换 | 用户态→内核态 |
| 开销 | 小 | 大 |
| 执行位置 | 用户空间 | 内核空间 |

### 系统调用的过程

以 Linux 的 read() 系统调用为例：

```c
// 用户程序
int fd = open("/tmp/test.txt", O_RDONLY);
char buf[100];
ssize_t n = read(fd, buf, 100);  // 系统调用
```

执行过程：

```
1. 用户程序调用 read()
   ↓
2. C 库函数（glibc）准备参数
   - 将系统调用号放入寄存器（read 的调用号）
   - 将参数放入寄存器（fd, buf, 100）
   ↓
3. 执行 trap 指令（如 x86 的 int 0x80 或 syscall）
   - 触发软中断
   - CPU 切换到内核态
   ↓
4. 内核接管：
   - 根据系统调用号找到对应的内核函数
   - 验证参数合法性
   - 执行实际的读取操作
   ↓
5. 内核返回：
   - 将结果放入寄存器
   - 切换回用户态
   ↓
6. C 库函数返回结果给用户程序
```

### 常见系统调用

进程管理：
```c
fork()      // 创建进程
exec()      // 执行程序
exit()      // 退出进程
wait()      // 等待子进程
getpid()    // 获取进程 ID
```

文件操作：
```c
open()      // 打开文件
read()      // 读文件
write()     // 写文件
close()     // 关闭文件
lseek()     // 移动文件指针
```

内存管理：
```c
brk()       // 修改数据段大小
mmap()      // 内存映射
munmap()    // 解除映射
```

进程通信：
```c
pipe()      // 创建管道
socket()    // 创建套接字
shmget()    // 获取共享内存
```

信号处理：
```c
signal()    // 设置信号处理函数
kill()      // 发送信号
```

## 用户空间与内核空间

### 内存布局

在 32 位 Linux 系统中，典型的内存布局：

```
4GB  ┌─────────────────┐
     │   内核空间       │  1GB
     │  (Kernel Space) │  内核态可访问
1GB  ├─────────────────┤
     │                 │
     │   用户空间       │  3GB
     │  (User Space)   │  用户态可访问
     │                 │
0GB  └─────────────────┘
```

用户空间（User Space）：每个进程独立的虚拟地址空间。包含：代码段、数据段、堆、栈等。用户态只能访问这部分。

内核空间（Kernel Space）：所有进程共享，包含内核代码、内核数据结构、设备驱动等。只有内核态才能访问。
### 划分原则

* **安全隔离**：用户程序不能直接访问内核空间，防止破坏内核数据。

* **共享内核**：所有进程共享同一个内核，节省内存空间。

* **快速切换**：系统调用时不需要切换页表，只需要切换特权级。

## 常见问题

**用户态 vs 内核态**

| 对比维度 | 用户态 | 内核态 |
|---------|--------|--------|
| 权限 | 受限 | 最高 |
| 可执行指令 | 非特权指令 | 所有指令 |
| 内存访问 | 用户空间 | 所有空间 |
| 运行代码 | 应用程序 | 操作系统内核 |
| 崩溃影响 | 当前进程 | 整个系统 |

**什么操作会触发用户态到内核态的切换**

会触发：
- 系统调用（如 read、write、fork）
- 异常（如缺页异常、除零错误）
- 外部中断（如时钟中断、I/O 中断）

不会触发：
- 普通函数调用
- 普通的算术运算
- 访问自己的内存空间

**系统调用一定会触发上下文切换吗**

不一定。

系统调用 ≠ 上下文切换：
- 系统调用：用户态 → 内核态（模式切换）
- 上下文切换：进程 A → 进程 B

示例：
```c
// 只有模式切换，没有上下文切换
int x = getpid();  // 系统调用，但不切换进程

// 既有模式切换，又有上下文切换
read(fd, buf, size);  // 如果需要等待 I/O，可能切换到其他进程
```

**用户态可以直接访问硬件吗**

不能。

原因：
- 硬件访问需要执行特权指令
- 用户态不能执行特权指令
- 必须通过系统调用请求内核

例外：某些系统允许用户态直接访问特定硬件（如显卡的帧缓冲），需要特殊的权限设置，仍然受到严格限制。

**内核态可以访问用户空间的内存吗**

可以。

- 内核态拥有最高权限
- 可以访问所有内存空间
- 系统调用时经常需要访问用户空间（如 read 写入用户缓冲区）

但需要注意：
- 内核需要验证用户空间地址的合法性
- 防止用户程序传入非法地址
- 使用 copy_from_user() 和 copy_to_user() 等安全函数

**模式切换 vs 上下文切换**

| 对比项 | 模式切换 | 上下文切换 |
|--------|----------|-----------|
| 定义 | 用户态↔内核态 | 进程 A → 进程 B |
| 是否切换进程 | 否 | 是 |
| 是否切换页表 | 否 | 是（进程切换） |
| 开销 | 较小 | 较大 |
| 触发方式 | 系统调用、中断、异常 | 时间片用完、阻塞等 |

关系：上下文切换一定包含模式切换，模式切换不一定包含上下文切换。

**为什么系统调用比普通函数调用慢**

原因：

1. 特权级切换：需要切换 CPU 特权级，保存和恢复寄存器
2. 栈切换：从用户栈切换到内核栈，需要额外的内存访问
3. 参数验证：内核需要验证用户传入的参数，防止恶意攻击
4. 缓存失效：可能导致 CPU 缓存失效，需要重新加载数据

典型数据：
- 普通函数调用：几个 CPU 周期
- 系统调用：几百到几千个 CPU 周期

**如何减少用户态和内核态切换的开销**

方法：

1. 批处理：一次系统调用处理多个操作，如 writev() 一次写入多个缓冲区
2. 缓存：在用户空间缓存数据，减少系统调用次数
3. 内存映射：使用 mmap() 映射文件到内存，避免频繁的 read/write 系统调用
4. 异步 I/O：使用异步 I/O 接口，减少阻塞等待
5. 用户态驱动：某些场景下使用用户态驱动，如 DPDK（数据平面开发套件）
