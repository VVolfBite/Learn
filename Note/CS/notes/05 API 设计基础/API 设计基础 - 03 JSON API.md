# 03 JSON API

JSON 是最常见的 API 数据载体，但真正重要的从来不是“能不能序列化成 JSON”，而是**这个 JSON 在长期演进中是否仍然清晰、稳定、可兼容、可排障**。
 很多接口刚开始看起来都能用：字段也返回了，前端也能解析，联调也能跑通。但只要进入多人协作、版本迭代、客户端灰度、第三方接入这些真实场景，JSON 结构是否稳定、字段语义是否明确、错误格式是否统一，就会迅速决定接口维护成本。

因此，JSON API 的核心问题不是“格式是什么”，而是“**同样的格式里承载了怎样的契约**”。一个结构混乱、字段漂移、错误不可判定的 JSON 接口，即使语法合法，也仍然是难用的接口；一个结构稳定、字段清晰、兼容策略明确的 JSON 接口，才真正适合作为长期协作边界。

---

## 字段与命名

### 命名风格

命名风格首先要解决的是**一致性**。`snake_case` 和 `camelCase` 本身没有绝对优劣，真正重要的是全站统一。
 一旦同一套 API 里同时出现 `user_name`、`userName`、`UserName` 这类混合风格，调用方不仅要不断适配不同接口，还很容易在代码生成、序列化映射、文档理解上出错。

命名一旦选定，就应长期坚持，而不是“某个新模块顺手换一种”。
 JSON API 最怕的不是某种风格“看起来不够现代”，而是同一系统里风格不断漂移。

### 布尔语义

布尔字段最好使用肯定语义，例如 `is_active`、`enabled`、`is_deleted` 这类形式。
 这样做的原因，是为了减少阅读歧义。字段名如果设计成否定式或双重否定，例如 `not_disabled`、`is_not_banned`，调用方在判断逻辑时很容易读错，条件分支也会变得更绕。

布尔字段的理想状态是：
 字段名本身就能让人一眼看出“true 代表什么，false 代表什么”。

### 时间表达

时间字段最好统一为 **ISO 8601 且显式包含时区信息**，例如 `2026-03-11T10:30:00Z`。
 时间格式一旦不统一，就会出现一系列很隐蔽的问题：

- 有的接口返回时间戳，有的返回字符串
- 有的按 UTC，有的按本地时间
- 有的精确到秒，有的精确到毫秒
   这些问题在单机调试时往往不明显，但一旦进入跨时区、多端、多语言环境，就非常容易引发排序错误、展示错误和比较错误。

因此，时间字段最重要的不是“哪种写法更短”，而是**同一系统里始终采用一致的时区与格式策略**。

## 关键结论

### 向后兼容

JSON API 最重要的原则之一是**向后兼容优先**。
 只要客户端已经上线，就不能假设它们会和服务端同步升级。因此，新增字段通常是安全的，因为旧客户端可以忽略它；但删除字段、修改字段语义、修改字段类型，往往会直接破坏已有客户端。

也就是说，JSON 结构设计时必须提前考虑“未来会不会加字段、会不会改状态、会不会扩展错误信息”，而不是只满足当前版本。

### 枚举扩展

枚举字段不能被设计成“永远只会有这几个值”。
 今天可能只有 `PENDING / PAID / FAILED`，明天就可能新增 `PROCESSING`、`CANCELLED`、`REFUNDED`。如果客户端把枚举写死，没有为未知值保留默认处理路径，那么一旦服务端扩展状态，旧客户端就可能直接崩溃或进入未定义行为。

因此，枚举设计必须遵守一个基本原则：
 **客户端对未知枚举值要能容忍，服务端对枚举扩展要保持谨慎。**

### 错误结构

成功返回和错误返回都应当有稳定的结构。
 成功时调用方应该知道去哪拿业务数据；失败时调用方应该知道去哪拿错误码、错误信息、请求 ID。
 如果同一套 API 里有的接口返回 `data`，有的接口直接返回对象；有的错误返回 `message`，有的返回 `msg`，有的根本没有结构，那调用方就很难形成统一处理逻辑。

因此，JSON API 不只是字段值要稳定，**返回外层 envelope 也要稳定**。

### 数值精度

JSON 本身对数值精度的处理并不总是安全，尤其是在金额、汇率、大整数 ID 这类场景里。
 如果直接用浮点表示金额，例如 `12.34`，在跨语言、跨平台、前后端多次计算过程中，很容易出现精度误差。
 因此，金额字段通常更稳妥的方式有两种：

- 用最小货币单位整数，例如 `amount_cent = 1234`
- 用字符串表示金额，再配合 `currency`

核心目标不是“好看”，而是**避免把精度问题埋进接口契约**。

## 常见结构（推荐模板）

### 成功响应

成功响应最重要的是：**业务数据位置固定，辅助信息位置固定**。
 常见而稳定的做法是使用统一 envelope，把真正的业务结果放在 `data` 里，把请求追踪信息放在 `request_id` 里。

对象型响应通常可以写成：

```
{ "data": { ... }, "request_id": "..." }
```

列表型响应通常可以写成：

```
{ "data": [ ... ], "page": { ... }, "request_id": "..." }
```

这种结构的好处是：
 无论返回的是单对象还是列表，调用方都知道 `data` 是主数据区，`request_id` 是排障辅助区，`page` 是列表补充信息区。
 结构一旦固定，客户端封装和文档说明都会简单很多。

### 错误响应

错误响应同样需要固定结构，通常会把错误内容统一放在 `error` 对象中，例如：

```
{
  "error": {
    "code": "...",
    "message": "...",
    "details": { ... }
  },
  "request_id": "..."
}
```

这里最关键的不是字段名本身，而是语义要长期稳定：

- `code` 用于程序判断
- `message` 用于人阅读
- `details` 用于补充上下文
- `request_id` 用于排障定位

一旦错误结构漂移，客户端的异常处理、监控告警、问题定位都会变得碎片化。

## 常见坑点

### 字段漂移

最难排查的问题之一，就是**同名字段在不同接口里语义不同**。
 例如 `status` 在某些接口里表示业务状态，在另一些接口里表示处理结果；`type` 在一个接口里是资源类型，在另一个接口里是操作类型。
 这种问题表面上看不严重，因为字段名都“合法”，但实际会极大增加理解成本，并且很容易在多端接入时出现误用。

### null 语义

`null` 不是一个简单的“空值”，它必须有明确语义。
 例如：

- `null` 表示“当前未知”
- `null` 表示“确实没有这个值”
- `null` 表示“允许调用方清空该字段”

如果文档里不写清楚，调用方往往无法判断 `null` 到底意味着什么，前后端的业务逻辑也会逐渐分裂。
 因此，`null` 最怕的不是出现，而是**出现了却没有定义**。

### 分页结构漂移

列表接口最容易失控的地方之一，就是分页字段各写各的。
 有的接口返回 `page/pageSize`，有的返回 `offset/limit`，有的返回 `cursor/next_token`，有的干脆把分页信息散在顶层字段里。
 短期看每个接口都“能用”，长期看却会让客户端适配逻辑不断膨胀。
 因此，分页结构应尽量抽成统一块，而不是让每个列表接口自行发挥。

## 快问快答

- **Q：返回里要不要带 request_id？**  
  强烈建议带。它把日志/链路/错误关联起来，排障效率指数级提升。

- **Q：金额字段怎么设计更稳？**  
  用 `amount_cent`（整数）或 `amount`（字符串）+ `currency`，避免浮点。
