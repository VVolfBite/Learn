# 09 OpenAPI 规格

OpenAPI 的意义，不只是“给接口写一份说明书”，而是把原本分散在聊天记录、口头约定、代码注释和经验记忆里的接口契约，收敛成一份**机器可读、团队可共享、流程可自动化**的正式描述。
 一旦接口规模变大、参与方变多，单靠口头同步或临时文档几乎不可能长期稳定。参数叫什么、哪些字段必填、返回结构长什么样、错误码有哪些、鉴权怎么传，只要其中任意一项表达不清，联调成本就会迅速上升。

因此，OpenAPI 真正的价值不在“能不能生成文档页面”，而在于它把 API 契约从“人类理解的说明”推进成“**既能给人看，也能给工具消费**”的结构化定义。
 只要契约被结构化，就可以进一步做自动校验、自动生成、自动测试、自动评审，从而把很多原本只能靠人工对齐的工作，前置到流程中解决。

---

## 能解决什么

### 契约对齐

OpenAPI 首先解决的是“接口到底长什么样”这件事。
 路径、方法、参数位置、字段类型、响应结构、错误码、鉴权方式，这些原本容易在不同文档、不同人理解里出现偏差的内容，都可以统一落在一份结构化规格中。
 这样做的好处是：前端、后端、测试、SDK 维护者看到的是同一份定义，而不是各自脑子里的一套理解。

### 自动化基础

一旦契约是结构化的，它就不再只是“文档”，而是工具链可以直接使用的输入。
 可以基于它自动生成接口文档、生成客户端 SDK、生成服务端代码骨架、生成 mock server、做 schema 校验、做契约测试。
 也就是说，OpenAPI 的真正价值在于把接口设计从“靠记忆和沟通”推进到“靠规范和自动化”。

### 联调降本

很多联调问题并不是代码复杂，而是“双方理解的不一样”。
 例如前端以为字段可空，后端以为必填；后端以为错误用 409，前端却按 400 处理。
 OpenAPI 的存在，就是把这些歧义提前暴露出来，让接口定义尽量在开发前、评审时、CI 阶段被发现，而不是等到联调或上线后才暴露。

## 必写内容清单

### 路径与方法

每个接口至少要清楚说明：

- 路径是什么
- 使用什么 HTTP 方法
- 资源对象是谁
- 这个操作的语义是什么

没有路径和方法的统一定义，后面的参数与响应再完整，也只是零散片段。

### 参数位置与类型

参数不仅要写“有哪些”，还要写清楚**它们出现在哪里**：

- query
- path
- header
- body

同时要明确类型、必填性、枚举值、格式限制、默认值等。否则客户端虽然知道有这个参数，但并不知道怎么正确传。

### 响应结构

成功返回和错误返回都需要定义。
 不仅要写 2xx 成功长什么样，还要写 4xx、5xx 的错误结构，说明字段含义和可能出现的业务错误码。
 很多接口文档只写成功返回，这会导致调用方在失败路径上完全失去预期。

### 错误码枚举

错误码如果只是“示意几个”，实际价值会很有限。
 更完整的做法是列出关键业务错误码、语义说明和常见场景示例，让调用方知道哪些是参数错误，哪些是资源冲突，哪些可以重试，哪些不应重试。

### 鉴权方式

鉴权必须写清楚。
 Bearer Token、API Key、OAuth、签名校验、权限范围，这些都属于接口契约的一部分。
 如果鉴权只写一句“需要登录”或“需要 token”，调用方实际上仍然不知道该如何接入。

### 分页约定

分页参数、排序参数、过滤参数如果不在 OpenAPI 中显式说明，列表接口通常最容易在联调中出问题。
 尤其是 cursor/limit 这种模式，必须说明 cursor 格式、limit 范围、next_cursor 的含义以及返回结构约定。

## 常见坑点

### 文档漂移

OpenAPI 最大的风险不是“没人写”，而是“写了但不再跟实现保持一致”。
 如果代码已经改了，文档还停留在旧状态，那么 OpenAPI 反而会从帮助变成误导。调用方以为看到的是官方契约，实际上拿到的是过期描述。

因此，真正危险的不是没有文档，而是**有一份看起来权威、实际上不可信的文档**。

### 示例缺失

字段定义再完整，如果没有示例，请求方仍然可能不知道具体该怎么组织数据。
 例如一个复杂 body 结构，光看 schema 类型并不能让人立刻理解真实请求长什么样。
 示例的价值在于把抽象结构变成可直接模仿的具体样本，尤其对联调和第三方接入非常重要。

### 错误结构不统一

很多团队在文档里会认真写成功返回，却对错误返回一笔带过，或者每个接口各写各的。
 结果就是：文档虽然存在，但调用方最关心的“失败时会收到什么”反而没有统一答案。
 一旦错误结构不统一，自动生成的 SDK 也很难形成统一异常处理模型。

## 推荐实践

### 单一事实来源

OpenAPI 最好的使用方式，是把它当作接口契约的**单一事实来源（SSOT）**。
 也就是说，文档、SDK、mock、校验、评审都尽量围绕这份定义展开，而不是再维护多套互相不一致的说明材料。

### CI 校验

如果 OpenAPI 只停留在仓库里静态存在，它迟早会漂移。
 真正有效的做法，是把它接入 CI：一旦有人提交了破坏性 schema 变更、删了必填字段、改了错误结构，CI 应该能自动发现并阻止合并。
 这样 OpenAPI 才不只是“描述现状”，而是“约束未来”。

### 契约测试

关键接口可以基于 OpenAPI 做契约测试，例如用 schema validation 校验响应结构，或基于 mock server 做联调测试。
 这样做的目的，是把“实现和契约是否一致”这个问题从人工检查变成自动检查。

### 流程嵌入

OpenAPI 只有进入团队流程，才会真正被用起来。
 如果设计评审看它、联调用它、CI 校验它、SDK 从它生成，那么它就会成为真实工作流的一部分；
 如果只是写完放在那里，那么再完整的规格最终也会沦为“摆设文档”。

## 快问快答

- **Q：OpenAPI 的最大收益是什么？**  
  契约对齐与自动化：减少口头约定，降低联调成本。

- **Q：文档写了但没人看怎么办？**  
  让它进入流程：评审/CI 校验/生成 SDK，才会“被用起来”。
