# 10 推送与实时

“服务端主动通知客户端”这件事，看起来只是把消息从服务端发出去，实际上涉及三个核心问题：

- **通知对象是谁**
- **连接是单次触发还是持续保持**
- **消息丢了、重复了、断线了以后怎么办**

因此，推送与实时通信并不是简单地在 Webhook、SSE、WebSocket 之间三选一，而是要先判断业务的交互形态：是服务端通知另一个服务，还是通知浏览器；是只需要单向推送，还是需要双向实时交互；是更关心低延迟，还是更关心投递可靠性。
 不同协议的选择，本质上是在**交互方式、可靠性、复杂度、资源占用**之间做平衡。

## 推送语义

### 主动通知

推送的本质，是服务端不再等客户端来轮询查询，而是在事件发生时主动把变化送出去。
 这种方式的直接价值是降低轮询延迟、减少无意义请求、让状态变化更快到达调用方。
 但一旦改成主动通知，系统就必须正面处理“消息可能重复、可能丢失、可能乱序、可能断线”的问题。也就是说，推送把“及时性”带来的收益，换成了“投递可靠性设计”的责任。

### 协议选择

不同推送方式并不是互相替代关系，而是分别适合不同通信对象和场景：

- Webhook 更像“服务通知另一个服务”
- SSE 更像“服务持续推送给浏览器”
- WebSocket 更像“客户端和服务之间持续双向交互”

因此，协议选择首先取决于“谁在接收消息”，其次才是技术偏好。

## Webhook

### 基本含义

Webhook 表示：当某个事件发生时，服务端主动向订阅方提供的 URL 发起一次 HTTP 请求。
 本质上，它是“**服务端到服务端的异步通知机制**”。
 与浏览器轮询不同，Webhook 的接收方通常是另一个后端系统，因此它更像系统间的事件推送，而不是前端实时刷新机制。

### 适用场景

Webhook 最典型的场景包括支付回调、订单状态变更通知、第三方事件推送、外部平台回调等。
 这些场景的共同特点是：

- 接收方通常是另一个服务，而不是浏览器页面
- 事件发生频率不一定特别高，但要求“尽快通知”
- 接收方通常需要基于事件继续执行业务逻辑

### 必备设计点

Webhook 真正难的不是“发一个 HTTP 请求”，而是保证这个通知机制在真实网络环境里仍然可靠。

**签名校验**是第一层防线。
 如果回调请求没有签名，任何人都可能伪造事件请求，接收方很难确认这是不是来自可信发送方。因此，支付通知、账务事件这类高敏感场景几乎都必须带签名或摘要校验。

**重试机制**是第二层保障。
 接收方可能临时不可用、网络可能抖动、响应可能超时，因此发送方通常不能发一次就算了，而需要配合退避重试。
 但只要存在重试，就必然会出现重复回调。

**幂等处理**因此成为第三层必要能力。
 接收方必须把同一事件的重复到达视为常态，而不是异常。否则一次支付成功通知重试，就可能变成重复记账、重复发货、重复发券。

**事件 ID**则是整套机制里的关键标识。
 它既用于去重，也用于日志追踪和问题定位。没有事件 ID，系统很难区分“这是同一事件重试了三次”还是“三个不同事件”。

**回调超时控制**也不能忽略。
 如果发送方每发一次 Webhook 都长期阻塞等待接收方，会很容易把自己的线程池、连接池拖满。
 因此，Webhook 的发送通常需要有严格超时、异步投递和重试队列，而不应把回调请求变成同步长等待链路。

## SSE

### 基本含义

SSE（Server-Sent Events）是基于 HTTP 长连接的单向推送机制，服务端可以在同一条连接上持续向浏览器发送事件流。
 它适合“服务端不断产出更新，客户端只需要持续接收”的场景。
 从交互方向上看，SSE 强调的是**服务端单向推送**，而不是客户端和服务端双向对话。

### 适用场景

SSE 非常适合浏览器端的实时通知、进度更新、日志流展示、简单实时看板等场景。
 这些场景的共同特点是：

- 客户端主要是浏览器
- 推送方向基本是单向
- 不需要像聊天、协作编辑那样频繁双向交互
- 希望实现比 WebSocket 更轻一些

### 关键点

SSE 最重要的问题是连接管理。
 连接一旦中断，客户端需要能够自动重连，并尽量知道“上次收到哪里了”，这就是 `Last-Event-ID` 之类机制存在的意义。
 否则断线重连后，客户端可能漏掉中间一段更新，或者只能从头再来。

另外，SSE 虽然实现相对轻，但它依然是长连接，服务端必须考虑连接数、心跳、空闲连接清理、代理层支持等问题。
 也就是说，SSE 并不是“没有状态成本”，只是相对于 WebSocket 更适合单向事件流。

## WebSocket

### 基本含义

WebSocket 是一种建立在长连接之上的双向实时通信机制。
 一旦连接建立，客户端和服务端都可以主动向对方发送消息，而不需要每次都重新发起 HTTP 请求。
 它适合“双方都可能随时发消息”的强交互场景。

### 适用场景

WebSocket 常见于 IM、协作编辑、多人在线协作、实时游戏、直播互动等场景。
 这些场景的特点是：

- 双方都需要低延迟发送消息
- 消息频繁往返
- 单向推送已经不够，需要真正的双向会话

### 关键点

WebSocket 的难点主要在于连接生命周期管理。

**心跳与断线重连**是第一层基础能力。
 连接建立后，并不能假设它会一直稳定存在。网络中断、代理超时、移动端切后台、服务重启，都可能导致连接失效。因此必须有心跳机制判断连接是否仍然存活，并在断开后支持重连。

**鉴权与连接升级**是第二层关键点。
 因为 WebSocket 建立后会维持较长时间，如果没有明确鉴权和续期机制，就容易出现长连接劫持、过期会话仍在使用等问题。

**广播与房间路由**是第三层复杂度来源。
 一旦进入 IM、协作编辑、直播间这类场景，消息往往不是发给单个连接，而是发给一组会话或某个房间内的所有连接。这时候系统就必须维护连接与房间、用户与会话之间的映射关系，通常还需要专门的网关层或状态同步机制。

所以 WebSocket 虽然功能最强，但也意味着状态管理最重、扩展复杂度最高。

## 常见坑点

### Webhook 伪造

如果 Webhook 不做签名校验，接收方几乎无法确认请求是否真的来自合法发送方。
 这意味着任何人都可能伪造“支付成功”“订单完成”之类事件，风险极高。
 因此，Webhook 没有签名时，最大的问题不是“不够规范”，而是“根本不可信”。

### Webhook 重复

Webhook 的发送方通常会做失败重试，因此重复回调几乎是常态。
 如果接收方没有幂等机制，就很容易在重试场景下出现重复发货、重复记账、重复更新状态的问题。
 所以 Webhook 的设计重点，从来不只是“怎么发”，还包括“接收方怎么把重复当正常情况处理”。

### WebSocket 状态丢失

WebSocket 最大的坑之一，是把连接状态、房间关系、订阅关系只保存在单个服务实例内。
 一旦服务重启、实例扩缩容、连接迁移，这些状态就可能瞬间丢失，导致客户端虽然重连成功，但已经不在原来的房间，也收不到原来的广播。
 因此，WebSocket 真正复杂的部分不是握手本身，而是长连接状态在多实例环境下如何恢复和同步。

## 快问快答

- **Q：支付回调用哪种？**  
  Webhook（服务到服务），必须签名 + 幂等 + 重试。

- **Q：浏览器要实时刷新列表用哪种？**  
  SSE 足够且实现成本低；强交互才上 WebSocket。
