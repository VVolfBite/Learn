# 07 分页过滤排序

列表接口通常最容易出现一种情况：**功能上能跑，但一旦数据量变大、并发变高、客户端接入变多，就会变得很难用**。
 原因在于，列表接口并不只是“查一批数据返回”，它同时承担了三件事：

- 如何把大量结果拆成可消费的小批次
- 如何让筛选条件表达得清楚且可控
- 如何保证结果顺序稳定、翻页可继续

因此，分页、过滤、排序看起来是普通的查询细节，实际上决定了列表接口的长期可用性。一个设计不好的列表接口，往往会出现这些问题：
 第一页和第二页重复、翻到后面越来越慢、相同请求每次顺序不一样、过滤条件越来越随意、客户端很难形成稳定预期。
 所以这一章真正要解决的，不是“列表怎么返回 20 条数据”，而是**如何让列表在数据变化中仍然稳定、可扩展、可维护**。

---

## 分页方式

### Offset / Limit

Offset / Limit 是最直观的分页方式。它的含义很简单：从第 `offset` 条开始，取 `limit` 条数据。
 例如：

```
GET /orders?offset=0&limit=20
```

这种方式最容易理解，也最容易在后台管理、小规模列表里快速落地。调用方只需要知道“从哪里开始、取多少条”，就能完成分页。

但 Offset/Limit 的问题也非常典型。
 第一，**越翻越慢**。因为很多数据库在处理大 offset 时，并不是直接跳到目标行，而是需要先扫描、跳过前面的大量结果，再取后面的记录。数据量越大，这种方式的成本越高。
 第二，**翻页不稳定**。如果在用户翻页过程中，列表前面发生了插入或删除，那么原来的 offset 所对应的位置就会发生偏移，结果可能出现重复数据或漏数据。
 也就是说，Offset/Limit 更像是“按当前位置切片”，而不是“沿着稳定顺序继续往后走”。

因此，Offset/Limit 适合小数据量、弱一致、对翻页稳定性要求不高的场景，但不适合作为大规模高频列表的默认方案。

------

### Cursor 分页

Cursor 分页的核心思想不是“从第几条开始取”，而是“**从上一次看到的位置继续往后取**”。
 例如：

```
GET /orders?cursor=...&limit=20
```

这里的 `cursor` 不是页码，也不是简单偏移量，而是一个“游标”，通常包含上一次最后一条记录的排序锚点。

Cursor 分页最大的好处有两个。
 第一，**翻页更稳定**。因为它不是按绝对位置分页，而是按排序键往后推进，所以即使前面有新数据插入，也不容易影响已经翻过的内容。
 第二，**大数据量性能更好**。数据库不必先跳过前面几万、几十万条记录，而是可以直接根据排序锚点继续往后查。

不过 Cursor 分页要想真正稳定，必须满足两个条件。
 一是 **cursor 必须与排序键绑定**。如果排序按 `created_at,id` 进行，那么 cursor 里就必须能够表达这两个值，否则无法精确知道“从哪里继续”。
 二是 **cursor 不能裸露成可随意伪造的明文参数**。实际系统中，cursor 通常会做编码，甚至带签名，目的是防止客户端随意篡改，破坏查询语义或触发异常路径。

所以 Cursor 分页不是简单地把 `offset` 改个名字，而是一套“围绕稳定排序推进”的分页机制。

## 排序规则

### 默认排序

列表接口必须有明确的默认排序。
 如果没有默认排序，同样的请求在不同时间、不同节点、不同数据库执行计划下，返回结果顺序都可能不同。这样一来，分页就失去了稳定基础，因为“第一页到底是哪 20 条”本身都不确定。

因此，默认排序不是可有可无的补充项，而是列表接口契约的一部分。调用方不指定排序时，服务端也必须知道该按什么顺序返回。

### 稳定排序

排序稳定的关键，在于排序键最好具备**唯一性或接近唯一性**。
 如果只按 `created_at` 排序，那么同一秒内创建的多条记录顺序就可能不确定，翻页时也可能出现重复或遗漏。
 因此，常见做法是把排序键设计成复合键，例如：

- `created_at DESC, id DESC`
- `updated_at DESC, id DESC`

这样做的目的，不是为了“排序更复杂”，而是为了让顺序在同值情况下仍然可以继续确定推进。
 分页的前提不是“能排序”，而是“**排序后顺序稳定且可继续**”。

## 过滤与查询参数

### 过滤原则

过滤的作用是让调用方从资源集合中筛出需要的数据，例如按状态、时间区间、用户、类型等条件筛选。
 但过滤能力不是越自由越好。过滤字段一旦无限开放，调用方就可能组合出大量难以优化、难以限流、难以缓存的查询，最终把数据库查询面暴露得过于宽泛。

因此，更稳妥的做法通常是**过滤字段白名单**。
 也就是：明确哪些字段允许过滤、允许怎样过滤，而不是让调用方任意传字段名和运算符。这样既便于后端优化索引，也便于做访问控制和查询预算限制。

### 查询参数表达

过滤、排序、分页这类能力，本质上属于“如何从资源集合里得到一个视图”，因此通常通过 query string 表达最自然，例如：

- `status=PAID`
- `created_from=...`
- `sort=-created_at`
- `limit=20`

如果过滤语法过于灵活，甚至允许调用方直接拼类 SQL 片段，那么虽然看起来很强大，实际上会迅速带来安全风险、性能失控和维护困难。
 因此，复杂过滤也应当使用固定语法，而不是把数据库语言半暴露给客户端。

## 返回结构

### 结构目标

列表接口的返回结构，目标不是“能把数组吐出来”就行，而是让调用方清楚地知道：

- 当前拿到的数据在哪
- 下一页如何继续
- 本次请求如何追踪

一个稳定的列表返回通常会把业务数据和分页信息明确分层，例如：

```
{
  "data": [...],
  "page": {
    "limit": 20,
    "next_cursor": "..."
  },
  "request_id": "..."
}
```

这里的意义很明确：

- `data` 是当前页结果
- `page` 是分页元信息
- `request_id` 用于排障追踪

结构一旦固定，客户端 SDK、前端分页组件、日志追踪都更容易复用。

### 分页信息表达

Cursor 分页里最重要的字段通常是 `next_cursor`。
 它告诉调用方：如果还想继续取下一页，应当把哪个游标带回去。
 相比之下，Cursor 模式下不一定总能轻易给出“总页数”或“总偏移量”，因为它强调的是顺序推进，而不是绝对位置切片。

因此，Cursor 分页的返回结构通常更关注“如何继续向后翻”，而不是“现在处于第几页”。

## 常见坑点

### Offset 漂移

Offset 最大的问题是数据变化导致位置漂移。
 列表翻页时，如果前面新增了一条记录，那么原本第二页的数据可能有一部分会被挤到第三页；如果前面删掉了一条记录，第二页又可能跳过某条本该看到的数据。
 结果就是：**重复和漏数据并存**。

### 排序不稳定

如果排序字段本身不唯一，例如只按 `created_at` 排序，那么多条同时间记录的前后顺序就可能在多次请求中发生变化。
 这会直接破坏分页稳定性，因为第二页的起点并不是一个可靠锚点。

### 过滤过宽

如果客户端可以自由指定任意字段过滤、任意组合条件、任意排序方式，那么后端很难控制查询复杂度，也很难提前设计索引与限流策略。
 这种“表面上很灵活”的接口，长期通常最难维护，也最容易演变成数据库性能黑洞。

## 快问快答

- **Q：什么时候还能用 offset？**  
  小数据量、后台管理、弱一致列表；或明确接受性能与稳定性问题。

- **Q：cursor 里要放什么？**  
  放排序推进所需的“最后一条记录的排序键”（通常是 created_at + id）。
