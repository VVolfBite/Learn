# 04 资源与 URI

资源建模决定了一套 API 到底是在“暴露一堆零散接口”，还是在“表达一个有结构的系统”。
 如果资源建模清楚，调用方看到路径时就能立刻理解：这是什么对象，它和谁有关，应该对它做什么操作；如果资源建模混乱，即使接口数量不多，使用者也会很快陷入“这个路径到底表示数据还是动作、对象之间是什么关系”的困惑。

URI 的价值并不只是“给接口取个地址”，而是把资源边界、从属关系和查询方式显式表达出来。好的 URI 不是为了好看，而是为了让调用方在不看实现的情况下，也能推断接口语义。
 因此，资源与 URI 的设计，本质上是在回答三个问题：

- **这是什么资源**
- **它和其他资源是什么关系**
- **请求到底是在定位资源，还是在检索资源**

## 资源建模

### 资源含义

资源是 API 世界里最核心的抽象单位。订单、支付单、用户、退款单、文章、评论，这些都属于典型的资源，因为它们在业务上有独立身份、有明确生命周期，也值得被单独读取、更新、追踪。

判断某个对象是否值得建模成资源，通常看三点：

- 是否有独立 ID
- 是否有相对稳定的状态与生命周期
- 是否会被单独查询、更新或引用

如果答案是肯定的，这个对象通常就适合成为资源。

### 子资源含义

子资源用于表达**从属关系**。
 当一个对象天然依附于另一个对象存在，或者只有在父资源语境下才有明确意义时，就适合建模成子资源。例如订单条目属于订单，评论属于文章，支付记录可能属于支付单或订单。

子资源的价值在于：把“属于谁”写进路径里，减少语义歧义。调用方一看到 `/orders/{id}/items`，就知道查的是“某个订单下面的条目”，而不是某个独立漂浮的对象集合。

### 关系表达

资源之间的关系不一定都要靠深层嵌套路径表达。
 有些关系更适合在资源内容里通过 `id`、链接或引用字段表示，而不是无限地在 URI 里继续往下嵌套。因为一旦路径层级过深，资源边界会越来越不清楚，调用方也要承担更高的拼接和理解成本。

所以资源建模的目标不是“把所有关系都写进 URI”，而是**只把最重要、最自然的从属关系写进 URI**，其他关系通过字段和链接表达即可。

## URI 规则（建议背）

### 名词复数

URI 一般推荐使用名词复数，例如 `/orders`、`/users`、`/payments`。
 这样做的语义很明确：路径首先表示“资源集合”，单个资源则通过集合下的 ID 来定位，例如 `/orders/{id}`。
 复数并不是语法强制，而是一种有助于统一理解的约定。它能让“集合”和“单对象”的关系天然清晰。

### 层级表达从属

URI 的层级最适合表达“谁属于谁”。
 例如 `/orders/{order_id}/items` 一看就知道 items 不是一个脱离上下文的条目集合，而是某个订单下的条目集合。
 层级的价值在于表达语境，但也正因为如此，层级应该节制使用，只在从属关系足够明确时使用。

### 避免动词

URI 更适合表达“对象是什么”，而不是“现在要执行哪个动作”。
 动作本身通常由 HTTP 方法、资源状态变化或子资源来表达。
 例如比起 `/cancelOrder`，更倾向于通过更新订单状态或创建一个取消记录来表达“取消”这件事。这样一来，资源边界与状态流转会更清晰，也更容易长期演进。

### 稳定 ID

路径中的 ID 必须具备**长期稳定、唯一定位**的能力。
 同一个 ID 不能今天表示订单，明天又表示退款，也不应在不同资源下重复承载不同行为语义。
 稳定 ID 的价值不仅在于查数据方便，还在于日志、监控、链路追踪、排障时能够准确定位同一对象。

### 查询参数

过滤、排序、分页这类能力，本质上不是“资源是谁”，而是“如何从资源集合里挑选和组织结果”。
 因此，它们更适合通过 query string 表达，例如 `status=PAID`、`limit=20`、`cursor=...`，而不是硬塞进路径里。
 路径用于定位资源，查询参数用于描述视图，这是 URI 设计里很重要的一条分工原则。

## 场景示例

### 订单查询

`GET /orders/{id}`
 这类路径的语义非常直接：定位某一个订单资源，并读取它的当前表示。

### 订单列表

`GET /orders?status=PAID&limit=20&cursor=...`
 这里路径仍然表示“订单集合”，而真正的筛选条件、分页条件放在查询参数里。这样既保持了资源层次的清晰，也让检索语义更自然。

### 订单条目

`GET /orders/{id}/items`
 这表示查询某个订单下面的条目集合。它适合用于父子关系明确、且调用方需要在父资源上下文里理解子资源的场景。

### 退款创建

`POST /refunds`
 或
 `POST /payments/{id}/refunds`

这两种设计都可能合理，关键在于业务上“退款”到底被视为一个独立资源，还是支付资源下的从属行为。如果退款有独立生命周期、状态流转、审计记录，那么独立建模成 `/refunds` 往往更稳；如果退款明显依附于支付单上下文，建模成子资源也可能更自然。

## 常见坑点

### 路径过深

路径过深往往意味着资源边界开始模糊。
 例如 `/users/{u}/orders/{o}/items/{i}/...` 这种层级，一方面说明多个对象被强绑定在同一条路径里，另一方面也会让客户端在拼接 URI 时承担更高复杂度。
 一般来说，超过 3 到 4 层之后，就应该认真反思：这里是不是把本该用资源关系表达的问题，硬塞进了层级结构里。

### 集合与动作混用

像 `/orders/{id}/cancel` 这类路径，表面上看很直观，但它把“资源”和“动作”混在了一起。
 更稳定的做法通常有两类：

- 把取消视为订单状态变化，用状态字段表达
- 把取消视为可记录、可审计的业务对象，建成 `POST /orders/{id}/cancellations`

后一种方式尤其适合“取消”这件事本身需要记录原因、操作者、发生时间等信息的场景，因为这时取消已经不只是一个动作，而更像一个有独立业务意义的资源。

### 关系过度嵌套

不是所有关系都应该写成 `/a/{id}/b/{id}/c/{id}`。
 如果一个资源已经有自己独立身份，并且常常需要脱离父资源单独被访问，那么继续把它埋在深层路径里，反而会让资源边界越来越不清楚。
 这时更好的做法，通常是让资源有自己独立路径，同时通过字段或链接表达与其他资源的关系。

## 快问快答

- **Q：取消订单到底是 PATCH status 还是 POST cancellation？**  
  只要团队统一都可以。若取消需要记录原因/时间/操作者，建子资源更自然。

- **Q：资源 ID 用自增还是 UUID？**  
  两者都行；重点是稳定、不可复用、能在日志/链路中追踪。
