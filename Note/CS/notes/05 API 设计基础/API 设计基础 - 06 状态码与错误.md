# 06 状态码与错误

错误设计的目标，从来不是“失败时随便返回一个 500”，而是让调用方在收到错误后，能够立刻回答三个问题：

- 这是什么类型的问题
- 这次是否值得重试
- 问题应该去哪里排查

如果这三件事回答不清楚，接口即使功能本身正确，接入体验也会很差。调用方会不知道该提示用户重试、该修改参数、还是该联系后端；运维和开发也会因为缺少稳定错误信息而难以定位问题。因此，错误设计本质上是一种**失败契约设计**：成功路径描述“怎么用”，失败路径描述“出问题时系统怎么表达自己”。

一个好的错误体系，通常至少要同时满足三点：
 第一，**分类明确**，让人和程序都知道错误属于哪一类；
 第二，**语义稳定**，避免同一个错误今天这样解释、明天那样解释；
 第三，**可观测**，出现问题后能通过错误码、请求 ID、上下文信息快速追到日志与链路。

## 状态码速记

### 2xx 成功

2xx 表示请求已经被成功接收、理解并处理。它传递的是“调用结果是成功的”这一层语义，而不是业务对象一定长什么样。

- **200**：最常见的成功返回，通常带响应体
- **201**：创建成功，常用于资源新建，配合 `Location` 表头更完整
- **204**：成功但没有返回体，常用于删除、空更新或无需额外内容的操作

2xx 的关键点在于：它应该只用于真正意义上的成功路径。只要语义上属于错误、拒绝或冲突，就不应该为了“统一返回 200”而硬塞到成功类别中。

------

### 4xx 客户端问题

4xx 表示：请求确实到达了服务端，但问题主要出在请求本身或调用上下文，而不是服务端内部崩溃。
 这类错误通常意味着：调用方需要修改参数、认证信息、权限条件、请求频率或资源状态，才能让下次请求有成功可能。

- **400**：参数格式错误、缺字段、字段类型不合法、请求语法不符合要求
- **401**：未认证，通常表示没有提供有效身份凭证
- **403**：已认证但没有权限访问该资源或执行该操作
- **404**：资源不存在，或不对当前调用方暴露该资源
- **409**：冲突，通常用于版本冲突、幂等重复创建、状态不允许推进
- **422**：请求格式合法，但语义校验不通过（很多团队也会统一并入 400）
- **429**：请求频率过高，被限流或节流策略拦截

4xx 最重要的特点是：**多数情况下不是“服务端坏了”，而是调用条件不满足**。
 因此，调用方看到 4xx 时，优先考虑修请求，而不是盲目重试。

------

### 5xx 服务端问题

5xx 表示：请求本身大体上是合理的，但服务端在处理过程中出现了内部异常，导致无法完成请求。
 这类错误通常意味着：问题出在服务端代码、依赖服务、数据库、网关、资源耗尽或链路异常，而不是调用方传错了参数。

- **500**：未知或未细分的服务端错误，应尽量保留给真正的内部异常
- **502 / 503 / 504**：更偏向网关、上游依赖、服务不可用、超时等场景，通常比裸 500 更有排障价值，也更适合驱动重试策略

5xx 的重要意义在于：它告诉调用方“请求不是语义上被拒绝，而是服务端没有正确处理成功”。
 因此，这类错误通常更可能进入重试、熔断、降级、告警路径。

## 错误响应格式

### 统一结构

一个稳定的错误结构，通常要把“类别”和“细节”同时表达出来。常见写法如下：

```
{
  "error": {
    "code": "ORDER_NOT_FOUND",
    "message": "order not found",
    "details": { "order_id": "123" }
  },
  "request_id": "..."
}
```

这里最重要的不是字段名一定长这样，而是**同类错误始终用同一结构表达**。
 只要错误结构统一，前端、SDK、日志采集、告警系统、排障平台都可以围绕它建立稳定处理逻辑。

------

### 字段语义

#### code

`code` 表示业务级错误码。它应该是**稳定、可枚举、可程序判断**的。
 调用方不应依赖 `message` 去写逻辑分支，而应依赖 `code`。因此，`code` 一旦上线，就不应该频繁改名或改语义。

#### message

`message` 是给人看的简短说明，用于快速理解问题。
 它的作用更偏向展示和排障提示，而不是机器判断逻辑。为了兼容国际化或多语言环境，`message` 最好不要承载不可替代的业务判断信息。

#### details

`details` 用于携带额外上下文，例如哪个字段校验失败、哪个资源 ID 不存在、哪个状态冲突。
 它的价值在于提升定位效率，但必须注意脱敏，不能把内部 SQL、堆栈、密钥、数据库连接信息直接暴露出去。

#### request_id

`request_id` 用于把这次失败和服务端日志、链路追踪、告警记录对应起来。
 没有 `request_id` 时，调用方只能说“刚刚有个请求失败了”；有了 `request_id`，才能精确追查是哪一次请求、在哪一层失败、关联了哪些下游调用。

## 关键结论

### 类别与原因分离

HTTP 状态码负责表达“错误属于哪一大类”，业务错误码负责表达“具体是哪一种原因”。
 这种分层设计能让调用方先粗分（比如要不要重试），再细分（具体失败在什么业务条件上）。

### 语义长期稳定

错误码一旦被客户端依赖，就必须保持长期稳定。
 如果今天 `ORDER_NOT_FOUND` 表示“订单不存在”，明天却拿它表示“订单不可见”，调用方很容易写出错误逻辑，排障和监控也会失真。

### 可重试性明确

错误设计必须让调用方能判断“这次到底能不能重试”。
 例如 429、502、503、504 通常可以配合退避重试；而余额不足、权限不足、参数不合法通常不应重试。
 如果这一层语义不清楚，调用方就只能靠猜，最终要么错过恢复机会，要么制造重试风暴。

### 结构全站统一

同一系统里的同类接口应使用同一套错误结构。
 如果 A 接口返回 `error.code`，B 接口返回 `err_no`，C 接口直接把报错写在 `message` 顶层，那么前端和 SDK 最终都会堆满特殊分支，维护成本会越来越高。

## 常见坑点

### 200 包错误

最糟糕的设计之一，是“无论成功失败都返回 200，然后在 body 里告诉调用方其实失败了”。
 这样做会直接破坏 HTTP 状态码本来的分类价值，调用方只能靠解析 message 或某个自定义字段来判断结果，代理层、网关、缓存、SDK、监控系统也都无法正确理解请求语义。

### 业务拒绝当 500

余额不足、库存不足、状态不允许推进、权限不足，这些本质上是**语义拒绝**，不是服务端内部崩溃。
 如果把它们都包成 500，调用方会误以为“服务端暂时出错，可以重试”，结果不仅误导用户，也可能放大系统压力。

### details 泄露内部信息

`details` 很适合放上下文，但不适合暴露底层实现细节。
 把 SQL 语句、堆栈、数据库异常、密钥信息直接返回给客户端，既增加安全风险，也会把内部实现细节暴露成对外契约的一部分。

## 快问快答

- **Q：参数校验失败用 400 还是 422？**  
  两者都可，关键是团队统一；很多团队用 400 覆盖校验失败即可。

- **Q：重复创建用什么？**  
  409 冲突 + 稳定业务码（如 `ORDER_ALREADY_EXISTS`），更清晰。
