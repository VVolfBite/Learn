# 08 版本与兼容

API 的长期成本，很多时候并不来自“第一次把接口做出来”，而是来自**接口上线之后如何继续修改**。
 只要接口已经被前端、移动端、第三方系统或其他服务接入，就不能再把它当成一段可随意重构的内部代码。任何字段、状态、错误码、必填规则的变化，都会沿着调用链传递出去，影响已有客户端的行为。

因此，版本与兼容要解决的核心问题不是“怎么给接口起 v1、v2 的名字”，而是：**如何在系统继续演进的同时，不把已上线客户端一起打碎**。
 一个成熟的接口设计，通常不是靠频繁开新版本来维持，而是优先通过兼容演进把变化消化掉；只有当变化已经无法兼容时，才谨慎引入新版本。

---

## 兼容性原则

### 新增字段

新增字段通常被认为是最安全的变更之一。
 原因在于，大多数客户端在解析 JSON 时，只会读取自己关心的字段，额外出现的未知字段通常可以被忽略。因此，只要新增字段不会改变旧字段语义，也不会把原来的可选变成必填，旧客户端一般可以继续工作。

但这里的“安全”有一个前提：客户端必须具备“忽略未知字段”的能力。如果客户端实现过于脆弱，连多一个字段都会出问题，那么接口契约本身就已经不够健壮。

### 删除字段

删除字段通常属于破坏性变更。
 因为一旦某个客户端仍然依赖这个字段，而服务端把它删掉，客户端就可能直接报错、展示异常，或者在更糟糕的情况下悄悄进入错误逻辑。
 所以删除字段不能被理解成“服务端已经不用了就删掉”，而必须考虑“外部是不是还在用”。

### 改变语义

改变字段语义往往比删除字段更危险。
 删除字段至少会让问题比较容易暴露：客户端拿不到值，通常会报错或明显异常；而语义改变则可能让客户端**看起来还能运行，但结果已经错了**。
 例如原本 `status=PAID` 表示支付完成，后来却被改成“支付处理中”；字段名没变，客户端也没报错，但业务行为已经静默偏离。
 这种“静默错”通常比显式报错更难排查，也更容易造成线上事故。

### 枚举扩展

枚举字段必须从一开始就按“未来可能增加新值”来设计。
 只要客户端把枚举写死，不允许出现未知值，那么服务端一旦增加新状态，旧客户端就可能崩溃、漏展示或走到意料之外的逻辑分支。
 因此，兼容性的一个基本要求是：**客户端必须容忍未知枚举值，服务端必须对枚举扩展保持谨慎**。

## 版本策略

### URL 版本

把版本写进路径，例如：

```
/v1/orders
```

这种方式最大的优点是直观。路径本身就告诉调用方现在访问的是哪一版接口，路由层实现也通常比较简单，文档和调试都容易理解。
 但它的缺点同样明显：一旦版本分裂，`/v1`、`/v2`、`/v3` 可能会长期并存，旧版本很难下线，最后形成多套接口长期维护的局面。
 所以 URL 版本虽然简单，但如果缺乏下线策略，很容易让版本债务长期堆积。

### Header 版本

把版本信息写进 Header，例如：

```
Accept: application/vnd.xxx.v1+json
```

这种方式更偏“契约化”，因为资源路径本身保持稳定，版本被放在内容协商层。
 它的优点是语义更纯粹：URI 仍然表示资源是谁，版本作为表达形式或协议契约的一部分存在。
 但它的缺点是可见性较差。浏览器直接调试、命令行访问、日志排查时，没有路径版本那么直观，接入与调试成本会略高。

### 不显式版本

不显式版本的核心思想是：**优先通过向后兼容的方式持续演进，而不是一有变化就开新版本**。
 例如新增字段、增加可选能力、扩展枚举、增加新的子资源，这些变化通常都可以在原有接口基础上自然演进，而不必立刻推出 `v2`。
 只有当出现真正无法兼容的破坏性变更时，才考虑启用新版本。

这种方式的好处是版本数量更少、系统更整洁；但前提是团队对兼容性边界必须足够清楚，否则“看起来没升版本”的变化，可能实际已经把旧客户端悄悄破坏掉。

## Expand/Contract（推荐方法论）

Expand / Contract 是处理兼容演进最经典的方法论。它的核心不是“怎么升级更快”，而是**怎么把一个可能破坏兼容性的变化拆成多个安全的小步骤**。

### Expand

扩展阶段先引入新能力，但不立即移除旧能力。
 例如加一个新字段、新资源、新响应结构，让旧客户端仍然可以继续使用旧路径，新的调用方则可以开始接入新能力。
 这一阶段的目标不是替换旧世界，而是让新旧世界先能并存。

### Dual-write

如果新旧结构需要同时生效，通常会进入双写阶段。
 也就是说，服务端在一段时间内同时写旧字段和新字段，确保从这一刻开始，新产生的数据在两套结构中都存在。
 双写的意义在于：为后续切换争取缓冲空间，避免切换时发现新结构中没有完整数据。

### Migrate

双写只能保证“新产生的数据”进入新结构，历史存量数据仍然需要迁移。
 这一步通常是后台批处理，把旧字段、旧表、旧状态逐步搬到新结构中。
 迁移过程必须可追踪、可恢复、可校验，否则一旦中断或出错，很容易留下半迁移状态。

### Switch

当新结构已经具备完整数据，且经过校验后，可以逐步把读路径和写路径切到新结构。
 切换不一定是一刀切，更稳妥的做法通常是灰度切换、采样对账、保留开关，确保出现问题时还能快速退回旧路径。

### Contract

收缩阶段才是真正删除旧字段、旧逻辑、旧结构的时候。
 这一步一定要放在足够长的兼容窗口之后，确保旧客户端、旧任务、旧依赖已经退出。
 Contract 的本质是“收尾”，而不是“新版本刚上线就顺手删旧字段”。

## 常见坑点

### 语义静默漂移

同名字段改语义，是最危险的一类兼容性问题。
 表面上接口仍然存在、字段也还在，客户端甚至不会直接报错，但业务解释已经变了，最终产生的是“悄悄错”，而不是“明显挂”。
 这种问题通常比删除字段更难排查。

### 版本使用不可见

版本切换如果没有观测能力，就会出现一个经典问题：旧版本到底还有没有人在用，根本不知道。
 这时如果贸然删除 `v1` 或停掉某条旧路径，很可能直接造成线上事故。
 因此，版本策略不能只设计“怎么发新版本”，还要设计“如何观察旧版本使用情况、如何安全下线”。

### 枚举新增崩溃

客户端如果把枚举状态写死，没有 default 分支或兜底逻辑，一旦服务端增加新值，旧客户端就可能直接崩溃、空白或显示错误。
 因此，枚举兼容性其实是版本兼容里最容易被忽略、但又最常见的坑之一。

## 快问快答

- **Q：什么时候一定要开新版本？**  
  破坏性变更（删除字段、改语义、改必填规则）且无法兼容演进时。

- **Q：如何判断能不能不升版本？**  
  如果只新增字段/新增可选能力，且旧客户端行为不变，通常不需要升版本。
