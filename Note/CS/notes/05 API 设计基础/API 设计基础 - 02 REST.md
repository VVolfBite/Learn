# 02 REST

REST 本质上不是一套“URL 写法规范”，也不等于“接口走 HTTP 就算 REST”。它更接近一种**围绕资源组织系统能力的设计方式**：先把业务世界里的核心对象抽象成资源，再用统一的方法去读取、创建、更新、删除这些资源。
 因此，REST 真正关心的不是“路径里是不是用了名词”，而是接口是否围绕清晰稳定的资源展开，是否让调用方容易理解“这是什么对象、能做什么、状态怎么变化”。

如果一个接口表面上用了 `GET /orders/123` 这样的路径，但实际语义混乱、状态推进靠一堆隐式动作、不同接口的行为完全不一致，那么它只是“长得像 REST”，并不真的具备 REST 的设计价值。
 REST 的核心收益在于：**把接口从“零散动作集合”整理成“围绕资源的统一系统”**。

---

## 核心概念

### 资源含义

资源（Resource）是业务对象在接口层的抽象表示。订单、支付单、用户、退款单、文章、评论，都可以被建模成资源。
 这里的关键不是“数据库里有没有这张表”，而是“这个对象在业务上是否值得被独立识别、独立寻址、独立操作”。
 资源一旦建模清楚，接口的很多问题就会自然收敛：路径怎么起名、方法怎么选、状态怎么表达、关联关系怎么组织，都会更统一。

### 统一接口

统一接口是 REST 最重要的思想之一。它强调不要为每个动作都发明一套全新的调用方式，而应当尽量使用一组有限且稳定的方法来表达常见操作，例如读取、创建、整体替换、部分更新、删除。
 统一接口的价值是：调用方不用记住大量风格各异的“特殊接口”，只需要理解资源和标准方法的组合，就能快速推断接口行为。

### 无状态

无状态指的是：**每个请求都应当携带完成本次处理所需的信息**。服务端不依赖某个连接上的临时会话状态来理解这次请求。
 这并不等于系统里完全没有状态，而是说“状态不寄存在服务端内存中的某个会话上下文里”。
 这样做的好处是扩展性和容灾性更强：请求落到哪台实例都能处理，不需要强绑定某个节点的会话。

## 关键结论

### 路径表达资源

路径最重要的职责是回答“访问的是哪个资源”。
 例如 `GET /orders/123` 一眼就能看出这是在获取编号为 123 的订单；相比之下，`/getOrder?id=123` 更像在调用一个动作函数，而不是在定位一个资源对象。
 REST 倾向于把“对象是谁”放进路径，把“对对象做什么”放进方法。

### 方法表达动作

REST 并不是不要动作，而是希望动作尽量通过标准方法来表达。
 同样是订单资源：

- `GET /orders/{id}` 表示查询
- `POST /orders` 表示创建
- `PATCH /orders/{id}` 表示局部更新
   这种方式的好处是行为模式更统一，调用方的预期也更稳定。

### 名词复数

资源路径通常推荐使用名词复数形式，如 `/orders`、`/users`、`/payments`。
 复数形式表达的是“资源集合”，单个资源则通过 ID 在集合下定位，例如 `/orders/{id}`。
 这样做不是为了形式好看，而是为了让资源集合与资源实例的层级关系更自然。

### 子资源关系

当一个资源天然从属于另一个资源时，使用子资源路径通常更清晰。例如：
 `/orders/{id}/items` 表示“某个订单下面的订单项集合”。
 子资源的价值在于把归属关系直接写进路径中，让调用方一眼就能看出对象之间的上下文。

### 状态字段表达

REST 风格通常更倾向于把业务流程表达为资源状态变化，而不是为每一步动作都单独创造一个接口。
 例如订单从 `PENDING` 变成 `PAID`，核心是订单资源的状态推进，因此可以通过状态字段表达，而不是上来就设计 `/payOrder`、`/confirmOrder`、`/finishOrder` 一连串动作式路径。
 这会让状态机更集中，也更容易统一管理。

## 常见场景

### 订单资源

订单是典型资源。

- `POST /orders`：创建订单
- `GET /orders/{id}`：查询订单
- `PATCH /orders/{id}`：更新订单部分状态或属性

如果订单下还有条目、支付记录、取消记录等，它们通常可以继续作为子资源组织。

### 支付与退款

支付单、退款单这类对象也适合独立建模成资源，因为它们通常有自己的生命周期、状态和审计需求。
 例如 `payments` 可以表示支付流程对象，`refunds` 表示退款流程对象。这样设计后，支付和退款就不只是“订单上的两个动作”，而是有自己边界和状态的业务实体。

## 常见坑点

### 动词路径泛滥

最常见的问题之一，是把接口设计成一堆动作方法：
 `/createOrder`、`/cancelOrder`、`/doPay`、`/closeRefund` ……
 这样短期实现可能很快，但长期会出现两个问题：
 第一，接口风格越来越难统一；第二，状态机被分散到大量动作接口中，后续扩展和兼容都很痛苦。

更稳妥的做法通常是：

- 能表达为资源状态变化的，优先表达为状态变化
- 需要单独记录的动作，建模成子资源
   例如取消订单可以设计成 `POST /orders/{id}/cancellations`，这样取消行为本身也有了明确边界和记录载体。

### 查询语义混乱

另一个常见问题是把本来只是“查询”的操作设计成 POST。
 如果只是普通检索、过滤、排序、分页，优先考虑 GET + 查询参数。这样更符合调用预期，也更利于缓存、观测和代理层处理。
 只有在参数过长、过滤条件过复杂、或者明确接受语义偏离时，才考虑用 POST 做查询，并且应当清楚说明：这类查询通常不具备 GET 那样天然的可缓存性和可预期性。

### 资源边界混乱

有些接口表面上看像 REST，实际资源边界并不清楚。例如订单、支付、退款、状态流转全塞在同一个对象里，最后调用方根本分不清“到底在操作哪个业务实体”。
 REST 真正难的地方，不在 URL 形状，而在资源划分是否合理。资源边界一旦混乱，后面的方法选择、状态设计、权限控制都会跟着混乱。

## 快问快答

- **Q：REST 一定要 HATEOAS 吗？**  
  理论上是 REST 的一部分，但工程里多数团队不强制；关键是资源建模与一致性。

- **Q：什么时候不建议强行 REST？**  
  纯 RPC 行为、强动作语义且难资源化的场景，可用 RPC/gRPC，但仍要保持一致的错误与契约。
