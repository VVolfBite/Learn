# 12 高频判断

把常见决策“口诀化”，用于接口评审与面试场景题。

---

## 资源建模

- **先资源，后动作**：路径尽量名词化，状态用字段表达
- **动作复杂可建子资源**：取消/退款/审批等需要记录时，用子资源更自然

---

## 幂等与重试

- **任何可能重试的写都要幂等**：超时、网关重试、客户端重试都会发生
- **幂等键 + 去重**：最通用的落地方案
- **增量更新要小心**：`delta` 不是幂等，需改“设置”或加幂等键

---

## 错误体系

- **HTTP 分类 + 业务码细化**
- **可重试性清晰**：4xx 通常不重试（除 429），5xx/网关类可重试但要退避
- **统一错误结构**：让 SDK/前端能统一处理

---

## 列表接口

- **默认排序必须稳定**：排序键最好包含唯一性
- **大数据量优先 cursor**：offset 在大页会慢且不稳定
- **过滤白名单**：避免无限组合把数据库打穿

---

## 版本演进

- **能不升版本就不升**：新增字段最安全
- **破坏性变更才升版本**：删字段/改语义/改必填
- **expand/contract**：线上变更的通用方法论

---

## 选择题风格 Q&A

- **Q：取消订单接口怎么设计更规范？**  
  需要记录取消原因/时间/操作者 → `POST /orders/{id}/cancellations`；  
  只是状态变更且简单 → `PATCH /orders/{id}` 修改 `status`（团队统一即可）。

- **Q：创建订单 POST 怎么保证重试安全？**  
  使用 `Idempotency-Key`（或 request_id）+ 去重表/唯一约束，重复请求返回同一结果。
