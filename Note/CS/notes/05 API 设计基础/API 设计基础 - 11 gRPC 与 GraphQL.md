# 11 gRPC 与 GraphQL

这一章的重点，不是去背协议细节或语法，而是弄清楚两件事：

- **它们分别解决什么问题**
- **引入它们以后要承担什么代价**

gRPC 和 GraphQL 经常被放在一起讨论，但它们并不是同一类东西。
 gRPC 更像是“服务与服务之间如何高效、强类型地通信”；
 GraphQL 更像是“客户端如何按需拿到自己真正需要的数据形状”。
 前者更偏**内部调用协议**，后者更偏**查询组织方式**。
 因此，二者的选择标准不在于“哪一个更先进”，而在于业务场景到底是在解决“服务间调用效率问题”，还是在解决“前端数据聚合与字段灵活性问题”。

------

## gRPC

### 基本含义

gRPC 可以理解为一种面向服务间调用的 RPC 机制，通常建立在 **HTTP/2 + Protobuf** 之上。
 它的核心目标不是“让接口更像浏览器可调试的 HTTP API”，而是让服务之间的调用在**性能、契约明确性、流式能力**上更强。
 也就是说，gRPC 主要优化的是机器与机器之间的通信体验，而不是人类手动调试的便利性。

### 契约特点

gRPC 的契约通常通过 `.proto` 文件定义。
 这意味着请求结构、响应结构、字段类型、服务方法在协议层就是显式的、强类型的。
 相比很多自由度更高的 JSON 接口，gRPC 的约束更强，因而在大型服务体系里更容易形成一致的接口定义和自动生成能力。

### 优点

gRPC 的一个重要优势是**性能更好**。
 它使用二进制序列化（Protobuf）而不是文本 JSON，传输体积更小；HTTP/2 支持多路复用，也更适合高并发短连接调用场景；同时它天然支持流式通信，可以更方便地表达服务端流、客户端流、双向流。

另一个优势是**契约强**。
 在微服务数量很多的系统里，接口如果全靠手写 JSON 和文档约定，很容易漂移；而 gRPC 通过 schema 先行，让生成代码、接口校验、类型对齐更自然。

### 缺点

gRPC 最大的短板不是“能力不够”，而是**浏览器与外部生态兼容性没那么自然**。
 浏览器直接调用并不方便，通常需要网关或额外适配层。
 此外，gRPC 调试体验往往不如 HTTP/JSON 直观：后者直接 curl 就能看懂，前者在没有合适工具时不够透明。

因此，gRPC 很适合内部调用，但如果需要大量面向外部开发者、浏览器直连或第三方接入，往往还要额外补一层 HTTP 网关或 API facade。

### 适用场景

gRPC 更适合：

- 微服务之间的大量内部调用
- 对延迟与吞吐要求较高的 RPC 场景
- 需要流式能力的服务间通信
- 希望依赖强类型契约统一接口定义的系统

换句话说，gRPC 解决的是“内部服务调用怎么更高效、更规范”的问题。

## GraphQL

### 基本含义

GraphQL 不是简单的“另一种接口协议”，而是一种**由客户端声明需要什么字段，服务端按该形状返回数据**的查询方式。
 它最核心的思想是：客户端不再只能被动接受服务端预先固定好的响应结构，而是可以在一定规则内，主动描述自己需要哪些字段、哪些关联对象、返回层级是什么样子。

这种机制特别适合前端页面复杂、字段组合变化频繁的场景。
 因为很多时候，前端真正痛苦的不是“接口没有”，而是“接口返回得太多、太少，或者需要多次调用才能拼出一个页面”。

### 优点

GraphQL 的最大优点是**按需取数**。
 客户端只声明需要的字段，理论上可以减少 over-fetching（返回太多没用字段）和 under-fetching（需要调多个接口才能拼齐数据）。
 这让前端在构建复杂页面时更灵活，尤其是在多端、多页面、字段需求差异大的场景里，服务端不必为每种页面组合都单独设计一个 REST 接口。

另一个优势是**聚合能力强**。
 一个查询可以跨多个资源对象组织返回结果，这对前端尤其有吸引力，因为页面想要的通常不是“某张表的单一对象”，而是一组带关联的聚合视图。

### 缺点

GraphQL 的问题不是“查不到”，而是**太容易查太多**。
 一旦把查询自由度放给客户端，后端就必须承担新的复杂度：

- 权限控制不能只按接口做，还要按字段和查询深度做
- 缓存不再像 REST 那样天然围绕 URI 工作
- 查询代价可能很难预估
- resolver 一旦设计不好，很容易引入 N+1
- 一个看似正常的查询可能在底层展开成极重的数据库与下游访问链路

因此，GraphQL 并不是“前端更方便”这么简单，它实际上是把一部分“数据组织复杂度”从前端接口组合层转移到了后端查询执行层。

### 适用场景

GraphQL 更适合：

- 多端页面复杂，字段需求经常变化
- 前端需要灵活聚合多个资源
- 页面构成差异大，不希望为每个页面形状都维护独立 REST 接口
- 团队有能力治理查询复杂度、权限与缓存策略

换句话说，GraphQL 解决的是“客户端如何更灵活地拿到自己真正需要的数据形状”的问题。

## 常见坑点

### GraphQL 成本失控

GraphQL 最典型的坑，是没有对查询复杂度做限制。
 客户端如果可以任意嵌套字段、无限展开关联对象，一个请求就可能在服务端触发大量数据库查询、远程服务调用和内存聚合，最终把后端打爆。
 因此，GraphQL 通常必须配套 query cost、深度限制、字段级权限和超时控制，否则“灵活”很容易演化成“失控”。

### GraphQL 的 N+1

GraphQL 的 resolver 天然容易写成“访问一个字段就查一次数据”的模式。
 这和 ORM 世界里的 N+1 很像：表面上只是逐层解析字段，实际上底层查询数量迅速膨胀。
 因此，GraphQL 几乎总要配套 batch / loader 之类的批量装载机制，否则稍复杂一点的查询就可能性能很差。

### gRPC 的外部暴露问题

gRPC 非常适合内部服务间调用，但如果系统还需要对浏览器、第三方、移动端直暴露接口，那么往往不能只靠 gRPC。
 这时就需要额外的网关、转译层或外部 HTTP API。
 如果一开始只考虑内部服务调用，没有配套外部访问策略，后续演进时很容易出现“内部协议很好，外部接入却很痛苦”的问题。

---

## 快问快答

- **Q：内部服务调用优先选什么？**  
  gRPC（性能与契约更强），但要配套网关与观测。

- **Q：前端页面字段组合多，接口难维护怎么办？**  
  GraphQL 可能更合适，但必须配成本控制与缓存策略。
