# 05 方法与幂等

HTTP 方法不是“随便挑一个能发请求的动词”，而是接口契约的一部分。方法一旦选定，就会影响调用方对这个接口的基本预期：
 这个请求是在读还是在改？
 可不可以缓存？
 失败后能不能安全重试？
 重复请求会不会造成额外副作用？

因此，方法选择的核心，不只是“功能上能不能实现”，而是**语义是否稳定、重复请求是否安全、调用方是否容易形成正确预期**。
 这也是为什么这一章真正的重点不只是 GET、POST、PUT、PATCH、DELETE 的表面区别，而是它们背后的三个关键词：**幂等、重试、副作用**。

---

## 方法语义（最常用）

### GET 语义

GET 用于读取资源，核心语义是“获取当前表示”，而不是“触发一次动作”。
 按照约定，GET 应尽量无副作用，并天然适合缓存、代理和重试。这里的“无副作用”不是说服务端绝对不会做任何事情，而是说**对业务状态不应产生可观察的改变**。
 例如记录访问日志通常不算违背 GET 语义，但“GET 一下就生成订单、扣库存、发短信”显然是不合适的。

### POST 语义

POST 通常用于创建资源，或者用于表达一类不容易自然落在其他方法语义里的非幂等操作。
 它的特点不是“比 GET 更强大”，而是**默认不具备天然幂等性**。
 也就是说，同样的 POST 请求发两次，系统很可能会执行两次副作用，例如创建两笔订单、发起两次扣款、产生两条记录。

因此，POST 并不是不能重试，而是**重试前必须先设计好幂等机制**。

### PUT 语义

PUT 的典型语义是“把这个资源整体变成某个确定状态”。
 它关注的是“最终结果是什么”，而不是“执行了几次动作”。也正因为如此，PUT 通常被认为是幂等的：同一个 PUT 请求执行一次和执行十次，只要请求内容相同，资源最终都会变成同样的样子。
 需要注意的是，PUT 更适合整体替换而不是局部补丁。如果接口只想修改少数字段，而不是替换整个表示，通常更适合 PATCH。

### PATCH 语义

PATCH 表示部分更新，重点是“只改其中某些字段”。
 它是否幂等，不取决于方法名本身，而取决于“补丁内容的语义”。
 如果 PATCH 表达的是“把昵称设置为某个值”，那通常是幂等的；如果 PATCH 表达的是“余额加 10”“库存减 1”，那就不是天然幂等，因为重复执行会不断累加副作用。

所以，PATCH 最容易出问题的地方在于：**看起来像普通更新，实际上可能是隐含动作语义**。

### DELETE 语义

DELETE 表示删除资源。它通常被视为幂等，因为同一个资源删一次和删多次，最终结果都应当是“这个资源不存在”或“对外视为不可用”。
 这里的重点不在于第二次删除时是否返回和第一次一模一样的状态码，而在于**重复执行不会让业务状态继续发生新的变化**。

## 幂等与重试

### 幂等定义

幂等表示：**同一个请求执行多次，最终效果与执行一次相同**。
 幂等关心的不是“请求会不会重复到达”，而是“重复到达后会不会产生新的副作用”。
 在网络环境里，请求重复到达是常态：用户重复点击、客户端自动重试、网关超时重放、服务间重试都会发生。所以 API 设计不能假设“每个请求只会来一次”。

### 重试现实

重试不是异常情况，而是分布式系统里的常见行为。
 网络超时、连接断开、网关 502/503/504、上游服务短时不可用，这些都可能触发自动或手动重试。
 因此，接口设计必须提前回答一个问题：**如果同一个请求再来一次，会不会把副作用重复执行一遍？**

### 副作用约束

只要一个操作会改变真实业务状态，就必须考虑副作用的唯一性。
 订单创建、扣款、发货、发券、库存扣减，这些都不是“重复一下也无所谓”的操作。
 因此，这类副作用要么天然幂等，要么必须通过幂等键、条件更新、去重表等机制实现“结果只生效一次”。

## 幂等实现套路（工程最常用）

### 幂等键

幂等键（例如 `Idempotency-Key` 或 `request_id`）是最常见也最实用的做法。
 它的基本思想是：给同一次业务操作分配一个稳定标识，服务端用这个标识识别“这是第一次请求，还是同一请求的重复到达”。

常见做法包括：

- 在业务表上对 request_id 建唯一约束
- 使用单独的去重表记录 request_id、状态和结果
- 重复请求不再执行业务动作，而是直接返回第一次的处理结果，或返回“处理中/已完成”的状态

幂等键的重点在于：**多次重试必须带同一个键**。如果每次重试都生成新键，那服务端根本无法识别它们属于同一业务操作。

### 条件更新

条件更新（CAS）适用于状态更新型接口。
 它的思路是：更新时不只按主键定位，还要求“当前版本或当前状态必须满足条件”。只有条件成立时才更新成功，否则说明数据已被并发修改、已被其他请求处理，当前请求应当失败或返回冲突。

这种方式特别适合订单状态推进、余额版本更新、库存版本控制等场景，因为它能把“只允许推进一次”的规则直接固化到数据库层。常见坑点

- **POST 重试导致重复创建**：没幂等键就会“多一单/多扣一次”
- **PATCH 语义不清**：`PATCH /balance {delta:+10}` 不是幂等的  
  若要幂等，应改成设置值（set）或带请求幂等键
- **DELETE 的 404 语义**：重复删除应返回 204 或 404？关键是保持一致与可解释

## 快问快答

- **Q：为什么 PUT 被认为幂等？**  
  因为它表达“把资源变成这个样子”，重复执行结果一致。

- **Q：PATCH 一定幂等吗？**  
  不一定。取决于你是“设置字段”还是“增量更新”。

- **Q：幂等键放哪里？**  
  Header（如 `Idempotency-Key`）最常见；也可放 body，但要标准化。
