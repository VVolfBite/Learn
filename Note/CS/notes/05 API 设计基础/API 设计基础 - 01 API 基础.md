# 01 API 基础

API 的本质，不只是“系统提供了一组可调用的接口”，而是系统对外暴露能力时所做出的**长期承诺**。

API 设计真正关注的，不是“这次请求能不能成功调通”，而是接口在长期使用中是否仍然**容易理解、行为一致、可以平滑演进、出了问题能快速定位**。这也是为什么 API 设计常被概括为四个关键词：**清晰、一致、可演进、可观测**。这四点看似抽象，实际上对应的是接口生命周期里最常见的四类成本：理解成本、接入成本、升级成本和排障成本。

## 核心概念

### 契约含义

API 首先是一种**契约**。所谓契约，不只是“参数名和返回值长什么样”，而是一次调用中所有重要约定的总和，包括：

- 输入字段代表什么含义
- 哪些字段必填，哪些可选
- 成功时会返回什么
- 失败时会返回什么错误以及错误语义
- 请求是否允许重试
- 同一个操作是否需要幂等保证
- 状态如何推进、结果何时可见

契约一旦被调用方依赖，就不能随意改变。字段名改了、错误码改了、返回结构改了，看起来只是后端改了一点实现，实际上可能会直接导致调用方兼容失败。

### 稳定边界

API 还是一种**稳定边界**。系统内部实现可以不断变化：数据库表可以重构，服务可以拆分，缓存与队列可以新增，内部状态机可以调整。但只要对外行为保持稳定，调用方就不需要跟着这些内部变化一起重写。
 因此，一个设计得好的 API，应该尽量把内部复杂性藏在边界后面，而不是把内部结构原样暴露给外部。边界稳定，内部才有演进空间；边界不稳定，系统每改一次内部实现，外部调用方就得跟着一起改。

### 抽象层次

API 还需要合适的**抽象层次**。抽象层次太低时，接口会像数据库表的直接镜像：字段名、状态、结构都紧贴存储模型，短期看开发很快，长期看却会把存储细节强行暴露给外部。
 抽象层次太高时，接口又可能过于空泛，调用方很难理解具体行为。
 合适的抽象层次应该围绕“业务能力”展开，而不是围绕“内部表结构”展开。也就是说，API 应表达的是“可以创建订单、查询支付状态、发起退款”，而不是“这里有一张 orders 表、一张 payments 表、一张 refunds 表”。

## 关键结论

### 接口即产品

接口不是后端给别人“顺手提供的一段访问入口”，而是会被长期依赖的产品部件。字段命名是否清楚、状态是否可理解、错误是否可判断，这些问题都会直接影响调用体验。很多接口问题并不是功能缺失，而是语义模糊：字段看着都在，但没人能确定它到底表示什么。

### 资源优先

设计 API 时，通常优先围绕资源建模，而不是围绕动作命名。订单、支付单、退款单、用户资料，这些都是稳定的业务对象；它们比一组零散的动作更适合作为接口设计的核心。这样做的好处是语义更统一、路径更清晰、后续扩展状态与关系也更自然。

### 一致性优先

同一套 API 里，相似问题最好有相似答案。请求结构、响应结构、分页格式、错误格式、时间字段写法、金额字段写法，如果每个接口都各搞一套，调用方最后会面对一个“看似属于同一系统、实际上行为风格完全不一致”的接口集合。
 一致性不是为了好看，而是为了降低接入成本和出错概率。

### 演进优先

API 一旦上线，就会被不同节奏的调用方长期使用。新增字段、增加状态、增加可选能力，通常都比“删字段、改语义、改结构”安全得多。因此，API 设计时就应该为未来扩展留空间，而不是只追求当前最省事的写法。
 真正危险的不是“新增东西”，而是“让原来依赖这个接口的调用方突然不再兼容”。

### 可观测优先

只要接口进入线上，就一定会出问题。问题出现时，能不能快速定位，取决于接口是否留足了观测信息。请求 ID、稳定错误码、上下游日志字段、调用链路信息，这些都不是“额外装饰”，而是线上排障的必要条件。
 没有可观测性的接口，即使功能本身设计得不错，出了问题也很难维护。

## 常见场景

### 多端协作

前端、移动端、小程序、第三方集成通常都依赖 API 作为统一入口。在这种场景里，接口的清晰度直接决定联调效率。
 如果字段语义模糊、分页风格不统一、状态变化没有说明，调用方就只能靠猜测接入，联调成本会非常高。

### 多服务协作

在微服务或服务化架构中，API 不只是对外接口，也可能是服务与服务之间的协作边界。此时边界不清晰带来的问题会更严重：上游服务不断依赖下游的内部字段，下游一改实现，上游就跟着出问题，最终调用链会变成“字段拼图”，很难演进。

### 第三方接入

面向外部伙伴或生态时，API 的契约性要求更高。因为第三方通常无法快速感知内部变更，也很难和服务端保持完全同步的发布节奏。
 这类场景里，文档质量、错误语义、兼容策略会直接决定接入成功率。

## 常见坑点

### 表结构镜像

最常见的问题之一，是把数据库表结构几乎原样暴露为 API。这样做短期省事，长期代价很大：字段语义会混乱、内部重构会直接影响外部、存储层限制被外部调用方感知到，最终把内部技术债转嫁给所有调用方。

### 风格不一致

同一系统中，A 接口用 `offset/limit`，B 接口用 `page/pageSize`；A 返回 `error_code`，B 返回 `code`；A 时间是时间戳，B 时间是字符串。
 这种不一致会让调用方不断切换理解方式，增加学习成本和接入错误率。

### 错误不可读

如果接口只返回 `500` 加一句模糊报错，调用方几乎无法判断问题是参数错了、权限不够、资源不存在，还是系统暂时不可用。
 错误一旦不可读，重试策略、用户提示、自动恢复、排障定位都会受到影响，成本会迅速上升。

## 快问快答

- **Q：API 设计最重要的第一原则？**  
  清晰与一致。调用方能理解、行为能预期，比“功能齐全”更优先。

- **Q：为什么不要暴露数据库结构？**  
  因为内部实现必然演进；暴露了就把未来改动成本强行外溢给调用方。
