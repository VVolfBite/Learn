# 可观测性基础 - 链路追踪与埋点

系统还是单体时，排障往往只要进一台机器看日志；系统一旦拆成多服务，请求会穿过网关、应用服务、缓存、数据库、消息系统，再回到调用方，原来那种“翻一份日志看到底”的方法很快就不够用了。链路追踪和埋点，就是为这种场景准备的。

## Tracing

链路追踪关心的是一次请求在系统里的完整路径。入口收到请求时发生了什么，调用了哪些下游，在哪一段耗时最长，哪一跳报了错，是否出现了重试或回退，这些都属于 tracing 想要回答的问题。

把 tracing 放在可观测性里看，它最重要的作用不是画漂亮的瀑布图，而是把“同一次请求”的上下文维持住。没有这层串联，日志再多，也很容易变成散点。

### Trace 与 Span

一条完整请求路径通常会被看成一个 trace，路径中的每一段操作则对应一个 span。入口服务创建 trace，下游调用继续沿着同一套上下文传递，各段 span 再拼成整条调用链。这样看一次超时，就不只是知道“超时了”，而是能继续看出“究竟慢在认证、数据库、下游 RPC，还是消息发送”。

## Instrumentation

Instrumentation 可以直接理解成埋点。系统不会自动知道哪些信息对排障有价值，很多关键观测点必须由开发者主动补上。请求进入时要不要记录入口信息，调用外部服务时要不要记录目标、耗时和返回状态，数据库查询要不要记慢 SQL，这些都属于埋点设计。

埋点不是随便多打几行日志。真正有价值的埋点，往往集中在几个位置：请求入口、外部依赖、数据库访问、缓存读写、消息生产消费、状态变更和异常分支。抓住这些位置，才能让运行时行为变得可追。

## Request ID and Correlation ID

Request ID 和 Correlation ID 常常一起出现，是因为它们都在解决“上下文怎么串起来”这个问题，但关注面稍有不同。

Request ID 更像一次请求在当前链路里的唯一标识。一个 HTTP 请求进入系统之后，沿路的日志、追踪和部分指标标签都可以带上它，这样查日志时就能把同一次请求关联起来。

Correlation ID 往往更强调跨边界关联。比如一次下单动作会拆成同步请求、异步消息、库存扣减和通知发送，这些步骤未必都在同一条同步链路里，但它们可能都属于同一个业务动作。Correlation ID 就适合把这种更宽的业务上下文串起来。

## 为什么上下文传递是关键

很多线上问题最后查不下去，不是因为没有日志，而是因为上下文断了。入口有一条日志，下游服务也有一条日志，但两边没有共同标识，只能靠时间猜。只要并发一高，这种猜测就很不可靠。

上下文传递做得好，日志、追踪和监控之间就能互相跳转：从一条告警可以找到异常请求，再从请求标识找到整条链路，最后落到具体服务的错误日志。这才是分布式场景下可观测性的真正价值。
