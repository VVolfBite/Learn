# 可观测性基础 - 总览

系统上线之前，人对它的理解主要来自代码、设计图和测试；系统上线之后，人对它的理解主要来自运行时暴露出来的信息。可观测性就是这层运行时理解能力。它回答的不是“系统有没有在跑”，而是“系统一旦异常，我们能不能从外部现象追回内部状态”。

## Observability Basics

很多人会把 observability 和 monitoring 混在一起。两者有关，但不是一回事。监控更像预先定义好的观察面：CPU 高不高、错误率有没有超过阈值、接口延迟有没有升高。可观测性更进一步，它关心的是当系统出现预料外的问题时，现有数据能不能支持继续向下分析。

所以可观测性不是某一个产品功能，而是一种整体能力。它要求系统留下足够多、足够连贯的线索，让人能从“出问题了”继续走到“问题发生在哪，为什么发生，影响到了谁”。

## Monitoring

监控是可观测性的一部分。它偏向持续观察，适合回答“系统现在稳不稳”“某个指标有没有越线”“趋势是上升还是下降”。

如果说可观测性是一套观察系统的方法，监控就是其中最先被搭起来的一层。它把一些关键量长期收集起来，再通过面板和告警把变化暴露给人。没有监控，问题常常是在用户投诉之后才被发现；只有监控，没有足够细的上下文信息，又会卡在“知道出事了，但不知道为什么”。

## Telemetry

Telemetry 可以直接理解成遥测数据。应用、容器、网关、数据库和各种中间件在运行时都会持续产生数据，这些数据被采集、传输、聚合，再送到不同的存储或分析系统里。日志、指标和链路追踪，本质上都是遥测数据的不同形态。

把 telemetry 放在总览里讲，是为了先建立一个整体视角：日志不是孤立文件，指标也不是孤立图表，它们都是系统运行时向外暴露的信息。后端写代码时，其实也在决定将来会产生什么样的遥测信息。

## 可观测性为什么不是后期补丁

很多团队一开始把可观测性当成上线后再补的事情，于是代码能跑就先跑，异常信息写得很随意，请求上下文也不传，关键依赖没有埋点，最后只剩一堆零碎日志。真正出了问题，排障就会变成盲查。

可观测性更像设计的一部分。接口入口要不要打请求标识，数据库访问要不要记录耗时，下游调用失败时要不要留下目标服务、错误码和重试次数，这些看起来像实现细节，实际上都决定了系统将来是否可被理解。

## 从后端视角看可观测性

后端最常遇到的问题不是“有没有数据”，而是“数据能不能连起来”。一次接口超时，往往不是单点事件，而是入口服务、认证逻辑、缓存、数据库、外部 RPC、消息消费链路共同作用的结果。单看某一处局部日志，经常不够。

所以后端学习可观测性，重点不是先背工具，而是先理解三类信息各自擅长什么：日志适合看事件，指标适合看趋势，链路追踪适合看一条请求的跨服务路径。它们合在一起，才接近真正可用的运行时观察面。
