## 为什么需要版本控制

这一节看起来往往容易变成一串要点，其实先理解“为什么需要版本控制”在软件工程基础里的作用会更顺。后面的内容可以理解成围绕这个问题展开的几种常见情况。

**没有版本控制的痛苦**：
- 不知道谁改了什么、为什么改
- 改错了不知道怎么回滚
- 多人同时改代码，互相覆盖
- 不敢删代码，只能注释掉

**有版本控制的好处**：
- 记录每次修改，知道谁改了什么
- 改错了可以回滚
- 多人协作，自动合并
- 可以放心删代码，需要时可以找回

**版本控制是团队协作的基础。**

## 分支协作模型

### Git Flow

读到“Git Flow”时，先别急着记结论。更稳的方式是先弄清它在“软件工程基础 - 版本控制与协作”这张卡片里到底想解决什么问题，这样后面的分类和判断就不会显得太散。

**分支类型**：
- `master`：生产环境代码
- `develop`：开发环境代码
- `feature/xxx`：功能分支
- `hotfix/xxx`：修复分支

**适用场景**：定期发布（如每月发布一次）

**优点**：流程清晰  
**缺点**：分支多，复杂

### GitHub Flow

这里真正要说明的是“GitHub Flow”为什么值得单独拿出来讲。把它放回“软件工程基础 - 版本控制与协作”这张卡片的上下文里看，后面的分点会更容易读顺。

**分支类型**：
- `main`：主分支
- `feature/xxx`：功能分支

**流程**：
1. 从 `main` 创建 `feature/xxx`
2. 开发完成后提交 Pull Request
3. Code Review 通过后合并到 `main`
4. 自动部署到生产环境

**适用场景**：持续交付（如每天发布多次）

**优点**：简单  
**缺点**：不适合定期发布

### Trunk-Based Development

**分支类型**：
- `main`：主分支
- 短期功能分支（1-2 天）

**适用场景**：持续集成，高度自动化测试

**优点**：最简单  
**缺点**：需要 Feature Toggle，需要高度自动化测试

## 提交规范

### 为什么需要提交规范

- 提交信息清晰，容易理解
- 方便查找历史记录
- 方便生成 Changelog

### 提交信息格式

这一节看起来往往容易变成一串要点，其实先理解“提交信息格式”在软件工程基础里的作用会更顺。后面的内容可以理解成围绕这个问题展开的几种常见情况。

```
<type>(<scope>): <subject>
```

**type**：
- `feat`：新功能
- `fix`：修复 bug
- `docs`：文档
- `refactor`：重构
- `test`：测试

**示例**：
```
feat(order): 添加订单支付功能
fix(user): 修复用户登录失败的问题
refactor(payment): 重构支付模块
```

### 提交频率

- 小步提交，每完成一个小功能就提交
- 不要一次提交太多代码
- 每次提交不超过 500 行

## 合并冲突

### 什么是合并冲突

多人同时修改同一个文件的同一行代码，Git 无法自动合并。

### 如何避免

- 及时拉取最新代码
- 小步提交，减少冲突范围
- 沟通协作，避免同时修改同一个文件

### 如何解决

读到“如何解决”时，先别急着记结论。更稳的方式是先弄清它在“软件工程基础 - 版本控制与协作”这张卡片里到底想解决什么问题，这样后面的分类和判断就不会显得太散。

1. 拉取最新代码：`git pull`
2. 解决冲突：手动编辑冲突文件
3. 标记已解决：`git add`
4. 提交：`git commit`

## Code Review

### 为什么需要 Code Review

这一节主要把“为什么需要 Code Review”放回“软件工程基础 - 版本控制与协作”这张卡片的主线里看。先抓住它在整个软件工程基础知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

- 发现 bug
- 提高代码质量
- 知识共享
- 统一代码风格

**Code Review 不是找茬，而是帮助提高代码质量。**

### Code Review 看什么

这里真正要说明的是“Code Review 看什么”为什么值得单独拿出来讲。把它放回“软件工程基础 - 版本控制与协作”这张卡片的上下文里看，后面的分点会更容易读顺。

**功能正确性**：
- 是否实现了需求
- 是否有 bug
- 边界条件是否处理

**代码质量**：
- 命名是否清晰
- 函数是否短小
- 是否有重复代码
- 是否符合设计原则

**性能**：
- 是否有性能问题
- 是否有慢查询
- 是否有 N+1 查询

**安全**：
- 是否有安全漏洞
- 是否有 SQL 注入
- 是否有 XSS 攻击

**测试**：
- 是否有测试
- 测试是否充分

### Code Review 流程

读到“Code Review 流程”时，先别急着记结论。更稳的方式是先弄清它在“软件工程基础 - 版本控制与协作”这张卡片里到底想解决什么问题，这样后面的分类和判断就不会显得太散。

1. 提交 Pull Request
2. 自动运行测试
3. 指定 Reviewer
4. Reviewer 审查代码
5. 提出修改意见
6. 作者修改代码
7. Reviewer 再次审查
8. 通过后合并

### Code Review 最佳实践

这一节看起来往往容易变成一串要点，其实先理解“Code Review 最佳实践”在软件工程基础里的作用会更顺。后面的内容可以理解成围绕这个问题展开的几种常见情况。

**小步提交**：
- 每次 PR 不超过 500 行
- 容易 Review
- 容易发现问题

**及时 Review**：
- 不要拖延
- 尽快给出反馈

**友好沟通**：
- 不要指责
- 提出建设性意见
- 说明为什么要改

**自动化检查**：
- 使用 Lint 工具
- 使用自动化测试
- 减少人工检查

## 协作最佳实践

### 1. 保持主分支稳定

- 主分支永远是可发布的
- 不要直接在主分支上开发
- 合并前要经过测试和 Review

### 2. 功能分支短期存在

- 功能分支不要存在太久
- 及时合并到主分支
- 避免分支偏离太远

### 3. 及时同步代码

- 每天至少拉取一次最新代码
- 避免冲突
- 及时发现问题

### 4. 提交信息清晰

- 说明做了什么
- 说明为什么做
- 关联 issue

### 5. Code Review 认真

- 不要走形式
- 认真看代码
- 提出有价值的意见
