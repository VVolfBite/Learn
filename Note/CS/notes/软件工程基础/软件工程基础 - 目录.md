## 这个模块是什么

这不是项目管理教材，不是测试教程，也不是 Git 手册。

这是**后端开发工程认知笔记**，帮你理解：
- 为什么要这样写代码
- 为什么要做测试
- 为什么要重构
- 为什么要协作规范

面试时，面试官不会问你"敏捷开发有几个阶段"，但会问你"你怎么保证代码质量"、"你怎么处理技术债"、"你怎么做代码评审"。

## 主线

软件工程的核心是**把代码写好、改好、测好、交付好**。

### 第一层：理解全局
**软件开发生命周期**
- 从需求到上线的完整流程
- 各阶段之间的关系
- 后端开发在其中的位置

### 第二层：日常开发
**需求、设计与开发流程**
- 需求澄清：理解要做什么
- 接口设计：定义怎么做
- 开发实现：写代码
- 联调上线：交付

### 第三层：代码质量
**代码质量与可维护性**
- 可读性：别人能看懂
- 可维护性：能改得动
- 可扩展性：能加功能
- 可测试性：能写测试
- 高内聚低耦合：模块划分

**设计原则**
- SOLID 原则
- 单一职责、开闭原则、依赖倒置等
- 不是背诵，而是理解为什么

### 第四层：持续改进
**重构与技术债**
- 为什么要重构
- 什么是技术债
- 什么时候还债
- 重构 vs 重写

**测试基础**
- 单元测试、集成测试、回归测试
- 为什么测试重要
- 测试覆盖率不是唯一目标

### 第五层：团队协作
**版本控制与协作**
- Git 分支协作
- 提交规范
- 合并冲突
- Code Review

**文档、评审与发布**
- 为什么要写文档
- 代码评审看什么
- 发布与回滚
- 变更风险控制

### 第六层：综合判断
**高频判断**
- 可读性 vs 可维护性
- 重构 vs 重写
- 单元测试 vs 集成测试
- 设计原则不是越多越好
- 覆盖率高不等于测试有效

## 核心理念

### 1. 工程不是流程
不要背诵"需求分析 → 设计 → 开发 → 测试 → 上线"这种流程图。

要理解：
- 为什么需要需求澄清（避免做错）
- 为什么需要设计（避免改不动）
- 为什么需要测试（避免出 bug）
- 为什么需要评审（避免埋坑）

### 2. 质量不是检查出来的
代码质量不是靠测试检查出来的，而是靠**写好**的。

测试只是最后一道防线，真正的质量来自：
- 清晰的需求理解
- 合理的设计
- 可读的代码
- 及时的重构

### 3. 协作不是制度
Git 分支、Code Review、提交规范不是为了管理，而是为了**降低协作成本**。

没有规范的团队，每次合并都是灾难。

### 4. 技术债不是罪恶
技术债是工程权衡的结果，不是一定不能欠。

关键是：
- 知道欠了什么债
- 知道什么时候还
- 不要让债务失控

## 面试中的软件工程

### 常见问题
1. 你怎么保证代码质量？
2. 你怎么处理技术债？
3. 你怎么做代码评审？
4. 你怎么做测试？
5. 你怎么协作开发？
6. 你遇到过什么技术难题，怎么解决的？
7. 你重构过代码吗？为什么重构？
8. 你怎么理解高内聚低耦合？
9. 你怎么理解单一职责原则？
10. 你怎么平衡开发速度和代码质量？

### 回答思路
不要背诵理论，要结合实际：
- 说清楚**为什么**这样做
- 举具体的**例子**
- 说明**权衡**和**取舍**

例如：
- 不要说"我会写单元测试"
- 要说"我会对核心业务逻辑写单元测试，因为这部分最容易出 bug，而且改动频繁。对于简单的 CRUD，我会写集成测试，因为单元测试的收益不高"
