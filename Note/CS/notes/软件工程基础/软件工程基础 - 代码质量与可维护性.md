
# 软件工程基础 - 代码质量与可维护性

## 定位

不是代码规范手册，不是最佳实践大全。

这是**代码质量的工程直觉**，帮你理解：
- 为什么要写可读的代码
- 为什么要写可维护的代码
- 如何判断代码质量好坏

## 代码质量的五个维度

### 1. 可读性：别人能看懂

**核心**：代码写一次，读无数次。

**关键点**：
- 命名清晰：`daysSinceCreation` 而不是 `d`
- 函数短小：一个函数只做一件事，不超过 20 行
- 注释准确：复杂逻辑要注释，简单逻辑不要注释

**面试表达**：
"可读性是指别人能看懂代码。主要通过清晰的命名、短小的函数、准确的注释来实现。比如变量名用 `username` 而不是 `s`，函数名用 `validateUser` 而不是 `process`。"

### 2. 可维护性：能改得动

**核心**：代码要能改，改了不出错。

**关键点**：
- 单一职责：一个类只负责一件事
- 避免重复：DRY 原则（Don't Repeat Yourself）
- 避免魔法数字：用常量代替硬编码

**面试表达**：
"可维护性是指能改得动代码。主要通过单一职责、避免重复、避免魔法数字来实现。比如一个类只负责用户业务逻辑，不负责数据库操作和邮件发送。"

### 3. 可扩展性：能加功能

**核心**：加新功能不影响旧功能。

**关键点**：
- 开闭原则：对扩展开放，对修改关闭
- 依赖倒置：依赖抽象，不依赖具体实现
- 接口隔离：接口职责单一

**面试表达**：
"可扩展性是指能加功能。主要通过开闭原则和依赖倒置来实现。比如支付方式，定义支付接口，每种支付方式实现这个接口，加新支付方式时只需要加新的实现类。"

### 4. 可测试性：能写测试

**核心**：代码要能测试，测试要容易写。

**关键点**：
- 依赖注入：通过构造函数或 setter 注入依赖
- 避免静态方法：静态方法难以 Mock
- 避免全局状态：全局状态难以隔离

**面试表达**：
"可测试性是指能写测试。主要通过依赖注入来实现。比如 OrderService 依赖 OrderDao，通过构造函数注入，测试时可以注入 Mock 对象。"

### 5. 高内聚低耦合：模块清晰

**高内聚**：模块内部联系紧密，只做一件事。

**低耦合**：模块之间依赖少，修改一个模块不影响其他模块。

**面试表达**：
"高内聚是指模块内部联系紧密，只做一件事。低耦合是指模块之间依赖少。比如 UserService 只负责用户业务逻辑，不依赖 OrderService，这样修改 UserService 不会影响 OrderService。"

## 代码质量的实践

### 命名

这一节看起来往往容易变成一串要点，其实先理解“命名”在软件工程基础里的作用会更顺。后面的内容可以理解成围绕这个问题展开的几种常见情况。

**变量命名**：
- 有意义：`username` 而不是 `s`
- 完整单词：`daysSinceCreation` 而不是 `d`
- 布尔变量：`isValid`、`hasPermission`、`canDelete`

**函数命名**：
- 动词开头：`validateUser`、`cancelOrder`、`sendEmail`
- 准确表达：不要用 `process`、`handle` 等模糊的词

**类命名**：
- 名词：`UserService`、`OrderRepository`
- 不要用 `Manager`、`Helper`、`Util` 等模糊的词

### 函数

读到“函数”时，先别急着记结论。更稳的方式是先弄清它在“软件工程基础 - 代码质量与可维护性”这张卡片里到底想解决什么问题，这样后面的分类和判断就不会显得太散。

**函数要短**：
- 一个函数只做一件事
- 不超过 20 行（建议）
- 如果太长，拆分成多个小函数

**参数要少**：
- 不超过 3 个（建议）
- 如果太多，封装成对象

**避免副作用**：
- 函数名说做什么，就只做什么
- 不要在函数里偷偷做其他事情

### 注释

这一节主要把“注释”放回“软件工程基础 - 代码质量与可维护性”这张卡片的主线里看。先抓住它在整个软件工程基础知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

**什么时候写注释**：
- 复杂的业务逻辑
- 不明显的算法
- 临时的 workaround

**什么时候不写注释**：
- 代码本身就很清晰
- 注释和代码重复

**好的注释**：
```java
// 这里使用缓存，因为用户信息查询频繁
User user = userCache.get(userId);
```

**不好的注释**：
```java
// 获取用户
User user = userDao.getById(userId);
```

### 避免重复

**DRY 原则**：Don't Repeat Yourself

**重复的坏处**：
- 修改一处，要改多处
- 容易遗漏，导致 bug

**如何避免**：
- 提取公共函数
- 提取公共类
- 使用继承或组合

### 避免魔法数字

**魔法数字**：代码中直接写的数字，不知道什么意思。

**不好的代码**：
```java
if (user.getAge() > 18) { ... }
if (order.getStatus() == 2) { ... }
```

**好的代码**：
```java
private static final int ADULT_AGE = 18;
private static final int ORDER_STATUS_PAID = 2;

if (user.getAge() > ADULT_AGE) { ... }
if (order.getStatus() == ORDER_STATUS_PAID) { ... }
```

## 代码质量的权衡

### 可读性 vs 性能

**大部分情况**：先保证可读性，再优化性能。

**例外**：性能要求极高的场景（如高频交易），可以牺牲一些可读性。

### 简单 vs 灵活

**大部分情况**：先保持简单，等需求明确后再增加灵活性。

**例外**：需求变化频繁的场景，可以提前增加灵活性。

### 抽象 vs 具体

**大部分情况**：先具体，再抽象。

**例外**：依赖不稳定的场景，可以提前抽象。

## 面试重点

### 什么是代码质量？
可读性、可维护性、可扩展性、可测试性、高内聚低耦合

### 什么是可读性？
别人能看懂代码，主要通过清晰的命名、短小的函数、准确的注释

### 什么是可维护性？
能改得动代码，主要通过单一职责、避免重复、避免魔法数字

### 什么是可扩展性？
能加功能，主要通过开闭原则和依赖倒置

### 什么是可测试性？
能写测试，主要通过依赖注入

### 什么是高内聚低耦合？
高内聚是模块内部联系紧密，低耦合是模块之间依赖少

### 如何提高代码质量？
清晰的命名、短小的函数、单一职责、避免重复、避免魔法数字、依赖注入

### 可读性和可维护性的区别？
可读性是局部（别人能看懂），可维护性是整体（能改得动）

### 你在项目中如何保证代码质量？
举具体例子，说明如何命名、如何拆分函数、如何避免重复
