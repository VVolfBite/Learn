
# 软件工程基础 - 文档、评审与发布

## 定位

不是文档模板，不是流程手册。

这是**工程交付的工程直觉**，帮你理解：
- 为什么要写文档
- 为什么要做评审
- 为什么要灰度发布

## 为什么要写文档

读到“为什么要写文档”时，先别急着记结论。更稳的方式是先弄清它在“软件工程基础 - 文档、评审与发布”这张卡片里到底想解决什么问题，这样后面的分类和判断就不会显得太散。

**代码不能表达的**：
- 为什么这样设计
- 为什么不那样设计
- 有哪些坑
- 如何使用

**文档的价值**：
- 新人上手快
- 减少沟通成本
- 避免重复踩坑
- 方便维护

**好的代码是最好的文档，但有些东西代码表达不了。**

**什么时候不写注释**：

## 代码评审

### 评审的目的

这一节主要把“评审的目的”放回“软件工程基础 - 文档、评审与发布”这张卡片的主线里看。先抓住它在整个软件工程基础知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

- 发现 bug
- 提高代码质量
- 知识共享
- 统一代码风格

**评审不是找茬，而是帮助提高代码质量。**

### 评审什么

这里真正要说明的是“评审什么”为什么值得单独拿出来讲。把它放回“软件工程基础 - 文档、评审与发布”这张卡片的上下文里看，后面的分点会更容易读顺。

**功能正确性**：
- 是否实现了需求
- 是否有 bug
- 边界条件是否处理

**代码质量**：
- 命名是否清晰
- 函数是否短小
- 是否有重复代码
- 是否符合设计原则

**性能**：
- 是否有性能问题
- 是否有慢查询
- 是否有 N+1 查询

**安全**：
- 是否有安全漏洞
- 是否有 SQL 注入
- 是否有 XSS 攻击

**测试**：
- 是否有测试
- 测试是否充分

### 评审流程

这里真正要说明的是“评审流程”为什么值得单独拿出来讲。把它放回“软件工程基础 - 文档、评审与发布”这张卡片的上下文里看，后面的分点会更容易读顺。

1. 提交 Pull Request
2. 自动运行测试
3. 指定 Reviewer
4. Reviewer 审查代码
5. 提出修改意见
6. 作者修改代码
7. Reviewer 再次审查
8. 通过后合并

### 评审最佳实践

**小步提交**：每次 PR 不超过 500 行

**及时评审**：不要拖延，尽快给出反馈

**友好沟通**：不要指责，提出建设性意见，说明为什么要改

**自动化检查**：使用 Lint 工具、使用自动化测试、减少人工检查

## 发布与回滚

### 为什么需要规范发布

- 降低风险
- 快速回滚
- 可追溯

### 发布流程

这一节主要把“发布流程”放回“软件工程基础 - 文档、评审与发布”这张卡片的主线里看。先抓住它在整个软件工程基础知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

1. 代码评审：Code Review 通过、测试通过
2. 合并代码：合并到主分支、打 tag
3. 构建部署：自动构建、自动部署到测试环境
4. 测试验证：冒烟测试、回归测试
5. 灰度发布：先发布小流量（1%、5%、10%）
6. 全量发布：发布到所有机器
7. 监控告警：接口成功率、接口耗时、错误日志

### 灰度发布

这一节看起来往往容易变成一串要点，其实先理解“灰度发布”在软件工程基础里的作用会更顺。后面的内容可以理解成围绕这个问题展开的几种常见情况。

**为什么需要灰度发布**：
- 降低风险
- 快速发现问题
- 快速回滚

**灰度策略**：
- 按流量：1% → 5% → 10% → 50% → 100%
- 按用户：内部用户 → 白名单用户 → 所有用户
- 按地区：某个城市 → 某个省 → 全国

**灰度时间**：
- 每个阶段观察 30 分钟到 1 小时
- 无问题后进入下一阶段

### 回滚

这一节主要把“回滚”放回“软件工程基础 - 文档、评审与发布”这张卡片的主线里看。先抓住它在整个软件工程基础知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

**什么时候回滚**：
- 出现严重 bug
- 接口成功率下降
- 接口耗时增加
- 错误日志增多

**如何回滚**：
- 回滚到上一个版本
- 重新部署

**回滚时间**：
- 5 分钟内完成回滚

### 变更风险控制

**小步发布**：每次发布改动少，容易定位问题

**自动化测试**：单元测试、集成测试、回归测试

**灰度发布**：先小流量，观察无问题后放量

**监控告警**：实时监控，异常告警

**回滚方案**：准备回滚方案，快速回滚
