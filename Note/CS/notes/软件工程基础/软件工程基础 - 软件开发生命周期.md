## 生命周期的本质

软件开发生命周期不是背诵的流程图，而是**从想法到产品的完整过程**。

理解生命周期的意义：
- 知道自己在做什么阶段的事
- 知道上下游是谁，需要什么
- 知道为什么要这样做

## 典型阶段

### 1. 需求分析
**做什么**：理解要解决什么问题

**后端视角**：
- 产品说"要做一个用户登录功能"
- 你需要问清楚：支持哪些登录方式、登录态怎么维护、需要支持多端登录吗、安全要求是什么

**如何做好**：
- 多问"为什么"和"如果...怎么办"
- 画出核心流程图
- 确认输入输出和边界条件

### 2. 设计
**做什么**：决定怎么实现

**后端视角**：
- 接口设计：URL、参数、返回值
- 数据库设计：表结构、索引、关系
- 架构设计：分层、模块划分
- 技术选型：用什么框架、中间件

**如何做好**：
- 先画出核心流程和数据流
- 定义清晰的接口和数据结构
- 考虑常见的扩展场景
- 不要过度设计

### 3. 开发
**做什么**：写代码实现功能

**如何做好**：
- 遵循代码规范
- 及时提交，提交信息清晰
- 核心逻辑写单元测试
- 复杂逻辑加注释

### 4. 测试
**做什么**：验证功能是否正确

**后端视角**：
- 单元测试：测试单个函数或类
- 集成测试：测试接口和数据库交互
- 联调测试：和前端联调
- 回归测试：确保没有破坏已有功能

### 5. 部署上线
**做什么**：把代码发布到生产环境

**如何做好**：
- 代码评审通过后再合并
- 先灰度小流量，观察无问题再全量
- 配置监控和告警
- 准备回滚方案

### 6. 维护迭代
**做什么**：修复 bug，优化性能，迭代功能

**如何做好**：
- 定期重构，不要让技术债失控
- 监控性能，及时优化
- 修改代码前先写测试
- 保持代码质量

## 各阶段的关系

### 不是瀑布流
传统的瀑布模型：需求 → 设计 → 开发 → 测试 → 上线，每个阶段完成后才进入下一个阶段。

**问题**：
- 需求变化快，等到开发完成需求可能已经变了
- 测试阶段才发现设计问题，成本很高
- 上线后才发现需求理解错了

### 是迭代循环
现代开发模式（敏捷）：小步快跑，快速迭代。

**特点**：
- 每个迭代都是一个完整的生命周期
- 快速交付，快速反馈
- 持续改进

### 各阶段的依赖

这一节主要把“各阶段的依赖”放回“软件工程基础 - 软件开发生命周期”这张卡片的主线里看。先抓住它在整个软件工程基础知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

- 测试不充分，上线就有风险
- 上线没监控，出问题不知道

## 后端开发的位置

### 你的上游
- **产品**：提供需求
- **前端**：定义接口
- **架构师**：提供技术方案

### 你的下游
- **测试**：验证功能
- **运维**：部署上线
- **用户**：使用产品

- 维护代码，持续优化

## 常见开发模式

### 瀑布模型
- 严格按阶段进行，每个阶段完成后才进入下一个
- 适合需求稳定、变化少的项目
- 问题：不灵活，反馈慢

### 敏捷开发

这一节主要把“敏捷开发”放回“软件工程基础 - 软件开发生命周期”这张卡片的主线里看。先抓住它在整个软件工程基础知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

- 小步快跑，快速迭代
- 每个迭代都是一个完整的生命周期
- 适合需求变化快的项目
- 问题：需要团队配合好

### DevOps

这里真正要说明的是“DevOps”为什么值得单独拿出来讲。把它放回“软件工程基础 - 软件开发生命周期”这张卡片的上下文里看，后面的分点会更容易读顺。

- 开发和运维一体化
- 自动化构建、测试、部署
- 持续集成、持续交付
- 适合快速迭代的项目

2. 典型阶段：需求分析、设计、开发、测试、部署上线、维护迭代
