## 索引

索引是一种数据结构，用于快速查找数据。就像书的目录，通过目录可以快速找到某一章，而不用从头翻到尾。

本质是一种用空间换时间的策略，通过额外的存储空间和维护成本，换取查询速度的提升。

索引可以大幅提高查询速度，加速排序和分组操作，而且唯一索引可以保证数据唯一性；但是占用额外的存储空间，也会降低写操作的速度（INSERT、UPDATE、DELETE 都需要维护索引），整体来说索引过多会增加查询优化器的选择成本。

### 索引加速查询

没有需要全表扫描，逐行比对。

```sql
SELECT * FROM user WHERE id = 100;
-- 需要扫描所有行，时间复杂度 O(n)
```

有索引则通过索引与查询算法快速定位，比如二分查找。

```sql
-- 在 id 上建立索引
CREATE INDEX idx_id ON user(id);

SELECT * FROM user WHERE id = 100;
-- 通过索引快速定位，时间复杂度 O(log n)
```

### 索引的数据结构

MySQL InnoDB 使用 **B+ 树**作为索引的核心思想是：通过**平衡多路树结构**保证查询路径尽可能短，从而实现快速查找；内部节点只存储索引信息用于快速定位，叶子节点存储实际数据，并通过链表串联支持范围查询和顺序遍历。这样，B+ 树既能高效定位单条记录，又能高效支持范围查询和排序操作，同时保持树的平衡以保证插入和删除操作的性能稳定。B+树有以下特点：

1. **所有数据都在叶子节点**：非叶子节点只存索引，不存数据
2. **叶子节点有序**：叶子节点通过指针连接，支持范围查询
3. **树矮**：一个节点可以存多个索引，减少磁盘 IO 次数

```
        [10, 20, 30]           ← 非叶子节点，只存索引
       /    |    |    \
     /      |    |      \
[1,5,8] [10,15] [20,25] [30,35] ← 叶子节点，存数据
   ↔       ↔       ↔       ↔     ← 叶子节点通过指针连接
```

**对比其他数据结构**：

| 数据结构 | 查询复杂度 | 为什么不用 |
|---------|-----------|-----------|
| 数组 | O(n) | 太慢 |
| 二叉搜索树 | O(log n) | 可能退化成链表 |
| 平衡二叉树 | O(log n) | 树太高，磁盘 IO 次数多 |
| 哈希表 | O(1) | 不支持范围查询 |
| B 树 | O(log n) | 非叶子节点也存数据，浪费空间 |
| B+ 树 | O(log n) | 正确 树矮、支持范围查询、叶子节点有序 |

### 聚簇索引与非聚簇索引

#### 聚簇索引

​	在聚簇索引中，索引和数据存储在一起，叶子节点存储的是完整的数据行。由于数据按照索引顺序在物理上存储，一张表只能有一个聚簇索引（通常是主键），查询时无需回表即可直接获取数据。在 InnoDB 中，如果表有主键，则主键就是聚簇索引；如果没有主键，会选择第一个非空唯一索引；若都不存在，InnoDB 会自动生成一个隐藏的 `row_id` 作为聚簇索引。

#### 非聚簇索引（二级索引）

​	非聚簇索引（也称二级索引）将索引和数据分开存储，叶子节点存储的是主键值而非完整数据行。通过非聚簇索引查询时，需要先通过索引找到对应的主键，然后再回表通过主键获取实际数据。一张表可以拥有多个非聚簇索引，这种设计使得索引灵活性更高，但查询时可能增加一次回表操作。

#### 聚簇索引 vs 非聚簇索引

| 维度 | 聚簇索引 | 非聚簇索引 |
|------|---------|-----------|
| 叶子节点存储 | 完整数据行 | 主键值 |
| 数量 | 一张表只有一个 | 一张表可以有多个 |
| 是否需要回表 | 不需要 | 需要 |
| 查询速度 | 快 | 慢（需要回表） |

#### 回表

通过非聚簇索引查询时，需要两次查询：
1. 在非聚簇索引中找到主键值
2. 通过主键值在聚簇索引中找到完整数据

**例子**：

```sql
-- 假设在 name 字段上有非聚簇索引
SELECT * FROM user WHERE name = 'Alice';

-- 查询过程：
-- 1. 在 name 索引中找到 name='Alice' 的主键 id=100
-- 2. 在主键索引中找到 id=100 的完整数据
```

**回表的代价**：需要两次 B+ 树查询，性能较差。

### 覆盖索引

如果查询的列都在索引中，就不需要回表，这叫覆盖索引。其避免了回表，查询速度快。

**例子**：

```sql
-- 假设有索引 idx_name_age(name, age)
SELECT name, age FROM user WHERE name = 'Alice';
-- 不需要回表，因为 name 和 age 都在索引中
```

### 联合索引

在多个列上建立的索引。当执行 `CREATE INDEX idx_name_age ON user(name, age);` 时，MySQL 会为每一行取出索引列值和主键（或 row_id）组成一个索引元组，然后将这些元组按照顺序插入 B+ 树中，构建出联合索引结构。B+ 树的内部节点用于快速定位，叶子节点存储索引列值加主键，方便后续回表获取完整数据，从而加速按索引列的查询操作。

```sql
CREATE INDEX idx_name_age ON user(name, age);
```

#### 最左前缀原则

联合索引遵循最左前缀原则，即查询条件必须从索引的最左列开始，且不能跳过中间列，只有查询条件中包含索引最左列时才能有效利用索引。比如在索引 `idx_name_age_city(name, age, city)` 上，查询 `WHERE name='Alice'` 或 `WHERE name='Alice' AND age=20` 或 `WHERE name='Alice' AND age=20 AND city='Beijing'` 都能充分利用索引，而查询 `WHERE age=20` 或 `WHERE city='Beijing'` 就无法使用索引，因为跳过了最左列；查询 `WHERE name='Alice' AND city='Beijing'` 时，只能利用 `name` 列的索引，`city` 列则无法直接利用索引加速。

**例子**：

```sql
-- 假设有索引 idx_name_age_city(name, age, city)

-- 能用到索引
SELECT * FROM user WHERE name = 'Alice';
SELECT * FROM user WHERE name = 'Alice' AND age = 20;
SELECT * FROM user WHERE name = 'Alice' AND age = 20 AND city = 'Beijing';

-- 不能用到索引
SELECT * FROM user WHERE age = 20;  -- 跳过了 name
SELECT * FROM user WHERE city = 'Beijing';  -- 跳过了 name 和 age

-- 部分用到索引
SELECT * FROM user WHERE name = 'Alice' AND city = 'Beijing';
-- 只能用到 name 部分，city 用不到（跳过了 age）
```

### 索引失效的常见情况

#### 违反最左前缀原则

```sql
-- 索引 idx_name_age(name, age)
SELECT * FROM user WHERE age = 20;  -- 失效
```

#### 在索引列上使用函数

```sql
SELECT * FROM user WHERE UPPER(name) = 'ALICE';  -- 失效
```

#### 在索引列上进行计算

```sql
SELECT * FROM user WHERE age + 1 = 21;  -- 失效
```

#### 使用不等于（!= 或 <>）

```sql
SELECT * FROM user WHERE name != 'Alice';  -- 可能失效
```

#### 使用 LIKE 以通配符开头

```sql
SELECT * FROM user WHERE name LIKE '%Alice';  -- 失效
SELECT * FROM user WHERE name LIKE 'Alice%';  -- 不失效
```

#### 类型不匹配

```sql
-- name 是 varchar 类型
SELECT * FROM user WHERE name = 123;  -- 失效（隐式类型转换）
```

#### 使用 OR 连接

```sql
-- 如果 OR 的两边有一个没有索引，整个查询就不走索引
SELECT * FROM user WHERE name = 'Alice' OR age = 20;
-- 如果 age 没有索引，整个查询不走索引
```

### 建立索引的原则

#### 适合建索引

- 频繁作为查询条件的列
- 频繁用于排序和分组的列
- 频繁用于连接的列（JOIN）
- 区分度高的列（值的种类多）

#### 不适合建索引

- 频繁更新的列
- 区分度低的列（如性别，只有男/女两种值）
- 数据量很小的表（全表扫描更快）
- 很少使用的列

### 索引的类型

#### 按功能分类

这一节主要把“按功能分类”放回“数据库原理 - 索引基础”这条主线里看。先抓住它在整个数据库原理知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

- **主键索引**：PRIMARY KEY，唯一且非空
- **唯一索引**：UNIQUE，值唯一但可以为空
- **普通索引**：INDEX，最常用的索引
- **全文索引**：FULLTEXT，用于全文搜索

#### 按列数分类

- **单列索引**：在一个列上建立索引
- **联合索引**：在多个列上建立索引

## 追问

**Q: 索引为什么能加速查询？**
因为索引是有序的数据结构，可以快速定位数据，避免全表扫描。

**Q: 为什么 MySQL 使用 B+ 树而不是哈希表？**
因为哈希表不支持范围查询和排序，而 B+ 树支持。

**Q: 为什么 MySQL 使用 B+ 树而不是 B 树？**
因为 B+ 树的非叶子节点不存数据，可以存更多索引，树更矮，磁盘 IO 次数更少。而且 B+ 树的叶子节点有序连接，支持范围查询。

**Q: 聚簇索引和非聚簇索引的区别？**
聚簇索引的叶子节点存完整数据，非聚簇索引的叶子节点存主键值。聚簇索引不需要回表，非聚簇索引需要。

**Q: 什么是回表？**
通过非聚簇索引查询时，先在非聚簇索引中找到主键，再通过主键在聚簇索引中找到完整数据，这个过程叫回表。

**Q: 什么是覆盖索引？**
查询的列都在索引中，不需要回表，这叫覆盖索引。

**Q: 什么是最左前缀原则？**
联合索引的查询条件必须从最左列开始，且不能跳过中间列。

**Q: 为什么索引列上使用函数会失效？**
因为索引存储的是原始值，使用函数后需要计算，无法利用索引的有序性。

**Q: 为什么 LIKE '%xxx' 会导致索引失效？**
因为通配符在开头，无法利用索引的有序性，必须全表扫描。

**Q: 索引越多越好吗？**
不是。索引会占用空间，降低写操作速度，增加查询优化器的选择成本。

**Q: 如何判断是否需要建索引？**
看该列是否频繁作为查询条件、排序、分组、连接，以及区分度是否高。

**Q: 为什么 MySQL 使用 B+树而不是 B 树？**
B+树的非叶子节点不存数据，只存索引，可以存更多索引，树更矮，磁盘 IO 次数更少。而且 B+树的叶子节点通过指针连接，支持范围查询和顺序遍历。

**Q: 联合索引的最左前缀原则为什么不能跳过中间列？**
因为联合索引是按照从左到右的顺序排序的。就像电话簿先按姓排序，再按名排序。如果你不知道姓，就无法利用这个顺序快速查找。

**Q: 为什么索引列上使用函数会失效？**
因为索引存储的是原始值，使用函数后需要对每一行计算函数结果，无法利用索引的有序性，必须全表扫描。

**Q: 什么情况下即使有索引也会全表扫描？**
当查询条件匹配的行数太多（通常超过表的 20-30%），优化器会认为全表扫描比走索引更快，因为走索引需要大量回表。
