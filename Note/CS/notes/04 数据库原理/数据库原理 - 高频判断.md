## 事务与 ACID

**正确**
- 一致性是 ACID 的核心目标，其他三个特性是手段
- 原子性通过 undo log 实现
- 持久性通过 redo log 实现
- 无并发场景下，只要满足原子性就能保证一致性
- 有并发场景下，需要同时满足原子性和隔离性才能保证一致性

**错误**
- ACID 四个特性是平级关系（实际上一致性是目标，其他是手段）
- 事务提交后还能回滚（提交后就无法回滚了）
- 原子性和一致性是一回事（原子性是操作层面，一致性是数据层面）
- 单事务场景下也需要隔离性（单事务不需要隔离性）

## 并发一致性问题

**正确**
- 脏读是读到未提交的数据
- 不可重复读是同一条数据多次读取值不同
- 幻读是同一范围查询多次读取行数不同
- 不可重复读由 UPDATE 引起，幻读由 INSERT/DELETE 引起
- 丢失更新在所有隔离级别下都可能发生

**错误**
- 脏读和不可重复读是一回事（脏读是读到未提交的，不可重复读是读到已提交的）
- 幻读只由 INSERT 引起（DELETE 也会导致幻读）
- 不可重复读和幻读没有区别（前者关注值变化，后者关注行数变化）
- 幻读一定是 SELECT 引起的（UPDATE/DELETE 也可能遇到幻读）

## 锁

**正确**
- 共享锁用于读，排他锁用于写
- 读读不互斥，读写互斥，写写互斥
- 行锁粒度小、并发高、开销大
- 表锁粒度大、并发低、开销小
- 意向锁是表锁，用于快速判断表中是否有行被锁
- 乐观锁适合读多写少，悲观锁适合写多读少
- 两段锁协议能保证可串行化，但可能导致死锁

**错误**
- 共享锁和排他锁可以同时持有（它们是互斥的）
- 意向锁是行锁（意向锁是表锁）
- 乐观锁不需要版本号（乐观锁通常通过版本号或时间戳实现）
- 死锁只能通过超时解决（还可以通过检测等待图、按固定顺序访问资源等方式）
- 一级封锁协议能解决脏读（一级只能解决丢失更新，二级才能解决脏读）

## 隔离级别

**正确**
- MySQL InnoDB 默认隔离级别是 Repeatable Read
- RC 每次读都生成新的 ReadView
- RR 事务开始时生成 ReadView
- RC 能解决脏读，不能解决不可重复读
- RR 能解决脏读和不可重复读
- Serializable 能解决所有并发问题，但并发性能最差
- 隔离级别越高，数据一致性越好，但并发性能越低

**错误**
- MySQL 默认隔离级别是 Read Committed（实际是 Repeatable Read）
- RR 不能解决幻读（MySQL InnoDB 通过 Next-Key Lock 解决了）
- RC 和 RR 都能解决不可重复读（RC 不能解决）
- 隔离级别能解决丢失更新（丢失更新需要应用层加锁）
- 提高隔离级别一定能提高性能（隔离级别越高，性能越低）

## MVCC

**正确**
- MVCC 通过保存多个版本实现读写不互斥
- MVCC 用于 RC 和 RR 两种隔离级别
- RC 每次 SELECT 生成 ReadView，RR 事务开始时生成 ReadView
- 快照读使用 MVCC，不加锁
- 当前读不使用 MVCC，需要加锁
- MVCC 和锁是配合使用的，不是对立关系
- undo log 既用于回滚，也用于 MVCC

**错误**
- MVCC 能完全解决幻读（只能解决快照读的幻读，不能解决当前读的幻读）
- MVCC 不需要锁（当前读和写操作仍需要锁）
- 所有 SELECT 都是快照读（SELECT FOR UPDATE 是当前读）
- MVCC 在所有隔离级别下都使用（RU 和 Serializable 不使用）
- ReadView 在事务结束时生成（RC 是每次读时生成，RR 是事务开始时生成）

## Next-Key Lock 与幻读

**正确**
- Next-Key Lock = Record Lock + Gap Lock
- 间隙锁用于防止其他事务在间隙中插入数据
- Next-Key Lock 锁定左开右闭区间
- 间隙锁只在 RR 级别及以上存在
- 快照读不会出现幻读（MVCC 保证）
- 当前读可能出现幻读（需要 Next-Key Lock）
- 等值查询如果记录不存在，会退化为 Gap Lock

**错误**
- 间隙锁锁定的是记录本身（间隙锁锁定的是记录之间的间隙）
- RC 级别也有间隙锁（RC 没有间隙锁）
- Next-Key Lock 锁定左闭右开区间（实际是左开右闭）
- 所有 SELECT 都会加 Next-Key Lock（只有当前读才会加）
- MVCC 能解决所有幻读（只能解决快照读的幻读）

## 日志与恢复

**正确 正确**
- undo log 保证原子性，redo log 保证持久性
- undo log 记录旧值，redo log 记录新值
- WAL 是先写日志再写磁盘
- redo log 是顺序写，比随机写快
- checkpoint 用于加快崩溃恢复速度
- binlog 是 Server 层的逻辑日志，redo log 是 InnoDB 的物理日志
- 两阶段提交保证 redo log 和 binlog 的一致性

**错误 错误**
- undo log 保证持久性（undo log 保证原子性，redo log 保证持久性）
- redo log 记录旧值（redo log 记录新值，undo log 记录旧值）
- redo log 和 binlog 是一回事（它们是不同层次的日志，作用不同）
- undo log 会立即删除（需要等所有事务都不需要时才删除）
- WAL 是先写磁盘再写日志（实际是先写日志再写磁盘）

## 索引

**正确 正确**
- 索引能加速查询，但会降低写操作速度
- MySQL InnoDB 使用 B+ 树作为索引结构
- B+ 树的所有数据都在叶子节点
- 聚簇索引的叶子节点存完整数据，非聚簇索引存主键值
- 一张表只能有一个聚簇索引
- 非聚簇索引查询需要回表
- 覆盖索引不需要回表
- 联合索引遵循最左前缀原则
- 在索引列上使用函数会导致索引失效
- LIKE '%xxx' 会导致索引失效

**错误 错误**
- 索引越多越好（索引会占用空间，降低写性能）
- 哈希表比 B+ 树更适合做索引（哈希表不支持范围查询）
- B 树比 B+ 树更适合做索引（B+ 树更矮，支持范围查询）
- 聚簇索引和非聚簇索引可以同时存在多个（聚簇索引只能有一个）
- 所有查询都需要回表（覆盖索引不需要回表）
- 联合索引可以跳过中间列（不能跳过，必须从最左列开始）
- LIKE 'xxx%' 会导致索引失效（不会失效，只有 '%xxx' 才会失效）

## 查询优化

**正确 正确**
- 查询慢的本质是扫描的数据量太多
- EXPLAIN 可以查看 SQL 的执行计划
- type 为 ALL 表示全表扫描，性能最差
- Using filesort 表示需要额外排序
- Using temporary 表示需要临时表
- Using index 表示使用了覆盖索引
- 深分页很慢，因为需要先查出前 N 行
- SELECT * 无法使用覆盖索引
- 子查询可能导致临时表
- UNION ALL 比 UNION 快

**错误 错误**
- EXPLAIN 会实际执行 SQL（EXPLAIN 只是分析执行计划，不会实际执行）
- type 为 index 表示使用了索引，性能很好（index 是全索引扫描，性能不好）
- rows 越大越好（rows 是扫描的行数，越小越好）
- Using where 表示性能差（Using where 只是表示使用了 WHERE 过滤，不一定差）
- 有索引就一定快（索引可能失效，或者回表次数多）

### 一致性 vs 隔离性（核心易混点）

| 维度 | 一致性 | 隔离性 |
|------|--------|--------|
| 关注点 | 数据是否符合约束 | 事务之间的可见性 |
| 层面 | 数据库约束 + 业务规则 | 并发控制 |
| 谁保证 | 数据库 + 应用程序 | 数据库（锁、MVCC） |
| 关系 | 目标 | 手段（保证一致性的手段之一） |

**关键区别**：
- 一致性：转账前后总金额不变（业务规则）
- 隔离性：转账过程中，其他事务看不到中间状态（并发控制）

### 原子性 vs 一致性

| 维度 | 原子性 | 一致性 |
|------|--------|--------|
| 关注点 | 操作是否完整 | 数据是否正确 |
| 层面 | 操作层面 | 数据层面 |
| 保证方式 | undo log | 原子性 + 隔离性 + 持久性 |
| 关系 | 手段 | 目标 |

**关键区别**：
- 原子性：操作要么全做，要么全不做
- 一致性：数据符合完整性约束和业务规则

## 易混辨析

### 脏读 vs 不可重复读
- **脏读**：读到未提交的数据（可能被回滚）
- **不可重复读**：读到已提交的数据（同一事务内多次读取值不同）

### 不可重复读 vs 幻读
- **不可重复读**：同一条数据的值变了（UPDATE）
- **幻读**：数据的行数变了（INSERT/DELETE）

### 共享锁 vs 排他锁
- **共享锁**：读锁，多个事务可以同时持有
- **排他锁**：写锁，只有一个事务可以持有

### 行锁 vs 表锁
- **行锁**：粒度小、并发高、开销大
- **表锁**：粒度大、并发低、开销小

### 乐观锁 vs 悲观锁
- **乐观锁**：不加锁，提交时检查，适合读多写少
- **悲观锁**：操作前加锁，适合写多读少

### 聚簇索引 vs 非聚簇索引
- **聚簇索引**：叶子节点存完整数据，不需要回表
- **非聚簇索引**：叶子节点存主键值，需要回表

### 快照读 vs 当前读
- **快照读**：使用 MVCC，不加锁，读快照
- **当前读**：不使用 MVCC，加锁，读最新数据

### undo log vs redo log
- **undo log**：记录旧值，用于回滚，保证原子性
- **redo log**：记录新值，用于恢复，保证持久性

### RC vs RR
- **RC**：每次读生成 ReadView，能读到其他事务的提交
- **RR**：事务开始时生成 ReadView，读的是快照

### 间隙锁 vs 临键锁
- **间隙锁**：锁定间隙，不包括记录本身
- **临键锁**：锁定间隙 + 记录，左开右闭区间

### binlog vs redo log
- **binlog**：Server 层，逻辑日志，用于主从复制
- **redo log**：InnoDB 层，物理日志，用于崩溃恢复

### 覆盖索引 vs 回表
- **覆盖索引**：查询的列都在索引中，不需要回表
- **回表**：通过非聚簇索引查询，需要回到聚簇索引获取完整数据

### UNION vs UNION ALL
- **UNION**：会去重，需要排序和比较
- **UNION ALL**：不去重，性能更好
