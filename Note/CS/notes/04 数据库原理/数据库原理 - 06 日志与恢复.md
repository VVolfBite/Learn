## 日志

​	数据库需要保证事务的 ACID 特性，其中**原子性**要求操作要么全做，要么全不做，**持久性**要求提交后的修改不能丢失。如果没有日志，事务执行一半崩溃，无法回滚（破坏原子性），事务提交后崩溃，修改可能丢失（破坏持久性）。日志就是为了解决这两个问题。

MySQL InnoDB 有两种关键日志：

- **undo log**：回滚日志，保证原子性
- **redo log**：重做日志，保证持久性

## 回滚日志 Undo Log

记录事务修改前的数据，用于回滚。每次修改数据前，先把旧值写入 undo log，如果事务失败或主动回滚，就根据 undo log 恢复数据。

**例子**：

```sql
START TRANSACTION;
UPDATE account SET balance = 200 WHERE id = 1;  -- 原值 100
-- undo log 记录：id=1, balance=100
ROLLBACK;  -- 根据 undo log 恢复 balance=100
```

### Undo Log 的内容与清理

- 对于 INSERT：记录主键，回滚时删除这条记录
- 对于 DELETE：记录整行数据，回滚时插入这条记录
- 对于 UPDATE：记录修改前的值，回滚时恢复旧值

undo log 不会立即删除，因为可能有其他事务需要读取历史版本。MySQL 有 purge 线程，定期清理不再需要的 undo log（所有事务都不需要的旧版本）。

### Undo Log 与 MVCC

undo log 不仅用于回滚，还用于 MVCC。通过 undo log 中的版本链，事务可以读取到历史版本的数据，实现快照读。

## 重做日志 Redo Log

记录事务修改后的数据，用于崩溃恢复。

如果事务提交后，数据还在内存中没写入磁盘，此时崩溃，数据就丢失了。因此先把修改写入 redo log（顺序写，很快），再慢慢把数据写入磁盘（随机写，较慢）。崩溃后，根据 redo log 重新执行修改。

### WAL 机制

WAL（Write-Ahead Logging）：先写日志，再写磁盘。

**流程**：

1. 事务修改数据，数据在内存中（Buffer Pool）
2. 修改记录写入 redo log（磁盘，顺序写）
3. 事务提交，redo log 持久化
4. 后台线程异步将内存中的数据写入磁盘（随机写）

Redo Log 是顺序写，速度快；而之后的数据页是随机写，可以批量写，提高效率；另外即使崩溃，也能根据 Redo Log 恢复。

### Redo Log 的内容与写入

记录的是数据页的物理修改，例如：在某个数据页的某个偏移量，写入某个值

**例子**：
```
页号: 100, 偏移: 200, 新值: 500
```

**三个阶段**：
1. 事务执行时，redo log 写入内存的 log buffer
2. 事务提交时，log buffer 写入磁盘的 redo log file
3. 后台线程定期将 redo log 刷盘

**参数控制**：`innodb_flush_log_at_trx_commit`
- `0`：每秒刷盘一次（性能最好，但可能丢失 1 秒数据）
- `1`：每次提交都刷盘（最安全，默认值）
- `2`：每次提交写入 OS 缓存，每秒刷盘（折中）

### Undo Log vs Redo Log

| 维度 | undo log | redo log |
|------|----------|----------|
| 作用 | 回滚，保证原子性 | 恢复，保证持久性 |
| 记录内容 | 修改前的数据（逻辑日志） | 修改后的数据（物理日志） |
| 使用时机 | 事务回滚、MVCC | 崩溃恢复 |
| 何时写入 | 修改数据前 | 修改数据后 |
| 何时删除 | 所有事务都不需要时 | checkpoint 后 |

## 检查点 Checkpoint

定期将内存中的脏页（已修改但未写入磁盘的数据页）刷入磁盘，并记录 checkpoint 位置。如果没有 checkpoint，崩溃恢复时需要从头重放所有 redo log，非常慢。然而有了 checkpoint，只需要从最近的 checkpoint 开始重放，大大加快恢复速度。

我们可以简单理解为游戏的存档或者Windows的还原点。

- **严格检查点 sharp checkpoint**：将所有脏页刷入磁盘，数据库通常需要停止服务使用
- **模糊检查点 fuzzy checkpoint**：只刷部分脏页，数据库运行期间使用

## 崩溃恢复流程

1. 找到最近的 Checkpoint
2. 从 Checkpoint 开始，读取 Redo Log
3. 重放 Redo Log 中的修改（redo）
4. 对于未提交的事务，根据 Undo Log 回滚（undo）

**例子**：

```
时间线：
T1: 事务 A 开始
T2: 事务 A 修改数据，写 undo log 和 redo log
T3: checkpoint
T4: 事务 A 提交
T5: 事务 B 开始
T6: 事务 B 修改数据，写 undo log 和 redo log
T7: 崩溃

恢复流程：
1. 从 T3 的 checkpoint 开始
2. 重放 T4 的 redo log（事务 A 已提交，需要恢复）
3. 重放 T6 的 redo log（先恢复，再回滚）
4. 根据 undo log 回滚事务 B（未提交）
```

## BinLog 二进制日志

记录所有修改数据的 SQL 语句，用于主从复制和数据恢复。

### 两阶段提交

为了保证 redo log 和 binlog 的一致性，InnoDB 使用两阶段提交：

**流程**：
1. 写入 redo log，状态为 prepare
2. 写入 binlog
3. 提交事务，redo log 状态改为 commit

### binlog vs redo log

| 维度       | binlog               | redo log               |
| ---------- | -------------------- | ---------------------- |
| 层次       | MySQL Server 层      | InnoDB 引擎层          |
| 作用       | 主从复制、数据恢复   | 崩溃恢复               |
| 记录内容   | 逻辑日志（SQL 语句） | 物理日志（数据页修改） |
| 写入时机   | 事务提交时           | 事务执行时             |
| 是否循环写 | 否，追加写           | 是，循环写             |

## ACID 的实现与保障

**ACID** 包括：

1. **原子性（Atomicity）**
   - 每个事务要么全部执行成功，要么全部回滚
   - **实现机制**：Undo Log
     - 当事务执行失败时，可以通过 Undo Log 回滚已经修改的内容，保证操作原子性
2. **一致性（Consistency）**
   - 事务执行前后，数据必须保持一致性约束
   - 依赖数据库的完整性约束和事务逻辑
3. **隔离性（Isolation）**
   - 多事务并发执行时，各事务的操作互不干扰
   - **实现机制**：
     - **MVCC（多版本并发控制）**
       - 支持不同隔离级别：Read Uncommitted、Read Committed、Repeatable Read
       - 通过版本控制保证读操作看到的数据稳定
     - **锁机制**
       - 包括行锁、间隙锁、Next-Key Lock
       - 进一步保证可串行化（Serializable）隔离级别
4. **持久性（Durability）**
   - 一旦事务提交，修改的数据必须永久保存
   - **实现机制**：
     - **Redo Log**：记录事务操作的重做信息
     - **Binlog**：用于复制和持久化
     - **Checkpoint**：定期刷脏页到磁盘，加快恢复
     - **恢复流程**：服务崩溃时，先通过 checkpoint 找到最近还原点，再通过 Redo Log 重放完成持久化

## 追问

**Q: undo log 和 redo log 的区别？**
undo log 记录旧值用于回滚，保证原子性；redo log 记录新值用于恢复，保证持久性。

**Q: 为什么需要 redo log？**
因为数据修改后可能还在内存中，如果崩溃就丢失了。redo log 先持久化，崩溃后可以根据 redo log 恢复。

**Q: 为什么 redo log 比直接写数据快？**
因为 redo log 是顺序写，而数据页是随机写。顺序写比随机写快很多。

**Q: WAL 是什么？**
Write-Ahead Logging，先写日志再写磁盘。保证即使崩溃，也能根据日志恢复。

**Q: checkpoint 的作用是什么？**
定期将脏页刷入磁盘，加快崩溃恢复速度（只需从最近的 checkpoint 开始重放）。

**Q: undo log 什么时候删除？**
当所有事务都不再需要某个旧版本时，purge 线程会删除对应的 undo log。

**Q: redo log 是循环写的吗？**
是的。redo log 有固定大小，写满后会覆盖旧的日志（已经 checkpoint 的部分）。

**Q: binlog 和 redo log 的区别？**
binlog 是 Server 层的逻辑日志，用于主从复制；redo log 是 InnoDB 的物理日志，用于崩溃恢复。

**Q: 为什么需要两阶段提交？**
保证 redo log 和 binlog 的一致性，避免主从数据不一致。如果只写 redo log 不写 binlog，主库恢复了但从库没有这条记录；如果只写 binlog 不写 redo log，主库崩溃后丢失了但从库有这条记录。

**Q: 如果事务提交后，redo log 写入成功但 binlog 写入失败，会怎样？**
事务会回滚，因为两阶段提交保证了 redo log 和 binlog 要么都成功，要么都失败。具体流程：redo log 先写入 prepare 状态，binlog 写入成功后，redo log 才改为 commit 状态。

**Q: WAL 为什么能提高性能？**
因为 redo log 是顺序写，而数据页是随机写。顺序写比随机写快很多（机械硬盘上差距更明显）。而且可以批量将多个数据页的修改一次性刷盘，减少磁盘 IO。

**Q: redo log 写满了会怎样？**
redo log 是循环写的，写满后会覆盖旧的日志。但覆盖前必须确保对应的数据页已经刷盘（通过 checkpoint）。如果数据页还没刷盘，就需要等待，这时会阻塞写操作。

**Q: undo log 除了回滚还有什么用？**
用于 MVCC，通过 undo log 中的版本链，事务可以读取历史版本的数据。

**Q: 崩溃恢复的流程是什么？**
从最近的 checkpoint 开始，重放 redo log 恢复已提交的事务，根据 undo log 回滚未提交的事务。
