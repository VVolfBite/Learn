## 扫描

### 全表扫描

没有索引或索引失效时，需要扫描所有行。

```sql
-- 假设表有 100 万行
SELECT * FROM user WHERE age = 20;  -- 没有索引，扫描 100 万行
```

### 索引扫描

通过索引快速定位，只扫描少量行。

```sql
-- 在 age 上建立索引
CREATE INDEX idx_age ON user(age);

SELECT * FROM user WHERE age = 20;  -- 通过索引，可能只扫描 1000 行
```

建立合适的索引,避免索引失效，必要使用覆盖索引避免回表，优化查询条件，缩小范围

## 回表

### 回表的代价

通过非聚簇索引查询时，需要回表获取完整数据。

```sql
-- 假设在 name 上有索引，查询需要返回所有列
SELECT * FROM user WHERE name = 'Alice';
-- 1. 在 name 索引中找到 100 个 Alice
-- 2. 回表 100 次，获取完整数据
```

如果匹配的行数很多，回表的代价就很大。

### 避免回表

**使用覆盖索引**：

```sql
-- 建立联合索引
CREATE INDEX idx_name_age ON user(name, age);

-- 只查询索引中的列
SELECT name, age FROM user WHERE name = 'Alice';
-- 不需要回表，直接从索引中获取数据
```

## 排序

### 排序的代价

如果排序字段没有索引，需要在内存或磁盘中排序。

```sql
SELECT * FROM user WHERE age > 18 ORDER BY create_time;
-- 如果 create_time 没有索引，需要先查出所有 age>18 的数据，再排序
```

### 优化排序

**方法 1：利用索引的有序性**

```sql
-- 在 create_time 上建立索引
CREATE INDEX idx_create_time ON user(create_time);

SELECT * FROM user WHERE age > 18 ORDER BY create_time;
-- 如果能用到 create_time 索引，数据本身就是有序的，不需要额外排序
```

**方法 2：联合索引**

```sql
-- 建立联合索引
CREATE INDEX idx_age_create_time ON user(age, create_time);

SELECT * FROM user WHERE age > 18 ORDER BY create_time;
-- 利用联合索引，既能快速过滤，又能避免排序
```

## 临时表

当GROUP BY 的列没有索引、DISTINCT 的列没有索引、UNION 操作或者子查询等情况时会产生临时表

### 临时表的代价

- 需要额外的内存或磁盘空间
- 需要额外的数据复制和处理时间

### 避免临时表

- 在 GROUP BY 和 DISTINCT 的列上建立索引
- 优化子查询，改用 JOIN
- 避免不必要的 UNION（如果可以用 UNION ALL 就不要用 UNION）

## EXPLAIN 分析查询

EXPLAIN 可以查看 SQL 的执行计划，帮助分析查询性能。

```sql
EXPLAIN SELECT * FROM user WHERE name = 'Alice';
```

### 关键字段

**type**：访问类型，从好到坏：
- `system`：表只有一行
- `const`：通过主键或唯一索引查询，最多返回一行
- `eq_ref`：JOIN 时使用主键或唯一索引
- `ref`：使用非唯一索引
- `range`：范围查询
- `index`：全索引扫描
- `ALL`：全表扫描（最差）

**key**：实际使用的索引
- 如果是 NULL，说明没有使用索引

**rows**：预计扫描的行数
- 越少越好

**Extra**：额外信息
- `Using index`：使用了覆盖索引，不需要回表（好）
- `Using where`：使用了 WHERE 过滤
- `Using filesort`：需要额外排序（差）
- `Using temporary`：需要临时表（差）

### 例子

```sql
EXPLAIN SELECT * FROM user WHERE name = 'Alice';

-- 结果：
-- type: ALL（全表扫描，差）
-- key: NULL（没有使用索引）
-- rows: 1000000（扫描 100 万行）
-- Extra: Using where

-- 优化：在 name 上建立索引
CREATE INDEX idx_name ON user(name);

EXPLAIN SELECT * FROM user WHERE name = 'Alice';

-- 结果：
-- type: ref（使用索引，好）
-- key: idx_name（使用了索引）
-- rows: 100（只扫描 100 行）
-- Extra: NULL
```

## 查询优化的常见手段

###  建立合适的索引

- 在频繁查询的列上建立索引
- 使用联合索引覆盖多个查询条件
- 避免索引过多

### 避免索引失效

- 不在索引列上使用函数
- 不在索引列上进行计算
- LIKE 不以通配符开头
- 避免类型不匹配

### 使用覆盖索引

- 只查询需要的列，不要 SELECT *
- 让查询的列都在索引中

### 优化 JOIN

- 小表驱动大表
- 在 JOIN 的列上建立索引
- 避免多表 JOIN（超过 3 个表）

### 分页优化

**问题**：深分页很慢

```sql
SELECT * FROM user ORDER BY id LIMIT 1000000, 10;
-- 需要先查出 1000010 行，再取最后 10 行
```

**优化**：使用子查询

```sql
SELECT * FROM user 
WHERE id >= (SELECT id FROM user ORDER BY id LIMIT 1000000, 1)
LIMIT 10;
-- 先通过覆盖索引找到起始 id，再查询数据
```

### 避免 SELECT *

- 增加网络传输量
- 无法使用覆盖索引
- 可能查询不需要的列

### 使用 LIMIT

- 限制返回的行数
- 减少网络传输和内存占用

### 避免子查询

子查询可能导致临时表，性能较差。

```sql
-- 差
SELECT * FROM user WHERE id IN (SELECT user_id FROM order WHERE amount > 100);

-- 好
SELECT u.* FROM user u 
JOIN order o ON u.id = o.user_id 
WHERE o.amount > 100;
```

### 使用 UNION ALL 代替 UNION

UNION 会去重，需要额外的排序和比较。

```sql
-- 如果确定没有重复，用 UNION ALL
SELECT * FROM user WHERE age < 18
UNION ALL
SELECT * FROM user WHERE age > 60;
```

## 慢查询日志

MySQL 可以记录执行时间超过阈值的 SQL，帮助定位性能问题。

### 开启慢查询日志

```sql
-- 查看是否开启
SHOW VARIABLES LIKE 'slow_query_log';

-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';

-- 设置阈值（秒）
SET GLOBAL long_query_time = 2;
```

### 分析慢查询日志

使用 mysqldumpslow 工具分析慢查询日志，找出最慢的 SQL。

## 查询优化的思路

1. **定位慢查询**：通过慢查询日志或监控工具
2. **分析执行计划**：使用 EXPLAIN 查看执行计划
3. **找出问题**：是否全表扫描、是否回表、是否排序、是否临时表
4. **优化**：建立索引、改写 SQL、调整表结构
5. **验证**：再次 EXPLAIN，对比优化前后的差异

## 深分页为什么慢

**问题**：

```sql
SELECT * FROM user ORDER BY id LIMIT 1000000, 10;
```

1. MySQL 需要先扫描前 1000010 行
2. 然后丢弃前 1000000 行
3. 只返回最后 10 行

即使 id 上有索引，也需要回表 1000010 次（如果 SELECT *）。

**优化方案 1：子查询**

```sql
SELECT * FROM user 
WHERE id >= (SELECT id FROM user ORDER BY id LIMIT 1000000, 1)
LIMIT 10;
```

先通过覆盖索引找到起始 id（不需要回表），再查询数据（只回表 10 次）。

**优化方案 2：记录上次位置**

```sql
-- 第一页
SELECT * FROM user WHERE id > 0 ORDER BY id LIMIT 10;  -- 最后一条 id=10

-- 第二页
SELECT * FROM user WHERE id > 10 ORDER BY id LIMIT 10;  -- 最后一条 id=20

-- 第三页
SELECT * FROM user WHERE id > 20 ORDER BY id LIMIT 10;
```

每次记录上一页的最后一条 id，下一页从这个 id 开始查询。

## 追问

**Q: 查询慢的常见原因有哪些？**
没有索引、索引失效、回表次数多、需要排序、使用临时表、锁等待。

**Q: 如何优化查询？**
建立合适的索引、避免索引失效、使用覆盖索引、优化 JOIN、避免子查询、使用 LIMIT。

**Q: EXPLAIN 的 type 字段有哪些值？**
从好到坏：system、const、eq_ref、ref、range、index、ALL。

**Q: 什么是覆盖索引？**
查询的列都在索引中，不需要回表。

**Q: 如何避免回表？**
使用覆盖索引，只查询索引中的列。

**Q: 为什么不要 SELECT *？**
增加网络传输量、无法使用覆盖索引、可能查询不需要的列。

**Q: 深分页为什么慢？**
因为需要先查出前 N 行，再取最后几行，扫描的数据量很大。

**Q: 如何优化深分页？**
使用子查询，先通过覆盖索引找到起始 id，再查询数据。

**Q: 子查询为什么慢？**
可能导致临时表，而且优化器可能选择不好的执行计划。

**Q: UNION 和 UNION ALL 的区别？**
UNION 会去重，需要额外的排序和比较；UNION ALL 不去重，性能更好。

**Q: 如何定位慢查询？**
开启慢查询日志，使用 mysqldumpslow 分析，或使用监控工具。

**Q: EXPLAIN 的 Extra 字段中，哪些是不好的？**
Using filesort（需要排序）、Using temporary（需要临时表）。

**Q: 如何判断索引是否生效？**
使用 EXPLAIN，查看 type 和 key 字段。如果 type 是 ALL 或 key 是 NULL，说明没有使用索引。

**Q: 为什么有索引还是慢？**
可能是索引失效、回表次数多、扫描的行数多、需要排序或临时表。
