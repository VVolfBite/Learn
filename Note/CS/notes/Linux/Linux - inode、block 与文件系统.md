## 什么是 inode

**inode（索引节点）**：存储文件的元信息。

### inode 存储的信息

这一节看起来往往容易变成一串要点，其实先理解“inode 存储的信息”在Linux里的作用会更顺。后面的内容可以理解成围绕这个问题展开的几种常见情况。

这里真正要说明的是“inode 存储的信息”为什么值得单独拿出来讲。把它放到“Linux - inode𰀛lock 与文件系统”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

- 文件大小
- 文件所有者（UID）
- 文件所属组（GID）
- 文件权限（rwx）
- 时间戳：
  - atime：最后访问时间
  - mtime：最后修改时间（内容）
  - ctime：最后改变时间（元信息）
- 硬链接数
- 文件数据块的位置（block 指针）

### inode 不存储的信息

- **文件名**：文件名存储在目录中
- **文件内容**：文件内容存储在 block 中

### 查看 inode

```ash
ls -i file.txt            # 查看 inode 号
# 123456 file.txt

stat file.txt             # 查看详细信息
# File: file.txt
# Size: 1024        Blocks: 8          IO Block: 4096
# Inode: 123456     Links: 1
# Access: 2024-01-01 12:00:00
# Modify: 2024-01-01 12:00:00
# Change: 2024-01-01 12:00:00

df -i                     # 查看 inode 使用情况
# Filesystem      Inodes  IUsed   IFree IUse% Mounted on
# /dev/sda1      1000000  50000  950000    5% /
```

**面试重点**：
- inode 存储文件元信息，不存储文件名和内容
- 每个文件有唯一的 inode 号
- inode 数量有限，可能用完

## 什么是 block

**block（数据块）**：存储文件的实际内容。

### block 的特点

- 固定大小，通常是 4KB
- 文件内容分散存储在多个 block 中
- block 的位置记录在 inode 中

### 查看 block

```ash
stat file.txt             # 查看 block 数量
# Blocks: 8               # 占用 8 个 block

du -h file.txt            # 查看磁盘占用
# 4.0K file.txt           # 即使文件只有 1 字节，也占用 1 个 block

df -h                     # 查看 block 使用情况
# Filesystem      Size  Used Avail Use% Mounted on
# /dev/sda1       100G   50G   50G  50% /
```

**面试重点**：
- block 存储文件实际内容
- block 大小固定，通常是 4KB
- 小文件也占用至少 1 个 block

## 文件系统的组成

Linux 文件系统（如 ext4）由以下部分组成：

### 1. 超级块（Superblock）

读到“1. 超级块（Superblock）”时，先别急着记结论。更有效的方式是先弄清它在“Linux - inode𰀛lock 与文件系统”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

- 存储文件系统的整体信息
- inode 总数、block 总数
- 文件系统类型、挂载时间
- 非常重要，损坏会导致文件系统无法使用

### 2. inode 表

- 存储所有 inode
- 固定大小，创建文件系统时确定
- inode 用完就不能创建新文件，即使还有空间

### 3. 数据块（Data Blocks）

- 存储文件实际内容
- 占用文件系统的大部分空间

### 4. 块位图（Block Bitmap）

- 记录哪些 block 已使用，哪些空闲
- 用于快速分配 block

### 5. inode 位图（Inode Bitmap）

- 记录哪些 inode 已使用，哪些空闲
- 用于快速分配 inode

```
+-------------+-------------+-------------+-------------+
| Superblock  | Inode Bitmap| Block Bitmap| Inode Table |
+-------------+-------------+-------------+-------------+
|                    Data Blocks                        |
+-------------------------------------------------------+
```

## 文件读取过程

理解文件读取过程是面试高频考点。

### 读取文件的步骤

这一节主要把“读取文件的步骤”放回“Linux - inode、block 与文件系统”这张卡片的主线里看。先抓住它在整个Linux知识体系中的位置，再看后面的分点和对比，会更容易把零散结论串起来。

这里真正要说明的是“读取文件的步骤”为什么值得单独拿出来讲。把它放到“Linux - inode𰀛lock 与文件系统”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

1. **根据文件路径找到目录的 inode**
2. **从目录的 inode 找到目录的 block**
3. **从目录的 block 中找到文件名对应的 inode 号**
4. **根据 inode 号找到文件的 inode**
5. **从文件的 inode 找到文件的 block 指针**
6. **根据 block 指针读取文件内容**

### 例子：读取 /home/user/file.txt

```
1. 读取根目录 / 的 inode（inode 号固定为 2）
2. 从根目录的 block 中找到 home 的 inode 号
3. 读取 home 目录的 inode
4. 从 home 目录的 block 中找到 user 的 inode 号
5. 读取 user 目录的 inode
6. 从 user 目录的 block 中找到 file.txt 的 inode 号
7. 读取 file.txt 的 inode
8. 从 file.txt 的 inode 找到 block 指针
9. 读取 file.txt 的 block，获取文件内容
```

**面试重点**：
- 文件名存储在目录中，不在 inode 中
- 目录是特殊文件，存储文件名和 inode 的映射
- 读取文件需要多次 inode 和 block 访问

## 为什么删除文件可以恢复

理解这个问题需要理解删除文件的过程。

### 删除文件的过程

读到“删除文件的过程”时，先别急着记结论。更稳的方式是先弄清它在“Linux - inode、block 与文件系统”这张卡片里到底想解决什么问题，这样后面的分类和判断就不会显得太散。

这一节看起来往往容易变成一串结论，其实先抓住“删除文件的过程”在Linux里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

1. **从目录的 block 中删除文件名和 inode 的映射**
2. **inode 的硬链接数减 1**
3. **如果硬链接数为 0，标记 inode 为空闲**
4. **标记 block 为空闲**

**关键**：
- 删除文件只是标记 inode 和 block 为空闲
- 实际数据还在磁盘上，没有被清除
- 只要 block 没有被覆盖，数据就可以恢复

### 数据恢复的条件

这一节看起来往往容易变成一串结论，其实先抓住“数据恢复的条件”在Linux里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**可以恢复**：
- 删除后立即恢复
- block 还没有被新数据覆盖
- 文件系统没有被格式化

**不能恢复**：
- block 已被新数据覆盖
- 文件系统被格式化
- 使用了安全删除工具（如 shred）

### 安全删除

如果要彻底删除文件，不让别人恢复：

```ash
shred -u file.txt         # 多次覆盖后删除
dd if=/dev/zero of=file.txt bs=1M  # 用零覆盖
```

**面试重点**：
- 删除文件只是标记空闲，数据还在
- 数据恢复的关键是 block 没有被覆盖
- 安全删除需要覆盖数据

## 磁盘碎片

### 什么是磁盘碎片

- 文件的 block 不连续，分散在磁盘各处
- 读取文件需要多次寻道，降低性能

### 为什么会产生碎片

- 文件频繁创建和删除
- 文件大小变化
- 磁盘空间不足

### Linux 的磁盘碎片

这一节主要把“Linux 的磁盘碎片”放回“Linux - inode𰀛lock 与文件系统”这条主线里看。先抓住它在整个Linux知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**Linux 文件系统（ext4）碎片较少**：
- 预分配策略：创建文件时预留空间
- 延迟分配：写入时才分配 block
- 块组：将相关文件放在一起

**Windows 文件系统（NTFS）碎片较多**：
- 分配策略不同
- 需要定期整理碎片

**面试重点**：
- Linux 文件系统碎片较少，通常不需要整理
- 磁盘碎片会降低性能

## inode 和 block 的关系

### 一对多关系

- 一个 inode 对应一个文件
- 一个文件可以占用多个 block
- 一个 block 只能属于一个文件

### inode 指针

读到“inode 指针”时，先别急着记结论。更有效的方式是先弄清它在“Linux - inode𰀛lock 与文件系统”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**直接指针**：
- inode 直接指向 block
- 通常有 12 个直接指针
- 可以存储小文件（12 * 4KB = 48KB）

**间接指针**：
- inode 指向一个 block，这个 block 存储更多 block 指针
- 一级间接指针：指向存储指针的 block
- 二级间接指针：指向存储一级指针的 block
- 三级间接指针：指向存储二级指针的 block

**结构示意**：
```
inode
├─ 直接指针 1 → block 1
├─ 直接指针 2 → block 2
├─ ...
├─ 直接指针 12 → block 12
├─ 一级间接指针 → block → [指针1, 指针2, ...]
├─ 二级间接指针 → block → [block → [指针1, 指针2, ...]]
└─ 三级间接指针 → block → [block → [block → [指针1, ...]]]
```

**面试重点**：
- 小文件用直接指针，快速访问
- 大文件用间接指针，支持大文件

## 常见问题

**Q: inode 和 block 有什么区别？**  
A: inode 存储文件元信息，block 存储文件内容。

**Q: 文件名存储在哪里？**  
A: 文件名存储在目录的 block 中，不在 inode 中。

**Q: 为什么 inode 会用完？**  
A: 创建大量小文件会消耗 inode。即使磁盘有空间，inode 用完也不能创建新文件。

**Q: 如何查看 inode 使用情况？**  
A: 使用 \df -i\ 命令。

**Q: 删除文件后为什么可以恢复？**  
A: 删除文件只是标记 inode 和 block 为空闲，实际数据还在磁盘上。

**Q: 如何彻底删除文件？**  
A: 使用 \shred\ 命令覆盖数据后删除。

**Q: 为什么小文件也占用 4KB？**  
A: block 大小固定为 4KB，即使文件只有 1 字节，也占用 1 个 block。

**Q: Linux 需要整理磁盘碎片吗？**  
A: 通常不需要，Linux 文件系统（ext4）碎片较少。
