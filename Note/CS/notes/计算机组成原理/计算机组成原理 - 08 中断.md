## 中断

中断(Interrupt)是指 CPU 在执行程序的过程中,遇到某些紧急或特殊情况,暂停当前程序的执行,转去处理这些情况,处理完毕后再返回继续执行原程序就好比你正在看书(CPU 执行程序)，突然电话响了(中断请求)，你放下书,接电话(中断处理)，通话结束,继续看书(中断返回)。

**为什么 CPU 需要中？**因为 I/O 设备(键盘、硬盘、网卡)速度远慢于 CPU,如果 CPU 一直等待 I/O,会浪费大量时间。中断让 I/O 设备完成后主动通知 CPU,CPU 可以在等待期间做其他事情,利用率提高 100 倍以上。

### 中断的来源

#### 外中断(硬件中断)

**定义:** 由 CPU 外部的硬件设备产生的中断。异步发生,与当前程序无关，可屏蔽(某些可以被禁止)。

#### 内中断(异常)

**定义:** 由 CPU 内部事件引起的中断,通常是程序执行错误。同步发生,与当前指令相关，不可屏蔽。

**故障(Fault):**可以修复的异常、修复后重新执行引起异常的指令

**陷阱(Trap):**有意触发的异常、用于调试(断点)

**终止(Abort):**严重错误,无法恢复、程序必须终止

#### 软中断(陷入)

**定义:** 程序主动触发的中断,用于请求操作系统服务。

**Linux 系统调用:**

```c
// 用户程序
int fd = open("/tmp/file.txt", O_RDONLY);
```

**底层实现:**

```assembly
mov rax, 2          ; 系统调用号(open = 2)
mov rdi, filename   ; 参数 1: 文件名
mov rsi, O_RDONLY   ; 参数 2: 打开模式
syscall             ; 触发软中断,陷入内核
```

**为什么需要软中断:**

- 用户程序运行在用户态,权限受限
- 某些操作(文件操作、网络操作)需要内核态权限
- 通过软中断陷入内核,请求操作系统服务

### 中断的特点

| 特性 | 说明 |
|------|------|
| 随机性 | 中断发生的时间不确定 |
| 异步性 | 中断与当前程序执行无关 |
| 可屏蔽性 | 某些中断可以被禁止 |
| 优先级 | 不同中断有不同的优先级 |
| 可嵌套性 | 中断处理过程中可以被更高优先级中断打断 |

### 中断的基本作用

#### 处理异步事件

**异步事件:** 发生时间不确定的事件。

**示例 1: 键盘输入**

```
没有中断:
  CPU 不断检查键盘是否有输入(轮询)
  while (true) {
      if (键盘有输入) {
          处理输入;
      }
  }
  问题: CPU 大部分时间在空等,浪费

有中断:
  CPU 执行其他任务
  用户按键 → 键盘发出中断 → CPU 处理输入 → 继续执行任务
  优势: CPU 不需要主动检查,有事件时才处理
```

**示例 2: 网络数据到达**

```
没有中断:
  CPU 不断检查网卡是否有数据
  浪费 CPU 时间

有中断:
  网卡收到数据 → 发出中断 → CPU 处理数据
  CPU 可以做其他事情,数据到达时才处理
```

#### 提高 CPU 利用率

**I/O 设备和 CPU 的速度差距:**
- 硬盘读取: 5 毫秒 = 5,000,000 纳秒  CPU 运算: 1 纳秒
- 速度差距: 500 万倍

如果 CPU 等待硬盘,相当于一个人工作 1 秒,然后发呆 58 天,再工作 1 秒。这显然是巨大的浪费。

**没有中断的情况:**

```
程序执行流程:
  1. CPU 发出读硬盘命令
  2. CPU 空等硬盘读取完成 (5ms = 500 万个时钟周期)
  3. 硬盘读取完成
  4. CPU 继续执行

问题: CPU 在步骤 2 空等 5ms,这段时间本可以执行 500 万条指令!
```

**有中断的情况:**

```
程序执行流程:
  1. CPU 发出读硬盘命令
  2. CPU 切换到其他任务,继续执行指令
  3. 硬盘读取完成,发出中断信号
  4. CPU 响应中断,处理读取的数据
  5. 继续执行任务

优势: CPU 在等待 I/O 的 5ms 内可以执行 500 万条指令,利用率接近 100%
```

这就是为什么所有现代操作系统都基于中断机制。没有中断,CPU 大部分时间在空等 I/O,利用率不到 1%。

**效率对比:**

```
单任务,无中断:
  任务 A: [CPU][等待 I/O 5ms][CPU]
  CPU 利用率: 低

多任务,有中断:
  任务 A: [CPU]              [CPU]
  任务 B:      [CPU][CPU][CPU]
  I/O:     [5ms 硬盘读取]
  CPU 利用率: 高
```

#### 实现多任务

**时钟中断实现分时:**

```
操作系统使用时钟中断实现多任务:
  1. 时钟每隔一段时间(如 10ms)产生一次中断
  2. 中断处理程序检查当前任务是否用完时间片
  3. 如果用完,切换到下一个任务
  4. 如果没用完,继续执行当前任务

效果:
  任务 A: [10ms][10ms][10ms]...
  任务 B:       [10ms][10ms]...
  任务 C:             [10ms]...
  
  用户感觉多个任务"同时"运行
```

#### 处理异常情况

**异常:** 程序执行过程中出现的错误或特殊情况。

**常见异常:**

**除零异常:**

```c
int a = 10;
int b = 0;
int c = a / b;  // 触发除零异常(中断)
1. CPU 检测到除零操作
2. 触发除零异常(中断)
3. 操作系统接管,终止程序或显示错误信息
```

**缺页异常:**

```c
int* ptr = malloc(1000000);
ptr[500000] = 10;  // 可能触发缺页异常
1. 访问的页面不在内存中
2. 触发缺页异常(中断)
3. 操作系统从硬盘加载页面到内存
4. 返回,重新执行访问指令
```

**非法指令:**

```assembly
0xFF 0xFF 0xFF 0xFF  // 非法指令
1. CPU 无法识别指令
2. 触发非法指令异常
3. 操作系统终止程序
```



### 中断处理流程

#### 中断请求

**硬件中断:**I/O 设备完成操作、设备控制器向 CPU 发送中断信号、信号通过中断控制器(如 8259A、APIC)传递给 CPU

**软件中断:**程序执行特殊指令(如 `int 0x80`, `syscall`)、CPU 内部产生中断信号

#### 中断响应

**CPU 响应中断的条件:**
- 中断使能(IF 标志位为 1)
- 当前指令执行完毕
- 没有更高优先级的中断正在处理

**响应步骤:**

```
1. 保存现场:
   - 保存程序计数器(PC)
   - 保存状态寄存器(FLAGS)
   - 保存通用寄存器(可选,由中断处理程序决定)

2. 关闭中断(可选):
   - 防止中断嵌套
   - 某些中断处理程序会关闭中断

3. 识别中断源:
   - 读取中断向量号
   - 确定是哪个设备或事件触发的中断

4. 跳转到中断处理程序:
   - 根据中断向量号查找中断向量表
   - 获取中断处理程序的地址
   - 跳转到中断处理程序
```

#### 中断处理

**中断处理程序(Interrupt Handler):**也称为中断服务程序(ISR, Interrupt Service Routine)、处理中断事件的代码

**示例: 键盘中断处理**

```c
void keyboard_interrupt_handler() {
    // 1. 保存可能被修改的寄存器
    save_registers();
    
    // 2. 读取键盘扫描码
    char scancode = inb(KEYBOARD_PORT);
    
    // 3. 转换为字符
    char c = scancode_to_char(scancode);
    
    // 4. 放入键盘缓冲区
    keyboard_buffer[buffer_index++] = c;
    
    // 5. 通知设备中断处理完成
    outb(PIC_EOI_PORT, 0x20);  // 发送 EOI(End Of Interrupt)
    
    // 6. 恢复寄存器
    restore_registers();
    
    // 7. 返回
    iret();  // 中断返回指令
}
```

**中断处理程序的要求:**尽量短小精悍、不能执行耗时操作、不能调用可能阻塞的函数

#### 中断返回

**返回步骤:**

```
1. 恢复现场:
   - 恢复通用寄存器
   - 恢复状态寄存器(FLAGS)
   - 恢复程序计数器(PC)

2. 开启中断:
   - 如果之前关闭了中断,重新开启

3. 返回原程序:
   - 执行中断返回指令(如 x86 的 iret)
   - CPU 继续执行被中断的程序
```

**中断返回指令:**
- x86: `iret` (Interrupt Return)
- ARM: `movs pc, lr` (从链接寄存器恢复 PC)

### 中断优先级

#### 优先级

为每个中断分配优先级。高优先级中断可以打断低优先级中断处理

**从高到低:**

```
1. 硬件故障(电源故障、内存错误)
   - 最高优先级,必须立即处理

2. 程序性异常(除零、非法指令)
   - 与当前程序相关,需要及时处理

3. 外部中断(I/O 设备)
   - 时钟中断 > 硬盘中断 > 网络中断 > 键盘中断

4. 软中断(系统调用)
   - 最低优先级
```

#### 中断嵌套

**定义:** 中断处理过程中,被更高优先级的中断打断。

```
时间轴:
  程序 A 执行
    ↓
  键盘中断(优先级 3)
    ↓
  处理键盘中断
    ↓
  硬盘中断(优先级 2,更高)
    ↓
  暂停键盘中断处理,处理硬盘中断
    ↓
  硬盘中断处理完成,返回
    ↓
  继续处理键盘中断
    ↓
  键盘中断处理完成,返回
    ↓
  继续执行程序 A
```

## 高频判断

### 判断 1: 中断是 CPU 暂停当前程序去处理其他事件

**正确。** 中断的本质就是打断当前程序,处理紧急事件,然后返回。

### 判断 2: 所有中断都可以被屏蔽

**错误。** 外中断(硬件中断)可以被屏蔽,但内中断(异常)通常不可屏蔽。

### 判断 3: 中断可以提高 CPU 利用率

**正确。** 中断避免了 CPU 轮询等待,可以在等待 I/O 期间执行其他任务。

### 判断 4: 时钟中断用于实现多任务调度

**正确。** 操作系统利用时钟中断实现时间片轮转,让多个任务分时执行。

### 判断 5: 系统调用通过软中断实现

**正确。** 系统调用通过特殊指令(如 `syscall`, `int 0x80`)触发软中断,陷入内核。

### 判断 6: 中断处理程序可以执行耗时操作

**错误。** 中断处理程序应该尽量短小,不能执行耗时操作,否则影响系统响应。

### 判断 7: 缺页异常是一种中断

**正确。** 缺页异常属于内中断(异常),由 CPU 内部事件触发。

### 判断 8: 高优先级中断可以打断低优先级中断

**正确。** 这称为中断嵌套,高优先级中断可以抢占低优先级中断的处理。

### 判断 9: 中断发生时,CPU 会自动保存所有寄存器

**错误。** CPU 通常只保存 PC 和 FLAGS,其他寄存器由中断处理程序根据需要保存。

### 判断 10: 外中断是异步的,内中断是同步的

**正确。** 外中断由外部设备触发,发生时间不确定(异步);内中断由当前指令触发,与程序执行同步。

### 判断 11: 中断向量表存储中断处理程序的地址

**正确。** 中断向量表是一个数组,每个元素存储对应中断处理程序的入口地址。

### 判断 12: 中断返回后,程序从被中断的位置继续执行

**正确。** 中断处理完成后,恢复 PC 的值,程序从被中断的位置继续执行。

### 判断 13: 除零异常发生后,程序可以继续执行

**错误。** 除零异常通常是致命错误,操作系统会终止程序。

### 判断 14: 轮询比中断更高效

**错误。** 轮询会浪费大量 CPU 时间,中断方式更高效。

### 判断 15: 中断处理程序运行在内核态

**正确。** 中断处理程序需要访问硬件和内核数据结构,必须运行在内核态。
