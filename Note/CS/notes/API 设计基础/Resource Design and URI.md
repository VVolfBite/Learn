# Resource Design and URI

> 资源设计决定了 API 看起来像一套清晰能力，还是像一堆杂乱动作。URI 命名只是表面，背后真正重要的是你如何理解系统中的“资源”。

## 背景与作用

资源设计是把系统中的核心对象、集合和关系组织成接口路径的过程。URI 则是这些资源的外部标识。设计资源时，开发者需要先想清楚系统对外暴露的到底是什么实体、这些实体之间有什么从属关系，再决定路径应如何表达。

如果接口只按动作来命名，很快就会出现 `/getUserInfo`、`/updateUserStatus`、`/queryOrderList` 这种风格不一的路径。它们短期也能工作，但长期理解成本高，扩展也容易失控。资源导向的设计能让调用方更容易猜到接口长什么样，也让不同操作围绕同一对象自然聚合。

## 场景与边界

资源设计是 API 风格感的直接体现。面试问它，往往是在看你是否真的理解 REST 风格而不是只会背术语。工作里当接口不断增多时，资源组织得好不好，会直接影响团队沟通成本和后续文档质量。

最常见的误区是把路径设计成动作大全。动作不是不能出现，但应当是例外，不应成为主导。另一个误区是追求路径形式统一，却没有想清资源本身边界。路径是否优雅，最终取决于资源模型是否清楚，而不是是否用了复数名词。

## 面试表达

资源设计的核心是先定义系统对外暴露的实体和关系，再围绕它们组织 URI。这样做的好处是接口更一致、更好理解，也更容易扩展。相比之下，路径到处堆动作虽然短期方便，但长期很容易失去语义稳定性。

## 快速回忆

- 先想资源，再想 URI。
- 资源导向比动作堆砌更稳。
- 好路径来自清楚的资源模型。
- 命名统一只是表层，语义清楚才是关键。
