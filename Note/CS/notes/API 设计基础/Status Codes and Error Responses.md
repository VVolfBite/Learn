# Status Codes and Error Responses

> 状态码只能告诉调用方“发生了哪一类结果”，真正让接口好用的是错误响应是否清楚、一致、可定位。

## 背景与作用

HTTP 状态码是协议层对结果的大类表达，比如成功、客户端错误、服务端错误。错误响应则是在这个大类之下，进一步告诉调用方具体出了什么问题、是否可重试、参数哪里不对、需要怎样修正。二者一起构成接口反馈机制。

如果只有模糊的状态码和一句含糊报错，调用方很难知道问题到底在鉴权、参数、资源状态还是服务异常。良好的错误响应能显著降低联调成本、排障成本和误用成本。它不是为了把内部实现全部暴露出去，而是为了给调用方足够的信息去判断下一步该怎么做。

## 场景与边界

状态码几乎是所有 API 的共同语言，所以面试会问。工作里更常见的痛点则来自错误响应设计不统一：有的接口只返回字符串，有的接口返回一堆内部堆栈，有的接口对同类错误使用完全不同的格式。设计错误响应，看起来是细节，实际上直接影响 API 的可维护性。

一个误区是把状态码背成号码清单。对入门阶段来说，掌握常见成功、参数错误、未认证、无权限、未找到、冲突和服务异常这几类语义已经很重要。另一个误区是把内部异常原样透出。错误响应需要帮助定位问题，但不该泄露过多内部实现和敏感信息。

## 面试表达

状态码负责表达结果的大类语义，错误响应负责提供更具体的可读信息。一个设计好的 API 不会只停留在“返回 400 或 500”，而是会用一致的结构告诉调用方错误类型、关键信息和可能的处理方向，同时避免暴露不必要的内部细节。

## 快速回忆

- 状态码给大类，错误体给细节。
- 重点不是记号码，而是保持语义一致。
- 错误响应要帮助排障，但不能泄露过多内部信息。
- 同类问题应有稳定的响应结构。
