# API 设计基础 - 高频判断

## 主线判断

- API 是系统边界上的契约，不是数据库直接外放。
- 好的 API 首先追求清晰、一致、可演进。
- 用 HTTP 不等于天然符合 REST。

## 语义与返回

- HTTP 方法表达语义，幂等性表达重复请求的稳定性。
- 状态码只给大类，错误响应要提供可理解细节。
- JSON 常见，不代表字段风格可以随意。

## 列表与演进

- 分页、过滤、排序是列表接口的基础能力。
- 参数形式有常见约定，但团队统一更重要。
- 版本化服务于兼容性，不服务于表面规范感。

## 扩展风格

- OpenAPI 描述契约，不替代设计。
- Webhook、SSE、WebSocket 解决的是不同通知场景。
- gRPC、GraphQL 和 REST 通常是并存，不是简单替代。
