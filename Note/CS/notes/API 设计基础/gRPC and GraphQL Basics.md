# gRPC and GraphQL Basics

> REST 不是唯一答案。gRPC 和 GraphQL 之所以出现，是因为不同场景对接口表达、性能和查询灵活性的要求并不一样。

## 背景与作用

gRPC 是一种更强调高性能、强契约和服务间通信体验的 RPC 框架，常见于内部服务调用。GraphQL 则是一种查询语言和运行时，让客户端可以更精细地声明自己想要哪些字段和关联数据。它们都不是 REST 的“升级版”，更准确地说，是在不同问题上给出了不同取向。

gRPC 主要解决服务间调用时的契约表达、传输效率和代码生成问题，特别适合内部系统之间对性能和一致接口体验要求较高的场景。GraphQL 则更适合前端对数据形状有很强控制需求的场景，它减少了“为不同页面反复定制多个接口”的压力。两者出现的背景不同，所以不适合简单横向比较谁更先进。

## 场景与边界

随着系统架构变复杂，REST 之外的风格越来越常见。面试会借 gRPC 和 GraphQL 看候选人是否知道“接口不是只有一种设计方式”。工作里如果不了解它们，就很容易在需要高效服务间通信或需要灵活查询时仍然机械套 REST。

一个误区是把 gRPC、GraphQL 和 REST 看成替代关系。真实项目常常是并存：对外开放 API 仍然用 REST，对内服务调用用 gRPC，面向复杂前端聚合需求时引入 GraphQL。另一个误区是忽略更早期的风格。SOAP 提醒我们“强协议和企业集成”曾是核心需求，HATEOAS 则强调资源状态和可发现性，这些概念即使今天不常作为主流实现，也仍然有助于理解 API 设计演化。

## 面试表达

gRPC 和 GraphQL 的出现说明 API 设计会随场景变化。gRPC 更偏内部服务通信和强契约，GraphQL 更偏客户端按需取数。它们和 REST 不是简单替代关系，而是不同约束下的不同选择。理解这点，比机械比较哪种技术“更高级”更重要。

## 快速回忆

- gRPC 更偏服务间通信和强契约。
- GraphQL 更偏按需取数和数据聚合。
- REST、gRPC、GraphQL 经常会并存。
- SOAP 和 HATEOAS 是理解演化的背景点。
