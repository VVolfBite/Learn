# JSON APIs

> 绝大多数 API 选择 JSON，不是因为它完美，而是因为它在可读性、生态和跨语言协作上达到了足够好的平衡。

## 背景与作用

JSON 是一种轻量的数据交换格式。它和 API 结合得非常紧密，因为字段结构清晰，文本形式便于调试，几乎所有语言和平台都有成熟支持。对于大多数 Web 接口来说，JSON 足够表达常见对象、列表和嵌套结构，同时也容易被前端和其他服务消费。

API 需要一种让双方都能稳定理解的表示形式。JSON 的价值就在于低门槛和高通用性：人能看，工具能处理，网络上传输也足够方便。它让接口通信更标准化，减少了双方在数据表达上的额外成本。

## 场景与边界

JSON 已经是现代 HTTP API 的默认语言之一。面试问它，通常不是为了确认你知道大括号和方括号，而是想看你是否理解为什么接口要强调响应结构一致性。工作里真正影响体验的，不是“用没用 JSON”，而是字段语义、错误格式、空值处理、时间格式这些是否一致。

一个误区是把 JSON 看成“只要能返回就行”。实际上，即使都叫 JSON，风格混乱也会让调用方很难受。另一个误区是忽略它的边界。JSON 并不擅长表达复杂类型约束，也不天然保证字段语义稳定，所以需要额外的规范、文档和契约来补足。

## 面试表达

JSON 之所以成为常见 API 表示格式，主要是因为它轻量、可读、跨语言支持广。真正设计时需要关注的不只是用 JSON，而是保证字段命名、空值策略、错误响应和嵌套结构保持一致，让调用方能稳定理解。

## 快速回忆

- JSON 常见不是因为完美，而是因为足够通用。
- 重点在结构一致性，不在格式本身。
- 可读性强，但类型约束表达能力有限。
- 好的 JSON API 依赖稳定的字段语义。
