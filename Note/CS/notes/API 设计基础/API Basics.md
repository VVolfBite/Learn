# API Basics

> API 不是“把数据库内容暴露出来”的接口集合，而是系统边界上的协作契约。它决定别人如何理解你的系统，也决定你自己未来是否还能平稳演进。

## 背景与作用

API 是系统向外暴露能力和数据的方式。调用方不需要知道你的内部表结构、类设计和部署细节，只需要遵守一组稳定的输入输出约定。对后端来说，API 既是技术接口，也是组织协作界面：前端、第三方系统、其他服务都会通过它理解你的系统。

如果没有清晰 API，系统之间的调用会变成一堆临时约定，字段含义、错误处理和兼容规则都会越来越混乱。API 设计的意义在于把系统能力整理成可理解、可调用、可维护的边界，让使用者知道该怎样请求、会得到什么结果、出现问题时该怎样处理。

## 场景与边界

后端工作很大一部分就是设计、维护和演进 API。面试问 API 基础，并不是想听“接口是什么”，而是想看你是否理解接口设计其实影响着可读性、联调成本、错误排查和版本兼容。工作里越是多人协作、多服务协作，API 设计的重要性越明显。

一个误区是把 API 当成数据库镜像，把表结构几乎原样暴露出去。另一个误区是只追求当前能用，而不考虑一致性和演进。好的 API 不等于最炫的技术选型，它首先应当让调用方容易理解，让服务端在未来调整内部实现时仍有缓冲空间。

## 面试表达

API 是系统对外的协作契约，不只是技术调用入口。好的 API 设计要让能力表达清晰、行为一致、错误可理解，同时为后续演进留出空间。它的重点不是“能不能调通”，而是“长期是否容易用、容易改、容易维护”。

## 快速回忆

- API 是边界，不是数据库镜像。
- 首先追求清晰和一致。
- 设计不好，联调和演进成本都会上升。
- 可用只是起点，可维护才是长期价值。
