# Pagination Filtering Sorting

> 列表接口真正难的地方不在“把一堆数据返回出来”，而在于让调用方能有控制地拿到需要的那一部分结果。

## 背景与作用

分页、过滤和排序是列表接口最常见的三种控制能力。分页解决一次返回太多数据的问题，过滤帮助调用方缩小范围，排序让结果顺序具有明确语义。它们经常同时出现，因为真实系统中的列表查询几乎总会面临数据量和可读性问题。

如果没有分页，列表接口很容易把大量数据一次性压给服务端和客户端；如果没有过滤，调用方只能先全量拿到再自己筛；如果没有排序，结果的可预测性和稳定性都会下降。三者结合的价值，在于让接口既能控制成本，也能让调用方更高效地表达查询意图。

## 场景与边界

几乎所有后台、管理系统、搜索页面和数据接口都会碰到这三件事。面试问它，不是想让你背某套参数格式，而是看你是否理解列表接口应该如何在可用性和性能之间平衡。工作里也一样：很多接口问题并不是语法错，而是没有想清数据规模和调用场景。

一个误区是把某种参数命名方式当成行业标准。更准确的说法是：分页、过滤、排序有很多常见约定，但团队内部最重要的是统一。另一个误区是只关注调用方便，不考虑底层成本。过滤字段和排序字段如果没有配套索引或限制策略，很容易把列表接口变成性能风险入口。

## 面试表达

分页、过滤和排序是列表接口的基础能力，分别用于控制数据量、表达查询条件和确定结果顺序。设计时重点不是背某种参数形式，而是根据场景保证语义清楚、风格统一，同时控制底层查询成本。

## 快速回忆

- 分页控量，过滤筛选，排序定顺序。
- 参数形式有常见约定，但不算统一标准。
- 列表设计要同时考虑调用体验和查询成本。
- 一致性比花样更重要。
