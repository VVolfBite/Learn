# REST

> REST 不是协议，也不是一套必须逐条背诵的圣经，它更像一种接口设计取向：围绕资源组织能力，用统一语义暴露操作。

## 背景与作用

REST 由 Roy Fielding 在总结 Web 架构经验时提出。它不是为了发明一种新协议，而是把 Web 成功的一些关键特征抽象出来，比如资源标识、统一接口、无状态交互和可缓存响应。后来大量 HTTP API 借鉴 REST，正是因为这种设计方式天然容易理解，也容易和 HTTP 语义对齐。

在没有统一设计取向的情况下，接口很容易长成一堆杂乱的动作集合：命名风格不一，行为语义不清，学习成本和维护成本都会变高。REST 的价值在于鼓励开发者围绕资源来思考系统能力，让路径、方法、状态码和表示形式形成一套相对统一的表达方式。

## 场景与边界

REST 是大多数后端 API 讨论的共同语言。面试问它，是想看你是否知道“为什么很多 HTTP API 长这个样子”。工作里即使不刻意追求完全 RESTful，许多好的接口实践也都和 REST 的基本思想有关，比如资源导向、统一语义、无状态请求和缓存意识。

最大的误区是觉得“用了 HTTP 就是 REST”。更准确地说，HTTP 只是承载协议，REST 是一种设计风格。另一个误区是把无状态简单理解成“必须用 token，不能有 session”。这里更关键的含义是：每个请求应尽量自包含，服务端不依赖某个隐藏的交互上下文去理解请求。现实系统很少百分之百纯粹 REST，但这不妨碍我们借鉴它的设计价值。

## 面试表达

REST 是一种面向资源的接口设计风格，强调使用统一接口和清晰语义来操作资源，而不是随意堆动作接口。它常见于 HTTP API，但用了 HTTP 不等于天然就是 REST。理解 REST 的重点不是背约束，而是知道它为什么能让接口更一致、更容易理解。

## 快速回忆

- REST 是风格，不是协议。
- 围绕资源组织能力，而不是围绕动作堆接口。
- 无状态强调请求语义自包含，不是单纯等于 token。
- 用 HTTP 不等于天然 RESTful。
