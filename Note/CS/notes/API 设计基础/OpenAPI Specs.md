# OpenAPI Specs

> OpenAPI 的价值不在“自动生成了一堆文档”，而在于它把接口描述变成了一份可共享、可校验、可协作的契约。

## 背景与作用

OpenAPI 是用于描述 HTTP API 的规范。它用统一格式表达路径、方法、参数、请求体、响应结构、鉴权方式等内容，让接口不仅能被人阅读，也能被工具处理。对团队而言，这意味着接口定义可以成为一种独立资产，而不再只是散落在代码注释或口头约定里。

没有统一描述规范时，文档很容易过期，联调容易靠猜，代码和说明也容易脱节。OpenAPI 的价值就在于把“接口长什么样”这件事显式化，并为文档生成、Mock、代码生成和自动校验提供基础。它不能替代 API 设计本身，但能显著提升协作效率。

## 场景与边界

OpenAPI 已经成为很多团队的默认接口描述方式。面试问它，通常不是为了看会不会写 YAML，而是想确认你是否理解“接口契约要可共享、可验证”。工作里一旦接口多、团队多，OpenAPI 往往能明显降低沟通和联调成本。

一个误区是把 OpenAPI 等同于 API 设计。更准确地说，它描述 API，但不替你做设计决策。另一个误区是认为“有文档就够了”。如果文档和实现脱节，规范本身也会失去价值。OpenAPI 真正有效的前提，是团队愿意把它纳入设计和交付流程，而不是上线后再补。

## 面试表达

OpenAPI 是 HTTP API 的描述规范，帮助团队把接口契约显式化，并支持文档、联调、Mock 和部分代码生成。它不等于 API 设计本身，但能让设计结果更容易被共享、校验和持续维护。

## 快速回忆

- OpenAPI 描述的是接口契约。
- 它帮助文档、联调和自动化协作。
- 不等于 API 设计本身。
- 规范只有和实现持续同步才真正有价值。
