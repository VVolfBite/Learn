# Versioning

> API 版本化不是为了看起来规范，而是因为接口一旦被外部依赖，就必须认真对待兼容性和演进节奏。

## 背景与作用

Versioning 指 API 在功能变化、字段调整或语义修改时，如何让新旧调用方能够有序过渡。它关心的不是版本号写在哪里，而是系统如何在演进中减少破坏性影响。只要 API 对外提供服务，版本化迟早会成为现实问题。

调用方和服务端迭代速度往往不同。如果接口在服务端升级时直接破坏旧约定，外部系统就会突然失效。版本化的意义在于给双方一个缓冲区，让新能力可以上线，旧调用方也有迁移时间，从而避免把每一次变更都变成硬切换事故。

## 场景与边界

面试问版本化，是为了看候选人是否意识到 API 是长期契约。工作里随着调用方增多，接口想改就改会越来越不现实。真正麻烦的通常不是加一个字段，而是删除旧字段、改变语义、调整认证方式这些会影响兼容性的变化。

一个误区是认为所有小变动都必须立刻升大版本。很多兼容性变化可以通过向后兼容设计平滑处理。另一个误区是把版本号当成全部。版本化只是手段，更关键的是提前考虑兼容策略、弃用周期和文档通知方式。

## 面试表达

API 版本化的核心是管理接口演进带来的兼容性问题，而不是机械地在路径或请求头里加版本号。是否版本化、怎样版本化，要看变更是否破坏旧约定，以及调用方是否需要明确迁移窗口。

## 快速回忆

- 版本化服务于兼容性，不服务于形式。
- 不是所有变化都需要立刻升版本。
- 真正关键的是迁移策略和弃用节奏。
- API 一旦被依赖，就要按长期契约来管理。
