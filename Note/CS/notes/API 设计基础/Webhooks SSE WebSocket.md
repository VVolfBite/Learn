# Webhooks SSE WebSocket

> 这三种机制都和“服务端主动通知”有关，但它们解决问题的方式完全不同。选型的关键不是流行度，而是通信方向、实时性和控制权。

## 背景与作用

Webhook 是“事件发生后由服务端主动回调到另一个服务”的模式，适合系统与系统之间的异步通知。SSE 是服务器向浏览器持续单向推送文本事件的机制，更适合简单实时更新。WebSocket 则建立长连接，支持真正的双向通信，适合聊天、协同编辑、实时互动等场景。

轮询虽然简单，但会带来延迟和额外请求。Webhook、SSE 和 WebSocket 都是在尝试减少这种浪费，让信息在更合适的时机主动到达调用方。不同点在于：Webhook 更偏服务间异步集成，SSE 更偏服务端到浏览器的单向事件流，WebSocket 则适合双方都需要频繁主动说话的场景。

## 场景与边界

现代系统越来越重视实时感知和系统间事件协作，所以这三种机制经常被拿来比较。面试问它们，是想看你能否根据场景选型，而不是只会喊“实时就上 WebSocket”。工作里选错方案，后续在复杂度、可靠性和运维成本上都会吃亏。

最大的误区是把所有“实时”问题都交给 WebSocket。实际上，如果只是外部系统收到事件通知，Webhook 往往更自然；如果只是页面单向接收更新，SSE 常常更轻。另一个误区是忽略可靠性设计。尤其是 Webhook，本质上仍是跨系统通知，需要认真处理重试、签名校验和幂等。

## 面试表达

Webhook、SSE 和 WebSocket 都能减少轮询带来的浪费，但适用边界不同。Webhook 更适合服务间异步回调，SSE 适合服务端到浏览器的单向流式更新，WebSocket 适合真正双向、低延迟互动。选型重点是通信方向、实时性要求和可接受的复杂度。

## 快速回忆

- Webhook：服务到服务的事件回调。
- SSE：服务端到浏览器的单向推送。
- WebSocket：双向长连接。
- 不是所有实时需求都该上 WebSocket。
