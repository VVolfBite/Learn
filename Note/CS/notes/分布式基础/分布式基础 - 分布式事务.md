## 为什么需要分布式事务

### 本地事务的局限

**本地事务是什么**
- 单个数据库的事务
- 满足 ACID 特性
- 通过数据库的事务机制保证

**为什么本地事务不够**

**场景 1：跨数据库操作**
```
用户下单：
1. 订单库：插入订单记录
2. 库存库：扣减库存
3. 积分库：增加积分

问题：三个数据库，本地事务只能管一个
```

**场景 2：跨服务操作**
```
订单服务 → 库存服务 → 支付服务

问题：三个独立的服务，各自有自己的数据库
```

**场景 3：分库分表**
```
订单表按用户 ID 分片：
- 用户 A 的订单在数据库 1
- 用户 B 的订单在数据库 2

问题：跨分片的操作需要分布式事务
```

### 分布式事务的本质困难

读到“分布式事务的本质困难”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 分布式事务”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**网络不可靠**
- 消息可能丢失、延迟
- 无法确定操作是否成功

**节点可能故障**
- 协调者或参与者可能故障
- 需要故障恢复机制

**性能开销大**
- 需要多轮通信
- 锁定资源时间长
- 影响系统吞吐量

**一致性与可用性的权衡**
- 强一致性影响可用性
- 需要根据业务场景选择

## 2PC（两阶段提交）

### 基本思想

通过引入协调者（Coordinator）来协调参与者（Participant）的行为，分两个阶段完成事务。

**角色**
- **协调者（Coordinator）**：事务管理器，协调整个事务
- **参与者（Participant）**：资源管理器，执行本地事务

### 执行过程

这一节主要把“执行过程”放回“分布式基础 - 分布式事务”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**阶段一：准备阶段（Prepare）**

1. 协调者向所有参与者发送 Prepare 请求
2. 参与者执行事务操作，但不提交
3. 参与者将 Undo 和 Redo 信息写入日志
4. 参与者向协调者返回 Yes（可以提交）或 No（无法提交）

**阶段二：提交阶段（Commit）**

**情况 1：所有参与者都返回 Yes**
1. 协调者向所有参与者发送 Commit 请求
2. 参与者提交事务，释放资源
3. 参与者向协调者返回 Ack
4. 协调者收到所有 Ack，事务完成

**情况 2：任一参与者返回 No**
1. 协调者向所有参与者发送 Rollback 请求
2. 参与者回滚事务，释放资源
3. 参与者向协调者返回 Ack
4. 协调者收到所有 Ack，事务取消

### 2PC 的四大问题

这一节看起来往往容易变成一串结论，其实先抓住“2PC 的四大问题”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**问题 1：同步阻塞**
- 参与者在等待协调者指令时处于阻塞状态
- 锁定资源，无法处理其他请求
- 影响系统吞吐量

**为什么这是问题**
- 准备阶段，参与者锁定资源
- 等待协调者的 Commit 或 Rollback
- 这段时间内，资源被锁定，其他事务无法访问

**问题 2：单点问题**
- 协调者是单点，故障会导致整个事务阻塞
- 特别是在提交阶段，参与者会一直等待

**为什么这是问题**
- 协调者发送 Prepare 后故障
- 参与者不知道是提交还是回滚
- 只能一直等待

**问题 3：数据不一致（最严重）**
- 提交阶段，协调者发送 Commit 后故障
- 部分参与者收到 Commit，部分未收到
- 导致数据不一致

**为什么这是问题**
- 协调者发送 Commit 给参与者 A，成功
- 发送 Commit 给参与者 B 时，网络故障
- 参与者 A 提交了，参与者 B 没提交
- 数据不一致

**问题 4：太过保守**
- 任一参与者失败，整个事务失败
- 没有容错机制

### 2PC 的适用边界

这一节看起来往往容易变成一串结论，其实先抓住“2PC 的适用边界”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**适合**
- 对一致性要求极高的场景
- 金融系统、核心交易
- 参与者数量少（2-3个）

**不适合**
- 高并发场景（性能差）
- 参与者数量多（阻塞时间长）
- 对可用性要求高的场景

## 3PC（三阶段提交）

### 基本思想

在 2PC 基础上增加了 CanCommit 阶段和超时机制，减少阻塞时间。

### 执行过程

这一节主要把“执行过程”放回“分布式基础 - 分布式事务”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**阶段一：CanCommit（询问阶段）**

1. 协调者向所有参与者发送 CanCommit 请求
2. 参与者判断是否可以执行事务
3. 参与者返回 Yes 或 No
4. **不执行事务，不锁定资源**

**阶段二：PreCommit（准备阶段）**

**情况 1：所有参与者都返回 Yes**
1. 协调者向所有参与者发送 PreCommit 请求
2. 参与者执行事务操作，但不提交
3. 参与者将 Undo 和 Redo 信息写入日志
4. 参与者向协调者返回 Ack

**情况 2：任一参与者返回 No 或超时**
1. 协调者向所有参与者发送 Abort 请求
2. 参与者中断事务

**阶段三：DoCommit（提交阶段）**

**情况 1：所有参与者都返回 Ack**
1. 协调者向所有参与者发送 DoCommit 请求
2. 参与者提交事务，释放资源
3. 参与者向协调者返回 Ack

**情况 2：任一参与者返回 No 或超时**
1. 协调者向所有参与者发送 Abort 请求
2. 参与者回滚事务，释放资源

**超时机制**
- 参与者在 PreCommit 后等待 DoCommit
- 如果超时未收到，**自动提交**（假设网络问题）
- 减少阻塞时间

### 3PC vs 2PC

这一节主要把“3PC vs 2PC”放回“分布式基础 - 分布式事务”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**改进**
- 增加 CanCommit 阶段，提前发现问题
- 引入超时机制，减少阻塞
- 参与者可以在超时后自动提交

**仍存在的问题**
- 网络分区时，超时自动提交可能导致数据不一致
- 性能开销更大（三阶段）
- 实际应用较少

**为什么 3PC 仍可能不一致**
- 网络分区，部分参与者收到 DoCommit，部分未收到
- 未收到的参与者超时后自动提交
- 但如果协调者实际是发送 Abort，就不一致了

## TCC（Try-Confirm-Cancel）

### 基本思想

TCC 是应用层的两阶段提交，将业务逻辑分为 Try、Confirm、Cancel 三个操作。

**核心概念**
- **Try**：尝试执行，预留资源
- **Confirm**：确认执行，使用预留资源
- **Cancel**：取消执行，释放预留资源

### 执行过程

这一节主要把“执行过程”放回“分布式基础 - 分布式事务”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**阶段一：Try**
1. 协调者调用所有参与者的 Try 接口
2. 参与者执行业务检查，预留资源
3. 参与者返回成功或失败

**阶段二：Confirm 或 Cancel**

**情况 1：所有 Try 都成功**
1. 协调者调用所有参与者的 Confirm 接口
2. 参与者使用预留资源，完成业务操作

**情况 2：任一 Try 失败**
1. 协调者调用所有参与者的 Cancel 接口
2. 参与者释放预留资源

### TCC 示例：转账

这里真正要说明的是“TCC 示例：转账”为什么值得单独拿出来讲。把它放到“分布式基础 - 分布式事务”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**Try 阶段**
- 账户 A：冻结 100 元（余额不变，冻结金额 +100）
- 账户 B：预留 100 元额度（余额不变，预留金额 +100）

**Confirm 阶段**
- 账户 A：扣减 100 元（余额 -100，冻结金额 -100）
- 账户 B：增加 100 元（余额 +100，预留金额 -100）

**Cancel 阶段**
- 账户 A：解冻 100 元（冻结金额 -100）
- 账户 B：释放预留额度（预留金额 -100）

### TCC 的工程问题

这里真正要说明的是“TCC 的工程问题”为什么值得单独拿出来讲。把它放到“分布式基础 - 分布式事务”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**问题 1：空回滚**
- Try 阶段因网络超时未执行
- 但协调者认为失败，调用 Cancel
- Cancel 需要处理"没有 Try 就 Cancel"的情况

**问题 2：幂等性**
- Try、Confirm、Cancel 都可能重复调用
- 需要保证幂等性

**问题 3：悬挂**
- Cancel 先于 Try 执行
- Try 执行后，资源被冻结，但不会被释放

**解决方案**
- 记录事务状态
- Try 前检查是否已 Cancel
- Cancel 时记录，Try 时检查

### TCC vs 2PC

| 特性 | 2PC | TCC |
|------|-----|-----|
| 实现层 | 数据库层 | 应用层 |
| 依赖 | 数据库事务 | 业务实现 |
| 性能 | 中 | 高 |
| 复杂度 | 中 | 高 |
| 锁定时间 | 长 | 短 |
| 业务侵入 | 低 | 高 |

## 最终一致性方案

### Saga 长事务

读到“Saga 长事务”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 分布式事务”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**基本思想**
- 将长事务拆分为多个本地短事务
- 每个短事务都有对应的补偿操作
- 某个短事务失败，执行之前所有成功事务的补偿操作

**执行过程**
- 正常流程：T1 → T2 → T3 → 完成
- 异常流程（T3 失败）：T1 → T2 → T3 失败 → C2 → C1 → 回滚

**特点**
- 适合长事务
- 不需要锁定资源
- 不保证隔离性，可能读到中间状态

### 本地消息表

这一节看起来往往容易变成一串结论，其实先抓住“本地消息表”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**基本思想**
- 利用本地事务保证消息一定会发送
- 通过消息队列保证最终一致性

**执行过程**
1. 业务操作和插入消息在同一个本地事务中
2. 定时任务扫描消息表，发送到消息队列
3. 消费者从消息队列接收消息，执行业务操作

**特点**
- 实现简单
- 最终一致性
- 需要额外的消息表和定时任务

## 分布式事务的权衡

### 如何选择

读到“如何选择”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 分布式事务”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**第一原则：优先避免分布式事务**
- 通过业务设计避免
- 合并服务，减少跨服务调用
- 异步化，降低一致性要求

**第二原则：根据业务特点选择**
- 核心业务：TCC
- 非核心业务：Saga、本地消息表
- 长事务：Saga
- 短事务：TCC

**第三原则：大部分场景用最终一致性**
- 通过补偿、重试、幂等保证可靠性
- 强一致性只用于核心业务

### 实际应用建议

读到“实际应用建议”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 分布式事务”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**1. 能不用就不用**
- 分布式事务复杂度高，性能开销大
- 优先通过业务设计避免

**2. 大部分场景用最终一致性**
- 消息队列 + 幂等性
- 本地消息表
- 定时任务补偿

**3. 核心业务才用 TCC**
- 金融交易、库存扣减
- 需要团队有足够的技术能力

**4. 不要用 2PC/3PC**
- 性能差，可能不一致
- 有更好的选择

## 面试问法

这一节看起来往往容易变成一串结论，其实先抓住“面试高频问法”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**为什么需要分布式事务？**
- 分布式系统中，数据分散在多个节点
- 跨节点操作需要保证一致性
- 本地事务无法跨节点

**2PC 有什么问题？**
- 同步阻塞：参与者等待时锁定资源
- 单点问题：协调者故障导致阻塞
- 数据不一致：提交阶段部分成功
- 太过保守：任一失败则全部失败

**3PC 相比 2PC 有什么改进？**
- 增加 CanCommit 阶段，提前发现问题
- 引入超时机制，减少阻塞
- 但仍可能数据不一致

**TCC 和 2PC 的区别？**
- TCC 是应用层实现，2PC 是数据库层
- TCC 需要业务实现三个接口
- TCC 性能更好，锁定时间短
- TCC 业务侵入性强

**Saga 适合什么场景？**
- 长事务
- 对隔离性要求不高
- 业务允许补偿
- 不需要锁定资源

**如何保证最终一致性？**
- 消息队列：保证消息可靠传递
- 重试机制：失败后重试
- 幂等性：重复执行结果相同
- 补偿机制：失败后回滚

**实际项目中如何选择分布式事务方案？**
- 优先避免分布式事务
- 核心业务用 TCC
- 非核心业务用 Saga 或本地消息表
- 大部分场景用最终一致性

## 易混点

这一节看起来往往容易变成一串结论，其实先抓住“易混淆点”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**2PC vs 3PC**
- 3PC 增加了 CanCommit 和超时机制
- 3PC 减少阻塞但仍可能不一致
- 实际应用 2PC 更多（虽然都不推荐）

**TCC vs Saga**
- TCC 是两阶段，Saga 是多阶段
- TCC 需要预留资源，Saga 不需要
- TCC 保证隔离性，Saga 不保证

**强一致性 vs 最终一致性**
- 强一致性实时一致，最终一致性允许延迟
- 强一致性性能差，最终一致性性能好
- 根据业务选择

**分布式事务 vs 分布式锁**
- 分布式事务保证多个操作的原子性
- 分布式锁保证多个进程的互斥性
- 解决不同问题
