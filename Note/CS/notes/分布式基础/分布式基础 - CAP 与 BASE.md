## CAP 定理

### 什么是 CAP

CAP 定理是分布式系统的基础理论，由 Eric Brewer 在 2000 年提出。它指出分布式系统不可能同时满足以下三个特性：

- **C（Consistency）一致性**：所有节点在同一时刻看到的数据是一致的
- **A（Availability）可用性**：每个请求都能在有限时间内得到响应（成功或失败）
- **P（Partition Tolerance）分区容忍性**：系统在网络分区时仍能继续工作

### 为什么会有 CAP 问题

**根本原因：网络不可靠**

分布式系统的节点通过网络通信，而网络是不可靠的。消息可能延迟、丢失、重复，节点之间可能无法通信（网络分区）。这是物理世界的客观现实，无法避免。

**单机系统没有这个问题**

单机系统不需要网络通信，所有数据在一个节点上，天然满足一致性和可用性。但单机系统有单点故障风险，无法扩展。

**分布式系统的两难**

为了高可用和可扩展，我们需要多个节点。但多个节点就需要通过网络同步数据，而网络不可靠就导致了 CAP 问题。这是分布式系统的本质矛盾。

### 一致性（Consistency）

所有节点在同一时刻看到的数据是一致的。写入数据后，立即读取能得到最新值，就像操作单机数据库一样。

**例子**：银行转账，A 转账给 B，两个账户余额必须同时更新。如果 A 扣款了但 B 没收到，就违反了一致性。

### 可用性（Availability）

每个请求都能在有限时间内得到响应，无论成功还是失败。系统不会拒绝服务，不会无限期等待。

**例子**：电商网站，即使部分服务器故障，用户仍能浏览商品。如果系统直接返回"服务不可用"，就违反了可用性。

### 分区容忍性（Partition Tolerance）

网络分区指节点之间无法通信。分区容忍性要求系统在遇到网络分区时，仍然能对外提供服务。

**例子**：机房之间的网络中断，导致两个机房的节点无法通信。系统必须能够容忍这种情况，而不是直接崩溃。

## CAP 不是三选二

### 最容易误解的地方

很多人认为 CAP 是"三选二"，即可以选择 CA、CP 或 AP。**这是错误的。**

**为什么不是三选二？**

网络分区（P）是客观存在的，无法避免。只要是分布式系统，就必须容忍网络分区。所以实际上是在 C 和 A 之间做权衡，而不是三选二。

**CA 系统不存在**

如果不容忍分区，就不是分布式系统。单机系统可以同时满足 CA，但那不是分布式系统。分布式系统必须选择 CP 或 AP。

### 网络分区下的权衡

当网络分区发生时，系统必须在一致性和可用性之间做出选择。

**选择一致性（CP）**：为了保证一致性，拒绝服务部分请求，等待网络恢复后再提供服务。牺牲了可用性。

**选择可用性（AP）**：为了保证可用性，允许读取旧数据，不同分区的数据可能不一致。牺牲了一致性。

![](https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/691f6399-6b0d-4d5c-b69f-4b2d6c9a8d3e.png)

**为什么必须二选一？**

假设网络分区发生，节点 A 和节点 B 无法通信。此时客户端向节点 A 写入数据，向节点 B 读取数据。

- 如果要保证一致性（C），节点 B 必须等待与节点 A 同步后才能响应，但网络分区导致无法同步，所以节点 B 只能拒绝服务（违反 A）。
- 如果要保证可用性（A），节点 B 必须立即响应，但无法与节点 A 同步，所以返回的是旧数据（违反 C）。

这就是 CAP 的本质：网络分区下，一致性和可用性不可兼得。

## CP 与 AP 的权衡

### CP 系统

这里真正要说明的是“CP 系统”为什么值得单独拿出来讲。把它放到“分布式基础 - CAP 与 BASE”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**特点**
- 保证强一致性
- 网络分区时部分节点不可用
- 等待网络恢复后才能提供服务

**适用场景**
- 对一致性要求高的场景
- 金融系统、库存系统
- 可以容忍短暂不可用

**典型系统**
- ZooKeeper：选举过程中不可用
- HBase：Region Server 故障时该 Region 不可用
- Redis Cluster：主节点故障时该分片不可用（直到选出新主）
- Etcd：基于 Raft，保证强一致性

**权衡**
- 优点：数据一致性有保证
- 缺点：可用性降低，用户体验可能受影响

### AP 系统

这一节主要把“AP 系统”放回“分布式基础 - CAP 与 BASE”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**特点**
- 保证高可用性
- 网络分区时仍然提供服务
- 允许数据短暂不一致

**适用场景**
- 对可用性要求高的场景
- 社交网络、内容分发
- 可以容忍短暂不一致

**典型系统**
- Cassandra：最终一致性，高可用
- DynamoDB：可配置一致性级别
- Eureka：服务注册中心，AP 优先
- CouchDB：多主复制，冲突解决

**权衡**
- 优点：系统始终可用，用户体验好
- 缺点：可能读到旧数据，需要冲突解决机制

### 如何选择

读到“如何选择”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - CAP 与 BASE”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**业务特点**
- 金融、交易、库存：选择 CP
- 社交、内容、推荐：选择 AP
- 根据业务对一致性和可用性的要求权衡

**实际应用**
- 同一个系统的不同模块可以有不同选择
- 核心业务选择 CP，非核心业务选择 AP
- 可以通过配置动态调整

## BASE 理论

### 什么是 BASE

BASE 是对 CAP 的实践指导。CAP 告诉我们"不能同时满足"，BASE 告诉我们"怎么做权衡"。

核心思想：即使无法做到强一致性，也可以根据业务特点，采用适当的方式达到最终一致性。

BASE 是以下三个短语的缩写：
- **BA（Basically Available）基本可用**
- **S（Soft State）软状态**
- **E（Eventually Consistent）最终一致性**

### 基本可用（Basically Available）

分布式系统在出现故障时，允许损失部分可用性，保证核心功能可用。

**表现形式**：响应时间变长（正常 0.5 秒，故障时 2 秒）、功能降级（只提供核心功能）、部分用户受影响（引导到降级页面）。

**例子**：电商大促时，部分用户看到"系统繁忙，请稍后再试"；推荐系统故障时，展示默认推荐内容；评论功能故障时，暂时关闭评论。

### 软状态（Soft State）

允许系统中的数据存在中间状态，并认为该中间状态不会影响系统整体可用性。即允许系统不同节点的数据副本之间进行同步的过程存在时延。

**直观理解**：数据可以有中间状态，不同节点的数据可以不同步，这个不同步是暂时的。

**例子**：主从数据库之间存在复制延迟；缓存和数据库之间存在短暂不一致；分布式缓存的不同节点数据可能不同。

### 最终一致性（Eventually Consistent）

系统中所有的数据副本，在经过一段时间的同步后，最终能达到一致的状态。

**直观理解**：不要求实时一致，但保证最终会一致。这个"最终"可能是几秒、几分钟。

**例子**：DNS 更新，全球 DNS 服务器最终会同步；社交网络点赞数，不同用户看到的数字可能不同，但最终会一致；分布式缓存失效，不同节点的缓存最终会失效。

## ACID vs BASE

### 为什么会有 BASE

ACID 是传统数据库的事务特性，适合单机系统。但分布式系统中，严格的 ACID 会导致性能和可用性问题。BASE 是对 ACID 的放松，通过牺牲强一致性换取高性能和高可用。

**ACID（传统数据库）**：原子性、一致性、隔离性、持久性。强一致性，适合单机数据库，性能和可用性受限。

**BASE（分布式系统）**：基本可用、软状态、最终一致性。最终一致性，适合分布式系统，高性能和高可用。

**两者关系**：ACID 和 BASE 不是对立的。同一个系统可以同时使用，核心业务用 ACID，非核心业务用 BASE。根据业务特点灵活选择。

## 面试问法

读到“面试高频问法”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - CAP 与 BASE”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**CAP 定理是什么？**
- 一致性、可用性、分区容忍性不能同时满足
- 网络分区是必然的，所以实际是在 C 和 A 之间权衡
- CP 系统保证一致性，AP 系统保证可用性

**为什么说 CAP 不是三选二？**
- 网络分区是客观存在的，无法避免
- 分布式系统必须容忍分区
- 所以只能在 C 和 A 之间选择
- CA 系统不存在（单机除外）

**CP 和 AP 如何选择？**
- 看业务对一致性和可用性的要求
- 金融、交易、库存选 CP
- 社交、内容、推荐选 AP
- 同一系统不同模块可以不同选择

**BASE 理论是什么？**
- 对 CAP 的实践指导
- 基本可用、软状态、最终一致性
- 通过牺牲强一致性换取高可用
- 适合大规模分布式系统

**最终一致性如何实现？**
- 异步复制：主节点写入后异步同步到从节点
- 消息队列：通过消息保证最终一致
- 补偿机制：定时任务检查并修复不一致
- 版本号/时间戳：解决冲突

**举例说明 CP 和 AP 系统**
- CP：ZooKeeper、HBase、Etcd、Redis Cluster
- AP：Cassandra、DynamoDB、Eureka、CouchDB
- 可调：MongoDB、Riak（可配置一致性级别）

## 易混点

这一节看起来往往容易变成一串结论，其实先抓住“易混淆点”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**一致性（CAP 中的 C）vs 一致性（ACID 中的 C）**
- CAP 的 C：多个副本的数据一致
- ACID 的 C：事务前后数据满足约束
- 两者含义不同

**可用性 vs 高可用**
- 可用性：系统能响应请求
- 高可用：系统持续提供服务的能力
- 高可用是可用性的量化指标

**最终一致性 vs 弱一致性**
- 最终一致性：保证最终会一致
- 弱一致性：不保证会一致
- 最终一致性是弱一致性的一种

**BASE vs ACID**
- 不是对立关系
- ACID 适合单机，BASE 适合分布式
- 可以在同一系统中结合使用
