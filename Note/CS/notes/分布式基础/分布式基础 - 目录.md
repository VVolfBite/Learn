## 核心主线：从单机到分布式的代价

### 第一步：理解为什么要分布式

**单机的三大瓶颈**
1. **性能瓶颈**：单机 CPU、内存、磁盘、网络都有上限
2. **容量瓶颈**：数据量超过单机存储能力
3. **可用性瓶颈**：单点故障导致整个系统不可用

**分布式的收益**
- 水平扩展：加机器就能提升性能
- 高可用：一台机器挂了，其他机器继续服务
- 就近部署：用户访问最近的节点，降低延迟

**但分布式不是免费的**
- 网络不可靠：消息可能丢失、延迟、重复
- 节点会故障：需要检测和恢复
- 时钟不同步：无法用时间戳判断顺序
- 状态难一致：多个副本怎么保持同步？

[分布式系统基础](分布式基础%20-%20分布式系统基础.md)

### 第二步：理解一致性的代价

**CAP 不是三选二**
- 网络分区（P）是必然的，你躲不掉
- 真正的选择是：分区时，要一致性（C）还是可用性（A）？
- CP 系统：宁可拒绝服务，也要保证数据一致（ZooKeeper、Etcd）
- AP 系统：宁可返回旧数据，也要保证服务可用（Cassandra、Eureka）

**BASE 不是放弃一致性**
- 是放弃"强一致性"，选择"最终一致性"
- 允许短暂不一致，但保证最终会一致
- 大部分业务场景都可以接受

**一致性协议是干什么的**
- 让多个节点对某个值达成共识
- Paxos 是理论基础，难懂难实现
- Raft 是工程化方案，易懂易实现
- 面试知道 Leader 选举、日志复制就够了

### 第三步：理解分布式事务和锁

**为什么本地事务不够**
- 订单库和库存库在不同数据库
- 下单要同时扣库存，怎么保证原子性？
- 本地事务只能管一个数据库

**分布式事务的权衡**
- 2PC：强一致，但有阻塞、单点、数据不一致风险
- 3PC：改进了 2PC，但仍可能不一致
- TCC：应用层实现，性能好，但开发成本高
- Saga：适合长事务，但不保证隔离性
- 最终一致性：大部分场景的选择

**为什么需要分布式锁**
- 多个服务实例同时扣库存，怎么保证不超卖？
- JVM 锁只能管单个进程，跨进程需要分布式锁
- Redis 实现简单但可能丢锁
- ZooKeeper 实现可靠但性能较低

### 第四步：理解缓存和消息队列

**缓存是高频中的高频**
- 为什么要缓存？数据库太慢
- 三大问题：穿透（查不存在的）、击穿（热点过期）、雪崩（大量过期）
- 一致性问题：先更新数据库，再删除缓存
- 不要过度追求强一致性

**消息队列的三大作用**
- 解耦：服务之间不直接依赖
- 削峰：缓冲突发流量
- 异步：提高响应速度

**消息队列的代价**
- 系统复杂度增加
- 可能丢消息：需要保证可靠性
- 可能重复消费：需要保证幂等性
- 可能乱序：需要保证顺序性

### 第五步：理解高可用和服务治理

**负载均衡**
- 把请求分发到多个服务器
- 常见算法：轮询、加权轮询、最少连接、IP Hash
- Session 一致性：用 Session 服务器（Redis）

**服务治理是微服务的基础**
- 服务注册与发现：服务动态上下线
- 熔断：下游故障时快速失败
- 限流：控制流量，保护系统
- 降级：保证核心功能，牺牲非核心功能

- 缓存穿透 vs 击穿 vs 雪崩

## 常见误区

**误区1：CAP 是三选二**  
 正确：P 是必须的，实际是在 C 和 A 之间选择

 **误区2：BASE 是放弃一致性**  
 正确：是放弃强一致性，选择最终一致性

 **误区3：分布式锁可以完全避免死锁**  
 正确：仍可能死锁，需要超时机制

 **误区4：缓存一定要保证强一致性**  
 正确：大部分场景允许短暂不一致

 **误区5：消息队列可以保证强一致性**  
 正确：只能保证最终一致性

 **误区6：2PC 可以完全保证数据一致性**  
 正确：提交阶段可能部分成功，导致不一致

 **误区7：微服务一定比单体应用好**  
 正确：看场景，微服务增加了复杂度
