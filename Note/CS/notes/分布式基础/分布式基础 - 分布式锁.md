## 为什么需要分布式锁

### JVM 锁的局限

**JVM 锁是什么**
- synchronized、ReentrantLock
- 只能在单个 JVM 进程内生效
- 多线程竞争同一资源

**为什么 JVM 锁不够**

**场景 1：库存扣减**
```
电商系统部署了 3 个实例：
- 实例 A：查询库存 10，扣减 1
- 实例 B：查询库存 10，扣减 1
- 实例 C：查询库存 10，扣减 1

问题：JVM 锁只能管单个实例，无法跨实例
结果：库存应该是 7，实际是 9（超卖）
```

**场景 2：定时任务**
```
定时任务部署了 3 个实例：
- 每天凌晨 1 点执行数据统计

问题：3 个实例都会执行，重复处理
需要：只让一个实例执行
```

**场景 3：缓存更新**
```
缓存失效，多个实例同时查询数据库：
- 实例 A：查询数据库
- 实例 B：查询数据库
- 实例 C：查询数据库

问题：数据库压力大（缓存击穿）
需要：只让一个实例查询
```

### 分布式锁的本质

这里真正要说明的是“分布式锁的本质”为什么值得单独拿出来讲。把它放到“分布式基础 - 分布式锁”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**跨进程的互斥**
- 多个进程竞争同一资源
- 同一时刻只有一个进程能持有锁
- 依赖外部存储（Redis、ZooKeeper、数据库）

**不是 JVM 锁的简单扩展**
- 网络不可靠：消息可能丢失、延迟
- 节点可能故障：持有锁的进程可能崩溃
- 时钟不同步：无法用时间戳判断超时

## 分布式锁的基本要求

### 必须满足的特性

这一节看起来往往容易变成一串结论，其实先抓住“必须满足的特性”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**1. 互斥性（Mutual Exclusion）**
- 同一时刻只有一个客户端能持有锁
- 这是锁的基本要求

**2. 防死锁（Deadlock Free）**
- 即使持有锁的客户端崩溃，锁也能被释放
- 通过超时机制实现

**3. 高可用（High Availability）**
- 锁服务高可用，不能成为单点
- 获取锁和释放锁的成功率要高

**4. 可重入（Reentrant）**
- 同一个客户端可以多次获取同一把锁
- 避免死锁

### 可选的特性

这一节看起来往往容易变成一串结论，其实先抓住“可选的特性”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**公平性（Fairness）**
- 按照请求顺序获取锁
- 避免饥饿

**阻塞与非阻塞**
- 阻塞：获取不到锁时等待
- 非阻塞：获取不到锁时立即返回

**自动续期**
- 业务执行时间超过锁超时时间
- 自动延长锁的有效期

## 基于 Redis 的实现

### 基本实现

**获取锁**
```java
// SET key value NX EX seconds
String result = redis.set(key, value, "NX", "EX", 30);
if ("OK".equals(result)) {
    // 获取锁成功
}
```

**NX 参数**
- SET if Not eXists
- 只有 key 不存在时才设置
- 保证互斥性

**EX 参数**
- 设置过期时间（秒）
- 防止死锁

**释放锁**
```java
// 使用 Lua 脚本保证原子性
String script = 
    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
    "    return redis.call('del', KEYS[1]) " +
    "else " +
    "    return 0 " +
    "end";
redis.eval(script, Collections.singletonList(key), Collections.singletonList(value));
```

### Redis 锁的五大问题

**问题 1：锁超时**

**问题描述**
- 业务执行时间超过锁的过期时间
- 锁自动释放，其他客户端获取到锁
- 两个客户端同时持有锁

**为什么会发生**
```
时间线：
T1: 客户端 A 获取锁，过期时间 30 秒
T2: 客户端 A 执行业务，耗时 40 秒
T31: 锁自动过期，客户端 B 获取锁
T41: 客户端 A 执行完成，释放锁（释放了客户端 B 的锁）
```

**解决方案 1：设置合理的过期时间**
- 根据业务执行时间设置
- 留有余量（如业务 10 秒，设置 30 秒）

**解决方案 2：自动续期（Watchdog）**
```java
// Redisson 的 Watchdog 机制
RLock lock = redisson.getLock("myLock");
lock.lock(); // 默认 30 秒，每 10 秒自动续期
try {
    // 业务逻辑，无论多久都不会超时
} finally {
    lock.unlock();
}
```

**Watchdog 原理**
- 获取锁时启动一个定时任务
- 每隔 1/3 过期时间（如 10 秒）检查一次
- 如果锁还在，延长过期时间
- 释放锁时停止定时任务

**问题 2：误释放**

**问题描述**
- 客户端 A 的锁超时，客户端 B 获取锁
- 客户端 A 执行完成，释放了客户端 B 的锁

**为什么会发生**
```
时间线：
T1: 客户端 A 获取锁，value = "A"
T31: 锁超时，客户端 B 获取锁，value = "B"
T41: 客户端 A 执行完成，DEL key（删除了客户端 B 的锁）
```

**解决方案：释放锁时检查 value**
```java
// 错误做法
redis.del(key);

// 正确做法：使用 Lua 脚本保证原子性
String script = 
    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
    "    return redis.call('del', KEYS[1]) " +
    "else " +
    "    return 0 " +
    "end";
redis.eval(script, Collections.singletonList(key), Collections.singletonList(value));
```

**为什么要用 Lua 脚本**
- GET 和 DEL 不是原子操作
- 可能 GET 后，锁被其他客户端获取
- Lua 脚本在 Redis 中原子执行

**问题 3：不可重入**

**问题描述**
- 同一个客户端无法多次获取同一把锁
- 可能导致死锁

**为什么会发生**
```java
public void methodA() {
    lock.lock();
    try {
        methodB(); // 调用 methodB
    } finally {
        lock.unlock();
    }
}

public void methodB() {
    lock.lock(); // 无法获取锁，死锁
    try {
        // 业务逻辑
    } finally {
        lock.unlock();
    }
}
```

**解决方案：使用 Hash 结构记录重入次数**
```java
// Redisson 的可重入锁实现
// Hash 结构：key -> {clientId: count}
HINCRBY lock_key client_id 1  // 获取锁，重入次数 +1
HINCRBY lock_key client_id -1 // 释放锁，重入次数 -1
// 重入次数为 0 时删除锁
```

**问题 4：主从切换丢锁**

**问题描述**
- Redis 主从复制是异步的
- 主节点获取锁后，还未同步到从节点
- 主节点宕机，从节点升级为主节点
- 锁丢失，其他客户端可以获取锁

**为什么会发生**
```
时间线：
T1: 客户端 A 向主节点获取锁，成功
T2: 主节点还未同步到从节点
T3: 主节点宕机，从节点升级为主节点
T4: 客户端 B 向新主节点获取锁，成功（锁丢失）
```

**解决方案：RedLock 算法**
- 使用多个独立的 Redis 实例（不是主从）
- 只有在多数实例上获取锁才算成功

**问题 5：锁续期失败**

**问题描述**
- Watchdog 续期时，网络故障
- 续期失败，锁过期
- 其他客户端获取锁

**解决方案**
- 监控续期失败
- 业务逻辑中检查锁是否还在
- 如果锁丢失，停止执行或回滚

### RedLock 算法

这里真正要说明的是“RedLock 算法”为什么值得单独拿出来讲。把它放到“分布式基础 - 分布式锁”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**基本思想**
- 使用 N 个独立的 Redis 实例（推荐 5 个）
- 只有在多数实例（N/2 + 1）上获取锁才算成功

**执行过程**

1. 获取当前时间戳 T1
2. 依次向 N 个 Redis 实例请求锁
   - 使用相同的 key 和 value
   - 设置较短的超时时间（如 5-50ms）
3. 计算获取锁的耗时 T2 - T1
4. 如果满足以下条件，则获取锁成功：
   - 在多数实例（N/2 + 1）上获取到锁
   - 获取锁的耗时小于锁的有效时间
5. 如果获取锁失败，向所有实例释放锁

**为什么需要多数**
- 防止脑裂
- 保证同一时刻只有一个客户端能获取锁

**为什么要计算耗时**
- 获取锁耗时过长，锁可能已经过期
- 需要保证锁的有效时间

**RedLock 的争议**

**Martin Kleppmann 的质疑**
- RedLock 依赖时钟同步
- 时钟跳跃可能导致锁失效
- 不如使用 ZooKeeper 或 Chubby

**Antirez（Redis 作者）的回应**
- RedLock 不依赖绝对时钟
- 只依赖相对时间
- 适合大部分场景

**实际应用建议**
- 对可靠性要求极高：用 ZooKeeper
- 对性能要求高：用 Redis 单实例 + Watchdog
- 折中方案：用 RedLock

## 基于 ZooKeeper 的实现

### 临时顺序节点

读到“临时顺序节点”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 分布式锁”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**ZooKeeper 特性**
- 临时节点：会话结束自动删除
- 顺序节点：自动追加递增序号
- 监听机制：节点变化时通知客户端

**实现思路**

1. 创建锁目录 `/locks`
2. 客户端在 `/locks` 下创建临时顺序节点
3. 获取 `/locks` 下所有子节点
4. 如果自己的节点序号最小，则获取锁成功
5. 否则监听前一个节点，等待通知
6. 业务完成后删除节点，释放锁

**为什么监听前一个节点**
- 避免羊群效应
- 如果监听所有节点，一个节点释放锁，所有客户端都会被唤醒
- 只监听前一个节点，只有下一个客户端会被唤醒

### ZooKeeper 锁的特点

读到“ZooKeeper 锁的特点”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 分布式锁”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**优点**
- 可靠性高，ZooKeeper 保证强一致性
- 自动释放，会话结束节点自动删除
- 支持阻塞等待
- 避免羊群效应

**缺点**
- 性能较低，需要多次网络通信
- 依赖 ZooKeeper 集群
- 实现相对复杂

## 面试问法

这一节看起来往往容易变成一串结论，其实先抓住“面试高频问法”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**为什么需要分布式锁？**
- 分布式环境下，JVM 锁无法跨进程
- 多个实例竞争同一资源需要互斥
- 典型场景：库存扣减、定时任务、缓存更新

**分布式锁需要满足哪些特性？**
- 互斥性：同一时刻只有一个客户端持有锁
- 防死锁：锁能自动释放
- 高可用：锁服务高可用
- 可重入：同一客户端可多次获取

**Redis 实现分布式锁有什么问题？**
- 锁超时：业务执行时间超过锁过期时间
- 误释放：释放了其他客户端的锁
- 主从切换：主节点宕机可能丢锁
- 不可重入：需要自己实现

**如何解决 Redis 锁超时问题？**
- 设置合理的过期时间
- 自动续期（Watchdog）
- Redisson 的 RLock 自动续期

**如何解决 Redis 锁误释放问题？**
- 释放锁时检查 value 是否是自己的
- 使用 Lua 脚本保证原子性

**RedLock 是什么？**
- 使用多个独立 Redis 实例
- 在多数实例上获取锁才算成功
- 提高可靠性，避免单点故障

**Redis 和 ZooKeeper 实现分布式锁的区别？**
- Redis 性能高，ZooKeeper 可靠性高
- Redis 需要自己处理超时、续期，ZooKeeper 自动处理
- Redis 不支持阻塞等待，ZooKeeper 支持
- Redis 适合高并发，ZooKeeper 适合高可靠性要求

**实际项目中如何选择？**
- 高并发、低延迟：Redis
- 高可靠性、强一致性：ZooKeeper
- 简单场景：数据库
- 推荐使用成熟框架：Redisson、Curator

## 易混点

这里真正要说明的是“易混淆点”为什么值得单独拿出来讲。把它放到“分布式基础 - 分布式锁”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**分布式锁 vs JVM 锁**
- JVM 锁只能在单个进程内生效
- 分布式锁可以跨进程、跨机器
- 分布式锁依赖外部存储

**分布式锁 vs 分布式事务**
- 分布式锁保证互斥性
- 分布式事务保证原子性
- 解决不同问题

**Redis 锁 vs ZooKeeper 锁**
- Redis 性能高但可靠性相对低
- ZooKeeper 可靠性高但性能相对低
- 根据场景选择

**SETNX vs SET NX**
- SETNX 是旧命令
- SET NX 是新命令，支持同时设置过期时间
- 推荐使用 SET NX

**锁超时 vs 误释放**
- 锁超时：业务执行时间超过锁过期时间
- 误释放：释放了其他客户端的锁
- 两者有因果关系
