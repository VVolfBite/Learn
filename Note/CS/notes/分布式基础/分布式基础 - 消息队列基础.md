## 为什么需要消息队列

### 同步调用的问题

**场景：用户下单**
```
用户下单 → 订单服务 → 库存服务 → 支付服务 → 积分服务 → 通知服务
```

**同步调用的问题**
- **响应慢**：需要等待所有服务执行完成，用户等待时间长
- **强耦合**：订单服务依赖所有下游服务，任一服务故障都会失败
- **难扩展**：增加新服务需要修改订单服务代码
- **峰值压力**：秒杀时，所有服务都承受峰值压力

### 消息队列的作用

这一节主要把“消息队列的作用”放回“分布式基础 - 消息队列基础”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**解耦（Decoupling）**
- 服务之间不直接依赖
- 通过消息队列通信
- 增加新服务不需要修改现有代码

**削峰（Peak Clipping）**
- 缓冲突发流量
- 消费者按自己的速度处理
- 保护下游服务不被压垮

**异步（Asynchronous）**
- 不需要等待下游服务执行完成
- 快速返回给用户
- 提高响应速度

## 消息队列的基本模型

### 点对点模型（Queue）

读到“点对点模型（Queue）”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 消息队列基础”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**特点**
- 一条消息只能被一个消费者消费
- 消费后消息被删除
- 支持多个消费者，但每条消息只被消费一次

**适用场景**
- 任务分发
- 负载均衡

### 发布订阅模型（Topic）

**特点**
- 一条消息可以被多个消费者消费
- 每个消费者都有自己的消费进度
- 消费后消息不会被删除（保留一段时间）

**适用场景**
- 事件通知
- 数据同步
- 日志收集

## 消息可靠性

### 为什么会丢消息

读到“为什么会丢消息”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 消息队列基础”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**生产者丢失**
- 生产者发送消息后，网络故障
- 消息未到达 MQ

**MQ 丢失**
- 消息在内存中，MQ 宕机
- 消息未持久化

**消费者丢失**
- 消费者拉取消息后，还未处理就宕机
- 消息已被标记为消费

### 如何保证消息不丢失

**生产者确认（Producer Acknowledgement）**

```java
// 同步发送，等待确认
SendResult result = producer.send(message);
if (result.getSendStatus() == SendStatus.SEND_OK) {
    // 发送成功
}

// 异步发送，回调确认
producer.send(message, new SendCallback() {
    @Override
    public void onSuccess(SendResult result) {
        // 发送成功
    }
    
    @Override
    public void onException(Throwable e) {
        // 发送失败，重试或记录
    }
});
```

**MQ 持久化（Persistence）**
- 消息写入磁盘
- 主从复制，多副本
- 保证 MQ 宕机后消息不丢失

**消费者确认（Consumer Acknowledgement）**

```java
// 手动确认
consumer.subscribe("topic", new MessageListener() {
    @Override
    public void onMessage(Message message) {
        try {
            // 处理消息
            processMessage(message);
            // 确认消费
            consumer.ack(message);
        } catch (Exception e) {
            // 处理失败，不确认，消息会重新投递
        }
    }
});
```

**关键点**
- 先处理消息，再确认
- 处理失败不确认，消息会重新投递
- 保证消息至少被消费一次

### 消息可靠性的代价

这里真正要说明的是“消息可靠性的代价”为什么值得单独拿出来讲。把它放到“分布式基础 - 消息队列基础”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**性能下降**
- 同步发送等待确认，吞吐量降低
- 持久化写磁盘，性能下降

**可能重复消费**
- 消费者处理完消息，确认前宕机
- 消息会重新投递，导致重复消费

**需要权衡**
- 核心业务：保证可靠性
- 非核心业务：可以容忍少量丢失

## 重复消费与幂等性

### 为什么会重复消费

这一节主要把“为什么会重复消费”放回“分布式基础 - 消息队列基础”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**场景 1：消费者确认失败**
- 消费者处理完消息
- 发送确认时网络故障
- MQ 认为消息未消费，重新投递

**场景 2：消费者超时**
- 消费者处理消息时间过长
- MQ 认为消费者故障，重新投递
- 消费者处理完成，但消息已被重新投递

**场景 3：MQ 重启**
- MQ 重启后，未确认的消息重新投递

### 如何保证幂等性

读到“如何保证幂等性”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 消息队列基础”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**幂等性的定义**
- 重复执行结果相同
- 执行一次和执行多次效果一样

**方案 1：唯一 ID + 去重表（推荐）**

```java
@Transactional
public void processMessage(Message message) {
    String messageId = message.getId();
    
    // 检查是否已处理
    if (deduplicationTable.exists(messageId)) {
        return; // 已处理，直接返回
    }
    
    // 处理业务逻辑
    doBusinessLogic(message);
    
    // 记录已处理
    deduplicationTable.insert(messageId);
}
```

**方案 2：业务层面幂等**
- 更新操作：`UPDATE SET status = 1 WHERE id = ? AND status = 0`
- 插入操作：使用唯一索引，重复插入会失败
- 删除操作：删除不存在的记录不会报错

**方案 3：分布式锁**
- 使用消息 ID 作为锁的 key
- 获取锁成功才处理消息
- 处理完成后释放锁

**关键点**
- 消息必须有唯一 ID
- 去重表需要定期清理
- 业务逻辑要设计成幂等的

## 顺序消息

### 为什么会乱序

**场景：订单状态变更**
```
创建订单 → 支付订单 → 发货 → 完成
```

**乱序的原因**
- 多个消费者并行消费
- 网络延迟不同
- 消费者处理速度不同

**乱序的后果**
- 先收到"发货"，后收到"支付"
- 订单状态错乱

### 如何保证顺序

这里真正要说明的是“如何保证顺序”为什么值得单独拿出来讲。把它放到“分布式基础 - 消息队列基础”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**方案 1：单分区 + 单消费者（强顺序）**
- 所有消息发送到同一个分区
- 只有一个消费者消费
- 保证严格顺序

**缺点**
- 吞吐量低
- 无法并行消费
- 消费者故障影响大

**方案 2：按 Key 分区（局部顺序）**

```java
// 生产者：使用订单 ID 作为 key
producer.send(message, orderId);

// MQ 根据 key 的 hash 值分配到同一个分区
// 同一个订单的消息在同一个分区，保证顺序
```

**优点**
- 不同订单可以并行消费
- 吞吐量高

**缺点**
- 只保证同一个 key 的消息有序
- 不同 key 的消息可能乱序

**方案 3：业务层面处理**
- 消息带上版本号或时间戳
- 消费者根据版本号判断是否处理
- 乱序的消息暂存，等待前置消息

### 顺序消息的代价

这一节看起来往往容易变成一串结论，其实先抓住“顺序消息的代价”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**性能下降**
- 单分区吞吐量有限
- 无法充分利用并行消费

**可用性降低**
- 单消费者故障影响大
- 需要快速故障转移

**需要权衡**
- 大部分场景不需要严格顺序
- 可以通过业务设计避免顺序依赖

## 消息积压

### 为什么会积压

这里真正要说明的是“为什么会积压”为什么值得单独拿出来讲。把它放到“分布式基础 - 消息队列基础”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**生产速度 > 消费速度**
- 突发流量：秒杀、促销
- 消费者故障：宕机、处理慢
- 消费者性能差：数据库慢查询

**积压的后果**
- 消息延迟增加
- 内存占用增加
- 可能导致 MQ 故障

### 如何处理积压

这里真正要说明的是“如何处理积压”为什么值得单独拿出来讲。把它放到“分布式基础 - 消息队列基础”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**方案 1：扩容消费者（推荐）**
- 增加消费者实例
- 提高并行消费能力
- 快速消化积压

**方案 2：批量消费**
```java
// 一次拉取多条消息
List<Message> messages = consumer.poll(100);
for (Message message : messages) {
    processMessage(message);
}
```

**方案 3：降级处理**
- 非核心消息直接丢弃或延后处理
- 核心消息优先处理
- 保证核心功能可用

**方案 4：临时队列**
- 将积压消息转移到临时队列
- 慢慢消化，不影响新消息

**预防措施**
- 监控消息积压量
- 设置告警阈值
- 提前扩容

## 消息传递语义

### 三种语义

这里真正要说明的是“三种语义”为什么值得单独拿出来讲。把它放到“分布式基础 - 消息队列基础”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**至多一次（At Most Once）**
- 消息可能丢失
- 消息不会重复
- 性能最好

**至少一次（At Least Once）**
- 消息不会丢失
- 消息可能重复
- 需要保证幂等性
- 最常用

**恰好一次（Exactly Once）**
- 消息不会丢失
- 消息不会重复
- 实现复杂，性能差
- 很少使用

### 如何选择

这一节看起来往往容易变成一串结论，其实先抓住“如何选择”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**至多一次**
- 日志收集
- 监控数据
- 可以容忍少量丢失

**至少一次**
- 订单处理
- 支付通知
- 大部分业务场景

**恰好一次**
- 金融交易
- 对一致性要求极高的场景
- 实现复杂，慎用

**如何保证消息不丢失？**
- 生产者确认：等待 MQ 确认

- RabbitMQ：功能丰富，适合复杂路由
- RocketMQ：阿里开源，适合电商场景
- 根据场景选择

## 易混点

这一节主要把“易混淆点”放回“分布式基础 - 消息队列基础”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**消息可靠性 vs 幂等性**
- 消息可靠性：保证消息不丢失
- 幂等性：保证重复消费不会产生副作用
- 两者解决不同问题

**点对点 vs 发布订阅**
- 点对点：一条消息只能被一个消费者消费
- 发布订阅：一条消息可以被多个消费者消费

**同步发送 vs 异步发送**
- 同步发送：等待确认，可靠性高，性能低
- 异步发送：不等待确认，性能高，可靠性相对低

**消息积压 vs 消息丢失**
- 消息积压：消息在队列中，未被消费
- 消息丢失：消息不在队列中，永久丢失

**解耦 vs 异步**
- 解耦：服务之间不直接依赖
- 异步：不需要等待下游服务执行完成
- 两者是不同的收益
