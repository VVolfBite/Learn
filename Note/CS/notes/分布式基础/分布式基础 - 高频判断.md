## CAP 相关

### CAP 不是简单三选二

**错误**：CAP 让你在一致性、可用性、分区容忍性中选两个

**正确**：
- 网络分区（P）是客观存在的，你躲不掉
- 分布式系统必须容忍分区
- 真正的选择是：分区发生时，要一致性（C）还是可用性（A）？

**面试怎么答**：CAP 是在网络分区发生时，在一致性和可用性之间做权衡，不是简单的三选二，P 是必须容忍的。

### BASE 不是放弃一致性

**错误**：BASE 就是不要一致性

**正确**：BASE 是放弃"强一致性"，选择"最终一致性"，允许短暂不一致，但保证最终会一致。

**面试怎么答**：BASE 是对 CAP 的实践指导，通过牺牲强一致性换取高可用，适合大规模分布式系统。

### 强一致性 vs 最终一致性

**强一致性**：任何时刻读到的都是最新数据，写入后立即可见，代价是性能差、可用性低。例子：ZooKeeper、Etcd

**最终一致性**：允许短暂不一致，保证最终会一致，代价是可能读到旧数据。例子：Cassandra、DNS

**面试怎么答**：强一致性实时一致但性能差，最终一致性允许延迟但性能好，根据业务选择。

## 分布式事务相关

### 2PC 的问题

**错误**：2PC 可以完全保证数据一致性

**正确**：2PC 有四大问题：
1. **同步阻塞**：参与者等待时锁定资源
2. **单点问题**：协调者故障导致阻塞
3. **数据不一致**：提交阶段部分成功
4. **太过保守**：任一失败则全部失败

**面试怎么答**：2PC 不能完全保证数据一致性，提交阶段可能部分成功，实际场景中需要人工介入。

### 3PC vs 2PC

**错误**：3PC 完全解决了 2PC 的问题

**正确**：3PC 增加了 CanCommit 阶段和超时机制，减少了阻塞时间，但仍可能数据不一致。

**面试怎么答**：3PC 改进了 2PC，但仍可能不一致，实际应用中 2PC 更多。

### TCC vs 2PC

**2PC**：数据库层实现，依赖数据库事务，实现简单，性能一般

**TCC**：应用层实现，不依赖数据库事务，需要业务实现 Try、Confirm、Cancel，性能好但开发成本高

**面试怎么答**：TCC 是应用层的两阶段提交，性能比 2PC 好，但业务侵入性强。

### 分布式事务的选择

**错误**：分布式事务一定要用 2PC 或 TCC

**正确**：
- 优先避免分布式事务
- 大部分场景使用最终一致性
- 核心业务才用 TCC

**如何避免**：合并服务，减少跨服务调用；异步化，降低一致性要求；业务设计避免

**面试怎么答**：优先避免分布式事务，实际场景中大部分使用最终一致性，强一致性只用于核心业务。

## 分布式锁相关

### 分布式锁 vs JVM 锁

**JVM 锁**：只能在单个 JVM 进程内生效，性能高，实现简单。例子：synchronized、ReentrantLock

**分布式锁**：可以跨进程、跨机器，性能相对低，实现复杂，依赖外部存储（Redis、ZooKeeper）

**面试怎么答**：JVM 锁无法解决分布式环境下的并发问题，分布式锁依赖外部存储，单机环境优先使用 JVM 锁。

### 分布式锁不是万无一失

**错误**：分布式锁可以完全避免死锁

**正确**：分布式锁仍可能死锁，需要超时机制，需要处理锁超时、误释放、续期等问题。

**常见问题**：
1. **锁超时**：业务执行时间超过锁过期时间
2. **误释放**：释放了其他客户端的锁
3. **不可重入**：同一客户端无法多次获取
4. **主从切换**：Redis 主从切换可能丢锁

**面试怎么答**：分布式锁不能完全避免死锁，需要超时机制和自动续期，Redis 实现简单但可能丢锁。

### Redis 锁 vs ZooKeeper 锁

**Redis 锁**：性能高，实现简单，主从切换可能丢锁，适合高并发场景

**ZooKeeper 锁**：可靠性高，实现复杂，性能较低，适合高可靠性要求

**面试怎么答**：Redis 性能高但可能丢锁，ZooKeeper 可靠但性能低，根据场景选择。

## 缓存相关

### 缓存穿透 vs 击穿 vs 雪崩

**缓存穿透**：查询不存在的数据，缓存和数据库都没有，每次请求都打到数据库。解决：布隆过滤器、缓存空值

**缓存击穿**：热点 key 过期，大量请求同时查询，瞬间打到数据库。解决：互斥锁、永不过期

**缓存雪崩**：大量 key 同时过期或缓存宕机，大量请求打到数据库。解决：过期时间加随机值、集群、限流降级

**对比**：
- 穿透：数据不存在，单个请求，持续影响
- 击穿：热点过期，单个 key，瞬时影响
- 雪崩：大量过期/宕机，大量 key，瞬时影响

**面试怎么答**：穿透是数据不存在，击穿是热点过期，雪崩是大量过期，三者解决方案不同，都会导致数据库压力大。

### 缓存一致性无法零成本保证

**错误**：缓存一定要保证强一致性

**正确**：缓存和数据库很难保证强一致性，代价太大，性能差，大部分场景允许短暂不一致。

**最佳实践**：先更新数据库，再删除缓存；重试机制；设置过期时间兜底；允许短暂不一致

**面试怎么答**：缓存很难保证强一致性，大部分场景不需要，允许短暂不一致。

### 为什么删除缓存而不是更新缓存

**删除缓存**：实现简单，下次读取时再加载，推荐

**更新缓存**：可能浪费（数据可能不会被读取），实现复杂（需要重新计算），不推荐

**面试怎么答**：删除缓存简单，下次读取时再加载；更新缓存可能浪费，实现复杂。

## 消息队列相关

### 消息可靠性 vs 幂等性

**消息可靠性**：保证消息不丢失，消息一定会被消费。解决：生产者确认、持久化、消费者确认

**幂等性**：重复执行结果相同，保证重复消费不会产生副作用。解决：唯一 ID + 去重表、业务层面幂等

**关系**：解决不同问题，消息可靠性可能导致重复消费，幂等性解决重复消费问题

**面试怎么答**：消息可靠性和幂等性解决不同问题，保证消息可靠性可能导致重复消费，幂等性解决重复消费问题。

### 消息队列不是零代价

**错误**：消息队列只有好处，没有坏处

**正确**：消息队列增加了系统复杂度，可能丢消息、重复消费、乱序，需要权衡。

**代价**：
1. 系统复杂度增加
2. 可能丢消息
3. 可能重复消费
4. 可能乱序
5. 可能积压

**面试怎么答**：消息队列不是零代价，增加了系统复杂度，需要权衡。

### 消息队列不能保证强一致性

**错误**：消息队列可以保证强一致性

**正确**：消息队列只能保证最终一致性，消息有延迟，不能保证实时一致。

**面试怎么答**：消息队列只能保证最终一致性，不能保证强一致性。

## 服务治理相关

### 熔断 vs 限流 vs 降级

**熔断**：下游服务故障时快速失败，被动触发，保护上游服务

**限流**：控制请求速率，主动控制，保护系统不被压垮

**降级**：保证核心功能，牺牲非核心功能，主动或被动，提高可用性

**关系**：熔断是降级的一种，限流可以触发降级，三者配合使用

**面试怎么答**：熔断是快速失败，限流是控制流量，降级是保证核心；熔断是被动触发，限流是主动控制；三者配合使用效果最好。

## 其他高频判断

### 分布式系统一定比单机好吗？

**错误**：分布式系统一定比单机系统性能好

**正确**：不一定，分布式有通信开销，需要处理一致性、故障等问题，小规模系统单机可能更好。

### 微服务一定比单体应用好吗？

**错误**：微服务一定比单体应用好

**正确**：看场景，微服务增加了系统复杂度，需要服务治理、分布式事务等，小团队、小项目单体应用可能更好。

### 水平扩展一定比垂直扩展好吗？

**错误**：水平扩展一定比垂直扩展好

**正确**：看场景，水平扩展需要解决数据分片、负载均衡等问题，初期垂直扩展更简单，大规模系统才需要水平扩展。

## 快速记忆口诀

**CAP**：分区必然，C 和 A 二选一

**缓存三大问题**：穿透查不存在，击穿热点过期，雪崩大量过期

**熔断限流降级**：熔断快速失败，限流控制流量，降级保证核心

**分布式事务**：2PC 两阶段，3PC 三阶段，TCC 应用层；优先避免，大部分用最终一致性

**消息队列**：解耦削峰异步，可靠幂等顺序

**分布式锁**：互斥防死锁，超时要续期
