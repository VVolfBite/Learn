## 为什么需要负载均衡

### 单机的局限

这一节看起来往往容易变成一串结论，其实先抓住“单机的局限”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**性能瓶颈**
- 单机 QPS 有上限
- CPU、内存、网络带宽都有限
- 无法应对高并发

**可用性问题**
- 单点故障导致服务不可用
- 维护升级需要停机

**扩展性差**
- 垂直扩展（升级硬件）成本高，有天花板
- 无法灵活应对流量变化

### 负载均衡的作用

这一节主要把“负载均衡的作用”放回“分布式基础 - 负载均衡与集群”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**提高性能**
- 将请求分发到多台服务器
- 充分利用每台服务器的资源
- 提高整体吞吐量

**提高可用性**
- 单台服务器故障，其他服务器继续服务
- 实现故障转移（Failover）
- 提高系统可用性

**水平扩展**
- 通过增加服务器数量提升性能
- 成本低，理论上可以无限扩展
- 灵活应对流量变化

## 负载均衡算法

### 轮询（Round Robin）

这一节主要把“轮询（Round Robin）”放回“分布式基础 - 负载均衡与集群”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**原理**
- 按顺序依次分配请求
- 第一个请求给服务器 1，第二个给服务器 2，以此类推
- 循环往复

**优点**
- 实现简单
- 请求分配均匀

**缺点**
- 不考虑服务器性能差异
- 不考虑服务器负载情况

**适用场景**
- 服务器性能相同
- 请求处理时间相近

### 加权轮询（Weighted Round Robin）

这里真正要说明的是“加权轮询（Weighted Round Robin）”为什么值得单独拿出来讲。把它放到“分布式基础 - 负载均衡与集群”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**原理**
- 根据服务器性能分配权重
- 性能好的服务器权重高，分配更多请求
- 按权重比例分配

**示例**
```
服务器 A：权重 3
服务器 B：权重 2
服务器 C：权重 1

分配顺序：A A A B B C A A A B B C ...
```

**优点**
- 考虑服务器性能差异
- 充分利用高性能服务器

**缺点**
- 需要手动配置权重
- 不能动态调整

**适用场景**
- 服务器性能不同
- 性能差异已知

### 最少连接（Least Connections）

这一节看起来往往容易变成一串结论，其实先抓住“最少连接（Least Connections）”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**原理**
- 将请求分配给当前连接数最少的服务器
- 动态考虑服务器负载

**优点**
- 考虑服务器实时负载
- 适合长连接场景

**缺点**
- 需要维护连接数
- 实现相对复杂

**适用场景**
- 请求处理时间差异大
- 长连接场景

### IP Hash

读到“IP Hash”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 负载均衡与集群”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**原理**
- 根据客户端 IP 的 hash 值分配服务器
- 同一个 IP 的请求总是分配到同一台服务器

**优点**
- 实现简单
- 同一用户的请求在同一台服务器，适合有状态服务

**缺点**
- 可能分配不均匀（IP 分布不均）
- 服务器增减会导致大量请求重新分配

**适用场景**
- 需要 Session 一致性
- 有状态服务

### 一致性哈希（Consistent Hashing）

这一节主要把“一致性哈希（Consistent Hashing）”放回“分布式基础 - 负载均衡与集群”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**原理**
- 将服务器和请求都映射到一个环上
- 请求顺时针找到最近的服务器

**优点**
- 服务器增减时，只影响相邻节点
- 减少数据迁移

**缺点**
- 实现复杂
- 可能分配不均匀（需要虚拟节点）

**适用场景**
- 分布式缓存
- 服务器频繁增减

### 随机（Random）

这一节看起来往往容易变成一串结论，其实先抓住“随机（Random）”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**原理**
- 随机选择一台服务器

**优点**
- 实现简单
- 请求量大时，分配相对均匀

**缺点**
- 请求量小时，可能不均匀

**适用场景**
- 服务器性能相同
- 请求量大

## Session 一致性问题

### 问题描述

**场景**
- 用户登录后，Session 存储在服务器 A
- 下次请求被负载均衡到服务器 B
- 服务器 B 没有用户的 Session，认为用户未登录

### 解决方案

这里真正要说明的是“解决方案”为什么值得单独拿出来讲。把它放到“分布式基础 - 负载均衡与集群”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**方案 1：Session 复制**
- 所有服务器之间同步 Session
- 任一服务器都有完整的 Session

**优点**
- 实现简单
- 无需额外组件

**缺点**
- 同步开销大
- 服务器数量多时，性能差
- 浪费内存

**方案 2：Session 粘滞（Sticky Session）**
- 使用 IP Hash 或 Cookie
- 同一用户的请求总是分配到同一台服务器

**优点**
- 实现简单
- 无需修改应用

**缺点**
- 负载可能不均衡
- 服务器故障时，Session 丢失

**方案 3：Session 服务器（推荐）**
- 使用 Redis 等集中存储 Session
- 所有服务器从 Redis 读取 Session

**优点**
- 负载均衡
- 服务器故障不影响 Session
- 易于扩展

**缺点**
- 需要额外的 Redis
- 增加网络开销

**方案 4：无状态（最佳）**
- 使用 JWT 等 Token
- 不在服务器存储 Session
- 客户端携带 Token

**优点**
- 完全无状态，易于扩展
- 无需 Session 存储

**缺点**
- Token 无法主动失效
- Token 可能较大

## 集群与高可用

### 什么是集群

这一节看起来往往容易变成一串结论，其实先抓住“什么是集群”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**定义**
- 多台服务器提供相同的服务
- 通过负载均衡分发请求
- 提高性能和可用性

**类型**
- **负载均衡集群**：提高性能
- **高可用集群**：提高可用性
- **高性能计算集群**：提高计算能力

### 主从架构

这一节主要把“主从架构”放回“分布式基础 - 负载均衡与集群”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**特点**
- 一个主节点，多个从节点
- 主节点处理写请求
- 从节点处理读请求
- 主从复制保证数据一致

**优点**
- 读写分离，提高性能
- 从节点可以做备份

**缺点**
- 主节点是单点
- 主从延迟

**适用场景**
- 读多写少
- MySQL 主从复制

**特点**
- 多个副本，地位平等
- 通过一致性协议保证数据一致
- 任一副本故障，其他副本继续服务

**优点**
- 无单点故障
- 高可用

**缺点**

- 性能开销

**适用场景**

### 故障转移（Failover）

读到“故障转移（Failover）”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 负载均衡与集群”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**什么是故障转移**
- 主节点故障时，自动切换到备节点
- 保证服务持续可用

**实现方式**
- **主备模式**：主节点故障，备节点接管
- **主主模式**：两个主节点都提供服务，互为备份

**关键问题**
- **故障检测**：如何快速发现故障
- **自动切换**：如何自动切换到备节点
- **脑裂**：如何避免两个主节点同时工作

### 健康检查

这里真正要说明的是“健康检查”为什么值得单独拿出来讲。把它放到“分布式基础 - 负载均衡与集群”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**为什么需要健康检查**
- 及时发现故障节点
- 避免将请求分配到故障节点
- 实现故障转移

**检查方式**
- **心跳检测**：定期发送心跳，超时认为故障
- **HTTP 检查**：定期发送 HTTP 请求，检查响应
- **TCP 检查**：检查 TCP 连接是否正常

**检查策略**
- 检查间隔：如 5 秒
- 超时时间：如 3 秒
- 失败次数：连续失败 3 次认为故障

## 面试问法

读到“面试高频问法”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 负载均衡与集群”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**为什么需要负载均衡？**
- 单机性能有限，无法应对高并发
- 单点故障导致服务不可用
- 通过负载均衡实现水平扩展和高可用

**水平扩展和垂直扩展的区别？**
- 垂直扩展：升级硬件，成本高，有天花板
- 水平扩展：增加服务器，成本低，理论上无限扩展

**常见的负载均衡算法有哪些？**
- 轮询：按顺序分配
- 加权轮询：根据性能分配
- 最少连接：分配给连接数最少的服务器
- IP Hash：根据 IP 分配，保证 Session 一致性
- 一致性哈希：减少服务器增减时的影响

**如何解决 Session 一致性问题？**
- Session 粘滞：同一用户分配到同一台服务器
- Session 服务器：使用 Redis 集中存储（推荐）
- 无状态：使用 JWT Token（最佳）

**什么是故障转移？**
- 主节点故障时，自动切换到备节点
- 保证服务持续可用

**如何实现高可用？**
- 负载均衡：多台服务器提供服务
- 健康检查：及时发现故障节点
- 故障转移：自动切换到备节点
- 多副本：数据多副本存储

## 易混点

读到“易混淆点”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 负载均衡与集群”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**负载均衡 vs 反向代理**
- 负载均衡：将请求分发到多台服务器
- 反向代理：代理服务器，隐藏后端服务器
- 负载均衡通常通过反向代理实现

**主从 vs 主备**
- 主从：主节点处理写，从节点处理读
- 主备：主节点提供服务，备节点待命

**集群 vs 分布式**
- 集群：多台服务器提供相同服务
- 分布式：多台服务器协同完成一个任务

**故障转移 vs 故障恢复**
- 故障转移：切换到备节点，快速恢复服务
- 故障恢复：修复故障节点，恢复正常
