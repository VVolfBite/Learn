## 为什么需要缓存

### 数据库的瓶颈

这一节主要把“数据库的瓶颈”放回“分布式基础 - 缓存基础”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**性能瓶颈**
- 数据库查询涉及磁盘 I/O，慢
- 内存访问比磁盘快 100 倍以上
- 复杂查询需要大量计算

**并发瓶颈**
- MySQL 单机 QPS 一般几千
- Redis 单机 QPS 可达 10 万+
- 数据库连接数有限

**成本问题**
- 数据库扩展成本高
- 缓存服务器成本低

### 缓存的作用

这里真正要说明的是“缓存的作用”为什么值得单独拿出来讲。把它放到“分布式基础 - 缓存基础”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**提升性能**
- 内存访问，毫秒级响应
- 大幅降低响应时间

**降低数据库压力**
- 缓存承担 80% 以上的读请求
- 数据库只处理缓存未命中的请求

**提高系统吞吐量**
- 支持更高的并发
- 提升用户体验

### 缓存的代价

这一节看起来往往容易变成一串结论，其实先抓住“缓存的代价”在分布式基础里的作用会更稳。后面的内容可以理解成围绕这个问题展开的几种常见判断和做法。

**一致性问题**
- 缓存和数据库可能不一致
- 需要权衡一致性和性能

**复杂度增加**
- 需要处理缓存穿透、击穿、雪崩
- 需要考虑缓存更新策略

**额外成本**
- 需要缓存服务器
- 需要运维

## 缓存的三大问题

### 缓存穿透

**问题描述**
查询一个不存在的数据，缓存和数据库都没有，每次请求都打到数据库。

**为什么会发生**
- 恶意攻击：大量查询不存在的 ID
- 业务 bug：查询条件错误
- 数据被删除：缓存和数据库都没有

**危害**
- 数据库压力大，可能崩溃
- 缓存失去意义
- 影响正常业务

**解决方案**

**方案 1：缓存空值（简单有效）**
```java
String value = redis.get(key);
if (value == null) {
    value = db.query(key);
    if (value == null) {
        // 缓存空值，设置较短的过期时间（如 60 秒）
        redis.set(key, "NULL", 60);
        return null;
    }
    redis.set(key, value, 3600);
}
return value.equals("NULL") ? null : value;
```

优点：实现简单，能解决大部分问题  
缺点：占用缓存空间，可能被恶意攻击（大量不同的 key）

**方案 2：布隆过滤器（推荐）**
```java
// 初始化布隆过滤器，将所有可能存在的 key 加入
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    100000, 0.01); // 10万个元素，1%误判率

// 查询时先检查布隆过滤器
if (!bloomFilter.mightContain(key)) {
    return null; // 一定不存在，直接返回
}
// 可能存在，查询缓存和数据库
```

优点：内存占用小，性能高  
缺点：有误判率（可能存在但实际不存在），无法删除元素

**方案 3：参数校验 + 限流**
- 对请求参数进行校验，拒绝明显不合法的请求
- 对单个 IP 或用户限流
- 加入黑名单

**面试追问**
- 布隆过滤器的误判率是多少？可以调整吗？
  - 可以调整，误判率越低，内存占用越大
- 布隆过滤器能删除元素吗？
  - 不能，因为多个元素可能映射到同一个位
- 缓存空值的过期时间怎么设置？
  - 根据业务场景，一般几十秒到几分钟

### 缓存击穿

**问题描述**
一个热点 key 过期，大量请求同时查询这个 key，瞬间打到数据库。

**为什么会发生**
- 热点数据过期
- 秒杀、热门商品
- 缓存重启

**危害**
- 数据库瞬时压力大，可能崩溃
- 影响其他请求

**解决方案**

**方案 1：互斥锁（推荐）**
```java
String value = redis.get(key);
if (value == null) {
    // 获取分布式锁，只让一个请求查询数据库
    String lockKey = "lock:" + key;
    if (redis.setnx(lockKey, "1", 10)) { // 10秒超时
        try {
            // 再次检查缓存（双重检查）
            value = redis.get(key);
            if (value == null) {
                // 查询数据库
                value = db.query(key);
                if (value != null) {
                    redis.set(key, value, 3600);
                }
            }
        } finally {
            redis.del(lockKey);
        }
    } else {
        // 没获取到锁，等待一段时间后重试
        Thread.sleep(50);
        return get(key); // 递归重试
    }
}
return value;
```

优点：保证只有一个请求查询数据库  
缺点：其他请求需要等待，性能下降

**方案 2：热点数据永不过期（逻辑过期）**
```java
class CacheValue {
    Object value;
    long expireTime; // 逻辑过期时间
}

CacheValue cacheValue = redis.get(key);
if (cacheValue == null) {
    // 缓存不存在，正常查询
    return queryAndCache(key);
}

// 检查逻辑过期时间
if (cacheValue.expireTime < System.currentTimeMillis()) {
    // 逻辑过期，异步更新缓存
    threadPool.execute(() -> {
        Object newValue = db.query(key);
        redis.set(key, new CacheValue(newValue, 
            System.currentTimeMillis() + 3600000));
    });
}
// 返回旧值（即使过期）
return cacheValue.value;
```

优点：性能好，不会阻塞  
缺点：可能返回旧数据，实现复杂

**方案 3：提前更新**
- 监控热点数据的过期时间
- 在过期前主动更新
- 适合可预测的热点数据

**面试追问**
- 互斥锁的超时时间怎么设置？
  - 根据数据库查询时间，一般几秒到十几秒
- 逻辑过期会不会一直返回旧数据？
  - 不会，异步更新完成后就是新数据
- 如何识别热点 key？
  - 监控访问频率，或者业务上已知（如秒杀商品）

### 缓存雪崩

**问题描述**
大量缓存在同一时间过期，或者缓存服务宕机，大量请求打到数据库。

**为什么会发生**
- 缓存集中过期：批量设置了相同的过期时间
- 缓存服务器宕机：Redis 挂了
- 缓存重启：所有缓存丢失

**危害**
- 数据库压力骤增，可能崩溃
- 整个系统不可用
- 影响范围大

**解决方案**

**方案 1：过期时间加随机值（简单有效）**
```java
// 避免集中过期
int expireTime = 3600 + new Random().nextInt(300); // 3600~3900秒
redis.set(key, value, expireTime);
```

**方案 2：缓存集群 + 主从复制**
- 使用 Redis Cluster 或 Sentinel
- 主从复制，故障自动切换
- 提高可用性

**方案 3：限流降级**
```java
// 使用 Guava RateLimiter 限流
RateLimiter rateLimiter = RateLimiter.create(100); // 每秒100个请求
if (rateLimiter.tryAcquire()) {
    // 处理请求
    return queryData(key);
} else {
    // 限流，返回降级数据或错误
    return getDefaultValue();
}
```

**方案 4：多级缓存**
- 本地缓存（Guava Cache、Caffeine）+ 分布式缓存（Redis）
- 分布式缓存故障时，本地缓存仍可用
- 提高可用性

**方案 5：缓存预热**
- 系统启动时，提前加载热点数据到缓存
- 避免启动后大量请求打到数据库

**面试追问**
- 随机值加多少合适？
  - 根据过期时间，一般 5%-10%
- 本地缓存和分布式缓存的区别？
  - 本地缓存在应用内存，快但不共享；分布式缓存独立部署，慢但共享
- 如何做缓存预热？
  - 启动时查询热点数据，写入缓存

## 缓存一致性问题

### 问题的本质

这里真正要说明的是“问题的本质”为什么值得单独拿出来讲。把它放到“分布式基础 - 缓存基础”这张卡片的上下文里看，后面的要点和区别会更容易读顺。

**为什么会不一致**
- 缓存和数据库是两个独立的系统
- 更新操作不是原子的
- 并发更新可能导致顺序错乱

**一致性的代价**
- 强一致性：性能差，实现复杂
- 弱一致性：性能好，可能读到旧数据
- 需要权衡

### 更新策略对比

**策略 1：先更新数据库，再删除缓存（推荐）**

```java
// 推荐方案
db.update(data);
redis.del(key);
```

优点：实现简单，一致性较好  
缺点：删除缓存失败会导致不一致

**并发问题分析**
- 线程 A 更新数据库
- 线程 B 查询，缓存未命中，查询数据库（旧值），写入缓存
- 线程 A 删除缓存
- 缓存中是旧值

**为什么这个问题不严重？**
- 线程 B 查询数据库比线程 A 更新数据库慢（数据库有锁）
- 这个时间窗口很小，概率很低

**策略 2：先删除缓存，再更新数据库**

```java
redis.del(key);
db.update(data);
```

缺点：并发场景下容易不一致
- 线程 A 删除缓存
- 线程 B 查询，缓存未命中，查询数据库（旧值），写入缓存
- 线程 A 更新数据库
- 缓存中是旧值

**这个问题很严重**
- 时间窗口大，概率高
- 不推荐

**策略 3：延时双删**

```java
redis.del(key);
db.update(data);
Thread.sleep(500); // 延时
redis.del(key); // 再次删除
```

优点：解决并发问题  
缺点：延时时间难以确定，影响性能

**策略 4：异步更新（最终一致性）**

```java
// 更新数据库
db.update(data);
// 发送消息到消息队列
mq.send(new UpdateCacheMessage(key));

// 消费者异步删除缓存
consumer.onMessage(msg -> {
    redis.del(msg.getKey());
});
```

优点：解耦，不影响主流程，可靠性高  
缺点：实现复杂，可能延迟

### 最佳实践

这一节主要把“最佳实践”放回“分布式基础 - 缓存基础”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**1. 允许短暂不一致**
- 大部分场景可以接受短暂不一致
- 不要过度追求强一致性

**2. 先更新数据库，再删除缓存**
- 实现简单
- 一致性较好
- 推荐方案

**3. 重试机制**
- 删除缓存失败时重试
- 使用消息队列保证可靠性

**4. 设置过期时间**
- 即使删除失败，缓存也会过期
- 兜底方案

**5. 监控告警**
- 监控缓存命中率
- 监控缓存和数据库的不一致

**面试追问**
- 为什么是删除缓存而不是更新缓存？
  - 更新缓存可能浪费（数据可能不会被读取）
  - 更新缓存复杂（需要重新计算）
  - 删除缓存简单，下次读取时再加载
- 如何保证删除缓存一定成功？
  - 重试机制
  - 消息队列
  - 设置过期时间兜底
- 能保证强一致性吗？
  - 很难，代价太大
  - 大部分场景不需要

## Cache Aside 模式

### 什么是 Cache Aside

**定义**
应用层直接操作缓存和数据库，是最常用的缓存模式。

**读操作**
1. 查询缓存
2. 缓存命中，直接返回
3. 缓存未命中，查询数据库
4. 将数据写入缓存

**写操作**
1. 更新数据库
2. 删除缓存

### 其他缓存模式

读到“其他缓存模式”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 缓存基础”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**Read/Write Through**
- 应用只和缓存交互
- 缓存服务负责加载和更新数据库
- 实现复杂，很少使用

**Write Behind（异步写入）**
- 只更新缓存
- 异步批量写入数据库
- 写性能极高，但数据可能丢失

## 缓存淘汰策略

### 为什么需要淘汰

**内存有限**
- 缓存不能无限增长
- 内存满时需要淘汰部分数据

### 常见策略

这一节主要把“常见策略”放回“分布式基础 - 缓存基础”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**LRU（Least Recently Used）**
- 淘汰最近最少使用的数据
- 认为最近使用的数据未来也会被使用
- 最常用

**LFU（Least Frequently Used）**
- 淘汰使用频率最低的数据
- 统计访问次数
- 适合访问频率差异大的场景

**FIFO（First In First Out）**
- 先进先出
- 淘汰最早进入缓存的数据
- 适合时效性数据

- **noeviction**：不淘汰，内存满时拒绝写入
- **allkeys-lru**：所有 key，LRU 淘汰（推荐）
- **allkeys-lfu**：所有 key，LFU 淘汰

- **volatile-lru**：设置过期时间的 key，LRU 淘汰
- **volatile-lfu**：设置过期时间的 key，LFU 淘汰
- **volatile-random**：设置过期时间的 key，随机淘汰
- **volatile-ttl**：设置过期时间的 key，淘汰 TTL 最小的

## 面试问法

这一节主要把“面试高频问法”放回“分布式基础 - 缓存基础”这条主线里看。先抓住它在整个分布式基础知识体系中的位置，再去看后面的分类和结论，会更容易把零散的判断串起来。

**为什么要用缓存？**
- 提升性能：内存比磁盘快
- 降低数据库压力：缓存承担大部分读请求
- 提高系统吞吐量：支持更高并发

**缓存穿透、击穿、雪崩的区别？**
- 穿透：查询不存在的数据，请求打到数据库（布隆过滤器、缓存空值）
- 击穿：热点 key 过期，大量请求打到数据库（互斥锁、永不过期）
- 雪崩：大量 key 同时过期或缓存宕机（过期时间加随机值、集群、限流降级）

**如何保证缓存一致性？**
- 先更新数据库，再删除缓存
- 重试机制
- 设置过期时间
- 允许短暂不一致

**为什么是删除缓存而不是更新缓存？**
- 更新缓存可能浪费（数据可能不会被读取）
- 更新缓存复杂（需要重新计算）
- 删除缓存简单，下次读取时再加载

**缓存击穿和缓存雪崩的区别？**
- 击穿：单个热点 key 过期
- 雪崩：大量 key 同时过期或缓存宕机
- 击穿影响范围小，雪崩影响范围大

**如何解决缓存穿透？**
- 布隆过滤器（推荐）
- 缓存空值
- 参数校验 + 限流

**Redis 和 Memcached 的区别？**
- Redis 支持更多数据结构（String、Hash、List、Set、ZSet）
- Redis 支持持久化（RDB、AOF）
- Redis 支持主从复制
- Memcached 多线程，Redis 单线程（6.0 后支持多线程 I/O）

## 易混点

读到“易混淆点”时，先别急着记结论。更有效的方式是先弄清它在“分布式基础 - 缓存基础”这一张卡片里到底解决什么问题，这样后面的分点和对比才不至于显得太散。

**缓存穿透 vs 击穿 vs 雪崩**
- 穿透：数据不存在
- 击穿：单个热点 key 过期
- 雪崩：大量 key 过期或缓存宕机

**先删缓存 vs 先更新数据库**
- 先更新数据库，再删除缓存（推荐）
- 先删除缓存，再更新数据库（并发问题）

**删除缓存 vs 更新缓存**
- 删除缓存简单，推荐
- 更新缓存复杂，可能浪费

**LRU vs LFU**
- LRU：最近最少使用
- LFU：使用频率最低

**强一致性 vs 最终一致性**
- 强一致性：实时一致，代价大
- 最终一致性：允许短暂不一致，大部分场景够用
