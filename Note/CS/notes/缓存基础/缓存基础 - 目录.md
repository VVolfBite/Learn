# 缓存基础 - 目录

缓存几乎是后端最容易被“先用再懂”的主题。接口一慢，就想加缓存；数据库一忙，也想加缓存。它看起来像一次简单提速，真正落进业务之后却很快会遇到另一个问题：更快是换来了，但数据为什么开始变旧，更新为什么开始别扭，某些流量高峰为什么反而更危险了。

这一组笔记先不从具体产品入手，而是先把缓存当成一种系统手段来理解。先看缓存为什么存在，它通常放在什么位置，命中、失效和淘汰分别在解决什么；再看缓存和数据库之间的关系，理解一致性窗口为什么是缓存绕不开的代价；最后再回到穿透、击穿、雪崩、热点 key 和大 key 这些高频问题，把前面的主线重新落到工程判断里。

## 缓存的基本位置

缓存并不只是一层独立组件。浏览器可以缓存，网关和代理可以缓存，服务进程里可以做本地缓存，独立缓存服务也可以承担共享读路径。它们都在做同一件事：把高频读取尽量挡在更靠前的位置，减少对后端核心存储的直接压力。

## 这一模块的阅读顺序

先读《缓存基础 - 基本位置》，先把缓存为什么存在、适合放在哪里、为什么会带来额外复杂度看清楚。再读《缓存基础 - 一致性与常见问题》，把更新路径、失效策略和常见风险放到一条主线上理解。最后读《缓存基础 - 高频判断》，把容易混淆的地方压实，避免只记术语、不知道背后的取舍。
