# 缓存基础 - 一致性与常见问题

缓存一旦真正进入业务核心路径，问题就很少出在“能不能读到”，而更常出在“读到的是不是该读到的那份”。缓存和数据库之间天然不是一体的，只要两边各自维护一份状态，就一定会有时间差、顺序差和失败差。

## 缓存一致性

缓存一致性说的不是绝对同时，而是更新发生之后，系统在多长时间、以什么方式接受副本和源数据之间的差异。很多系统并不追求严格同步，而是接受短时间旧数据；问题不在于有没有旧数据，而在于业务能不能承受。

## 常见更新模式

### Cache Aside

最常见的做法是读时先查缓存，没命中再查数据库并回填；写时先更新数据库，再删除缓存。这种模式足够灵活，所以很常见，但它的难点也很明显：删除缓存和更新数据库之间并不是天然原子动作，时序处理不好就可能读到旧值。

### Read Through

应用不直接管缓存回填，而是把“读不到就回源”交给缓存层处理。这样接口更统一，但对底层缓存层要求更高。

### Write Through

写请求先到缓存层，再由缓存层同步写入数据库。它让写路径看起来更整齐，但写延迟和故障处理会更复杂。

### Write Back

先写缓存，稍后异步落库。这样写性能很好，但系统必须接受更强的异步性和更高的数据风险，通常不会在通用业务数据里随便采用。

## 缓存穿透

请求的数据本来就不存在，于是每次都穿过缓存直接打到数据库。单次看起来问题不大，流量一高就会把后端压住。它的本质不是缓存没工作，而是缓存层对“不存在”这件事没有形成保护。

## 缓存击穿

某个非常热的 key 正好失效，短时间内大量请求同时回源，把后端瞬间打爆。它和穿透不同，数据本身是存在的，问题出在热点和失效时刻撞在了一起。

## 缓存雪崩

大批 key 在短时间内一起失效，或者缓存服务本身发生故障，导致大面积请求同时回源。它影响的不是单个热点，而是一整片访问面，所以后果通常更大。

## 热点 Key 与大 Key

热点 key 的问题在于访问过于集中，会把单点缓存节点、单条网络链路甚至数据库热点都放大出来。大 key 的问题在于单个值太大，读写、序列化、网络传输和淘汰都更重。两者都不是“缓存没命中”，而是缓存内容本身的分布出了问题。

## 本地缓存与分布式缓存

本地缓存访问更快，但多实例之间不天然同步；分布式缓存共享范围更大，但多了一跳网络，也多了一层可用性与容量约束。选哪一种，不是看谁更先进，而是看当前业务更在意响应速度、共享一致性还是管理复杂度。

缓存这件事，最后很少落在某条技巧上，而总是落回一个判断：你愿意为更快的读路径付出多大的复杂度和多长的不一致窗口。这个判断做清楚，后面的设计才会稳。
