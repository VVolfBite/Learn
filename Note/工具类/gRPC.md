# gRPC

## 工具介绍

gRPC 是一个高性能 RPC 框架，常用于服务和服务之间的通信。它把“远程调用”做得更规范：接口先定义，数据结构先约定，再基于这些约定生成客户端和服务端代码。

如果你已经熟悉 RESTful API，可以把 gRPC 先理解成一种更强调契约和高效序列化的服务间调用方式。

## 核心机制

### Protobuf

gRPC 最核心的基础之一是 Protocol Buffers。你先写 `.proto` 文件，定义消息结构和服务方法，再由工具生成代码。

```proto
syntax = "proto3";

service UserService {
  rpc GetUser(GetUserRequest) returns (UserResponse);
}

message GetUserRequest {
  int64 id = 1;
}

message UserResponse {
  int64 id = 1;
  string name = 2;
}
```

### RPC 模型

gRPC 让调用方像调本地方法一样去调远程服务，但本质上它仍然是网络通信。只是序列化、协议、接口契约这些都帮你规范好了。

### HTTP/2

gRPC 常基于 HTTP/2 工作，这也是它在多路复用、长连接和流式通信上的一个重要基础。

### 流式调用

gRPC 不只有简单请求响应，还支持服务端流、客户端流和双向流。这个特性在需要连续传输数据时很有用。

## 基本使用

### 定义 proto

先定义消息和服务。

### 生成代码

基于 proto 生成客户端和服务端代码。

### 实现服务与调用

服务端实现接口，客户端发起调用。刚入门时，先把一元 RPC 跑通最重要。

## 项目工程中的使用

### 服务间通信

微服务内部调用时，如果团队希望接口定义清晰、性能较好、代码生成成熟，gRPC 很常见。

### 强契约接口

当接口字段复杂、跨语言协作多时，proto 文件会比口头约定和零散文档可靠得多。

### 流式场景

日志流、消息流、长连接式数据推送场景里，gRPC 的流式能力会更有吸引力。

## 常见问题与注意点

第一，gRPC 不是浏览器公开 API 的默认首选，它更常见于服务内部通信。

第二，proto 设计很重要，字段编号、兼容性和版本演进要谨慎。

第三，不要因为“性能高”就机械替换所有 HTTP 接口，协议选择还是要看调用场景和团队生态。
