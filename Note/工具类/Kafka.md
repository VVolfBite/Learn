# Kafka

## 工具介绍

Kafka 是分布式消息系统，常用来做异步解耦、削峰填谷、日志采集和流式处理。它适合放在“请求先落地、后续再慢慢处理”的场景里，把同步调用链拉短。

对后端来说，学 Kafka 不是为了会背名词，而是为了理解：为什么有些事情不适合在接口请求里同步做完，为什么消息系统能帮系统扛住流量冲击，但又为什么它会带来顺序、重复消费和延迟可见性的问题。

## 核心机制

### Topic 与 Partition

Kafka 的消息会先进入 Topic，而 Topic 又会分成多个 Partition。分区的意义在于：
- 扩展吞吐
- 支持并行消费
- 让顺序性只在分区内成立

这也是 Kafka 最重要的理解点之一。你说“Kafka 有顺序”，通常要补上一句：**顺序一般只保证在单个分区里**。

### Producer 与 Consumer

Producer 负责发消息，Consumer 负责读消息。消费通常不是“读完就没了”，而是通过 offset 记录自己读到了哪里。

### Broker 与 Consumer Group

Kafka 集群由多个 broker 组成。消费者通常按 group 协作消费，同一个 group 内，一条消息一般只会被其中一个消费者实例处理。

### Offset

offset 可以理解成消费位置。它很重要，因为重启后能不能从上次位置继续读、能不能重放消息，很多都和它有关。

## 基本使用

### 最先要会的不是复杂集群，而是几个最基础动作

- 创建 topic
- 发送消息
- 消费消息
- 看分区和消费进度

伪代码层面，大致长这样：

```java
producer.send(new ProducerRecord<>("order-created", key, value));
```

```java
consumer.subscribe(List.of("order-created"));
while (true) {
    var records = consumer.poll(Duration.ofSeconds(1));
    for (var record : records) {
        // 处理消息
    }
}
```

### 日常最关心的操作

刚接项目时，真正会碰到的是：
- 某个 topic 有没有消息
- 消费者为什么没消费到
- offset 卡在哪
- 这条消息要不要重放

所以 Kafka 笔记里要有“概念 + 基础使用 + 排查入口”，不能只讲架构图。

## 项目工程中的使用

### 异步解耦

下单成功后，不一定要在接口返回前同步完成积分、短信、发券、日志写入。可以先把核心事务做完，再发消息给下游慢慢处理。

### 削峰填谷

流量高峰时，先把任务写入 Kafka，再由后端消费者按能力持续消费。

### 日志与事件流

操作日志、埋点事件、数据同步事件，也经常通过 Kafka 传递。

## 常见问题与注意点

第一，Kafka 不是“用了就高并发”，它只是帮你把同步压力转成异步处理。

第二，顺序性、重复消费、消息堆积是高频问题，不能只会发消息，不会看消费状态。

第三，消息系统不会自动替你保证最终业务一致性，业务侧仍然要考虑幂等和失败重试。
