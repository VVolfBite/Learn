# 01 API 基础

## API

API 是系统对外暴露能力时所做出的**长期承诺**。其设计关注的是接口是否容易理解、行为一致、可以平滑演进。这也是为什么 API 设计常被概括为四个关键词：**清晰、一致、可演进、可观测**。

### 契约含义

API 首先是一种**契约**。包括：输入字段代表什么含义、哪些字段必填，哪些可选、成功时会返回什么、失败时会返回什么错误以及错误语义、请求是否允许重试、同一个操作是否需要幂等保证、状态如何推进、结果何时可见等内容规定。

### 稳定边界

API 还是一种**稳定边界**。系统内部实现可以不断变化，但只要对外行为保持稳定，调用方应跟着变化一起重写。因此，一个设计得好的 API，应该尽量把内部复杂性藏在边界后面，而不是把内部结构原样暴露给外部。

### 抽象层次

API 还需要合适的**抽象层次**。抽象层次太低时，接口会像数据库表的直接镜像；抽象层次太高时，接口又可能过于空泛，调用方很难理解具体行为。合适的抽象层次应该围绕“业务能力”展开。

## 资源与 URI

### 资源建模

资源是 API 世界里最核心的抽象单位。订单、支付单、用户、退款单、文章、评论，这些都属于典型的资源，因为它们在业务上有独立身份、有明确生命周期，也值得被单独读取、更新、追踪。

判断某个对象是否值得建模成资源，通常看三点：

- 是否有独立 ID
- 是否有相对稳定的状态与生命周期
- 是否会被单独查询、更新或引用

子资源用于表达**从属关系**。当一个对象天然依附于另一个对象存在，或者只有在父资源语境下才有明确意义时，就适合建模成子资源。例如订单条目属于订单，评论属于文章，支付记录可能属于支付单或订单。

### URI

URI（Uniform Resource Identifier）统一资源标识符，它是一个**字符串用来标示抽象或物理资源**。Web上可用的每种资源（ HTML文档、图像、音频、视频片段、程序等）都由一个通用资源标识符（Uniform Resource Identifier, 简称”URI”）进行定位。URI的格式也由三部分组成：访问资源的命名机制、存放资源的主机名、资源自身的名称，由路径表示。一般命名有如下规则：

* 名词复数：URI 一般推荐使用名词复数，例如 `/orders`、`/users`、`/payments`。

* 层级表达从属：表达“谁属于谁”。例如 `/orders/{order_id}/items` 一看就知道 items 不是一个脱离上下文的条目集合，而是某个订单下的条目集合。

* 避免动词：URI 更适合表达“对象是什么”，而不是“现在要执行哪个动作”。动作本身通常由 HTTP 方法、资源状态变化或子资源来表达。

* 稳定 ID：路径中的 ID 必须具备**长期稳定、唯一定位**的能力。同一个 ID 不能今天表示订单，明天又表示退款，也不应在不同资源下重复承载不同行为语义。
* 查询参数：过滤、排序、分页这类能力，本质上不是“资源是谁”，而是“如何从资源集合里挑选和组织结果”。



## 幂等与去重方案

   幂等表示同一请求重复执行多次，最终效果与执行一次相同。幂等关注的是“副作用”的唯一性，而不是“请求次数”的唯一性：允许重复到达，但不允许重复扣款、重复扣库存、重复发货等。当网络与调用链不可靠：超时不等于失败，断线不等于未执行。只要存在重试或重复点击，写操作就可能被执行多次。幂等把不确定性收敛为确定性，使得重复请求返回同一结果。

### 幂等键
   幂等键（request_id/业务流水号）用于标识“同一次业务操作”。服务端以幂等键为依据完成去重：首次执行落库并记录结果；重复到达直接返回首次结果或处理中状态。幂等键必须在多次重试间保持稳定，否则无法归并为同一次操作。

#### 去重方案

##### 唯一约束方案  
   给每次“可能重试的写请求”一个稳定的 **幂等键**（比如 `request_id` / `Idempotency-Key`）。然后在数据库里规定 **UNIQUE（唯一约束/唯一索引）** 来做“幂等去重”。这条规定就是“唯一约束”。它解决的问题是**同一个请求被重复发送时，不要重复创建/重复扣减，而是“只成功一次”**。

##### 去重表方案  
单独建唯一去重表，比如建表：`dedup(request_id UNIQUE, status, result, created_at ...)`先插入 `request_id` 成功才继续执行业务；插入失败说明已处理/处理中，再返回已有结果或处理中状态。（这种是“去重表方案”，但本质仍依赖唯一约束。）

##### 条件更新方案  
更新时不光看主键 id，还要带上“前置条件”（版本号或当前状态）。只有条件满足才更新成功；条件不满足就更新失败（影响行数为 0），从而把并发冲突、重复请求、越界状态迁移挡在数据库层。典型的有：

* **版本号 CAS（乐观锁）**：表里有 `version` 字段。只有当当前 version 还是旧值时才更新，并顺便 version+1如果别的请求先更新了，version 已经变化，这次更新会失败（0 行受影响）。
* **状态机 CAS（基于状态的条件更新）**：表里有 `status` 字段，比如订单状态：`PENDING -> PAID -> SHIPPED`。只有当当前 status 是 `PENDING` 时才允许改成 `PAID`

### 重试策略

#### 退避抖动

   重试的第一目标不是“尽快再试一次”，而是**避免所有请求在同一时刻再次冲击系统**。当一次故障或抖动发生时，许多客户端会同时超时，如果它们立刻同步重试，就会形成明显的“同步脉冲”：第一次把系统打慢，第二次重试又把系统打得更慢，最终演化为重试风暴。

   指数退避的作用是让重试间隔按 1、2、4、8… 这样的节奏逐步变长，给系统留出恢复窗口；随机抖动的作用是把同一批请求的重试时间**打散**，避免所有请求在同一秒、同一毫秒再次集中到来。两者合用，才能把“同步冲击”变成“分散压力”。注意， 只做指数退避不加抖动，仍可能出现“整齐的波峰”；只加抖动不退避，可能变成“持续高压的小波峰”，依然难恢复。

#### 止损预算

   重试必须有“刹车”，否则会把一次局部问题放大成全链路拥塞。止损预算通常包含两层：**次数上限**与**时间上限**。次数上限用于防止无限循环；总超时预算用于保证一个请求不会在链路里耗上很久，拖垮线程池、连接池与队列。预算一旦耗尽，应当快速失败，把控制权交给更高层的降级或补偿机制，而不是继续在同一路径上硬撞。但是只限制重试次数但不限制总耗时，可能出现“每次都等很久”的慢性堵塞；只限制总耗时但允许过密重试，仍可能造成短时间内的压力尖峰。

#### 错误分类

并不是所有错误都值得重试。重试的本质是“期望下一次成功概率更高”，如果错误原因在短时间内不会变化，重试只是在制造无效负载。通常可重试的是**瞬时性故障**：网络超时、连接中断、网关类 5xx、临时过载等；通常不可重试的是**确定性拒绝**：参数非法、余额不足、库存不足、权限不足等。分类的目标是两件事：避免对“必然失败”的请求做无意义重试；把有限的系统资源留给真正有机会成功的请求。

## 方法语义

### GET 语义

GET 用于读取资源，核心语义是“获取当前表示”。按照约定，GET 应尽量无副作用，并天然适合缓存、代理和重试。这里的“无副作用”不是说服务端绝对不会做任何事情，而是说**对业务状态不应产生可观察的改变**。例如记录访问日志通常不算违背 GET 语义，但“GET 一下就生成订单、扣库存、发短信”显然是不合适的。

### POST 语义

POST 通常用于创建资源，或者用于表达一类不容易自然落在其他方法语义里的非幂等操作。它的特点是**默认不具备天然幂等性**。也就是说，同样的 POST 请求发两次，系统很可能会执行两次副作用，例如创建两笔订单、发起两次扣款、产生两条记录。**重试前必须先设计好幂等机制**。

### PUT 语义

PUT 用于把这个资源整体变成某个确定状态。也正因为如此，PUT 通常被认为是幂等的—同一个 PUT 请求执行一次和执行十次，只要请求内容相同，资源最终都会变成同样的样子。需要注意的是，PUT 更适合整体替换而不是局部补丁。如果接口只想修改少数字段，而不是替换整个表示，通常更适合 PATCH。

### PATCH 语义

PATCH 用于部分更新，重点是“只改其中某些字段”。它是否幂等，不取决于方法名本身，而取决于“补丁内容的语义”。

### DELETE 语义

DELETE 表示删除资源。它通常被视为幂等，因为同一个资源删一次和删多次，最终结果都应当是“这个资源不存在”或“对外视为不可用”。



## 错误响应

建议维持一个稳定的错误结构，**同类错误始终用同一结构表达**。通常要把“类别”和“细节”同时表达出来。常见写法如下：

```
{
  "error": {
    "code": "ORDER_NOT_FOUND",
    "message": "order not found",
    "details": { "order_id": "123" }
  },
  "request_id": "..."
}
 只要错误结构统一，前端、SDK、日志采集、告警系统、排障平台都可以围绕它建立稳定处理逻辑。
```

以下是一些常见的字段说明：

* `code` 表示业务级错误码。它应该是**稳定、可枚举、可程序判断**的。

* `message` 是给人看的简短说明，用于快速理解问题。

* `details` 用于携带额外上下文，提升定位效率，但必须注意脱敏，不能把内部 SQL、堆栈、密钥、数据库连接信息直接暴露出去。

* `request_id` 用于把这次失败和服务端日志、链路追踪、告警记录对应起来。

## 状态码

通常与 HTTP 状态码表达含义一致，包含了状态码以及原因短语，用来告知客户端请求的结果。

| 状态码 |               类别               |            含义            |
| :----: | :------------------------------: | :------------------------: |
|  1XX   |  Informational（信息性状态码）   |     接收的请求正在处理     |
|  2XX   |      Success（成功状态码）       |      请求正常处理完毕      |
|  3XX   |   Redirection（重定向状态码）    | 需要进行附加操作以完成请求 |
|  4XX   | Client Error（客户端错误状态码） |     服务器无法处理请求     |
|  5XX   | Server Error（服务器错误状态码） |     服务器处理请求出错     |

### 1XX 信息

-   **100 Continue**  ：表明到目前为止都很正常，客户端可以继续发送请求或者忽略这个响应。

### 2XX 成功

-   **200 OK**  

-   **204 No Content**  ：请求已经成功处理，但是返回的响应报文不包含实体的主体部分。一般在只需要从客户端往服务器发送信息，而不需要返回数据时使用。

-   **206 Partial Content**  ：表示客户端进行了范围请求，响应报文包含由 Content-Range 指定范围的实体内容。

### 3XX 重定向

- **301 Moved Permanently**  ：永久性重定向，要求用户访问新位置。

- **302 Found**  ：临时性重定向，资源暂时被迁移，之后应该还会恢复。

- **303 See Other**  ：和 302 有着相同的功能，但是 303 明确要求客户端应该采用 GET 方法获取资源。

  注：虽然 HTTP 协议规定 301、302 状态下重定向时不允许把 POST 方法改成 GET 方法，但是大多数浏览器都会在 301、302 和 303 状态下的重定向把 POST 方法改成 GET 方法。

- **304 Not Modified**  ：如果请求报文首部包含一些条件，例如：If-Match，If-Modified-Since，If-None-Match，If-Range，If-Unmodified-Since，如果不满足条件，则服务器会返回 304 状态码。这个通常和缓存有关，意味着用户之前请求过该资源，本地应该有缓存。

- **307 Temporary Redirect**  ：临时重定向，与 302 的含义类似，但是 307 要求浏览器不会把重定向请求的 POST 方法改成 GET 方法。

### 4XX 客户端错误

-   **400 Bad Request**  ：请求报文中存在语法错误。

-   **401 Unauthorized**  ：该状态码表示发送的请求需要有认证信息（BASIC 认证、DIGEST 认证）。如果之前已进行过一次请求，则表示用户认证失败。

-   **403 Forbidden**  ：请求被拒绝。

-   **404 Not Found**  ：资源未找到。

### 5XX 服务器错误

-   **500 Internal Server Error**  ：服务器正在执行请求时发生错误。

-   **503 Service Unavailable**  ：服务器暂时处于超负载或正在进行停机维护，现在无法处理请求。

## 查询参数与分页过滤排序

让接口表达分页、过滤、排序等查询细节，以增强接口的长期可用性。

### 分页

#### Offset / Limit

Offset / Limit 是最直观的分页方式。它的含义很简单：从第 `offset` 条开始，取 `limit` 条数据。

```
GET /orders?offset=0&limit=20
```

#### Cursor

Cursor 分页的核心思想不是“从第几条开始取”，而是“**从上一次看到的位置继续往后取**”。

```
GET /orders?cursor=...&limit=20
```

这里的 `cursor` 是一个“游标”，通常包含上一次最后一条记录的排序锚点。

### 排序

#### 默认排序

列表接口必须有明确的默认排序。如果没有默认排序，同样的请求在不同时间、不同节点、不同数据库执行计划下，返回结果顺序都可能不同。这样一来，分页就失去了稳定基础，因为“第一页到底是哪 20 条”本身都不确定。

#### 稳定排序

排序稳定的关键，在于排序键最好具备**唯一性或接近唯一性**。 如果只按 `created_at` 排序，那么同一秒内创建的多条记录顺序就可能不确定，翻页时也可能出现重复或遗漏。
 因此，常见做法是把排序键设计成复合键，例如：

- `created_at DESC, id DESC`
- `updated_at DESC, id DESC`

---

### 过滤

过滤的作用是让调用方从资源集合中筛出需要的数据，但过滤能力不是越自由越好。因此，更稳妥的做法通常是**过滤字段白名单**。明确哪些字段允许过滤、允许怎样过滤，而不是让调用方任意传字段名和运算符。

### 查询参数表达

过滤、排序、分页这类能力，本质上属于“如何从资源集合里得到一个视图”，因此通常通过 query string 表达最自然，例如：

- `status=PAID`
- `created_from=...`
- `sort=-created_at`
- `limit=20`

如果过滤语法过于灵活，甚至允许调用方直接拼类 SQL 片段。

---

## 返回结构

### 结构目标

列表接口的返回结构，目标不是“能把数组吐出来”就行，而是让调用方清楚地知道：

- 当前拿到的数据在哪
- 下一页如何继续
- 本次请求如何追踪

一个稳定的列表返回通常会把业务数据和分页信息明确分层，例如：

```
{
  "data": [...],
  "page": {
    "limit": 20,
    "next_cursor": "..."
  },
  "request_id": "..."
}
```

这里的意义很明确：

- `data` 是当前页结果
- `page` 是分页元信息
- `request_id` 用于排障追踪

结构一旦固定，客户端 SDK、前端分页组件、日志追踪都更容易复用。

### 分页信息表达

Cursor 分页里最重要的字段通常是 `next_cursor`。
 它告诉调用方：如果还想继续取下一页，应当把哪个游标带回去。
 相比之下，Cursor 模式下不一定总能轻易给出“总页数”或“总偏移量”，因为它强调的是顺序推进，而不是绝对位置切片。

因此，Cursor 分页的返回结构通常更关注“如何继续向后翻”，而不是“现在处于第几页”。

## 版本与兼容

### 版本策略

#### URL 版本

把版本写进路径，例如：

```
/v1/orders
```

这种方式最大的优点是直观。但缺点同样明显：一旦版本分裂，`/v1`、`/v2`、`/v3` 可能会长期并存。

#### Header 版本

把版本信息写进 Header，例如：

```
Accept: application/vnd.xxx.v1+json
```

语义更纯粹：URI 仍然表示资源是谁，版本作为表达形式或协议契约的一部分存在。但是可见性较差。浏

#### 不显式版本

不显式版本的核心思想是：**优先通过向后兼容的方式持续演进，而不是一有变化就开新版本**。例如新增字段、增加可选能力、扩展枚举、增加新的子资源，这些变化通常都可以在原有接口基础上自然演进，而不必立刻推出 `v2`。
只有当出现真正无法兼容的破坏性变更时，才考虑启用新版本。

### 兼容性原则

#### 新增字段

新增字段通常被认为是最安全的变更之一。

#### 删除字段

删除字段通常属于破坏性变更。因为一旦某个客户端仍然依赖这个字段，而服务端把它删掉，客户端就可能直接报错、展示异常，或者在更糟糕的情况下悄悄进入错误逻辑。

#### 改变语义

改变字段语义往往比删除字段更危险。删除字段至少会让问题比较容易暴露：客户端拿不到值，通常会报错或明显异常；而语义改变则可能让客户端看起来还能运行，但结果已经错了。

#### 枚举扩展

枚举字段必须从一开始就按“未来可能增加新值”来设计。因此，兼容性的一个基本要求是：**客户端必须容忍未知枚举值，服务端必须对枚举扩展保持谨慎**。

### Expand/Contract

Expand/Contract 是一种兼容性优先的数据库与接口演进策略，常用于上线变更时避免把旧调用方、旧代码或旧数据直接弄坏。

* Expand：扩展阶段先引入新能力，但不立即移除旧能力。

* Dual-write：如果新旧结构需要同时生效，通常会进入双写阶段。也就是说，服务端在一段时间内同时写旧字段和新字段，确保从这一刻开始，新产生的数据在两套结构中都存在。 

* Migrate：把旧字段、旧表、旧状态逐步搬到新结构中。

* Switch：当新结构已经具备完整数据，且经过校验后，可以逐步把读路径和写路径切到新结构。

* Contract： 收缩阶段确保旧客户端、旧任务、旧依赖已经退出后才能删除旧字段、旧逻辑、旧结构的时候。

## 设计理念

### 接口即产品

接口是会被长期依赖的产品部件。字段命名是否清楚、状态是否可理解、错误是否可判断，这些问题都会直接影响调用体验。

### 资源优先

设计 API 时，通常优先围绕资源建模，而不是围绕动作命名。订单、支付单、退款单、用户资料，这些都是稳定的业务对象；它们比一组零散的动作更适合作为接口设计的核心。

### 一致性优先

同一套 API 里，相似问题最好有相似答案，这主要是为了降低接入成本和出错概率。

### 演进优先

API 上线会被不同节奏的调用方长期使用，因此设计时就应该为未来扩展留空间。

### 可观测优先

观测信息帮助问题出现时的观测定位与维护。写好请求 ID、稳定错误码等信息是线上排障的必要条件。

## 快问快答

- **Q：API 设计最重要的第一原则？**  
  清晰与一致。调用方能理解、行为能预期，比“功能齐全”更优先。

- **Q：为什么不要暴露数据库结构？**  
  因为内部实现必然演进；暴露了就把未来改动成本强行外溢给调用方。
